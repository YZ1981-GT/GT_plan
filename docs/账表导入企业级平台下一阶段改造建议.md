# 账表导入企业级平台下一阶段改造建议

## 1. 文档目的

本文用于在现有大文件导入修复工作基础上，进一步从**企业级平台**视角重新审视账表导入链路，并提出下一阶段更系统的改造建议。

本文不是对单点 Bug 的补丁说明，而是面向以下目标的架构分析与实施建议：

- 将账表导入从“可用功能”升级为“平台级能力”
- 统一预览、导入、激活、回滚、事件联动的业务语义
- 让系统具备更强的可靠性、扩展性、可治理性与可运维性
- 为后续多实例部署、异步 worker 化、数据集版本化打基础

适用范围：

- `ledger/smart-preview`
- `ledger/smart-import`
- `account-chart/preview`
- `account-chart/import`
- `account-chart/import-async`
- `smart_import_engine.py`
- `ledger_import_upload_service.py`
- `import_queue_service.py`
- `event_handlers.py`
- `LedgerPenetration.vue`
- `AccountImportStep.vue`

---

## 2. 执行摘要

### 2.1 当前总体判断

当前系统已经完成了本轮 P0 修复中的关键基础工作：

- 大文件上传已支持先落临时存储再复用
- `smart_import_streaming(...)` 已具备批式处理与 staged activation 能力
- 导入完成后能够通过 `DATA_IMPORTED` 事件驱动下游重算与缓存失效
- `ImportQueueService` 已具备项目级并发控制和基础进度持久化
- CSV 链路的流式处理能力已具备较好的工程基础

这意味着：

**账表导入已经从“单次同步导入逻辑”升级为“具备平台化雏形的导入中心”。**

但从企业级平台标准看，当前仍存在几个结构性短板：

- 导入入口仍然分裂，存在新旧两套协议与实现
- 异步任务仍依赖 `asyncio.create_task(...)`，不具备 durable job 能力
- 当前只做了 `is_deleted` 级别的激活切换，还没有真正的数据集版本治理模型
- 上传产物仍依赖本地文件系统，不适合多节点部署
- `ImportBatch` 同时承担“用户作业”和“底层写入批次”两种职责，语义混杂
- 预览、校验、导入、激活之间还没有形成结构化治理闭环

### 2.2 核心结论

下一阶段不要再继续以“解析细节优化”为主线，而应把重点切换为三件事：

- **统一入口**
- **版本化数据集**
- **durable 作业系统**

如果这三件事不先落下来，后续即使继续优化 parser、映射、批大小，也很难把系统真正推向企业级稳定态。

## 3. 基于当前实现的再分析

## 3.1 当前已经做对的部分

### 3.1.1 上传产物复用方向正确

当前 `LedgerImportUploadService` 已支持：

- 分块落盘
- `upload_token` 复用
- 上传清单（manifest）
- TTL 过期控制

这是从“HTTP 请求持有大文件字节”向“上传产物引用化”转变的关键一步，也是后续 worker 化与共享存储化的前提。

### 3.1.2 新导入内核已具备 staged activation 雏形

当前 `smart_import_streaming(...)` 已实现：

- 新数据先以 `is_deleted=True` 写入
- 导入全程成功后统一激活
- 若导入失败则不覆盖旧有效数据

这已经符合企业级系统的核心数据安全原则之一：

**失败不能破坏当前线上有效账套。**

### 3.1.3 下游联动基础较好

当前 `DATA_IMPORTED` 事件已经驱动：

- 试算表全量重算
- 公式缓存失效
- 底稿预填数据标记过期

这说明当前系统并非孤立导入，而是在逐步形成“导入 -> 重算 -> 下游刷新”的平台事件闭环。

### 3.1.4 已有一定测试基础

当前 `backend/tests` 已覆盖：

- `smart_import_streaming(...)` 基础行为
- `ImportQueueService` 锁与状态逻辑
- 部分导入行为与回滚能力

虽然还远远不够，但说明系统并非完全没有工程化保障。

---

## 3.2 当前最关键的结构性问题

## 3.2.1 导入入口仍然分裂

当前至少有两套主要导入链路：

### 新链路

- `ledger/smart-preview`
- `ledger/smart-import`
- 前端入口：`LedgerPenetration.vue`

### 旧/并行链路

- `account-chart/preview`
- `account-chart/import`
- `account-chart/import-async`
- 前端入口：`AccountImportStep.vue`

问题在于：

- 两套路由协议并不一致
- 只有新链路支持 `upload_token` 文件复用
- `account-chart/preview` 仍保留了自己的手写预览解析逻辑
- `account-chart/import` / `import-async` 仍然在入口层 `await f.read()`，大文件仍可能顶满内存

这会导致以下后果：

- 同一个平台有两套导入语义
- 用户在不同页面体验不同
- 后续修复必须重复做两遍
- 未来必然出现“某条旧入口漏改”的回归问题

## 3.2.2 当前异步任务还不是企业级作业系统

当前 `smart-import` 后台任务通过 `asyncio.create_task(...)` 启动。

这在单实例或开发环境下可用，但不具备企业级稳态能力：

- 进程重启时任务会丢失
- 多 worker / 多实例时执行位置不可控
- 没有真正的 heartbeat
- 没有统一的 cancel / retry / resume 机制
- 上传产物若保存在本地磁盘，worker 与 Web 不同节点时无法共享

结论：

当前实现更接近“应用内后台任务”，而不是“平台级导入作业系统”。

## 3.2.3 当前仅有激活切换，没有数据集版本治理

当前四表和客户科目表依赖：

- `import_batch_id`
- `is_deleted`

实现 staging 与激活。

这能解决“失败不覆盖旧数据”，但还不能很好回答以下企业级问题：

- 当前线上有效的是哪一个完整数据集版本？
- 上一版是什么时候导入的？
- 这次导入包含哪些原始文件？
- 某个版本的诊断报告、错误、统计信息在哪里？
- 是否支持显式回滚到上一版？
- 是否能在 UI 上展示“导入历史版本”而不是只看若干 `ImportBatch`？

也就是说，当前有“批次”，但还没有真正的“业务级数据集版本”。

## 3.2.4 `ImportBatch` 职责混杂

当前 `ImportBatch` 同时承担：

- 导入作业对象（`__smart_import_job__`）
- 四表物理写入批次对象（`tb_balance` / `tb_ledger` 等）

这会带来几个问题：

- 查询语义变复杂
- 状态语义不纯粹
- 审计语义模糊
- 依赖特殊 `data_type` 哨兵值维护逻辑

对于企业级平台，建议把“作业状态”和“数据写入批次/数据集版本”明确拆开。

## 3.2.5 本地文件存储不适合多节点部署

当前 `LedgerImportUploadService` 基于本地文件系统保存 bundle。

单机部署可用，但多节点部署时存在以下问题：

- preview 请求可能落在 A 节点
- import 请求可能落在 B 节点
- B 节点无法访问 A 节点本地 bundle
- worker 化后也可能读取不到上传产物

因此，如果系统目标是企业级平台，上传产物最终必须迁到：

- 共享卷
- 或对象存储

## 3.2.6 预览与校验模型还不够平台化

当前已有 diagnostics，但整体还属于“解析诊断信息”的范畴，缺少更正式的校验治理模型：

- 没有统一的规则级 validation report
- 没有严格区分 `info` / `warning` / `error` / `fatal`
- 没有明确的 activation gate 规则清单
- 没有企业级导入验收口径

企业平台导入通常至少分三层：

- Schema validation
- Business validation
- Activation gate

当前这三层尚未被清晰建模。

## 3.2.7 代码级热点与建议改造边界

从下一轮落地角度看，建议优先按“职责收口”而不是“继续叠加逻辑”来拆分现有代码。

| 模块 | 当前主要问题 | 建议收口后的职责 |
|------|-------------|------------------|
| `backend/app/routers/ledger_penetration.py` | 已承担上传、预览、导入任务提交等多项编排职责 | 仅保留权限校验、参数解析、调用统一应用服务 |
| `backend/app/routers/account_chart.py` | 仍保留独立 preview/import 逻辑，且存在入口层整块读文件 | 作为兼容入口存在，内部统一转发到账表导入应用服务 |
| `backend/app/services/smart_import_engine.py` | 文件过大、职责过多，既做解析、转换、写库，也做激活与事件发布 | 逐步拆成 `ParseOrchestrator`、`RowConverter`、`BatchWriter`、`ActivationService` |
| `backend/app/services/ledger_import_upload_service.py` | 已具备可用能力，但当前仍绑定本地文件系统 | 抽象存储后端，支持本地/共享卷/对象存储三种实现 |
| `backend/app/services/import_queue_service.py` | 当前同时承担锁、进度、状态兜底 | 过渡期继续保留；长期只做状态门面，不做真实调度 |
| `audit-platform/frontend/src/views/LedgerPenetration.vue` | 已具备较新协议，但页面逻辑较重 | 保留为主导入页，逐步提炼 composable 与共享导入组件 |
| `audit-platform/frontend/src/components/wizard/AccountImportStep.vue` | 与主导入页重复维护预览、映射、导入流程 | 向共享导入组件靠拢，避免继续演化成第二套协议 |

这一步很关键，因为当前最大的风险不是单个函数慢，而是**多个入口、多个页面、多个服务都在各自演化自己的导入语义**。

## 4. 企业级平台目标架构

建议把账表导入升级为如下目标架构：

```text
前端页面/项目向导
    ↓
统一导入 API 门面
    ↓
LedgerImportApplicationService
    ├── ImportArtifactService（上传产物）
    ├── ImportJobService（导入作业）
    ├── ValidationService（校验报告）
    ├── ParseOrchestrator（多文件/多 sheet 编排）
    ├── DatasetService（数据集版本）
    ├── ActivationService（激活/回滚）
    └── EventPublisher（事件发布）
    ↓
Worker / Durable Job Runner
    ↓
分批写库 / 版本激活 / 下游事件
```

目标特征：

- 前端不再关心旧入口与新入口的差异
- 后端只有一个权威导入内核
- 作业与数据集版本解耦
- 上传产物可跨进程/跨节点访问
- 激活与回滚具备业务级语义
- 校验结果、诊断结果、导入历史可以长期留存

## 5. 下一步详细建议

## 5.1 建议一：统一所有账表导入入口

### 5.1.1 目标

建立唯一的账表导入应用服务，例如：

- `LedgerImportApplicationService`

所有入口都只负责：

- 权限校验
- 参数适配
- 调用统一服务

而不是各自维护自己的导入细节。

### 5.1.2 统一范围

应统一以下入口：

- `/ledger/smart-preview`
- `/ledger/smart-import`
- `/account-chart/preview`
- `/account-chart/import`
- `/account-chart/import-async`

建议保留外部 URL 以兼容前端，但内部都转发到同一套应用服务。

### 5.1.3 必须统一的协议项

至少统一：

- 上传产物引用方式：`upload_token` 或后续 `artifact_id`
- 自定义映射结构：统一 `file/sheet` 级语义
- 预览响应结构：统一 sheet diagnostics / summary / validation 格式
- 导入状态结构：统一 job status / progress / result / error
- 失败返回语义：前端不再猜测不同接口的异常格式

### 5.1.4 直接收益

- 减少一半以上重复维护成本
- 杜绝新老入口行为不一致
- 让项目向导也享受新链路的大文件保护能力
- 为后续 worker 化与 dataset 化打通基础边界

---

## 5.2 建议二：引入“数据集版本”模型

## 5.2.1 为什么必须引入

当前仅依赖 `is_deleted` 和 `import_batch_id` 的方案，难以支撑企业级治理。

建议引入业务级数据集表，例如：

- `ledger_datasets`

建议字段：

- `id`
- `project_id`
- `year`
- `status`：`staged` / `active` / `failed` / `rolled_back`
- `created_by`
- `created_at`
- `activated_by`
- `activated_at`
- `source_type`
- `source_summary`
- `validation_summary`
- `record_summary`
- `job_id`
- `previous_dataset_id`

并让四表与客户科目表挂接：

- `dataset_id`

### 5.2.2 与当前 `ImportBatch` 的关系

建议后续角色拆分为：

- `ImportJob`
  - 关注任务执行过程
- `LedgerDataset`
  - 关注业务级数据版本
- `ImportBatch`（可保留或逐步弱化）
  - 关注底层表写入统计/兼容过渡

### 5.2.3 带来的能力提升

引入数据集版本后，就可以自然支持：

- 当前有效版本查看
- 历史版本列表
- 一键回滚到上一版
- 导入文件清单追踪
- 导入报告审计
- 平台级“数据上线/回滚”能力

### 5.2.4 激活策略建议

建议激活动作从“翻转若干 `is_deleted` 标记”升级为：

- 旧 active dataset 标记为 `superseded` 或 `inactive`
- 新 staged dataset 标记为 `active`
- 数据查询统一只查询 active dataset

这样语义更干净，也更利于审计、回滚、运维定位。

### 5.2.5 查询读路径改造建议

引入 `LedgerDataset` 后，真正难的不是写入，而是**把现有读取链路从 `is_deleted=false` 迁移到“读取 active dataset”**。

建议采用以下过渡方式：

- 第一步：保留 `is_deleted`，同时新增 `dataset_id`
- 第二步：在查询层引入统一仓储/服务方法，例如 `get_active_dataset_id(project_id, year)`
- 第三步：四表查询逐步改为 `dataset_id == active_dataset_id`
- 第四步：仅在兼容阶段继续保留 `is_deleted=false` 作为兜底

不建议在业务代码中继续散落大量：

- `where is_deleted = false`

否则未来即使数据集版本模型建好了，读路径也会因为历史逻辑分散而难以彻底切换。

## 5.3 建议三：把导入任务升级为 durable job

## 5.3.1 当前问题

`asyncio.create_task(...)` 的最大问题不是“慢”，而是“不可靠”。

对企业平台，必须解决：

- 任务进程丢失
- 节点漂移
- 重试恢复
- 失败审计
- 取消任务

### 5.3.2 推荐目标能力

无论最终采用什么框架，至少要具备：

- 作业持久化
- 作业状态机
- 心跳与超时
- 最大重试次数
- 可取消
- 可恢复
- worker 独立于 Web 生命周期

### 5.3.3 推荐状态机

建议把作业状态显式定义为：

- `pending`
- `queued`
- `running`
- `validating`
- `writing`
- `activating`
- `completed`
- `failed`
- `canceled`
- `timed_out`

### 5.3.4 与现有 `ImportQueueService` 的过渡关系

建议不是立刻删除 `ImportQueueService`，而是逐步调整其职责：

现阶段职责：

- 锁 + 状态 + 持久化 + 轮询结果

目标职责：

- 只做“导入状态查询门面”
- 不再直接承担真实任务调度
- 真正的调度与执行交给 job runner / worker

---

## 5.4 建议四：把上传产物从本地临时文件升级为平台级 artifact

## 5.4.1 当前状态

当前 `LedgerImportUploadService` 已经把上传产物从“请求内存字节”变为“磁盘 bundle”，方向正确。

### 5.4.2 下一步目标

建议把上传产物抽象为平台级对象：

- `ImportArtifact`

建议字段：

- `artifact_id`
- `project_id`
- `upload_token`
- `storage_uri`
- `checksum`
- `size_bytes`
- `file_name`
- `mime_type`
- `created_by`
- `expires_at`
- `status`

### 5.4.3 存储建议

部署方式不同，建议也不同：

- 单机阶段：本地目录可继续使用
- 准生产 / 多副本阶段：共享卷
- 企业级平台阶段：对象存储优先

### 5.4.4 治理要求

上传产物建议增加：

- 文件数上限
- 总容量上限
- 单文件上限
- 扩展名白名单
- 内容哈希
- 重复上传去重能力（可选）
- 后台定时清理，而不是仅在访问时清理

---

## 5.5 建议五：建立正式的验证报告体系

### 5.5.1 建议的三层校验模型

#### 第一层：Schema Validation

检查：

- 文件类型
- 编码
- sheet 是否存在
- 必需列是否存在
- 表头是否可识别
- 自定义映射是否有效

#### 第二层：Business Validation

检查：

- 年度一致性
- 公司编码一致性
- 借贷平衡性
- 重复凭证
- 多文件覆盖关系
- 余额表与序时账期间关系
- 空值比例异常
- 维度拆分异常

#### 第三层：Activation Gate

明确哪些问题阻止激活：

- `fatal`：禁止激活
- `error`：默认禁止激活
- `warning`：允许激活但需明确提示
- `info`：仅提示

### 5.5.2 报告结构建议

建议统一返回结构：

- `file`
- `sheet`
- `rule_code`
- `severity`
- `message`
- `details`
- `sample_rows`
- `blocking`

### 5.5.3 产品价值

这样做之后：

- 预览不只是“看前几十行”，而是“看结构化导入报告”
- 导入失败不再只是一个字符串异常
- 前端可以对 warning / error 分区展示
- 导入历史页可以沉淀每次导入的质量报告

---

## 5.6 建议六：对下游事件语义做一次整理

### 5.6.1 当前优势

当前 `DATA_IMPORTED` 已经驱动：

- 试算表重算
- 公式缓存失效
- 底稿 stale 标记

这说明事件总线是有效的。

### 5.6.2 当前问题

`DATA_IMPORTED` 正在承担越来越多语义。

未来如果继续复用这个事件处理不同类型的“数据变化”，下游会越来越难维护。

### 5.6.3 建议的事件语义升级

建议逐步引入更明确的事件：

- `ledger.import_submitted`
- `ledger.import_failed`
- `ledger.dataset_validated`
- `ledger.dataset_activated`
- `ledger.dataset_rolled_back`

其中下游真正关心“数据已生效”的订阅方，应优先订阅：

- `ledger.dataset_activated`

而不是宽泛地订阅“导入发生过”。

### 5.6.4 迁移原则

- 短期：保留 `DATA_IMPORTED` 兼容现有订阅方
- 中期：新增新事件并让核心订阅方迁移
- 长期：收敛为更明确的事件语义体系

## 6. 建议的数据与服务分层

## 6.1 建议新增/重构的核心对象

### ImportArtifact

负责：

- 上传产物
- 存储位置
- 生命周期管理

### ImportJob

负责：

- 一次导入任务的执行状态
- 排队、运行、失败、取消、完成

### LedgerDataset

负责：

- 一次业务级导入版本
- 是否 active
- 验证报告与统计摘要

### ValidationReport

负责：

- 结构化校验结果
- 供前端与历史页查询

### ActivationRecord

负责：

- 某次激活动作的留痕
- 谁将哪个 dataset 激活为当前版本

## 6.2 建议的服务拆分

### LedgerImportApplicationService

作为导入总门面，负责协调整体流程。

### ArtifactService

负责上传产物创建、查询、过期清理、共享存储适配。

### ImportJobService

负责作业状态机、任务提交、重试、取消、超时。

### ValidationService

负责规则执行、分级报告、是否允许激活。

### DatasetService

负责数据集创建、状态更新、历史查询、回滚。

### ActivationService

负责旧版本失效、新版本生效、原子切换与事件发布。

## 6.3 建议的 API 演进方向

为避免未来继续维护“协议不同但语义相近”的多组接口，建议中期统一出一套更明确的导入 API。

建议目标接口可以是：

- `POST /api/projects/{project_id}/ledger-import/artifacts`
  - 创建上传产物，返回 `artifact_id` / `upload_token`
- `POST /api/projects/{project_id}/ledger-import/previews`
  - 基于上传产物或直接上传做预览，返回结构化 validation report
- `POST /api/projects/{project_id}/ledger-import/jobs`
  - 创建导入作业，返回 `job_id`
- `GET /api/projects/{project_id}/ledger-import/jobs/{job_id}`
  - 查询作业状态与阶段进度
- `GET /api/projects/{project_id}/ledger-import/datasets`
  - 查询数据集历史
- `POST /api/projects/{project_id}/ledger-import/datasets/{dataset_id}/rollback`
  - 回滚到指定版本或以前一版本为目标回滚

兼容策略建议：

- `ledger/smart-preview`、`ledger/smart-import` 先内部转发
- `account-chart/*` 路由保留，但逐步退化成兼容壳层
- 前端新页面与项目向导最终都只依赖上述统一 API 语义

## 6.4 建议的读取一致性治理

企业级平台里，“写入能成功”只是第一步，“所有读路径读到同一版本”才是真正的稳定性要求。

建议为账表查询统一建立以下约束：

- 任意项目年度读取前，先解析 active dataset
- 试算表重算、底稿预填、钻取明细、辅助余额汇总都依赖同一 active dataset 语义
- 禁止在下游服务中各自解释“当前有效数据”

建议最终形成统一约束：

- 导入侧负责产生 dataset
- 激活侧负责切换 active dataset
- 读取侧只认 active dataset

这样才能真正满足“同一 `project-year` 始终只有一个业务上生效的数据集”的平台要求。

## 8. 发布与迁移建议

## 8.1 发布策略

建议采用灰度与开关控制，而不是一次性大切换。

建议顺序：

- 第一步：统一服务内核，但外部路由暂时保留
- 第二步：项目向导切到统一协议
- 第三步：新数据集模型先并存，不立即移除旧逻辑
- 第四步：切换查询读路径到 active dataset
- 第五步：清理旧入口与旧模型依赖

## 8.2 回滚策略

每一步都要有回退点：

- 路由回退：旧路由仍可临时保留
- 写入回退：dataset 方案可与旧 `is_deleted` 方案短期并存
- 任务回退：worker 方案上线前保留现有轮询接口
- 事件回退：新事件发布同时保留旧事件兼容

## 8.3 风险控制建议

重点控制以下风险：

- 多入口同时改造导致前后端协议错位
- 数据集模型引入后查询链路读错版本
- durable job 改造时状态机不一致
- 共享存储切换后 artifact 访问失败
- 事件语义迁移造成下游漏订阅

## 9. 建议的验收标准

建议下一阶段验收至少覆盖以下口径：

### 架构口径

- 全部账表导入入口都通过统一应用服务进入
- 不再存在大文件入口层整块 `read()` 的主链路
- 导入任务不再依赖 Web 进程内 `create_task` 作为最终形态

### 数据口径

- 同一 `project-year` 始终只有一个业务上可见的有效版本
- 导入失败不会破坏当前有效账套
- 能识别并展示导入对应的数据集版本

### 运维口径

- 能查询导入作业历史
- 能查询导入数据集历史
- 有结构化导入报告
- 有最基本的指标与日志

### 产品口径

- 主导入页与项目向导导入体验一致
- 用户可以明确知道当前是“预览”“校验失败”“导入中”“已激活”“可回滚”中的哪一种状态

---

## 10. 最终建议

如果只选择一个下一阶段主题，我建议优先做：

### 第一优先级：统一入口内核

原因：

- 最直接降低维护成本
- 最容易消除新旧链路分裂
- 最能立刻让项目向导享受到本轮大文件修复成果

### 第二优先级：引入数据集版本模型

原因：

- 是企业级账套治理的核心能力
- 是后续回滚、历史版本、审计追踪的基础

### 第三优先级：作业 durable 化

原因：

- 决定系统是否真正能支撑生产级多实例环境

归纳为一句话：

**下一阶段的重点不应继续停留在 parser 细节，而应转向“统一入口 + 数据集版本 + durable 作业”的平台化收口。**

---

## 11. 推荐的直接行动项

建议按以下顺序启动下一轮工作：

- **动作 1**：输出统一导入应用服务设计稿，明确旧入口如何转发到新内核
- **动作 2**：设计 `LedgerDataset` / `ImportJob` / `ImportArtifact` 数据模型与迁移方案
- **动作 3**：制定 durable job 技术选型与迁移计划
- **动作 4**：补结构化 validation report 与前端展示草图
- **动作 5**：制定灰度发布与回滚策略

完成这五项后，再进入编码阶段，整体风险会明显低于继续边改边试。
