# 审计全流程断点修复计划

## 1. 文档目的

本文基于资深合伙人视角的系统全面评估，聚焦一个核心目标：

**让审计员能用这个系统完整走完一个真实的年审项目。**

不再以"API 数量"或"代码行数"衡量进度，而是以"审计员能否完成日常工作"为唯一验收标准。

### 3.1 P0：附注编辑器校验层做实（2 天）

**问题**：用户点"执行校验"只能看到 2 种结果，其他 6 种返回空。

| 校验器 | 功能 | 实现方案 |
|--------|------|----------|
| check_note_table_integrity | 附注表格内部纵向合计 | SQL 聚合子项 vs 合计行 |

| check_data_completeness | 数据完整性 | 非空率检查 |

**验收标准**：用户点"执行校验"后能看到 5+ 种不同类型的校验结果（LLM 相关的可降级为"服务不可用"提示）。

### 3.5 P1：复核流程端到端验证（2 天）

#### 3.5.1 待复核收件箱数据来源

**问题**：`wp_review_service` 有 add_comment/reply/resolve API，但 WorkpaperList.vue 的复核意见面板是否完整支持"查看→回复→解决"流程。

- 点击回复 → 弹窗输入 → 调用 reply API
- 点击解决 → 状态变为 resolved

#### 3.5.3 二级复核流转

**问题**：一级复核通过后是否自动流转到 pending_level2？二级复核人在哪里操作？

**修复**：验证 `update_review_status` 在 level1_passed 后是否自动设置 pending_level2，以及 ReviewInbox 是否能筛选二级待复核。

### 3.6 P1：操作 diff 审计记录（1 天）

**问题**：AuditLogMiddleware 记录了"谁在什么时候调了什么 API"，但没有记录"改了什么"（old_value → new_value）。

**修复范围**：对以下关键操作补充 diff 记录：
- 调整分录 CRUD（记录金额变化）
- 底稿状态变更（记录 old_status → new_status）
- 附注单元格编辑（记录 old_value → new_value）

**实现方案**：在 `adjustment_service.py` 的 update/delete 方法中，写入 `logs` 表的 `new_value` JSONB 字段增加 `_diff` 键。

### 3.7 P2：附注校验器 6 个 stub 做实（3 天）

详细实现方案：

```python
# check_note_table_integrity: 纵向合计
# 遍历附注表格，检查标记为 total 的行是否等于其上方 data 行之和

# check_formula_consistency: 期初+变动=期末
# 遍历变动表，检查 opening + increase - decrease == closing

# check_cross_table_consistency: 报表数 vs 附注合计
# 从 financial_report 取报表行金额，与附注表格合计行比对

# check_data_completeness: 非空率
# 检查附注表格中数值列的非空率，低于阈值报 warning

# check_policy_template_compliance: 会计政策比对
# 将当前会计政策文本与模板文本做相似度比对（difflib.SequenceMatcher）

# check_text_reasonableness: LLM 辅助
# 调用 llm_client.chat_completion 检查表述合理性（降级返回空）
```

## 5. 验收标准（全流程冒烟测试）

### 5.1 审计员视角（8 步）

- [ ] 上传 Excel 文件 → 预览 → 确认导入 → 四表有数据
- [ ] 查账页面 → 科目余额表有数据 → 双击穿透到序时账 → 看到凭证明细
- [ ] 新建 AJE → 借贷平衡 → 保存成功 → 试算表审定数变化
- [ ] 试算表 → 全量重算 → 一致性校验通过
- [ ] 底稿列表 → 选择底稿 → 在线编辑（或下载编辑）→ 上传 → QC 自检
- [ ] 附注编辑 → 选择科目 → 从底稿刷新 → 表格有数据 → 执行校验 → 看到结果
- [ ] 审计报告 → 生成 → 编辑段落 → 刷新财务数据 → 数据面板有值
- [ ] 导出 Word → 下载文件 → 打开确认格式正确

### 5.2 项目经理视角（复核流程）

- [ ] 审计员提交复核 → 项目经理收件箱看到待复核项
- [ ] 打开底稿 → 添加批注 → 退回
- [ ] 审计员看到退回原因 → 修改 → 重新提交
- [ ] 项目经理通过 → 状态变为 review_passed

### 5.3 合伙人视角（签字前检查）

- [ ] 合伙人看板 → 签字前检查 → 8 项全部通过
- [ ] 导出全套文档 → Word 包含审计报告+报表+附注
