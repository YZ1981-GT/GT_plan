## 一、全局组件接入一致性（前端呈现统一性）

### 现状实测数据

| 组件/composable | 应接入视图 | 实际接入 | 接入率 | 差距 |
|---|---|---|---|---|
| GtPageHeader | 83（排除 16 特殊） | 73（grep `<GtPageHeader`） | 88% | 10 视图缺失 |
| GtAmountCell | 所有含金额列视图 | 6（Ledger/Drilldown/Report/Misstatements/Adjustments + 1 组件） | ~15% | 其余表格裸 `{{ row.amount }}` |
| useEditMode | 所有可编辑视图 | 11 | 73% | 4 视图裸编辑 |
| useFullscreen | 数据密集型表格 | 6 | 60% | ConsolidationIndex 等缺失 |
| usePenetrate | 含科目/金额的视图 | 7 | 70% | AuxSummary/CFS 等缺失 |
| v-permission | 危险操作按钮 | 22 处 / 15 视图 | ~60% | 仍有按钮裸露 |
| handleApiError | catch 块 | ~53 视图 | ~59% | 其余用 ElMessage.error 或静默 |

### 建议


#### 1.1 � GtPageHeader 补齐剩余（P1，1 天）

**问题**：73/99 视图已接入 GtPageHeader（88%），剩余约 10 个视图缺失统一页头（排除 Login/NotFound/DevelopingPage 等特殊页面后）。

**方案**：
- 剩余 10 视图逐一分类：模式 A（标准页头 + actions slot）/ 模式 B（banner 横幅）/ 排除（子面板/嵌入组件）
- CI 卡点：`grep -c '<GtPageHeader' views/**/*.vue` 基线设为 80，只增不减
- 角色影响：全角色

#### 1.2 🔴 GtAmountCell 全量覆盖（P0，2 天）

**问题**：只有 5 个视图 + 1 个组件用了 GtAmountCell（6/99 视图=6%），其余表格金额列直接 `{{ row.amount }}`，导致：
- 千分位格式不统一（有的有逗号有的没有）
- 负数显示不统一（有的红色有的括号有的无标记）
- 万元/元切换只在部分视图生效
- 金额列字体不统一（应为 Arial Narrow + tabular-nums）

**方案**：
- 全局搜索 `align="right"` 的 el-table-column，凡是金额/数值列全部替换为 GtAmountCell
- GtAmountCell 内部统一读 displayPrefs store 的 amountUnit，不再需要 provide/inject
- 角色影响：全角色（合伙人看报表数字、助理看序时账、经理看汇总）

#### 1.3 🟡 handleApiError 残留清理（P1，0.5 天）

**问题**：约 11 处 ElMessage.error 裸用残留（handleApiError 已覆盖 53/99=59% 视图），catch 块错误处理风格不一致。用户看到的错误提示有的是中文有的是英文 detail，有的有重试建议有的没有。

**方案**：
- 机械替换：所有 catch 块内 `ElMessage.error(...)` → `handleApiError(e, '操作名')`
- handleApiError 内部统一：网络错误→"网络异常，请检查连接"；401→跳登录；403→"无权限"；409→冲突提示；5xx→"服务异常，请稍后重试"
- CI 基线从 11 降到 0

#### 1.4 🟡 v-permission 盘点补齐（P1，1 天）

**问题**：22 处 v-permission 覆盖了核心危险操作，但仍有遗漏：
- 项目删除按钮（Projects.vue）
- 批量操作按钮（WorkpaperList 批量分配/批量下载）
- 系统设置保存（SystemSettings）
- 报表重算触发

**方案**：
- 运行 `scripts/find-missing-v-permission.mjs` 更新盘点
- 按"操作不可逆程度"分级：删除/签字/归档 = 必加；编辑/导出 = 建议加；查看 = 不加
- 角色影响：质控（权限审计）、合伙人（风控）

## 二、数据穿透与联动闭环

### 现状

平台已建立事件驱动链路（event_handlers.py 订阅 25+ 事件），usePenetrate composable 覆盖 7 个视图，但**穿透链路存在断点**：

| 穿透路径 | 状态 | 问题 |
|---|---|---|
| 余额表 → 序时账 | ✅ 已通 | — |
| 序时账 → 凭证附件 | ❌ 缺失 | 无凭证影像关联 |
| 报表行 → 试算表科目 | ✅ 已通 | — |
| 试算表 → 底稿 | ⚠️ 半通 | wp_mapping 覆盖率低 |
| 调整分录 → 错报 | ✅ 已通 | — |
| 错报 → 报表影响 | ❌ 缺失 | 无"错报对报表影响金额"展示 |
| 附注行 → 底稿/报表 | ⚠️ 半通 | note_related_workpapers 端点已建但前端入口弱 |
| 底稿 → AI 建议 | ✅ 已通 | — |
| 报表 → 审计报告意见 | ❌ 缺失 | 无"报表数据支撑意见类型"可视化 |


### 建议

#### 2.1 🔴 穿透闭环补齐（P0，4 天）

**核心断点修复**：

1. **试算表 → 底稿**：当前 wp_mapping 只有 auto-match 生成的映射，覆盖率约 45%。建议：
   - 试算表每行科目右侧加"关联底稿"图标，点击弹出关联底稿列表
   - 未关联的科目标灰色提示"未关联底稿"
   - 角色影响：助理（快速定位工作底稿）、经理（检查底稿覆盖率）

2. **错报 → 报表影响**：Misstatements.vue 当前只显示金额，不显示该错报对哪些报表行产生影响。建议：
   - 错报详情弹窗增加"影响分析"Tab，展示受影响的报表行及调整前后金额
   - 角色影响：合伙人（判断错报是否重大）、质控（复核错报汇总）

3. **序时账 → 凭证影像**：审计底稿的核心是"抽凭"，当前序时账行无法跳转到原始凭证扫描件。建议：
   - 序时账右键菜单增加"查看原始凭证"（跳转附件管理，按凭证号过滤）
   - 长期：支持凭证影像 OCR 自动关联
   - 角色影响：助理（日常抽凭）、经理（复核抽凭结果）

#### 2.2 🟡 事件联动可视化（P1，2 天）

**问题**：后端 event_handlers.py 已订阅 25+ 事件实现级联更新（导入→试算表→报表→附注→审计报告），但前端用户完全看不到这条链路在工作。

**方案**：
- ProjectDashboard 增加"数据流水线"卡片，展示最近一次级联更新的时间线（哪些环节已刷新、哪些待刷新）
- 各视图 stale 横幅已有（R8 落地），但缺少"点击刷新"一键触发重算的按钮
- 角色影响：经理（确认数据是否最新）、合伙人（签字前确认全链路一致）

#### 2.3 🟡 穿透路径可视化地图（P1，1 天）

**问题**：用户不知道从当前页面能穿透到哪里。

**方案**：
- 帮助面板（F1/快捷键 `?`）增加"穿透地图"Tab，用流程图展示当前页面可达的所有穿透路径
- 或在 GtPageHeader 右侧加一个"🔗"图标，hover 显示可穿透目标列表
- 角色影响：新入职助理（学习系统）、经理（培训用）

## 三、前端呈现一致性与美观动感

### 现状

平台已建立 CSS 变量体系（gt-tokens.css 定义 50+ 变量），品牌色 #4b2d77 深紫已统一到顶栏，但**视觉一致性仍有明显断层**：

### 建议

#### 3.1 🔴 表格视觉统一规范（P0，2 天）

**问题**：
- 金额列宽度不统一：LedgerPenetration 用 width=200，TrialBalance 用 min-width=140，Adjustments 用 width=130
- 表头对齐方式不统一：有的 header-align="center" + align="right"，有的全 center
- 行高不统一：有的 stripe 有的不 stripe
- 操作列位置不统一：有的 fixed="right" 有的不 fixed

**方案**：
- 建立 `gt-table-presets.css`，定义 3 种表格预设：
  - `.gt-table-financial`：金额列 min-width=180 + right + Arial Narrow + nowrap + stripe + 操作列 fixed
  - `.gt-table-list`：常规列表（项目列表/人员列表），无特殊金额处理
  - `.gt-table-compact`：紧凑型（弹窗内/侧栏内），行高 36px
- 所有含金额的表格统一用 `.gt-table-financial` class
- 角色影响：全角色（视觉体验）

#### 3.2 🟡 动效与过渡统一（P1，1.5 天）

**问题**：
- 页面切换无过渡动画（直接替换 DOM）
- 弹窗打开/关闭无动效
- 表格数据加载无骨架屏（直接空白→数据出现）
- Tab 切换无滑动效果

**方案**：
- router-view 包裹 `<transition name="gt-fade" mode="out-in">`
- el-dialog 统一加 `:before-close` 淡出
- 数据加载中统一用 `v-loading` + 骨架屏（el-skeleton）
- 关键交互加微动效：按钮点击 scale(0.97)→scale(1)、卡片 hover 上浮 2px
- 角色影响：全角色（专业感知）

#### 3.3 🟡 空状态统一（P1，0.5 天）

**问题**：各视图空状态处理不一致——有的用 GtEmpty 组件，有的用 el-empty，有的直接一行文字"暂无数据"。

**方案**：
- 统一用 GtEmpty 组件（已有），传入 icon/title/description/action
- 按业务场景预设 5 种空状态：无数据/无权限/开发中/搜索无结果/加载失败
- 角色影响：全角色

#### 3.4 🟢 深色模式完善（P2，1.5 天）

**现状**：Phase 3 已落地暗色模式基础（gt-dark-mode.scss + gt-dark-overrides.css + useTheme.ts + html.dark 70+ CSS 变量 WCAG AA），顶栏已有切换入口。

**问题**：组件内仍有硬编码颜色值未替换为 CSS 变量，部分页面深色模式下对比度不足。

**方案**：
- 逐步替换组件内硬编码颜色为 CSS 变量引用
- 优先保证核心 5 页面深色模式视觉完整性
- 角色影响：加班审计人员（护眼）

## 四、五角色视角专项问题

### 4.1 审计助理视角

审计助理是系统最高频使用者，日常操作：导入账套→编制底稿→抽凭→填写调整分录→提交复核。

#### 4.1.1 🔴 底稿编制效率（P0，3 天）

**问题**：
- WorkpaperEditor 当前只有 Univer 表格编辑器，缺少"程序要求"侧栏（审计程序步骤引导）
- 无"上年底稿参考"面板（助理需要对照上年数据填写）
- 无"科目穿透"快捷入口（编辑底稿时想看某科目明细需要切页面）
- 自动保存间隔 60s 过长（Univer 大表格操作频繁，30s 更合理）

**方案**：
- WorkpaperSidePanel 已有 9 Tab（R8 落地），确认"程序要求"Tab 是否有真实数据填充
- 增加"上年参考"Tab，调用 `get_prior_year_workpaper` 展示上年同编码底稿快照
- 底稿内金额单元格支持右键"穿透到序时账"（复用 usePenetrate）
- 自动保存间隔从 60s 改为 30s
- 角色影响：助理

#### 4.1.2 🟡 抽凭工作流（P1，4 天）

**问题**：当前系统无完整的"抽凭"工作流。审计助理的核心工作之一是从序时账中选取凭证、查看原始凭证影像、记录检查结果。

**方案**：
- 序时账表格增加行选择（checkbox）
- 右键菜单增加"抽凭到底稿"（将选中凭证信息写入指定底稿的抽凭记录区）
- 抽凭记录结构：凭证号/日期/金额/摘要/检查结论/备注
- 长期：对接凭证影像系统（OCR 识别凭证号自动关联）
- 角色影响：助理（日常工作）、经理（复核抽凭充分性）

#### 4.1.3 🟡 工时填报便捷性（P1，1 天）

**问题**：
- 工时填报需要手动选择项目（应自动带出当前项目）
- 无"快速填报"模式（一周 5 天批量填写）
- 无工时提醒（超过 18:00 未填报当天工时应弹提醒）

**方案**：
- 工时表单默认选中当前项目
- 增加"本周批量"模式：一次填 5 天相同项目的工时
- 前端定时检查（每天 18:00）是否已填当天工时，未填则顶栏铃铛提醒
- 角色影响：助理、经理

### 4.2 项目经理视角

项目经理关注：项目进度、人员分配、底稿完成率、复核状态、工时预算。

#### 4.2.1 🔴 项目仪表盘信息密度（P0，2 天）

**问题**：
- ManagerDashboard 当前只有异常告警区块（前端 computed 派生），缺少"项目全景"
- 无法一眼看到：底稿完成率/复核通过率/工时消耗率/逾期工单数
- 缺少"项目健康度"综合评分

**方案**：
- ManagerDashboard 顶部增加 4 个 KPI 卡片：底稿完成率 / 复核通过率 / 工时消耗比 / 逾期工单
- 中部增加"项目甘特图"（按审计阶段 planning→execution→reporting 展示进度）
- 底部保留异常告警（当前已有）
- 角色影响：经理

#### 4.2.2 🟡 人员工作量可视化（P1，1.5 天）

**问题**：
- 经理无法快速看到每个助理的工作负荷
- 无"人员-底稿"矩阵视图（谁负责哪些底稿、完成状态如何）

**方案**：
- ProjectDashboard 增加"团队负荷"Tab：横轴=人员，纵轴=底稿数/工时，柱状图
- 支持拖拽重新分配底稿（从 A 拖到 B）
- 角色影响：经理

#### 4.2.3 🟡 复核批注追踪（P1，1 天）

**问题**：
- ReviewWorkbench 已有复核功能，但经理缺少"全项目复核进度"汇总视图
- 无法看到"哪些底稿有未解决批注"的全局列表

**方案**：
- 复核收件箱增加"项目汇总"Tab：按底稿分组，显示每个底稿的未解决/已解决批注数
- 支持一键跳转到具体批注
- 角色影响：经理、合伙人

### 4.3 质控人员视角

质控关注：规则执行率、抽查覆盖率、问题整改闭环、年度质控报告。

#### 4.3.1 🟡 质控规则执行可视化（P1，2 天）

**问题**：
- QcRuleList 已有 22 条规则，但缺少"规则执行热力图"——哪些规则频繁触发、哪些从未触发
- 质控人员无法快速判断规则有效性

**方案**：
- QCDashboard 增加"规则执行统计"Tab：每条规则的触发次数/通过率/最近触发时间
- 支持按项目/时间段筛选
- 从未触发的规则标黄色警告（可能规则条件过严或场景不存在）
- 角色影响：质控

#### 4.3.2 🟡 整改闭环追踪（P1，1 天）

**问题**：
- IssueTicketList 已有工单列表，但缺少"整改超期"醒目提示
- 质控人员需要一眼看到哪些工单已超 SLA

**方案**：
- 工单列表增加"超期"列，超期行标红色左边框（Q 类工单已有，扩展到全部）
- 顶部增加"超期工单数"badge
- 角色影响：质控、合伙人

### 4.4 项目合伙人视角

合伙人关注：签字风险、项目全局风险、报告质量、归档完整性。

#### 4.4.1 🔴 签字决策面板完善（P0，2 天）

**问题**：
- PartnerSignDecision 已有三栏布局（就绪检查/报告预览/风险摘要），但：
  - 报告预览是 HTML 渲染（非 PDF 预览），排版与最终出具的报告差异大
  - 风险摘要缺少"历史对比"（与上年同期风险对比）
  - 无"签字前 checklist"强制确认

**方案**：
- 报告预览改为 PDF iframe（调用后端 `/export/preview-pdf` 生成临时 PDF）
- 风险摘要增加"同比"列（上年同期高风险工单数/错报金额）
- 签字按钮前增加 3 项强制确认：①已阅读报告全文 ②已确认风险摘要 ③已确认就绪检查全绿
- 角色影响：合伙人

#### 4.4.2 🟡 合伙人工作台精简（P1，1 天）

**问题**：
- PartnerDashboard 信息过多，合伙人只关心"需要我做决策的事项"
- 当前混杂了统计数据和待办事项

**方案**：
- 顶部只保留"待签字项目数/待审批工单数/高风险项目数"3 个数字
- 中部改为"决策队列"：按紧急程度排序的待办列表（签字>审批>知悉）
- 底部保留项目概览（可折叠）
- 角色影响：合伙人

### 4.5 独立复核合伙人（EQCR）视角

EQCR 关注：独立性、重大判断复核、备忘录、与项目组的信息隔离。

#### 4.5.1 🟡 EQCR 信息隔离强化（P1，1.5 天）

**问题**：
- EQCR 当前能看到项目组的所有底稿内容，但 EQCR 的独立笔记对项目组不可见（设计正确）
- 缺少"EQCR 专属视图"——EQCR 看到的项目页面应该与项目组成员不同（只看关键判断相关内容）

**方案**：
- EqcrProjectView 已有 10 Tab，确认每个 Tab 的数据来源是否正确隔离

## 五、前后端联动问题

### 5.1 🔴 API 响应结构不一致（P0，2 天）

**问题**：
- 后端 `ResponseWrapperMiddleware` 把 2xx 响应包装为 `{code, message, data}`，但 HTTPException 走 `http_exception_handler` 把 detail 放到 `message` 字段
- 前端 apiProxy 解包 `data` 字段，但错误响应读 `data?.detail?.error_code`（实际应读 `data?.message?.error_code`）
- 导致部分错误处理逻辑失效（如 409 冲突检测）

**方案**：
- 统一错误响应结构：`{ code: number, message: string, error_code?: string, details?: object }`
- 前端 apiProxy 的错误拦截器统一从 `response.data.error_code` 或 `response.data.message` 取错误码
- 编写 `docs/API_RESPONSE_CONTRACT.md` 明确契约

### 5.2 🟡 前端轮询 vs SSE 双通道冗余（P1，1 天）

**问题**：
- ThreeColumnLayout 同时维护 SSE 连接（`/events/stream`）和 active-job 轮询（每 5-10s）
- 功能重叠：SSE 已能推送 IMPORT_PROGRESS 事件，轮询是历史遗留

**方案**：
- 短期：保持双通道，SSE 断连时自动降级为轮询（当前已是此逻辑）
- 中期：SSE 稳定后移除 active-job 轮询，改为 SSE 推送 + 首次加载时查一次
- 角色影响：全角色（减少无效请求，降低服务器压力）

### 5.3 🟡 前端 Store 职责边界模糊（P1，1 天）

**问题**：
- 9 个 store 中，`collaboration` store 与 `auth` store 职责重叠（都管 user/token）
- `drilldown` store 只被 1 个视图使用，不应该是全局 store
- `wizard` store 只服务 ProjectWizard，应改为组件内状态

**方案**：
- 合并 collaboration → auth（统一认证状态管理）
- drilldown/wizard 降级为组件内 composable 或 ref
- 保留 5 个真正全局的 store：auth / project / dict / displayPrefs / roleContext / addressRegistry
- 角色影响：维护者（降低心智负担）

### 5.4 🔴 年度切换联动不完整（P0，1.5 天）

**问题**：
- projectStore.changeYear 已 emit `year:changed` 事件（R8 落地），但只有部分视图订阅了此事件
- 用户在顶栏切换年度后，部分页面数据不刷新（需要手动 F5）
- 特别是：LedgerPenetration / Adjustments / Misstatements 切换年度后数据不变

**方案**：
- 所有依赖 year 的视图必须 watch `route.query.year` 或订阅 `year:changed` 事件
- 建立规约：凡是 URL 含 `?year=` 的视图，year 变化时必须重新 fetch
- CI 检查：grep 所有含 `year` query 的路由，确认对应视图有 watch
- 角色影响：全角色

## 九、核心原则总结

1. **一致性优先于功能丰富**：宁可 60 个页面风格统一，不要 90 个页面各自为政

3. **角色决定视图**：同一个数据，合伙人看风险、经理看进度、助理看操作入口
4. **前后端契约先行**：API 响应结构、错误码、枚举值必须有单一真源文档
5. **CI 卡点防退化**：每项改进落地后必须有 grep 基线或测试守护，否则下一轮又会回退
6. **维护成本纳入设计**：208 个组件如果没有文档和使用规范，就是 208 个维护负担

## 十、二轮深度复盘补充建议（2026-05-25）

经过对中间件、CI、错误处理、可观测性、无障碍等隐藏维度的二次扫描，发现以下未覆盖的关键问题。

### 10.1 后端可观测性与运维

#### 10.1.1 🔴 中间件双份代码（P0，0.5 天）

**问题**：实测 `backend/app/middleware/` 同时存在 `audit_log_middleware.py` 和 `audit_log.py` 两个文件，类名都是 `AuditLogMiddleware`。代码看似一致但同时存在两份是历史遗留的死代码。

**方案**：
- grep `from app.middleware.audit_log` 确认实际被 main.py 引用的是哪个
- 删除未引用的那个，避免新人改错文件
- 角色影响：维护者

#### 10.1.2 🟡 慢查询日志缺失（P1，1 天）

**问题**：
- ObservabilityMiddleware 已采集请求耗时，但只输出到结构化日志，无慢查询/慢请求告警
- YG2101 导入耗时 ~660s 这种性能瓶颈只能事后从日志翻找，无主动告警
- 数据库 slow query log 未启用（PG `log_min_duration_statement` 默认 -1）

**方案**：
- ObservabilityMiddleware 增加阈值：API > 5s 写 warning 日志，> 30s 写 error 日志并发通知
- PG 启用 `log_min_duration_statement = 1000`（>1s SQL 入慢查询日志）
- 新增 `/api/health/slow-queries` 端点暴露最近 100 条慢查询
- 角色影响：运维、维护者

#### 10.1.3 🟡 异常吞没导致 bug 隐藏（P1，5-8 天）

**问题**：grep `except Exception` 命中 **1162 处**，其中部分是正常的"非关键路径降级"（如 SSE 推送失败不阻断主流程），但部分是粗糙的吞没：
- `attachment_service.py` 多处 `except Exception: pass` 完全不记日志
- `assignment_service.py:170` 仅 `pass  # SSE 推送失败不阻断主流程` 不记日志
- 某些 service 内 `except Exception` 后只 `return None`，调用方无法区分"正常返回空"和"出错降级"

**方案**：
- 规约：所有 `except Exception` 必须至少 `logger.warning(..., exc_info=True)`
- 静默失败必须有明确注释说明原因（如"# 缓存失效不阻断主流程，调用方有 fallback"）
- 新增 ruff/flake8 自定义规则检查裸 `pass` 在 except 块内
- 角色影响：维护者（debug 效率）

#### 10.1.4 🟢 健康检查端点不完备（P2，0.5 天）

**问题**：当前只有 `/api/health/ledger-import` 单一健康端点，缺少全平台健康检查。

**方案**：
- 新增 `/api/health` 总检查端点，聚合：
  - DB 连接池状态
  - Redis 连通性
  - vLLM 服务状态
  - Ollama 状态
  - Worker 心跳（5 个 worker 是否都活）
  - 磁盘剩余空间
- Kubernetes/Docker healthcheck 直接调此端点
- 角色影响：运维

### 10.3 安全与权限

#### 10.3.1 ✅ 前端硬编码权限判断（已清零）

**现状**：grep 实测 `role ===` 在 .vue 文件中零命中，v-permission 指令已全面覆盖。此项已完成。

#### 10.3.2 🔴 Token 安全（P0，0.5 天）

**问题**：
- auth.ts 当前用 sessionStorage 存 token（关闭标签页失效，比 localStorage 安全）
- 但 token 仍是 JWT 明文，XSS 攻击可直接读取
- 无 refresh token 机制（token 过期需重新登录）

**方案**：
- 短期：保持 sessionStorage（已是最佳实践）
- 中期：实现 refresh token，access token 缩短到 15 分钟
- 长期：核心操作（签字、归档）走 httpOnly cookie + CSRF token
- 角色影响：合伙人（合规要求）

#### 10.3.3 🟡 API 速率限制（P1，1 天）

**问题**：当前只有 LLM 端点有 LLMRateLimitMiddleware，其他端点无速率限制。
- 用户误操作可能瞬间发起 100+ 请求（如循环点击导入按钮）
- 无暴力破解防护

**方案**：
- 全局速率限制：每用户每分钟 600 请求（10 req/s）
- 写操作单独限制：每用户每分钟 60 写请求
- 登录端点：每 IP 每分钟 5 次失败尝试后封禁 10 分钟
- 基于 Redis 实现，无 Redis 时降级 in-memory（本地开发）
- 角色影响：运维、合伙人

- 角色影响：年龄偏大的合伙人、未来无障碍合规要求

#### 10.4.2 🟢 国际化框架就位但未使用（P2，n/a）

**方案**：

### 10.5 大表格性能

#### 10.5.1 🔴 序时账大表格性能（P0，2 天）

**问题**：
- LedgerPenetration 序时账可能超过 60 万行（YG2101 实测 65 万）
- 当前是 el-table 全量渲染（即使有分页，前端仍可能加载全部数据）
- 用户拖动滚动条会卡顿

**方案**：

- 前端禁止"加载全部"操作（即使用户要求）

- 角色影响：助理（日常查账）、经理（复核）

- 角色影响：运维

## 十二、审计行业专项建议（合规与专业层）

前两轮覆盖了"系统层"问题（组件一致性、穿透、运维），本章补全**审计行业特有的合规/专业问题**。这些是平台与"普通办公软件"的本质区别。

### 12.1 🔴 金额精度全面 Decimal 化（B1，P0，5 天）

**问题**：实测 `note_validation_engine.py` / `note_formula_generator.py` / `prefill_engine.py` 等 60+ 处用 `float(...)` 做金额计算，`abs(expected - closing) > 0.01` 这种容差判断。

**审计行业铁律**：
- 浮点数累加丢分位（10 万行序时账每行 0.01 误差，累加可能差几千元）
- 监管/法庭场景下，金额必须是确定值
- 致同审计报告若因系统精度问题出错 → 法律责任

**方案**：
- 全平台金额统一走 Decimal（DB 已是 NUMERIC，service 层禁用 float）
- 前端引入 `decimal.js`（10KB），所有金额计算走 BigDecimal
- 校验容差从硬编码 `0.01` 改为按金额规模动态（已在 ledger validator 实现，扩展到全平台）
- 新增 ruff 规则禁止 service 层出现 `float(...)` 处理金额字段
- CI grep 卡点：`backend/app/services/` 下 `float(...amount)` 命中数 ≤ baseline
- **验收**：所有金额计算路径 Decimal 化 + 单元测试覆盖小数累加场景
- 角色影响：全角色（合规底线）

### 12.2 🔴 AI 内容全工作产物溯源（B2，P0，3 天）

**现状**：`ai_contribution_watermark` 仅用于归档报告页脚 + QC 年度报告。底稿、调整分录摘要、错报描述、附注内容等若由 AI 生成，**无水印无审核轨迹**。

**审计行业铁律**：
- 注协 / PCAOB 要求审计师对所有专业判断负责
- AI 生成内容必须有"已人工审核确认"标记并入归档
- 抽查时要能回溯"哪段是 AI 写的、谁审核的、几点几分"

**方案**：
- 所有调用 LLM 的端点强制返回 `wrap_ai_output(...)` 结构（含 id/generated_at/confirm_action/revised_content）
- 前端展示 AI 内容时必须带 🤖 标签 + "确认/修订/拒绝"按钮（已有 AiContentConfirmDialog，需全量接入）
- DB 新增 `ai_content_log` 表记录每段 AI 内容的生成/确认轨迹
- 归档时自动汇总该项目所有 AI 内容清单到独立章节（章节 05 AI 贡献明细）
- 门禁规则：未确认的 AI 内容存在时禁止 sign_off（AIContentMustBeConfirmedRule 已注册，确认全链路触发）
- **验收**：抽查任意底稿/报告，能 5 秒内回溯所有 AI 内容来源
- 角色影响：合伙人（合规风险）、质控（监管检查）

### 12.3 🔴 底稿索引体系前端呈现（B3，P0，3 天）

**问题**：致同有完整索引体系（A 序时表 / B 现金 / C 应收 / D 存货 / F 投资 / K 收入 / N 费用 ...），`backend/data/wp_account_mapping.json` 已有 88 条映射 v2025-R4。但 WorkpaperList **按编码字母排序展示**，索引树未可视化。

**实际场景**：审计助理用纸质底稿习惯按索引找（"翻到 B-100 看现金主表"），电子化后反而绕弯。

**方案**：
- WorkpaperList 左侧栏改为**索引树**（A/B/C/D/F/K/N 大类 → 中类 → 单底稿）
- 每个底稿显示完整索引号（如 "B-100 现金主表"），索引号字体加粗 monospace
- 索引号支持快捷跳转：Ctrl+G 输入 "B-100" 直接定位（复用 useShortcuts）
- 新增"索引体系总览"页（独立路由），展示完整 D/F/K/N 循环结构 + 接入率（每个索引位是否已有底稿）
- WorkpaperEditor 顶部面包屑显示"D 存货 > D-200 存货监盘"
- **验收**：审计助理能 3 秒内通过索引找到任一底稿
- 角色影响：助理（最日常操作）、经理（复核检查）

### 12.4 🔴 反向穿透闭环（B4，P0，3 天）

**实际场景**：合伙人复核报告时指着"应收账款 1.2 亿"问助理"这数字怎么来的"，助理需要翻 5 个页面拼凑。这是审计师**最高频的反向追溯**。

**方案**：完整 5 层链路一键穿透
```
报告行（1.2 亿）
  → 试算表科目（1122 应收账款）
    → 底稿计算过程（B-200 应收主表的公式）
      → 序时账明细（按月/按客户聚合）
        → 凭证扫描件（按凭证号定位附件）
```

**实现**：
- 每个金额单元格右键菜单加"📋 数字溯源"（复用 CellContextMenu）
- 弹出树形/时间线展示从下到上的完整链路（每层显示金额、占比、跳转链接）
- 后端新增 `GET /api/projects/{pid}/trace/amount?value=&context=` 聚合端点
- 前端复用 usePenetrate composable，扩展 `traceAmount()` 方法
- **验收**：合伙人 5 秒内反查任一报告金额来源
- 角色影响：合伙人（复核效率）、EQCR（独立验证）

### 12.5 🔴 函证完整工作流（B9，P0，独立 Sprint 8 天）

**实测**：`backend/app/routers/confirmations.py` 仍是 stub（`{"status": "developing"}`）。但函证是 BCDE 循环必备程序，每个项目都要做。

**完整工作流**（6 步）：
1. **函证清单生成**：按金额阈值/账龄从底稿筛选（应收/应付/银行/借款）
2. **函证文本生成**：模板 + 余额自动填充（多语言：中/英）
3. **发函登记**：邮寄/邮件/电话三种方式 + 跟踪号
4. **收函跟踪**：已收/未收/到期未回（超期高亮）
5. **差异调节**：账面 vs 函证 → 差异自动建议为错报
6. **替代程序**：未回函时记录替代检查程序（账龄分析/期后回款）

**方案**：
- 后端新建 `confirmation_service.py` + `confirmation_models.py`（4 张表：清单/发函/收函/差异）
- 前端 ConfirmationHub.vue（当前是占位）改为完整工作流页面
- 函证模板按业务循环预置（应收 / 应付 / 银行 / 借款 / 客户存款）
- 与 PBC 模块联动：客户提供联系人信息 → 自动填入函证
- **验收**：完整跑通"清单→发函→收函→错报建议"链路
- 角色影响：助理（执行）、经理（监控进度）、合伙人（看函证覆盖率）

### 12.6 🔴 抽样工作底稿专项（B10，P0，4 天）

**实测**：SamplingEnhanced 有 cutoff（截止性测试）/ aging（账龄分析）/ monthly_detail（月度明细）三 Tab，但**没有完整审计抽样工作底稿**。

**审计抽样底稿必备 6 要素**：
1. **抽样目标**：目标错报金额、控制偏差率、可接受信赖度
2. **抽样方法**：PPS（货币单位抽样）/ 统计抽样 / 非统计抽样
3. **抽样基础**：总体规模、容许误差、预期误差、样本量计算公式
4. **抽样过程**：随机种子（保证可复现）、抽出样本清单
5. **抽样结果**：实际错报金额、推断错报金额（按概率推算）
6. **抽样结论**：是否需要扩大抽样、对账户余额结论的影响

**方案**：
- SamplingEnhanced 新增第 4 Tab "完整抽样底稿"
- 后端新增 `sampling_workpaper_service.py` 实现样本量计算（统计公式）+ 推断错报算法
- 抽样过程必须记录随机种子（合规要求：抽样可复现）
- 抽样结果导出标准 Word 模板（致同抽样底稿格式）
- 与现有 cutoff/aging 联动：从这两个 Tab 一键转为正式抽样底稿
- **验收**：跑完一个完整抽样案例（如应收账款抽 30 个客户），生成符合致同标准的 Word 底稿
- 角色影响：助理（编制）、经理（复核样本量是否充分）

### 12.7 🟡 打印/导出合规格式（B11，P1，2 天）

**问题**：审计报告打印必须符合证监会格式（A4、宋体小四、行距 1.5、特定页边距、页码格式）。当前 PDF 导出走 LibreOffice headless，**默认格式不一定合规**。

**方案**：
- 审计报告专用 docx 模板（audit_report_templates_seed.json 已有），需对照《会计师事务所审计报告通用格式》逐项核验：
  - 页边距：上 2.54cm / 下 2.54cm / 左 3.17cm / 右 3.17cm
  - 字体：正文宋体小四（12pt）/ 标题黑体三号
  - 行距：1.5 倍
  - 页码：底部居中"第 X 页 共 Y 页"
  - 公章位置：尾页指定位置预留 5cm × 5cm 印章框
- 打印预览强制 A4 + 1:1 比例（el-dialog 内嵌 iframe 预览）
- 添加"合规打印"按钮（区别于普通"导出"），只输出符合监管格式的版本
- 长期：审计报告底稿/报告/附注合并为单一 PDF（合规归档要求）
- **验收**：打印一份完整审计报告，对照证监会模板逐项核验通过
- 角色影响：合伙人（最终交付物）、归档人员

**方案**：

- 离线时数据写入 IndexedDB
- 联网后自动同步（冲突走 R8 已实现的 OfflineConflictWorkbench）
- 优先级页面：WorkpaperEditor / LedgerPenetration / Adjustments 三个

### 13.4 风险评估工作底稿 RAS（C7，3 天）

**问题**：审计准则要求风险评估三层结构（财表层 / 认定层 / 重大账户循环），当前系统无对应模板。

**方案**：
- 新建 RAS（Risk Assessment Summary）专用底稿模板
- 三层结构：
  - 财表层风险（管理层凌驾、舞弊风险等）
  - 认定层风险（按 5 项认定：存在 / 完整 / 计价 / 权利义务 / 列报）
  - 重大账户循环风险（按 BCDE/D/F/K/N 循环逐项评估）
- 与重要性水平 / 抽样基础联动（高风险账户抽样比例提高）
