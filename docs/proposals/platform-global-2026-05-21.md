# 审计平台全局建议书

**编制日期**：2026-05-21  
**编制人**：资深合伙人（系统开发负责人）  
**版本**：v1.0  
**定位**：针对当前平台实际问题，从五角色视角出发的全局性改进建议

---

## 一、总体评价

平台已完成 11 个审计循环 spec（548/548 tasks）、400 条跨底稿引用、1035 个 prefill cells、53 条 VR 规则，技术架构成熟度高。但从"真实上线后日常使用"角度审视，仍存在若干系统性断层——这些断层不是单个 bug，而是需要全局统筹才能解决的体验/治理/运维问题。

## 二、五角色视角核心痛点

### 2.1 审计助理视角（日常高频操作者）

| 编号 | 问题 | 影响 | 建议 |
|------|------|------|------|
| A-1 | **底稿编辑器缺乏"我的待办"聚合入口** | 助理需在 WorkpaperList 逐个找自己负责的底稿，无法一眼看到"今天该做什么" | 新增 MyProcedureTasks 视图的"今日待办"卡片，按紧急度排序，显示 stale 底稿优先 |
| A-2 | **prefill 一键填充后无差异对比** | 填充后不知道哪些 cell 变了、变了多少，容易漏审 | 填充完成后弹出 diff 面板（旧值→新值），支持逐项确认或一键接受 |
| A-3 | **穿透跳转层级过深时迷路** | TB→AUX→Ledger→AuxLedger→底稿，5 层穿透后 Backspace 返回链不直观 | 顶部增加面包屑导航条，显示完整穿透路径，支持点击任意层级直接跳回 |
| A-4 | **批注/复核意见散落在各 cell** | 助理回复复核意见时需逐个 cell 找，效率低 | 增加"我的待回复批注"汇总面板（类似 ReviewWorkbench 但聚焦于"我被@的"） |
| A-5 | **工时填报与底稿编辑割裂** | 编辑底稿时无法顺手记录工时，需切换到 WorkHoursPage | 底稿编辑器侧边栏增加"计时器"小组件，保存底稿时自动累计工时 |

### 2.2 项目经理视角（进度管控+质量把关）

| 编号 | 问题 | 影响 | 建议 |
|------|------|------|------|
| M-1 | **项目进度缺乏"一张图"总览** | PartnerProjectDashboard 有 CycleProgressRing 但项目经理无独立的"我管的项目群"视图 | ManagerDashboard 增加多项目甘特图/燃尽图，按循环维度聚合进度 |
| M-2 | **复核分派无智能推荐** | 手动分配复核人，不考虑工时负载和专业领域 | 增加"推荐复核人"算法（基于历史复核记录+当前工时余量+循环专长） |
| M-3 | **底稿状态流转缺乏批量操作** | 10+ 个底稿需逐个"提交复核"或"退回修改" | WorkpaperList 增加批量状态变更（勾选→批量提交/批量退回），带确认弹窗 |
| M-4 | **SLA 超时预警不够前置** | sla_worker 检测超时后才告警，缺乏"即将超时"的黄色预警 | 增加 T-24h/T-8h 两级预警（黄色/橙色），推送到项目经理 NotificationCenter |
| M-5 | **工时审批与底稿进度脱钩** | 审批工时时看不到对应底稿的完成度，无法判断工时合理性 | 工时审批页面增加"关联底稿进度"列，显示该助理负责底稿的完成百分比 |

### 2.3 质控人员视角（独立复核+风险把控）

| 编号 | 问题 | 影响 | 建议 |
|------|------|------|------|
| Q-1 | **QC 复核缺乏"风险热力图"** | 质控人员需逐个底稿看 VR 结果，无法快速定位高风险区域 | QCDashboard 增加循环×风险等级热力图（blocking 红/warning 橙/info 灰） |
| Q-2 | **跨循环一致性检查无汇总视图** | 400 条 CWR 分散在各循环，质控人员无法一眼看到"哪些跨循环引用断裂" | ConsistencyDashboard 增加"跨循环断裂清单"Tab，按 severity 排序 |
| Q-3 | **复核意见模板缺失** | 每次写复核意见都从零开始，常见问题（如"请补充审计程序执行记录"）重复输入 | 增加"复核意见模板库"（知识库子模块），支持一键插入常用意见 |
| Q-4 | **质控报告导出格式不统一** | QC 报告需手动整理，无标准化 Word 导出 | 增加 QC 报告一键导出（Word 三线表格式，含风险汇总+意见清单+整改状态） |
| Q-5 | **VR 规则 53 条但覆盖面不均** | D 循环 4 条 vs M 循环 2 条，部分循环校验薄弱 | 制定 VR 规则覆盖度标准（每循环至少 3 条 blocking + 2 条 warning），补齐短板 |

### 2.4 项目合伙人视角（签字决策+风险兜底）

| 编号 | 问题 | 影响 | 建议 |
|------|------|------|------|
| P-1 | **签字决策面板信息密度不足** | PartnerSignDecision 仅展示基本状态，缺乏"签字前必须确认"的 checklist | 增加签字前 Gate（10 项强制确认：重要性/错报汇总/后续事项/独立性/...），全部 ✓ 才能签字 |
| P-2 | **多项目并行时缺乏优先级排序** | 合伙人管 5-10 个项目，无法快速判断哪个项目最紧急 | PartnerDashboard 增加"紧急度评分"（基于 SLA 剩余时间+blocking VR 数+未完成底稿比例） |
| P-3 | **年度对比分析缺失** | 无法快速看到"今年 vs 去年"的关键指标变动 | 项目仪表盘增加 YoY 对比卡片（重要性水平/错报汇总/调整分录数/工时对比） |
| P-4 | **合伙人无法"一键查看"所有 stale 底稿** | stale 传播后合伙人不知道影响范围 | LinkagePanoramaView 增加"仅显示 stale"过滤器，合伙人一键看到所有需要重新填充的底稿 |
| P-5 | **归档前缺乏完整性自检** | ArchiveWizard 流程中无"归档前自检报告" | 归档前自动生成完整性报告（缺失底稿/未签字底稿/未解决复核意见/stale 底稿清单） |

### 2.5 EQCR 独立复核合伙人视角（独立判断+形式合规）

| 编号 | 问题 | 影响 | 建议 |
|------|------|------|------|
| E-1 | **EQCR 工作台与主项目数据隔离不够** | EQCR 应只看到"已提交复核"的底稿快照，不应看到实时编辑中的草稿 | 增加"EQCR 快照"机制：提交 EQCR 复核时冻结当前版本，EQCR 只看冻结版 |
| E-2 | **独立复核意见无结构化模板** | 5 个判断维度（重大错报/持续经营/关键审计事项/其他信息/审计报告）缺乏标准化输入 | 每个判断维度提供结构化表单（结论+依据+引用底稿+风险等级），而非纯文本 |
| E-3 | **EQCR 与项目团队沟通无审计轨迹** | EQCR 提出问题后，项目团队回复散落在各处 | 增加"EQCR 问题单"独立模块（类似 IssueTicketList 但仅 EQCR↔项目团队可见） |
| E-4 | **EQCR 指标仪表盘缺乏历史趋势** | EqcrMetrics 仅显示当前状态，无法看到"本所 EQCR 通过率趋势" | 增加年度趋势图（通过率/平均复核天数/常见问题 Top 5） |

## 三、全局性功能设定建议

### 3.1 全局组件一致性

| 编号 | 问题 | 当前状态 | 建议 |
|------|------|---------|------|
| G-1 | **GtPageHeader 使用不一致** | 部分简单 CRUD 页面仍用紫色渐变横幅（memory 已标注"简单页面不用"） | 全局审查 96 个 views，统一规则：仅 Dashboard/项目级/仪表盘类用 GtPageHeader，其余用白色简洁工具栏 |
| G-2 | **表格字号控制方案不统一** | 部分用 `:style`（不生效），部分用动态 class + `!important` | 统一为 `gt-tb-font-${size}` class 方案，废弃所有 `:style="{ fontSize }"` 写法 |
| G-3 | **金额格式化入口分散** | 大部分用 `displayPrefs.fmt()`，但仍有个别组件自定义 fmt 函数 | grep 全仓库 `toFixed\|toLocaleString` 排查遗漏，统一收口到 `fmtAmountUnit` |
| G-4 | **el-dialog append-to-body 遗漏** | conventions 规定必须加，但新增弹窗可能遗漏 | 增加 ESLint 自定义规则：检测 `<el-dialog` 标签必须含 `append-to-body` |
| G-5 | **空状态组件不统一** | 部分用 `el-empty`，部分用自定义 `LoadingState`，部分用纯文字 | 统一为 `LoadingState` 组件（骨架屏+空状态+错误三态），废弃裸 `el-empty` |

### 3.2 数值处理与精度

| 编号 | 问题 | 建议 |
|------|------|------|
| N-1 | **前端浮点精度风险** | 金额计算统一用 `Decimal.js`（前端）或后端计算后下发，禁止前端 `Number` 直接运算超过 2 位小数的金额 |
| N-2 | **单位换算时机不明确** | 明确规则：DB 存"元"→API 返回"元"→前端 displayPrefs 仅在"显示层"换算，导出时还原为"元" |
| N-3 | **千分位格式与排序冲突** | el-table sortable 对格式化后的字符串排序错误 | 确保 sortable 列的 sort-method 基于原始数值而非格式化字符串 |
| N-4 | **负数显示格式不统一** | 部分用 `(1,234.56)`，部分用 `-1,234.56` | 统一为会计格式 `(1,234.56)` + 红色，displayPrefs 增加"负数格式"选项 |

### 3.3 权限与角色

| 编号 | 问题 | 建议 |
|------|------|------|
| R-1 | **路由守卫 meta.roles 覆盖不全** | 部分路由仅 `requireAuth` 无角色限制，依赖后端 403 | 全局审查路由表，敏感页面（签字/归档/QC/EQCR/用户管理）必须有前端 meta.roles 预筛 |
| R-2 | **项目级权限缓存刷新时机** | Redis 5min TTL 内角色变更不生效 | 角色变更时主动 invalidate Redis key（已有 SoD 撤销机制，扩展到普通角色变更） |
| R-3 | **按钮级权限与路由级权限双轨** | usePermission 管按钮，meta.roles 管路由，两套逻辑可能不一致 | 统一为 usePermission 驱动，路由守卫调用 `can(route.meta.permission)` 而非硬编码 roles 数组 |
| R-4 | **EQCR 双层判断（系统+项目）前端实现复杂** | 当前 `isEqcrEligible` 仅看系统角色，未校验项目级 assignment | 前端增加 `useProjectRole()` composable，从 `/api/projects/{id}/my-role` 获取项目级角色 |

### 3.4 四表联动穿透（TB↔报表↔底稿↔调整分录↔附注）

| 编号 | 问题 | 建议 |
|------|------|------|
| L-1 | **穿透方向单一** | 当前主要支持 TB→下钻，缺乏"从附注反向追溯到 TB 科目"的上钻 | 实现双向穿透：附注数字→点击→显示来源报表行→点击→显示 TB 科目→点击→显示明细账 |
| L-2 | **调整分录对报表的影响无实时预览** | 录入 AJE 后需手动刷新报表才能看到影响 | AJE 录入弹窗增加"影响预览"面板（实时显示受影响的报表行及变动金额） |
| L-3 | **底稿 prefill 与报表数据不一致时无告警** | 底稿填充的 TB 数据可能与报表显示的不一致（时间差） | 增加"一致性快照"：每次报表保存时记录 TB 快照，底稿 prefill 时对比快照，不一致则告警 |
| L-4 | **附注自动提数覆盖率不足** | 附注编辑器的 auto 模式仅覆盖部分科目表格 | 扩展附注公式引擎，支持从 TB/报表/底稿三源自动提数，覆盖率目标 ≥ 80% |
| L-5 | **穿透链路无审计轨迹** | 穿透操作不记录审计日志，无法追溯"谁在什么时候看了什么" | 穿透操作写入 audit_log（轻量级，仅记录 from→to + timestamp + user_id） |

### 3.5 编辑与显示体验

| 编号 | 问题 | 建议 |
|------|------|------|
| D-1 | **Univer 编辑器加载速度** | 大底稿（100+ sheet）首次加载慢 | 实现 sheet 懒加载（仅加载当前 active sheet），切换时按需加载 |
| D-2 | **编辑模式切换无过渡动画** | 查看→编辑切换时页面闪烁 | 增加 300ms fade 过渡，编辑模式下工具栏平滑展开 |
| D-3 | **全屏模式下快捷键冲突** | Ctrl+S 在全屏模式下可能被浏览器拦截 | 全屏模式下注册 Service Worker 拦截浏览器默认行为，确保快捷键优先级 |
| D-4 | **打印预览与实际打印不一致** | GtPrintPreview 的排版与浏览器打印有差异 | 使用 @media print CSS 精确控制打印样式，或直接生成 PDF 下载 |
| D-5 | **字体字号切换后表格列宽不自适应** | 切换到大字号后部分列内容溢出 | displayPrefs 字号变更时触发 el-table `doLayout()` 重新计算列宽 |

### 3.6 查询与搜索

| 编号 | 问题 | 建议 |
|------|------|------|
| S-1 | **缺乏全局搜索（Ctrl+K）** | 用户无法快速跳转到任意底稿/科目/报表行 | 实现全局 Command Palette（Ctrl+K），支持搜索：底稿编号/科目名称/报表行/人员/项目 |
| S-2 | **表格内搜索（Ctrl+F）体验不一致** | 部分模块用 useTableSearch，部分未接入 | 统一所有 el-table 页面接入 useTableSearch，确保 Ctrl+F 行为一致 |
| S-3 | **CustomQuery 自定义查询功能薄弱** | 当前仅支持简单条件，无法做复杂的跨表关联查询 | 增加"高级查询构建器"（可视化拖拽条件+预览 SQL+导出结果） |
| S-4 | **历史版本搜索缺失** | 无法搜索"某个底稿的历史版本中某个值" | 版本对比功能增加"搜索历史值"入口 |

### 3.7 知识库与 LLM 辅助

| 编号 | 问题 | 建议 |
|------|------|------|
| K-1 | **6 个 LLM stub 引擎未接入** | H/I/G/K/J/N 的 LLM 分析功能均为 stub | 制定 LLM 接入路线图：Phase 1 接入规则类（K 费用异常解释）→ Phase 2 接入计算类（H DCF/I 商誉）→ Phase 3 接入文档类（J/N 报告生成） |
| K-2 | **知识库与底稿编辑器联动不足** | useKnowledge 仅在附注/审计报告中使用 | 底稿编辑器增加"相关准则"侧边栏，根据当前底稿循环自动推荐相关 CAS/ISA 条文 |
| K-3 | **LLM 对话历史无项目隔离** | AI 对话可能泄露跨项目信息 | 对话历史严格按 project_id 隔离，LLM context 不跨项目注入 |
| K-4 | **审计判断辅助缺乏"解释链"** | LLM 给出建议但不解释推理过程 | 所有 LLM 输出必须附带"推理依据"（引用的准则条文+参考的历史案例+数据来源） |

### 3.8 工时管理

| 编号 | 问题 | 建议 |
|------|------|------|
| W-1 | **工时填报粒度过粗** | 当前按"天"填报，无法精确到"底稿级别" | 支持按底稿/循环/程序三级粒度填报，自动汇总到日/周/月 |
| W-2 | **工时预算与实际对比无可视化** | budget_alert_worker 仅后台告警 | WorkHoursPage 增加"预算 vs 实际"柱状图（按循环/按人员两个维度） |
| W-3 | **工时审批流程无移动端支持** | 项目经理出差时无法审批工时 | 优先实现工时审批的响应式布局（移动端可用），或提供微信/钉钉审批集成 |
| W-4 | **加班工时无自动识别** | 超过标准工时（8h/天）的部分需手动标记 | 系统自动识别超时部分并标记为"加班"，审批时高亮显示 |

### 3.9 复核流程

| 编号 | 问题 | 建议 |
|------|------|------|
| RV-1 | **复核层级不够灵活** | 当前固定为"助理→经理→合伙人"三级 | 支持自定义复核链（如"助理→高级→经理→合伙人"四级，或"助理→经理"两级） |
| RV-2 | **复核意见无优先级标记** | 所有复核意见同等对待，助理无法判断哪些必须改、哪些建议改 | 复核意见增加优先级（必须修改/建议修改/仅供参考），必须修改项未处理不能重新提交 |
| RV-3 | **复核进度无实时通知** | 提交复核后助理不知道经理是否已开始看 | 增加复核状态实时推送（已接收/复核中/已完成），SSE 推送到 NotificationCenter |
| RV-4 | **批量复核缺乏"通过模板"** | 经理复核 20 个底稿时，对无问题的底稿需逐个点"通过" | 增加"批量通过"功能（勾选多个底稿→一键通过+统一复核意见"已审阅，无异议"） |

### 3.10 枚举与字典管理

| 编号 | 问题 | 建议 |
|------|------|------|
| DT-1 | **枚举字典 sessionStorage 缓存无版本控制** | 后端更新枚举后前端可能使用旧缓存 | 枚举 API 增加 `version` 字段，前端启动时对比版本号，不一致则刷新 |
| DT-2 | **部分枚举硬编码在前端** | statusMaps.ts 中的状态映射与后端可能不同步 | 所有状态枚举统一从 `/api/system/dicts` 获取，statusMaps.ts 仅作为 fallback |
| DT-3 | **枚举管理无 UI 界面** | template-library-coordination spec 剩余 1 task（6.3 枚举字典 DB-backed）未完成 | 优先完成此 task，提供枚举管理 CRUD 界面（仅 admin 可见） |

### 3.11 年度切换与跨年数据

| 编号 | 问题 | 建议 |
|------|------|------|
| Y-1 | **年度切换后 prefill =PREV 公式依赖上年数据** | 如果上年项目未归档，=PREV 取数可能取到未审定的数据 | =PREV 公式增加"数据状态校验"：仅从已归档/已签字的上年底稿取数，否则标记为"待确认" |

## 四、前后端联动专项问题

| 编号 | 问题 | 影响 | 建议 |
|------|------|------|------|
| FE-1 | **SSE 连接稳定性** | ThreeColumnLayout 的 SSE 连接在弱网环境下频繁断开重连 | 增加指数退避重连策略（1s→2s→4s→8s→max 30s），断连期间显示"离线模式"横幅 |
| FE-2 | **API 响应体积过大** | 部分列表 API 返回全量字段（如 WorkpaperList 返回 parsed_data） | 实现 GraphQL-like 字段选择（`?fields=id,wp_code,status,updated_at`），或分离 list/detail 端点 |
| FE-3 | **乐观更新与服务端状态不一致** | 前端乐观更新后如果后端失败，UI 状态与实际不一致 | 关键操作（状态变更/删除/审批）禁止乐观更新，必须等后端确认后再更新 UI |
| FE-4 | **并发编辑冲突处理** | useEditingLock 基于 Redis 锁，但锁过期后可能两人同时编辑 | 增加"版本号"乐观锁：保存时携带 version，后端对比 version 不一致则返回 409 Conflict |
| FE-5 | **前端路由与后端权限不同步** | 前端 meta.roles 和后端 require_project_access 可能判断不一致 | 统一权限源：前端路由守卫调用 `/api/projects/{id}/my-permissions` 缓存结果，而非硬编码 roles |
| FE-6 | **批量操作无事务保证** | 批量删除/批量状态变更如果中途失败，部分成功部分失败 | 后端批量操作使用 SAVEPOINT，全部成功才 commit，任一失败全部回滚 |
| FE-7 | **WebSocket/SSE 事件类型无 TypeScript 类型约束** | eventBus 的事件名是字符串，容易拼写错误 | 定义 `EventMap` 类型（`type EventMap = { 'workpaper:saved': { wpId: string }; ... }`），编译期检查 |

## 十、总结

当前平台的**技术深度**已经非常扎实（548 tasks 全部完成、11 循环全覆盖、400 CWR、1035 prefill cells），但从"真实上线后 5 个角色每天使用 8 小时"的视角看，存在三类系统性问题：

1. **信息聚合不足**：各角色缺乏"一张图看全局"的能力（助理的待办、经理的进度、质控的风险、合伙人的决策）
2. **操作效率断层**：批量操作、快捷搜索、智能推荐等"提效"功能缺失，导致重复性操作多
3. **前后端联动的"最后一公里"**：后端能力已具备但前端未完全消费（如 stale 传播已实现但合伙人无法一键查看、VR 规则已 53 条但质控无热力图）

建议按"Phase 1 体验断层 → Phase 2 角色提升 → Phase 3 系统增强 → Phase 4 长期治理"的节奏推进，每个 Phase 结束后做一轮真实 UAT 验收（Playwright + 5 角色真人测试），确保改进真正落地到用户层。
