# 审计作业平台合伙人视角上线评审与全局改进建议（2026-06-06）

> 编制视角：大型会计师事务所系统建设负责人 / 资深项目合伙人  
> 评审对象：GT 审计作业平台当前代码与既有全局建议文档  
> 关注重点：前后端联动、五类用户实际使用效果、页面一致性与美观动感、全局组件应用、上线后维护便利性  
> 结论定位：这是面向当期平台实际问题的全新建议文档，不替代需求文档，也不等同于纯技术债清单。

---

## 0. 总体判断

这个平台已经不是一个单点工具，而是一个接近企业级审计操作系统的综合平台。它覆盖项目、人员、工时、底稿、附件、复核、报表、附注、合并、知识库、AI、事件治理、质量控制、独立复核、归档等多个闭环。从架构完整性看，已经具备上线试点的基础；从真实事务所规模化推广看，还存在若干必须在全局层面统一的关键问题。

我对当前平台的总体评价是：

| 维度 | 判断 | 主要原因 |
|---|---|---|
| 业务覆盖面 | 较强 | 已覆盖审计作业主链路和管理链路，尤其底稿、附注、合并、复核、QC、EQCR 模块齐全 |
| 工程成熟度 | 中上 | 有 FastAPI 服务层、路由域拆分、事件总线、迁移、Schema 漂移检测、测试与 PBT 思路 |
| 前端一致性 | 中等 | 已有 GT 组件和设计变量，但实际页面中仍存在裸 `el-table`、超大页面、局部样式不统一 |
| 用户体验闭环 | 中等偏上 | 多数功能能走通，但角色入口、跨模块穿透、编辑锁、状态提示、异常降级仍需统一 |
| 上线维护性 | 有风险 | 模块增长快、服务数量多、旧新双轨并存、部分文档规模数字漂移，后期维护压力较大 |
| 合伙人推广可控性 | 需加强 | 需要更明确的角色驾驶舱、质量门禁、异常提示、审批链和项目级指标追踪 |

我的核心建议是：下一阶段不要继续单纯堆功能，而要做一次“平台化收口”。即把已经做出来的功能，收敛成一致的使用体验、一致的数据口径、一致的全局组件和一致的维护规则。否则上线初期会显得功能丰富，但推广到多个项目组后，用户会感受到“入口多、规则多、页面风格多、数据说法多”。

## 1. 从五类角色看真实使用效果

### 1.1 审计助理

审计助理每天最关心的是：我今天做什么、资料在哪、底稿怎么填、数字从哪里来、提交后谁复核、问题怎么改。

当前平台已经具备较完整的助理工作面：

- `MyTodoCard`、任务树、底稿列表、底稿编辑器、附件、查账穿透、AI 辅助、注释与复核意见等功能已经成形。
- 底稿、试算表、序时账、调整分录、附注之间已有多处穿透和联动设计。
- 编辑锁、离线冲突、保存提示、版本历史等能力已有基础。

但从助理真实使用看，问题主要在“入口和反馈不够单一”：

| 问题 | 影响 | 建议 |
|---|---|---|
| 待办、底稿列表、任务树、流程进度、复核意见入口分散 | 助理不知道应该从哪里开始一天工作 | 建立“我的作业台”作为唯一主入口，聚合今日待办、被退回复核、即将截止、AI 建议、资料缺口 |
| 底稿内数字来源提示不统一 | 助理填表时不确定数字是 TB、序时账、调整还是人工 | 在所有金额单元格统一显示来源标记：TB、GL、AJE、RJE、MANUAL、AI |
| 复制粘贴、金额格式、负数显示在不同页面不完全一致 | 影响录入效率，容易产生小数误差 | 统一 `GtAmountCell`、`decimal.ts`、复制粘贴规则，禁止页面自行格式化金额 |
| AI 辅助目前有多处 stub 或降级 | 助理可能误以为是真 AI 结论 | AI 输出必须统一加“来源、模型、置信度、是否人工确认、是否可入底稿”标识 |
| 编辑器与底稿渲染方式复杂 | 用户不关心 Univer/OnlyOffice/HTML，只关心能不能填 | 前端只暴露“可编辑、只读、需刷新、被锁定、可穿透”五类状态，不暴露技术实现差异 |

对助理而言，最重要的体验不是功能多，而是“少想一步”。平台应把“我该做什么、数字从哪里来、下一步交给谁”放在所有页面最显眼的位置。

### 1.2 项目经理

项目经理关心的是：项目进度、人员负荷、复核质量、关键风险、延期事项、预算工时、底稿完成率。

当前平台已有 `ManagerDashboard`、`ProjectProgressBoard`、`WorkpaperList`、`ReviewWorkbench`、`WorkHoursPage`、`ProcedureTrimming` 等模块，基础足够。但经理视角仍需强化“项目经营驾驶舱”：

| 当前不足 | 具体表现 | 建议 |
|---|---|---|
| 进度指标和质量指标分散 | 进度在底稿，复核在复核台，工时在工时页，风险在其他页 | 建立经理首页四象限：进度、质量、预算、风险 |
| 工时管理偏记录，缺少预警 | 工时超预算、人员闲忙、关键任务滞后不够突出 | 工时页增加“预算消耗率、剩余小时、超支预测、人员负荷热力图” |
| 复核意见没有形成管理闭环 | 复核问题关闭率、平均关闭天数、重复问题缺少总览 | 复核模块增加问题 Aging、责任人、重复问题、逾期未回复指标 |
| 项目级配置入口多 | 项目、成员、权限、年度、准则、模板散在不同位置 | 建立“项目设置中心”，把年度、适用准则、模板、人员、权限、锁定策略集中管理 |

经理端应避免堆复杂操作，而应提供“红黄绿灯式”管理判断：哪些底稿卡住、哪些人负荷过高、哪些复核意见逾期、哪些报表/附注数字已 stale。

### 1.3 质控人员

质控人员关心的是：哪些项目风险高、哪些底稿复核不足、哪些程序缺失、哪些事项需要抽查、质量问题是否闭环。

当前平台的 QC 模块比较完整：`QcInspectionWorkbench`、`QcRuleEditor`、`QcCaseLibrary`、`QcAnnualReports`、`ClientQualityTrend`、规则引擎、评分 Worker、案例库等都已有基础。

主要问题是 QC 和业务作业之间的“证据链感”还可以更强：

| 问题 | 影响 | 建议 |
|---|---|---|
| QC 规则与底稿证据的链接不够显性 | 质控发现问题后，要跳多层才能看底稿证据 | 每条 QC 发现必须带“一键定位”：项目、循环、底稿、单元格、附件、复核记录 |
| QC 抽查与项目实时状态关联不足 | 质控看到的是结果，不一定看到过程 | QC 看板接入 stale、复核 Aging、AI 内容确认状态、关键程序完成度 |
| QC 规则编辑器是专业功能，但需要防误用 | 规则改错影响全平台评分 | QC 规则必须有草稿、模拟运行、审批发布、版本回滚 |
| 质量趋势应服务合伙人和培训 | 只看单项目不够 | 建立“问题类型库”：反复出现的审计程序遗漏、底稿说明不足、附件不充分等沉淀为培训材料 |

质控端的核心不是“再做一个检查页面”，而是把检查意见直接嵌回底稿和复核链路，形成可追踪、可整改、可复盘的闭环。

### 1.4 项目合伙人

项目合伙人关心的是：重大风险是否识别、关键判断是否复核、报表和附注是否一致、意见类型是否支持、签发前还有哪些红灯。

当前平台有 PartnerDashboard、PartnerProjectDashboard、PartnerSignDecision、风险摘要、报告、附注、归档、交付件中心等模块基础。但合伙人视角需要更“少而准”：

| 当前问题 | 合伙人实际感受 | 建议 |
|---|---|---|
| 信息过多且技术化 | 合伙人不会逐页看底稿 | 合伙人首页只保留重大事项、未解决复核、关键调整、报告意见、签发阻断项 |
| 报告/附注/调整/底稿一致性还需要更强提示 | 签发风险高 | 签发页增加“一致性清单”：四表、未审报表、调整分录、审定表、附注、报告正文 |
| AI 结论和人工判断边界需清楚 | 合伙人不应为未确认 AI 内容背书 | 所有 AI 辅助内容进入签发链前必须人工确认，并在签发页列示 |
| 归档与交付件状态需统一 | 签发后文件版本多、来源多 | 交付件中心应成为签发后的唯一文件真源，版本、预览、下载、签章、归档都从这里走 |

合伙人端不要做成“功能大全”。应做成“签发风险雷达”。合伙人只需要知道：还有什么不能签、为什么不能签、谁负责、多久能解决。

### 1.5 项目独立复核合伙人（EQCR）

EQCR 关心的是独立性、重大判断、关键审计事项、报告意见、估计和持续经营、关联方、集团审计、复核证据完整性。

当前 EQCR 模块较丰富：`EqcrWorkbench`、`EqcrProjectView`、`EqcrMemoEditor`、`EqcrRelatedParties`、`EqcrShadowCompute`、`EqcrReviewNotesPanel` 等组件已经具备专业化方向。

但 EQCR 最大的问题是“信息来源要可信、边界要独立”：

| 问题 | 风险 | 建议 |
|---|---|---|
| EQCR 与项目组作业数据共用，但独立复核痕迹要清楚 | 容易混同项目组复核和独立复核 | EQCR 所有批注、结论、退回必须单独标识，不进入普通经理复核流 |
| 重大判断来源需一键追溯 | EQCR 不应逐页翻找证据 | EQCR 页应聚合 KAM、重大估计、持续经营、关联方、集团范围、重大调整 |
| Shadow Compute 是亮点，但需解释口径 | 独立测算与项目组数字差异要有解释 | 每个 shadow 结果必须有计算口径、输入来源、差异阈值、处理意见 |
| EQCR 签出前缺少独立检查清单 | 管理层难证明复核充分 | 建立 EQCR 签出 checklist，并与报告签发联动 |

EQCR 端应强调“独立复核证据链”，不能只是项目资料的另一个入口。

## 2. 前后端联动的主要问题与建议

### 2.1 当前联动基础较好

平台已经形成了较完整的前后端分层：

- 后端按 `system / workpaper / report / collaboration / cycle_engines` 五大域注册路由。
- 前端按 `views / components / composables / services / stores / utils` 分层。
- `services/apiPaths` 已经把 API 路径按业务域拆分。
- `apiProxy` 统一解包，`http.ts` 处理 axios 和请求去重。
- 项目事件、SSE、stale、锁、版本、归档、导出任务等都有成体系设计。

这是优点。问题不在“有没有联动”，而在“联动口径是否唯一、页面是否都按同一方式联动”。

### 2.2 前后端联动断点

| 断点 | 表现 | 风险 | 建议 |
|---|---|---|---|
| 路由已实现但前端 `meta.developing` 拦截 | `AuxSummary`、`ConsolSnapshots`、`ReportFormatManager`、`CheckIns` 有页面但进不去 | 用户看到菜单或文档却无法使用 | 上线前清理 developing 路由：要么开放，要么隐藏入口，要么明确标 beta |
| 跨模块引用与冲突调解两套机制并行 | `useCrossModuleRefs` 读引用，`cross_module_conflicts` 处理冲突，二者未完全闭环 | 用户能看到引用但不一定能处理冲突 | 建立统一“联动中心”：引用、冲突、stale、穿透、影响范围共用一套数据结构 |
| 底稿 `wp_code` 深链未统一 resolve 到 `wp_id` | 部分跳转只带 `wp_code` 到列表页 | 用户需二次点击，穿透体验断 | 全局提供 `resolveWorkpaperRoute(projectId, wpCodeOrId)`，所有模块禁止手写底稿跳转 |
| 附注模板绑定存在新旧路径 | binding JSON 不存在时走 legacy fallback | 不同环境生成结果可能不同 | 启动健康检查列示“附注绑定模式”：binding-driven / legacy |
| Stale 传播存在静默跳过 | 报表 stale 字段缺失时 pass | 用户以为数字已刷新，实际报表未提示 stale | stale 字段、事件、前端 badge 需做三层契约测试 |
| WOPI/OnlyOffice 与 Univer 并存 | 用户不理解为什么有的文件在线编辑、有的是前端编辑器 | 体验割裂 | 对用户只展示“编辑模式”：表格编辑、文档编辑、预览，不展示技术名 |

### 2.3 建议建立“联动契约层”

当前四表、未审报表、审计表、底稿、调整分录、附注之间已经有很多服务，但缺少一个用户可理解、开发可复用的统一契约。

建议定义全局 `LinkageContract`：

| 字段 | 说明 |
|---|---|
| `source_type` | tb / ledger / audit_sheet / workpaper / adjustment / report / note / attachment / ai |
| `source_id` | 来源对象 ID |
| `source_cell` | 来源单元格或字段 |
| `target_type` | 目标类型 |
| `target_id` | 目标对象 ID |
| `amount` | 金额，Decimal 字符串 |
| `basis` | 取数口径 |
| `status` | current / stale / conflict / manual_override |
| `confidence` | system / manual / ai_suggested / ai_confirmed |
| `trace_url` | 前端可跳转路由 |
| `audit_log_id` | 留痕 ID |

所有穿透、溯源、冲突、stale、导出校验都围绕这个契约展开。这样才能避免每个页面自己拼跳转、自己判断 stale、自己解释金额来源。

## 3. 前端页面一致性、美观和动感

### 3.1 现状判断

平台已有 GT 视觉系统、CSS 变量、通用组件、暗色模式、表格组件指南。这是好的基础。但真实页面仍有三个问题：

1. 大页面太多，业务逻辑、状态管理、UI 布局混在一个 `.vue` 文件里。
2. 通用组件有，但使用不够强制，部分新页面仍可能裸用 Element Plus。
3. 动效和状态反馈不统一，有的页面有 skeleton，有的页面空白或只显示 loading。

### 3.2 全局 UI 标准建议

| 类别 | 建议标准 |
|---|---|
| 页面头部 | 一律使用 `GtPageHeader` + 项目/年度/模板/状态信息栏 |
| 工具栏 | 一律使用 `GtToolbar` 或业务域 Toolbar，不再每页手写按钮组 |
| 表格 | 展示用 `GtTableExtended`，编辑用 `GtFormTable`，旧 `GtEditableTable` 限期迁移 |
| 金额 | 一律用 `GtAmountCell` 或统一 formatter，禁止页面自行处理千分位、负数、小数 |
| 状态标签 | 一律走 `dictStore`，颜色和文案由字典控制 |
| 空状态 | 统一 `GtEmpty`，区分无数据、无权限、加载失败、开发中 |
| 异常提示 | 统一 `handleApiError`，禁止页面直接 `ElMessage.error(err.message)` |
| 加载状态 | 首屏 skeleton，局部刷新 v-loading，后台任务用 `AsyncJobProgress` |
| 跨模块跳转 | 统一用全局 navigation helper，不再手写路径 |
| 表单弹窗 | 统一确认、保存、取消、脏数据拦截语义 |

### 3.3 美观与动感的取舍

审计平台不是消费级应用，不应追求花哨动画。动感应服务于“状态变化可感知”：

- 数字刷新时，单元格短暂高亮，提示来源已更新。
- stale 解除时，状态 badge 平滑变为 current。
- 保存成功不要只弹 toast，应在页面头部或状态栏显示“已保存于 HH:mm”。
- 异步导入、导出、AI 生成必须有进度条和可取消/可重试状态。
- 长表格滚动时保留表头、金额列、关键操作列固定。

审计人员每天看 8 小时系统，界面要“稳、清楚、不刺眼”。建议少用大面积强色，多用层次、留白、状态标签和细节反馈。

## 4. 用户点名的全局功能逐项建议

### 4.1 项目

建议把项目维度作为全平台主上下文。任何页面都应清楚显示当前项目、年度、适用准则、审计范围、项目状态。

建议：

- 建立项目上下文条 `ProjectContextBar`，全页面统一。
- 项目切换后所有 store、缓存、SSE、编辑锁、字典上下文必须刷新。
- 项目设置中心集中管理：年度、准则、模板、成员、权限、归档状态、锁定状态。
- 项目状态进入 signed / archived 后，默认只读，需特殊权限才能修改。

### 4.2 人员与权限

当前有用户、员工、角色、项目权限、SoD、独立性等模块，但需要更清楚地区分“系统角色”和“项目职责”：

| 类型 | 示例 | 建议 |
|---|---|---|
| 系统角色 | admin、auditor、manager、partner、qc、eqcr | 决定系统菜单可见范围 |
| 项目职责 | 编制人、复核人、项目经理、合伙人、EQCR | 决定项目内操作权限 |
| 临时授权 | 替补复核、紧急解锁 | 必须有到期时间和审计日志 |

建议全局建立 `PermissionMatrix` 文档和 API，前端所有按钮显隐、后端所有操作授权都引用同一权限矩阵，避免前端能点、后端 403，或前端隐藏但 API 可调用。

### 4.3 工时与工时管理

工时不应只是填报模块，而应服务项目管理和事务所经营：

- 助理端：快速填报、从底稿任务带出工作内容。
- 经理端：预算消耗率、剩余预算、人员负荷。
- 合伙人端：项目毛利风险、超支预警。
- 质控端：异常工时与质量风险关联。

建议：

1. 工时记录与底稿任务、复核意见、项目阶段关联。
2. 工时审批页增加批量审批、异常工时提示、超预算提示。
3. 项目看板增加“预算燃尽图”。
4. 工时分类枚举纳入全局字典。

### 4.4 底稿

底稿是平台核心。当前底稿能力很丰富，但需要收口：

- 底稿入口统一到 `WorkpaperEditor`，旧子编辑器只作为内部素材，不再作为产品入口。
- 底稿页面统一显示：当前状态、锁定人、复核状态、数据来源、最后保存时间、stale 状态。
- 所有底稿金额单元格必须支持来源追溯。
- 底稿保存必须统一经过：校验、dirty 检查、版本记录、审计日志、事件发布。
- 底稿模板生成后不得出现“有底稿记录但无 parsed_data 内容”的空壳情况。

### 4.5 附件

附件是审计证据，不只是文件管理。

建议：

- 附件应支持证据属性：来源、取得日期、提供方、关联底稿、是否关键证据、是否被引用。
- 附件预览、OnlyOffice/WOPI、vue-office 预览应对用户统一为“预览/编辑/下载/引用”四个动作。
- 附件删除、替换必须提示影响范围：哪些底稿、哪些附注、哪些复核意见引用了该附件。
- OCR/AI 识别结果必须标明“机器识别，需人工确认”。

### 4.6 复核

复核应从“批注功能”升级为“质量闭环”：

- 每条复核意见必须有责任人、状态、到期日、关闭证据。
- 复核意见应能定位到底稿单元格、附件、报告段落、附注表格。
- 复核问题关闭前，相关底稿不应进入可签发状态。
- 经理和合伙人看板应展示复核 Aging。
- 重复出现的问题应沉淀为 QC 案例或培训材料。

### 4.7 枚举

当前枚举字典设计较健康，但覆盖范围还不够。

建议纳入全局字典治理的高频枚举：

- 审计循环 A-N
- 底稿状态、复核状态、归档状态
- 工时状态、审批状态
- 调整分录状态、错报状态
- 抵销分录类型、抵销审批状态
- 风险等级、QC 问题等级
- AI 内容确认状态

原则：枚举 value 不允许随意改，label/color 可由管理员配置。

### 4.8 年度

年度是审计平台最容易埋 bug 的维度。建议：

- 全局 `selectedYear` 统一由项目上下文管理。
- 页面不得各自硬编码年度默认值。
- 跨年度对比、上年数、期初数、上年附注导入都必须明确年度来源。
- 年度切换后，应清空当前底稿、附注、报表缓存，避免跨年串数据。

### 4.9 编辑与保存

建议统一编辑状态机：

| 状态 | 含义 |
|---|---|
| pristine | 未修改 |
| dirty | 有未保存修改 |
| saving | 保存中 |
| saved | 已保存 |
| conflict | 保存冲突 |
| readonly | 只读 |
| locked | 被他人锁定 |
| archived | 已归档不可改 |

所有编辑页面必须使用同一套状态栏，不再各自实现保存提示。

### 4.10 显示、字体字号

建议全局确立显示规范：

- 中文主字体、数字字体、表格字体分开定义。
- 金额列右对齐，负数格式统一。
- 表格默认字号、紧凑模式、宽松模式由 `displayPrefs` 控制。
- 深色模式应重点测试底稿、报表、附注、试算表四类高密度页面。
- 审计报告、附注导出样式与前端预览样式应建立映射表。

### 4.11 查询

高级查询是平台亮点，但应避免普通用户误用：

- 普通审计人员使用“业务查询模板”。
- 经理及以上使用“高级查询构建器”。
- 查询结果应支持保存为模板、分享给项目组、导出、引用到底稿。
- 任何回写型查询必须记录审计日志，并有预览确认。

### 4.12 知识库

知识库应从“文档存储”升级为“审计方法库 + 项目经验库 + 证据索引库”：

- 方法库：制度、准则、底稿编制指引。
- 项目库：历史项目问题、客户行业特点。
- 证据库：附件、合同、函证、会议纪要。
- AI 检索必须显示来源段落、文档版本和引用位置。

建议尽快收敛知识库多套实现，明确 DB 文档、文件系统、向量索引之间的主从关系。

### 4.13 对话聊天与 LLM 辅助

LLM 不能直接成为审计结论。建议全局分级：

| 级别 | 含义 | 是否可入底稿 |
|---|---|---|
| suggestion | 仅建议 | 不可直接入底稿 |
| draft | 可作为草稿 | 人工确认后可入 |
| confirmed | 已人工确认 | 可入底稿 |
| rejected | 已拒绝 | 保留痕迹但不使用 |

所有 AI 输出必须记录 prompt、模型、时间、上下文来源、确认人。对报告、附注、重大判断、EQCR 事项，AI 内容必须强制人工确认。

### 4.14 数值处理

数值是审计平台生命线。建议：

- 后端金额一律 Decimal，不允许 float。
- 前端计算一律走 `decimal.ts`，不得用 JS 原生浮点做金额汇总。
- 所有金额 API 建议传字符串或固定精度 Decimal 序列化。
- 四表、调整、底稿、附注之间的金额差异必须有统一容差规则。
- 所有求和、差异、重分类、抵销均应有 trace。

### 4.15 复制粘贴

审计人员大量依赖 Excel 习惯。复制粘贴需要全局设计：

- 支持从 Excel 粘贴到表格，保留行列结构。
- 金额粘贴自动识别千分位、括号负数、中文空格。
- 粘贴前预览差异，避免覆盖公式列。
- 粘贴操作进入撤销栈和审计日志。
- 从平台复制到 Excel 时保持金额格式和行标题。

### 4.16 四表、未审报表、审计表、底稿、调整分录、附注之间的联动

这是平台最有价值、也最容易出错的链路。建议建立统一数据脊梁：

```text
四表/序时账
  -> 试算表
  -> 未审报表
  -> 调整分录 / 重分类分录
  -> 审定表 / 底稿
  -> 审定报表
  -> 附注
  -> 审计报告
```

每一层都需要回答四个问题：

1. 这个数字从哪里来？
2. 是否被人工改过？
3. 上游变化后是否 stale？
4. 下游有哪些地方会受影响？

建议把穿透能力做成全局组件 `TraceSourcePopover` 的升级版，不只显示来源，还显示影响范围、当前状态、可跳转页面、最后刷新时间。

## 6. 上线后维护便利性建议

### 6.1 建立“平台治理四本账”

| 账本 | 内容 | 负责人 |
|---|---|---|
| API 契约账 | 路由、请求响应、权限、错误码 | 后端负责人 |
| 组件账 | 全局组件、适用场景、禁用组件、迁移计划 | 前端负责人 |
| 数据口径账 | 金额、年度、准则、状态、stale、来源 | 产品/审计负责人 |
| 质量账 | 测试、PBT、压测、线上缺陷、回归范围 | QA/平台负责人 |

没有这四本账，企业级平台上线后会迅速变成“谁写的谁懂”。

### 6.2 代码维护规则

建议立即建立以下硬规则：

1. 新增页面必须说明使用哪些全局组件。
2. 新增金额字段必须说明 Decimal 序列化方式。
3. 新增状态必须进入枚举字典或说明为何不进入。
4. 新增路由必须配权限、错误码、前端路径。
5. 新增 AI 功能必须配人工确认机制。
6. 新增穿透必须使用统一 LinkageContract。
7. 新增表格不得裸用 `el-table`，除非有豁免说明。
8. 新增服务不得重复已有 composable/service 能力。
9. 新增数据库列必须 DDL、ORM、service、测试同步。
10. 新增导出/归档文件必须进入交付件中心版本链。

### 6.3 文档维护

已有文档很多，但需要避免“文档越来越多，权威越来越少”：

- 每个模块只保留一个 current 文档，旧文档进 archive。
- 架构规模数字自动生成，不手工写。
- 组件指南必须和 CI baseline 联动。
- 每个 ADR 要有状态：proposed / accepted / superseded / deprecated。
- 用户手册按角色写，不按模块写。

## 8. 我认为最值得保留和强化的亮点

不能只看问题。这个平台有不少方向是正确的：

1. 五大路由域拆分是对的，避免了单一 router_registry 继续膨胀。
2. 事件总线、stale、版本链、编辑锁这些企业级能力方向正确。
3. PBT 和三层一致契约测试的意识很强，适合审计平台这种高一致性系统。
4. 附注、合并、报表、底稿之间的联动野心很大，一旦收口好，会明显优于传统 Excel 作业。
5. EQCR、QC、合伙人看板等角色化模块说明平台不是只服务助理填表，而是在搭事务所管理体系。
6. 全局组件、CSS 变量、字典 store、apiPaths 已有基础，只需要更强的执行约束。

## 9. 最终建议

如果我是负责这个系统上线的合伙人，我会给出以下决策：

1. 当前平台可以进入受控试点，但不建议直接全所铺开。
2. 试点前必须先做“全局收口”，重点是项目上下文、权限、金额、stale、AI 确认、穿透契约。
3. 试点范围建议选择 1 个标准单体项目 + 1 个有合并/附注复杂度的项目，覆盖真实痛点。
4. 试点期间不要再大规模新增功能，而是记录五类角色每天卡在哪里。
5. 技术团队下一阶段 KPI 不应是“新增多少模块”，而应是“减少多少断点、统一多少组件、沉淀多少契约、降低多少维护成本”。

一句话总结：这个平台已经具备企业级审计平台的骨架，但要从“功能集合”进化为“事务所级作业系统”，关键不在再加功能，而在统一口径、统一入口、统一体验、统一治理。只有这样，助理愿意每天用，经理愿意靠它管项目，质控愿意用它抽查，合伙人愿意基于它签发，EQCR 愿意把它作为独立复核证据链。
