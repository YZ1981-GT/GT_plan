# 底稿模块内容体系专项复盘与优化建议（2026-06-06）

> 视角：大型会计师事务所项目合伙人 / 底稿体系负责人 / 平台建设负责人  
> 范围：审计底稿内容、阶段、循环、科目、文档、sheet、底稿联动、公式、LLM、人机协同、调整分录、报表、附注、复核与溯源  
> 关系：本文不替代 `workpaper-module-refinement-2026-05-30.md` 与各循环 spec；本文专注“底稿内容如何成为平台的一部分”，而不是再泛谈架构或页面瘦身。

---

## 0. 总体判断

底稿模块是整个审计平台最大的工作量，也是最核心的资产。当前系统已经有很多基础：底稿列表、编辑器、渲染 schema、Univer/HTML 双轨、`GtAuditSheet`、`GtDForm`、`GtCNoteTable`、公式管理、附件、复核、底稿关联、stale、AI、各循环弹窗、D 销售循环部分优化等。但从专业底稿建设角度看，目前仍处在“把模板内容塞进平台”的阶段，尚未真正完成“底稿内容平台化”。

我对当前底稿模块的判断是：

1. **编辑器壳子已经有了，但底稿内容模型还不够清楚。**
2. **不同底稿形态没有被系统性分类，导致很多 sheet 只是被动渲染。**
3. **底稿之间的关系有 `cross_wp_references` 和 D 循环增强，但还没有成为全平台底稿导航骨架。**
4. **自动提取、公式、调整分录、报表、附注之间已有通道，但未形成统一“数据链”。**
5. **LLM 能力有入口和部分弹窗，但还没有深度融入底稿编制过程。**
6. **内控了解、控制测试、穿行测试、控制偏差、实质性程序之间的递进关系尚未产品化。**

这不是一个简单 UI 问题，而是要把致同多年沉淀的底稿方法论转成平台的内容模型、交互模型、数据模型和审计判断模型。

## 1. 当前底稿模块的真实基础

### 1.1 已有关键代码与配置

| 类别 | 现有位置 | 说明 |
|---|---|---|
| 底稿编辑入口 | `WorkpaperEditor.vue` | 项目底稿编辑主入口 |
| 渲染分发 | `GtWpRenderer.vue` | 根据 render schema 分发不同 sheet 组件 |
| 审定表 | `GtAuditSheet.vue` | 审定表、调整、重分类、审定数计算 |
| D 类表单 | `GtDForm.vue` 及子组件 | table / paragraph / qa / confirmation / review |
| 附注披露表 | `GtCNoteTable.vue` | 底稿中的附注披露 sheet |
| 渲染 schema | `backend/data/wp_render_schema/*.yaml` | 376 个 schema，含 generated 与披露专属 schema |
| schema 服务 | `wp_render_schema_service.py` | 按 wp_code 精确、C-{wp_code}-disclosure、prefix fallback 加载 |
| 底稿-科目映射 | `wp_account_mapping.json` | 206 条底稿-科目-报表-附注映射 |
| 跨底稿引用 | `cross_wp_references.json` | =WP 公式、stale、依赖图基础 |
| 模板元数据 | `wp_template_metadata_seed.json` 等 | 底稿编码、阶段、循环、科目、目标、认定 |
| D 销售循环导航 | `useDSalesCycleSheetGroups.ts` | D 循环 14 类 sheet 分组 |

这些基础说明，平台已经具备底稿内容平台化的入口。但目前缺的是更高层的“底稿内容分类与审计逻辑编排”。

### 1.2 D 销售循环已有一点正确方向

D 销售循环的 `useDSalesCycleSheetGroups` 已经体现出正确思路：不再让用户面对横向铺开的几十个 sheet，而是按业务语义分组：

- 索引
- 总控台
- 审定表
- 明细表
- 坏账与减值
- 分析
- 截止测试
- 检查表
- 关联方
- 监盘
- 访谈
- 附注披露
- 调整分录
- 历史遗留

这正是底稿平台化需要推广到所有循环的模式。底稿不是 Excel sheet 的集合，而是“审计程序、取数、判断、证据、结论”的组合。

## 2. 底稿内容应建立统一分类模型

当前不同底稿形态没有完全理顺。建议建立统一的 `WorkpaperContentType` 与 `SheetContentType`。

### 2.1 底稿级分类

| 类型 | 示例 | 核心能力 |
|---|---|---|
| 审定表 | 货币资金审定表、应收票据审定表 | 取数、调整、审定数、报表/附注联动 |
| 明细表 | 应收账款明细、固定资产明细 | 明细导入、筛选、分组、抽样、穿透 |
| 程序表 | 实质性程序、控制测试程序 | 程序裁剪、执行状态、证据、结论 |
| 内控了解 | 了解销售循环、采购循环、资金循环 | 流程记录、关键控制点、穿行测试 |
| 控制测试 | 控制设计/执行有效性测试 | 样本、偏差、结论、控制依赖 |
| 分析性程序 | 毛利率、增长率、集中度 | 自动计算、异常提示、LLM 草拟说明 |
| 函证/询证 | 银行函证、往来函证 | 样本、发函、回函、差异调节 |
| 盘点/监盘 | 存货监盘、固定资产盘点 | 盘点记录、差异、照片/附件 |
| 减值/估值模型 | ECL、DCF、公允价值 | 模型参数、敏感性、复核提示 |
| 附注披露表 | 科目附注披露 sheet | 附注取数、报表一致性、公式 |
| 调整分录表 | AJE/RJE 明细 | 调整生成、影响预览、审批 |
| 结论/备忘录 | 审计说明、判断备忘录 | LLM 草稿、人审、版本与证据 |

### 2.2 Sheet 级分类

每个底稿内部 sheet 应有明确语义：

| Sheet 类型 | 示例 | 默认权限 | 是否可自动取数 |
|---|---|---|---|
| `control_panel` | D2A 程序总控台 | 可编辑程序状态 | 部分 |
| `audit_sheet` | 审定表D1-1 | 可编辑调整说明 | 是 |
| `detail_table` | 应收账款明细表 | 可编辑/导入 | 是 |
| `analysis` | 账龄分析、增长率分析 | 多为自动生成 | 是 |
| `procedure` | 实质性程序表 | 可勾选、写结论 | 部分 |
| `control_understanding` | 了解内控 | 可编辑文字/流程 | LLM 辅助 |
| `control_test` | 控制测试 | 可编辑样本/偏差 | 部分 |
| `confirmation` | 函证明细/回函 | 可编辑状态 | 是 |
| `disclosure` | 附注披露表 | 默认只读或受控编辑 | 是 |
| `adjustment` | 调整分录 | 受权限控制 | 是 |
| `legacy` | 修订前/原 | 默认隐藏只读 | 否 |

建议所有 render schema 都明确 sheet content type。不能只靠 sheet 名正则推断。

### 2.3 科目工作包：平台主对象应是“科目”，不是“Excel 文件”

你补充的这一点是底稿平台化的核心。很多科目不是一个 Excel 文件就能承载，例如应收账款、营业收入、固定资产、存货、长期股权投资等，往往由多个 Excel、多个工作表、多个专项测试共同组成。如果平台仍以“上传了几个 Excel 文件”作为用户看到的主对象，就会出现：

- 同一科目的程序散在多个文件里，助理不知道从哪里开始。
- 审定表、明细表、函证、截止测试、分析程序、附注披露表之间缺少统一状态。
- 经理无法一眼判断“这个科目完成到什么程度”。
- 复核人需要在多个文件之间来回跳。
- 公式和关联引用难以统一管理。

因此建议底稿平台的主对象调整为：

```text
项目
  -> 循环
    -> 科目工作包 Account Workpaper Package
      -> 程序控制台 control_panel
      -> sheet / procedure / document / attachment / model
```

也就是说，用户进入“应收账款”时，看到的不是 D2 某个 Excel 文件，而是一个完整的“应收账款工作包”。这个工作包下面可以包含多个来源文件、多个 sheet、多个审计应对程序。

#### 2.3.1 Account Workpaper Package 建议结构

```json
{
  "account_package_id": "D2_accounts_receivable",
  "cycle": "D",
  "account_code": "1122",
  "account_name": "应收账款",
  "report_row": "BS-005",
  "note_section": "五、3",
  "primary_wp_code": "D2",
  "control_panel_sheet": "D2A",
  "source_documents": [
    "D2 应收账款审定表.xlsx",
    "D2-2 应收账款明细表.xlsx",
    "D2-3 坏账准备明细表.xlsx",
    "D0 函证结果汇总.xlsx"
  ],
  "sheets": []
}
```

#### 2.3.2 程序控制台是科目工作包的核心

每个科目的程序控制台不应只是原 Excel 中的一个 sheet，而应成为平台中的“汇总审计程序表”。它需要汇总该科目下所有文档和 sheet 的程序状态。

程序控制台应包含：

| 区块 | 说明 |
|---|---|
| 常规程序 | 标准实质性程序、审定、明细核对、分析程序 |
| 专项程序 | 根据风险、行业、交易特征追加的程序 |
| 备用程序 | 默认不启用，但可由经理/合伙人启用 |
| IPO/上市/重组程序 | 场景触发，非普通项目默认隐藏 |
| 自定义程序 | 项目组新增，需说明原因 |
| 前置依赖 | B/C 类内控、风险评估、函证、资料清单 |
| 完成状态 | 每项程序完成、证据、复核、问题 |
| 数据状态 | TB/明细/函证/调整/附注是否最新 |
| 结论状态 | 是否形成科目结论，是否可提交复核 |

这样，“程序控制台”才是真正的科目驾驶舱，而不是一个普通 sheet。

#### 2.3.3 多 Excel 文件应被收敛为一个科目包

以应收账款和收入为例：

```text
D2 应收账款工作包
  D2A 程序控制台（汇总所有程序）
  D2-1 审定表
  D2-2 明细表
  D2-3 坏账准备 / ECL
  D2-4 账龄分析
  D2-5 函证覆盖与回函差异
  D2-6 期后回款
  D2-7 关联方应收
  D2-8 附注披露表
  D2-9 调整分录
  D2-C 科目结论

D4 营业收入工作包
  D4A 程序控制台
  D4-1 审定表
  D4-2 收入明细
  D4-3 收入分类 / 产品 / 区域
  D4-4 毛利率分析
  D4-5 收入截止测试
  D4-6 主要客户集中度
  D4-7 合同抽查
  D4-8 客户访谈
  D4-9 附注披露表
  D4-10 调整分录
  D4-C 科目结论
```

多个 Excel 文件只是这些 sheet / procedure 的来源，不应成为前端的主导航层。

#### 2.3.4 前端导航建议

前端应从“文件导航”升级为“科目工作包导航”：

```text
销售循环
  ├─ 应收票据 D1
  ├─ 应收账款 D2
  ├─ 预收账款/合同负债 D3/D7
  ├─ 营业收入 D4
  ├─ 合同资产 D6
  └─ 应收款项融资 D5
```

进入某个科目后，顶部显示：

```text
科目：应收账款 | 报表行：BS-005 | 附注：五、3 | 责任人 | 复核状态 | 数据状态
```

左侧或顶部 tab 才显示该科目下的程序 sheet：

```text
总控台 / 审定表 / 明细 / 函证 / 坏账 / 分析 / 截止 / 附注披露 / 调整 / 结论
```

这样更符合审计人员“按科目完成底稿”的心智。

## 3. 按阶段构建底稿内容体系

### 3.1 承接与保持阶段（B 类）

典型内容：

- 业务承接/保持评估
- 前任会计师沟通
- 独立性与利益冲突
- 项目风险初判
- 质量控制委员会记录
- 业务约定书

建议平台化：

1. 表单化承接/保持判断，不再只是 docx/xlsx 展示。
2. 与客户、项目、人员、独立性模块联动。
3. LLM 可辅助整理客户背景和风险点，但必须人工确认。
4. 承接结论应成为后续风险评估和项目计划的输入。
5. B 类底稿结论应能被合伙人首页和 EQCR 查看。

### 3.2 风险评估与计划阶段（A/B/C/S 类）

典型内容：

- 财务报表支持
- 总体审计策略
- 重要性水平
- 重大错报风险识别
- 了解企业及环境
- 了解内控
- 各循环控制了解
- 特殊事项底稿

建议平台化：

1. 重要性水平、风险识别、循环计划应成为全局参数，不应停留在单张底稿里。
2. C 类内控了解应结构化为“流程、控制点、风险、控制目标、责任岗位、频率、证据”。
3. 控制测试底稿应从内控了解自动生成候选控制点。
4. 若控制测试结论无效，后续实质性程序应自动提示增加样本或扩展程序。
5. S 类特殊事项应作为风险标签，反向影响 A/B/C/D-N 各循环。

### 3.3 实质性程序阶段（D-N 循环）

每个循环都应建立统一模式：

```text
循环总控台
  -> 审定表
  -> 明细表
  -> 分析程序
  -> 专项测试
  -> 附注披露表
  -> 调整分录
  -> 结论
```

当前 D 循环已有一些雏形，应把它推广到 E/F/G/H/I/J/K/L/M/N。

### 3.4 完成与报告阶段（A/S/报告/附注）

底稿应与报告、附注和归档形成闭环：

- 底稿审定数 → 报表
- 报表项目 → 附注
- 底稿程序结论 → 审计报告关键段落
- 调整分录 → 报表和附注
- 重大判断 → KAM / 强调事项 / EQCR
- 归档前检查 → 底稿质量清单

## 4. 按循环和科目建设内容模型

### 4.1 A 类：总体与报告支持

重点：

- 报告支持
- 重要性
- 重大错报风险
- 合并/单体口径
- KAM
- 管理层声明
- 期后事项

建议：

- A 类底稿应成为“签发控制台”的数据源。
- 报告段落引用的底稿、附注、调整必须可追溯。
- A 类不应只是文档归档，应与签发 checklist 联动。

### 4.2 B 类：承接保持与项目初始化

重点：

- 客户接受/保持
- 前任会计师沟通
- 独立性
- 业务约定
- KAA/质量控制

建议：

- B 类以表单和审批为主。
- 结论应影响项目状态和风险标签。
- LLM 可辅助提炼客户背景，但不得替代承接判断。

### 4.3 C 类：内控了解与控制测试

重点：

- 各循环业务流程了解
- 关键控制点识别
- 穿行测试
- 控制设计有效性
- 控制执行有效性
- 控制偏差和补充程序

建议：

1. 内控了解应采用流程卡片：
   - 流程步骤
   - 责任岗位
   - 系统/单据
   - 风险点
   - 控制点
   - 控制频率
   - 证据

2. 控制测试应从控制点自动生成：
   - 样本范围
   - 抽样方法
   - 测试步骤
   - 偏差记录
   - 结论

3. 控制测试结论应影响实质性程序：
   - 控制有效 → 可提示减少部分实质性程序
   - 控制无效 → 自动提示扩展样本/增加程序

4. 前端必须提供层层递进确认：
   - 已了解流程？
   - 已识别风险？
   - 已识别控制？
   - 是否依赖控制？
   - 是否执行控制测试？
   - 是否存在偏差？
   - 是否调整实质性程序？

这部分不能只靠鼠标悬停说明，应使用步骤条、弹窗确认、风险提示和引导式问答。

### 4.4 D 销售循环

关键科目：

- 应收票据
- 应收账款
- 预收账款/合同负债
- 营业收入
- 合同资产
- 应收款项融资

关键文档和 sheet 类型：

- D0 函证
- D1 应收票据审定表
- D2 应收账款审定表
- D3 预收账款审定表
- D4 营业收入审定表
- D5/D6/D7 合同资产、合同负债、应收款项融资
- 明细表
- 坏账/ECL
- 账龄分析
- 收入截止测试
- 毛利率/增长率/集中度分析
- 客户访谈
- 附注披露表
- 调整分录

你提到应收票据前几个 sheet 已经能体现思路，这正是应该推广的模式：每个 sheet 不再是 Excel 原样展示，而应明确它在审计链条中的角色。

建议 D1 应收票据示例结构：

```text
D1 应收票据
  D1A 程序总控台
  D1-1 审定表
  D1-2 票据明细表
  D1-3 票据类型/承兑人分析
  D1-4 背书/贴现/质押检查
  D1-5 坏账或信用风险评估
  D1-6 截止测试
  D1-7 附注披露表
  D1-8 调整分录
  D1-C 结论与复核
```

每个 sheet 应具备：

- 目的说明
- 数据来源
- 自动填充字段
- 需人工判断字段
- 公式与校验
- 相关底稿跳转
- 相关附件
- LLM 可辅助事项
- 提交复核前检查

### 4.4.1 参照应收票据，形成 D 循环“科目工作包”标准

应收票据前几个 sheet 已经体现出正确方向：平台不应展示一堆 Excel 文件，而应把它们整合成一个“科目工作包”。这个原则应推广到应收账款、营业收入等存在多个 Excel 文档的科目。

平台中的主对象应是：

```text
销售循环
  -> 应收票据工作包 D1
  -> 应收账款工作包 D2
  -> 营业收入工作包 D4
```

而不是：

```text
D2-1.xlsx
D2-2.xlsx
D2-3.xlsx
D4-1.xlsx
D4-2.xlsx
```

Excel 文件只是来源，sheet 才是具体审计应对程序或审计证据承载单元。科目工作包中的程序控制台应是这个科目的核心功能表，也是该科目下所有文档和 sheet 的汇总审计程序表。

程序控制台必须汇总：

| 类型 | 说明 |
|---|---|
| 常规程序 | 致同标准实质性程序 |
| 专项程序 | 风险触发或科目特定程序 |
| 备用程序 | 默认隐藏，可按需启用 |
| IPO/上市程序 | 场景触发，普通项目默认不启用 |
| 自定义程序 | 项目组新增，需说明原因 |
| 前置依赖 | B/C 类风险评估、内控了解、控制测试 |
| 数据状态 | 试算表、明细、函证、调整、附注是否最新 |
| 复核状态 | 编制、复核、退回、关闭 |
| 证据状态 | 附件、函证、样本、访谈记录 |
| 下游影响 | 报表、附注、调整分录、KAM、签发 |

### 4.4.1A 函证模块不是底稿 sheet，而是独立模块与底稿双向联动

需要特别强调：函证已有独立模块，不应被降级为 D 循环某个普通 sheet。当前代码中已有：

- 后端 `backend/app/routers/confirmations.py`
- 后端 `backend/app/services/confirmation_service.py`
- 后端 `backend/app/models/confirmation_models.py`
- 前端 `ConfirmationHub.vue`
- 前端事件 `confirmation:received`
- 前端测试 `ConfirmationHub.spec.ts`
- AI 相关 `confirmationAI`

因此底稿中的 D0 函证、D1 应收票据函证、D2 应收账款函证，应当与函证模块形成双向联动：

```text
函证模块
  -> 函证清单、发函、回函、差异、状态、附件、AI识别
  -> 回写 D0 函证汇总
  -> 回写 D1/D2 科目工作包的函证覆盖、回函差异、已确认金额
  -> 影响审定表、调整分录、附注披露、复核结论
```

#### 4.4.1A.1 函证模块作为事实真源

函证模块应成为以下信息的事实真源：

| 信息 | 真源 |
|---|---|
| 函证对象 | Confirmation |
| 函证类型 | receivable/payable/bank/loan |
| 发函状态 | Confirmation.status |
| 回函金额 | Confirmation.confirmed_amount |
| 账面金额 | Confirmation.book_amount |
| 差异金额 | Confirmation.diff_amount |
| 回函附件 | Attachment linked to confirmation |
| AI 地址/回函识别 | confirmationAI |

底稿不应重复维护这些字段，只应引用和汇总。

#### 4.4.1A.2 底稿工作包中展示函证状态

在 D1/D2 科目工作包中，函证应以“外部模块卡片”的形式出现，而不是复制一份函证表：

```text
D2 应收账款工作包
  函证卡片
    - 应函证客户数
    - 已发函
    - 已回函
    - 未回函
    - 回函不符
    - 已调节差异
    - 未解决差异
    - 函证附件
    - 跳转函证模块
```

程序控制台中的“函证程序”状态应来自函证模块，而不是手工勾选。

#### 4.4.1A.3 函证回函后的联动

函证模块状态变化后，应发布事件：

```text
confirmation:created
confirmation:sent
confirmation:returned
confirmation:mismatch_detected
confirmation:resolved
```

底稿模块订阅后：

- 更新 D0 函证汇总。
- 更新 D1/D2 工作包中的函证覆盖率。
- 更新 D2 审定表“已函证金额”“未函证金额”。
- 产生或提示调整分录。
- 触发报表/附注 stale。
- 进入复核工作台提醒。

#### 4.4.1A.4 函证与底稿的边界

| 能力 | 应属于函证模块 | 应属于底稿模块 |
|---|---|---|
| 创建函证 | 是 | 否，仅跳转 |
| 发函/回函状态 | 是 | 展示摘要 |
| 回函 OCR/AI 识别 | 是 | 展示结果和差异 |
| 回函差异调节 | 是为主 | 在底稿形成审计结论 |
| 函证覆盖率 | 函证模块计算 | 底稿引用展示 |
| 审定金额影响 | 提供确认金额 | 底稿写入审定表/调整 |
| 附注披露影响 | 提供差异和状态 | 底稿披露表/附注引用 |

#### 4.4.1A.5 对 D0 的定位

D0 不应再被理解为“函证模块本体”，而应是销售循环中的“函证底稿视图 / 汇总视图”：

```text
ConfirmationHub = 函证业务模块
D0 = 销售循环函证审计底稿视图
```

D0 负责：

- 按审计程序要求展示函证覆盖。
- 记录函证程序执行结论。
- 汇总回函差异处理结果。
- 作为 D1/D2/D4 等科目工作包的函证数据来源。

这样既尊重独立函证模块，又保留审计底稿体系中的 D0 工作底稿。

### 4.4.3 D2 应收账款工作包建议

应收账款往往存在多个 Excel 文档，不能让用户在文件层面自行拼接。平台应统一成 D2 应收账款工作包。

```text
D2 应收账款工作包
  D2A 程序控制台
    - 标准程序
    - 函证程序
    - 账龄分析程序
    - 坏账准备程序
    - 期后回款程序
    - 关联方程序
    - IPO/上市专项程序
    - 自定义程序
  D2-1 审定表
  D2-2 应收账款明细表
  D2-3 坏账准备明细表
  D2-4 账龄分析
  D2-5 函证覆盖与回函差异
  D2-6 期后回款
  D2-7 关联方应收
  D2-8 客户集中度
  D2-9 附注披露表
  D2-10 调整分录
  D2-C 科目结论
```

应收账款程序控制台应汇总：

- D0 函证结果
- 应收账款明细导入状态
- 坏账准备测算状态
- 账龄分析是否完成
- 期后回款是否完成
- 关联方余额是否识别
- 审定数是否与报表一致
- 附注披露是否已同步

关键联动：

- `ConfirmationHub 函证结果` → `D0 函证汇总视图` → `D2-5 函证覆盖与差异`
- `D2-2 明细` → `D2-4 账龄分析`
- `D2-4 账龄` → `D2-3 坏账准备`
- `D2-3 坏账准备` → `D2-1 审定表`
- `D2-1 审定数` → `报表 BS-005`
- `D2-9 附注披露表` → 附注“应收账款”
- `D2-10 调整分录` → 调整模块

其中函证联动应遵循：

- 函证对象和回函状态来自函证模块。
- D2 工作包只展示函证覆盖、差异和处理结果。
- 回函金额与账面金额差异应进入 D2 程序控制台。
- 未解决回函差异应阻断 D2 科目结论或至少 warning。
- 回函差异若需调整，应一键进入调整分录模块。
- 函证附件应作为 D2 科目证据链的一部分。

LLM 可辅助：

- 解释应收账款变动原因
- 汇总主要客户集中度
- 草拟函证差异原因摘要
- 草拟坏账准备合理性说明
- 提醒异常长账龄客户

但坏账准备结论、重大客户判断、关联方判断必须人工确认。

### 4.4.4 D4 营业收入工作包建议

收入科目通常文档更多、程序更多、场景更多，应以 D4 工作包统一管理，而不是多个 Excel 各自散开。

```text
D4 营业收入工作包
  D4A 程序控制台
    - 标准收入程序
    - 分析性程序
    - 截止测试
    - 合同抽查
    - 主要客户访谈
    - 关联方交易
    - 毛利率分析
    - IPO/上市专项
    - 自定义程序
  D4-1 审定表
  D4-2 收入明细表
  D4-3 按产品/服务分类
  D4-4 按客户分类
  D4-5 按地区/渠道分类
  D4-6 毛利率分析
  D4-7 收入截止测试
  D4-8 合同抽查
  D4-9 客户访谈
  D4-10 关联方收入
  D4-11 附注披露表
  D4-12 调整分录
  D4-C 科目结论
```

收入程序控制台应能根据风险自动提示：

- 收入增长异常 → 增加分析程序
- 毛利率波动异常 → 增加毛利率分析
- 期末集中确认收入 → 增加截止测试
- 关联方收入占比高 → 增加关联方专项
- IPO/上市项目 → 增加收入确认专项
- 合同条款复杂 → 增加合同抽查和履约义务判断

关键联动：

- `D4-2 收入明细` → `D4-1 审定表`
- `D4-2 收入明细` → `D4-6 毛利率分析`
- `D4-7 截止测试差异` → 调整分录
- `D4-1 审定数` → 利润表营业收入
- `D4-11 附注披露表` → 附注“营业收入”
- `D4-C 科目结论` → 报告/KAM/风险摘要

### 4.4.5 控制台中的程序来源和状态

每个科目程序控制台中的程序应标明来源：

| 来源 | 示例 |
|---|---|
| `standard` | 致同标准程序 |
| `risk_triggered` | 风险触发程序 |
| `scenario_triggered` | IPO/上市/重组触发 |
| `control_dependency` | 控制测试结论触发 |
| `llm_suggested` | LLM 建议但未确认 |
| `custom` | 项目组自定义 |
| `backup` | 备用程序 |

每项程序应有状态：

```text
not_started -> in_progress -> prepared -> reviewed -> resolved
```

并显示：

- 编制人
- 复核人
- 截止日
- 证据数量
- 复核意见数量
- 是否阻断提交
- 是否影响报表/附注

## 5. 底稿之间的关联应成为主导航

当前已有 `cross_wp_references.json`，但用户体验上还没有变成主导航。

建议建立“底稿关系图谱”，至少包括：

| 关系类型 | 示例 |
|---|---|
| 数据流 | H 折旧 → K 费用 / F 成本 |
| 反向回填 | D0 函证结果 → D2 已函证金额 |
| 前置依赖 | B/C 完成 → D 实质性程序 |
| 证据依赖 | 附件 → 底稿结论 |
| 调整影响 | 调整分录 → 审定表/报表/附注 |
| 报表支持 | 底稿 → 报表项目 |
| 附注披露 | 底稿披露表 → 附注章节 |
| 复核依赖 | 关键底稿未复核 → 合伙人不能签发 |

前端应支持：

- 当前底稿依赖哪些底稿
- 哪些底稿依赖当前底稿
- 哪些底稿未完成会影响当前底稿
- 当前底稿保存后影响哪些报表/附注
- 一键跳转
- stale 提示

## 6. 自动提取与填充建议

底稿平台化的核心是“哪些字段可以自动填，哪些必须人判断”。

### 6.1 自动填充字段

适合自动填：

- 期初数
- 未审数
- 审定数
- 调整分录合计
- 重分类合计
- 明细表合计
- 函证回函金额
- 账龄分组
- 折旧测算
- 利息测算
- 附注披露数

### 6.2 人工判断字段

必须人工确认：

- 审计程序是否适用
- 样本是否充分
- 控制是否有效
- 异常是否合理
- 估计参数是否合理
- 审计结论
- 重大判断说明
- AI 草拟内容

建议每个字段标识：

```json
{
  "field_id": "audited_amount",
  "source": "trial_balance",
  "editable": false,
  "requires_confirmation": false,
  "traceable": true
}
```

## 7. 公式管理需求

底稿公式不是简单 Excel 公式，而是审计数据链。

建议公式分四类：

| 类型 | 示例 |
|---|---|
| 表内公式 | 合计、差异、变动率 |
| 跨 sheet 公式 | 明细表合计 → 审定表 |
| 跨底稿公式 | D0 函证 → D2 应收账款 |
| 跨模块公式 | 底稿 → 报表 / 附注 |

公式管理应具备：

- 公式来源
- 公式依赖图
- 最近执行结果
- 错误提示
- 手工覆盖
- 恢复自动
- 版本历史
- 公式影响范围

不建议让用户直接面对大量 Excel 公式字符串，而应提供“业务公式说明”：

```text
应收账款审定数 = 未审数 + 账项调整 + 重分类调整
数据来源：试算表 + 调整分录
```

## 8. LLM 应成为底稿编制副驾驶

LLM 不应直接写死结论，而应参与以下环节：

| 环节 | LLM 作用 |
|---|---|
| 内控了解 | 根据访谈纪要草拟流程描述 |
| 控制点识别 | 提示可能的控制点和风险 |
| 控制测试 | 根据偏差草拟影响分析 |
| 分析程序 | 解释波动原因候选 |
| 审计说明 | 草拟“目标-程序-发现-结论” |
| 复核回复 | 根据证据草拟回复 |
| 附件 OCR | 提取合同、函证、发票关键字段 |
| 异常提示 | 对大额、异常、反常趋势提出关注点 |

但所有 LLM 输出必须：

- 标记为 AI 草稿
- 显示引用数据来源
- 人工确认后才能进入底稿
- 可拒绝
- 留存 prompt、模型、上下文、确认人

## 9. 交互建议：层层递进的确认，而不是悬停提示

用户明确提到“必要提示不限于鼠标悬停说明”，这非常关键。审计流程中，很多判断必须强提示、强确认。

建议交互层级：

| 场景 | 交互方式 |
|---|---|
| 普通知识说明 | hover / tooltip |
| 操作提示 | inline help |
| 需要补资料 | banner |
| 影响下游 | drawer 影响范围 |
| 覆盖系统值 | confirm dialog |
| 裁剪程序 | stepper + 理由 |
| 控制无效 | blocking modal |
| 签发前异常 | checklist 阻断 |

示例：内控了解递进确认：

```text
1. 是否已了解流程？
2. 是否识别风险？
3. 是否识别控制？
4. 是否依赖控制？
5. 是否执行控制测试？
6. 是否存在偏差？
7. 是否调整实质性程序？
```

每一步都应有结果、责任人和证据。

## 11. 最终建议

底稿模块下一阶段的重点不是继续把 Excel 原样搬进系统，而是把底稿内容拆成平台可理解的对象：

```text
阶段 -> 循环 -> 科目 -> 底稿 -> sheet -> block -> field/cell -> evidence -> conclusion
```

每个对象都要有：

- 业务目的
- 数据来源
- 适用条件
- 是否可自动填充
- 是否需要人工确认
- 公式和校验
- 上下游关系
- 附件证据
- LLM 可辅助点
- 复核状态
- 报表/附注影响

这才是真正把“致同底稿财富”沉淀到平台里。否则平台只是在显示 Excel；做完这一步，平台才是在承载审计方法论。
