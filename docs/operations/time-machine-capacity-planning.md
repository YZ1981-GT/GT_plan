# 时光机快照容量规划 — V3 Req 11.9

> 起草日期：2026-05-28
> 对应 spec：`.kiro/specs/global-refinement-v3/` Sprint 3 Req 11
> 涉及表：`time_machine_snapshots`（V017 迁移）
> 实现：`backend/app/services/time_machine_service.py` + `useWorkpaperAutoSave.ts`

## 1. 业务前提

| 参数 | 值 | 来源 |
|------|---|------|
| 目标并发用户数 | **6000** | memory.md `用户偏好` |
| 快照间隔 | **5 分钟**（300_000ms） | `useWorkpaperAutoSave.ts:165` |
| 快照保留窗口 | **7 天** | `time_machine_service.cleanup`（每日 03:00） |
| 实例类型 | workpaper / adjustment / disclosure | `instance_type` 字段 |
| 编辑高峰活跃比例 | 30% 同时在编辑（保守上限） | 业务侧估算（旺季） |
| 单次 diff 平均体积 | **3 KB**（JSON Patch 反向 diff） | 见 §3 实测 |

## 2. 写入吞吐估算

### 2.1 峰值写入 QPS

```
活跃编辑用户 = 6000 × 30%       = 1800 用户
快照频率     = 1 / 300s         = 0.0033 /s/用户
峰值 QPS    = 1800 × 0.0033     = 6 QPS（持续写入）
```

**结论**：6 QPS 远低于 PG 16 单机的 INSERT 吞吐（典型 ≥ 10000 QPS），**不构成瓶颈**。

### 2.2 写入冲突 / 锁竞争

`time_machine_snapshots` 索引：
- `idx_time_machine_snapshots_instance(instance_type, instance_id, created_at DESC)`
- `idx_time_machine_snapshots_project(project_id, created_at DESC)`

每次 INSERT 触发 2 个 B-Tree 索引更新；6 QPS × 2 索引 = 12 索引页写。无表级锁，行级锁仅在同一 instance 的并发快照（极小概率，单用户 5 分钟一次）。

**结论**：无锁竞争风险。

## 3. 单条快照体积测算

### 3.1 各业务实例 diff 量级

| 实例类型 | 原始数据典型大小 | 单次编辑 diff 平均 | 备注 |
|---------|----------------|-------------------|------|
| workpaper（附注/底稿） | 50-500 KB（parsed_data JSONB） | **2-5 KB** | 一次编辑改 1-3 个 cell |
| adjustment（调整分录） | 5 KB | **0.5-1 KB** | 单笔分录 5-10 字段变化 |
| disclosure（附注章节） | 20-100 KB | **1-3 KB** | 段落级编辑 |

**取保守上限**：单条 diff_json **≤ 5 KB（JSONB 压缩前）**。PG JSONB 含字典压缩，落盘约 **2-3 KB**。

### 3.2 单实例快照链长度

每实例快照频率 = 编辑会话内 5 分钟一次。  
典型审计旺季单 wp 编辑会话 = 30 分钟（6 个快照）。  
非编辑期不写。  

## 4. 7 天累计存储估算

### 4.1 总写入数

```
6000 用户 × 30% 活跃 = 1800 编辑用户
平均日活跃时长 = 4 小时 = 240 分钟
平均日快照数/用户 = 240 / 5 = 48 个
1800 × 48 = 86,400 快照/日
7 天 = 604,800 快照
```

### 4.2 落盘容量

```
604,800 快照 × 3 KB（JSONB 压缩后）= 1.81 GB
+ 索引开销 30% = 0.54 GB
+ TOAST 大字段开销（少量）≈ 100 MB
─────────────────────────────────────
合计：约 2.5 GB（7 天滚动）
```

**结论**：7 天滚动存储 **≤ 3 GB**，对 PG 16 单实例（典型部署 100 GB+ data 盘）**占比 < 3%，可忽略**。

### 4.3 清理任务负载

`time_machine_service.cleanup` 每日 03:00 删除 >7 天快照：

```
日清理量 = 86,400 行
DELETE 速率（含索引维护）≈ 5,000 行/秒
完成耗时 ≈ 17 秒
```

**结论**：清理任务在 03:00 低峰窗口内 < 1 分钟完成，**不影响业务**。

## 5. 风险与缓解

| 风险点 | 评估 | 缓解措施 |
|-------|------|---------|
| 单 wp 持续编辑 8 小时 → 96 个快照 | diff 链增长 | 已有：service 层 `parent_snapshot_id` + `cleanup` 每日删 >7 天 |
| 旺季 6000 用户 100% 同时编辑 | QPS 翻 3.3x → 20 QPS | 仍远低于 PG 容量；如真发生可转 partition by date |
| diff_json 异常膨胀（如全量序列化） | 单条 > 100 KB | service 层应监控 `len(diff_json) > 50_000` 告警 |
| jsonpatch 库 OOM（历史超长） | 单实例链 > 100 节点 | 已有：1 小时滚动 + 30 分钟粒度合并（spec §11 AC 11.3） |
| 跨期数据迁移 | 年度切换时旧快照失效语义 | 7 天 TTL 自然清理 |

## 6. 验收数据点

下列阈值用于上线后监控（建议接入 Grafana）：

| 指标 | 告警阈值 | 严重阈值 |
|------|---------|---------|
| 单日新增快照数 | > 200,000 | > 500,000 |
| `time_machine_snapshots` 表大小 | > 5 GB | > 10 GB |
| 单条 diff_json 大小 | > 50 KB | > 200 KB |
| 单实例链长（仍未清理） | > 50 | > 200 |
| `cleanup_worker` 单次耗时 | > 60s | > 5 min |

## 7. 结论

**6000 用户场景下时光机机制安全可上线**，关键容量参数 **远低于 PG 16 单机吞吐与存储上限**：

- 写入 QPS：6 / 容量 10,000+
- 7 天存储：2.5 GB / 数据盘 100 GB+
- 清理耗时：17 秒 / 03:00 无压力窗口

无需引入额外架构（分库 / 归档冷存）。后续若用户规模扩到 60,000，再评估按 `project_id` 范围分表或迁冷数据到 OSS。
