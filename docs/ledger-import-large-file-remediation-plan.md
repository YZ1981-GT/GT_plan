# 账表大文档上传、识别、入库企业级修复方案

## 1. 文档目的

本文用于在正式开始修复前，统一大文档场景下的修复目标、分阶段步骤、注意事项、风险控制、验收标准与回滚策略。

适用范围：

- `Excel`、`CSV` 大文档上传
- 智能预览（`smart-preview`）
- 智能导入（`smart-import`）
- 多 sheet、多文件、辅助维度拆分、企业/年度识别
- 前端导入交互、后端解析、后台任务、写库与激活机制

本文基于当前实现进行分析，重点关注以下现有代码入口：

- `backend/app/routers/ledger_penetration.py`
- `backend/app/services/smart_import_engine.py`
- `backend/app/services/import_queue_service.py`
- `backend/app/services/import_engine/parsers.py`
- `audit-platform/frontend/src/views/LedgerPenetration.vue`

**说明：** 本文是修复前的设计与执行文档，不代表已经实施代码改造。

---

## 2. 当前现状与主要问题

### 2.1 前端现状

当前前端在 `LedgerPenetration.vue` 中通过 `FormData` 上传文件：

- 预览调用 `/ledger/smart-preview`
- 导入调用 `/ledger/smart-import`
- 导入后通过轮询 `/api/data-lifecycle/import-queue/{project_id}` 获取进度

当前优点：

- 浏览器未直接解析 Excel/CSV，大部分压力在服务端
- 导入走后台任务提交，避免前端长时间等待一个 HTTP 请求
- 预览只展示诊断与统计，不把全量数据返回前端

当前不足：

- 前端没有文件大小、数量、总容量的前置保护
- 没有取消导入机制
- 没有分片/断点续传能力
- 大文件预览与导入仍然是同步上传整文件，网络与服务端压力集中

### 2.2 后端现状

当前后端在 `ledger_penetration.py` 中对上传文件直接执行完整读取：

- 预览路径：`await f.read()` 后进入 `smart_parse_files(...)`
- 导入路径：`await f.read()` 后进入 `smart_import_streaming(...)`

这意味着：

- 请求入口先把整个文件读入内存
- 多文件并发时内存峰值会快速叠加
- 即使预览只读取前 `50` 行，上传文件本体也已经完整进入 Python 内存

### 2.3 解析与写库现状

#### CSV 路径

`_stream_csv_import(...)` 已具备较好的大文件处理能力：

- 流式解码
- 分批解析
- 分批转换
- `COPY` 写入数据库
- 每批提交一次，内存上界相对可控

#### Excel 路径

Excel 路径当前是主要短板：

- `smart_import_streaming(...)` 名义上是流式导入
- 但 `smart_parse_sheet(...)` 会先把一个 sheet 的所有行放入 `rows` 列表
- 后续再做 `convert_balance_rows(...)` / `convert_ledger_rows(...)`
- 实际仍是“整 sheet 先入内存，再转换写入”

这会在以下场景中产生高风险：

- 超大单 sheet 序时账
- 多个 ledger sheet 同时存在
- 辅助维度拆分后记录数膨胀
- 合并表头导致 workbook 进入 full mode

### 2.4 数据安全现状

当前 `smart_import_streaming(...)` 在导入早期就执行年度旧数据清理：

- 先清理旧数据
- 再开始解析与导入新数据

问题在于：

- 如果导入中途失败，旧数据已被清空
- 新数据未完整写入时，项目年度数据处于不一致状态
- 这不符合企业级系统对“失败不影响线上有效数据”的要求

### 2.5 任务可靠性现状

当前后台任务通过 `asyncio.create_task(...)` 启动。

现有优点：

- 简单直接
- 开发环境可用
- 已结合 `ImportQueueService` 做了项目级锁和进度持久化

现有问题：

- Web 进程重启时任务可能丢失
- 多 worker/多实例场景下不够可靠
- 任务执行与 Web 服务生命周期耦合过深

## 3. 修复目标与非目标

## 3.1 修复目标

本轮修复的核心目标是：

- **前端不可因大文件上传/预览/导入而明显卡死**
- **后端内存峰值可预测、可限制、可监控**
- **Excel 与 CSV 都具备真正的批式处理能力**
- **导入失败不影响当前生效账套数据**
- **后台任务与 Web 请求解耦，具备更高可靠性**
- **保留现有智能识别能力，不牺牲多 sheet、多表头、企业/年度提取能力**
- **保留辅助维度拆分能力，但避免在内存中一次性膨胀**

## 3.2 非目标

本轮不建议同时承诺以下事项：

- 兼容所有历史 `.xls` 老格式
- 一次性重做所有导入 UI
- 在第一阶段就实现完整断点续传与断点恢复
- 同时重写所有已有导入服务
- 将所有导入都统一到新架构后再发布

原因：

- 范围过大，周期过长
- 风险过于集中
- 不利于渐进式上线与回滚

---

## 4. 设计原则

修复必须遵循以下原则：

### 4.1 有界内存

任何阶段都不允许：

- 入口层无限制持有整批大文件字节
- 解析层无限制持有整 sheet 全量行对象
- 辅助维度拆分后一次性膨胀成超大列表

### 4.2 失败隔离

任何导入失败都不应影响当前线上有效数据。

### 4.3 任务解耦

上传请求、预览任务、导入任务、写库任务应具备清晰边界。

### 4.4 可回滚

每个阶段都必须有明确回滚点，而不是“改完后只能一路往前”。

### 4.5 可观测

没有指标与日志的重构，不允许直接上生产。

### 4.6 渐进发布

优先先让系统“稳住”，再追求更复杂的优化能力。

## 5. 关键架构决策与论证

## 5.1 文件接入：继续用内存字节，还是先落临时存储？

### 方案 A：继续使用 `await f.read()` 读成内存字节

优点：

- 改动小
- 当前逻辑兼容性最好

缺点：

- 请求入口内存风险极高
- 预览/导入重复持有大对象
- 多文件上传时内存峰值不可控

### 方案 B：上传后先落临时文件/临时存储，再把引用交给后续流程

优点：

- 显著降低请求层内存峰值
- 预览与导入可复用同一份上传产物
- 更适合异步任务和后续 worker 化

缺点：

- 需要补充临时文件生命周期管理
- 需要处理过期清理、失败清理、孤儿文件清理

### 结论

**推荐采用方案 B。**

这是整个修复的基础前提。如果入口层仍然整文件读入内存，那么后续所有“流式优化”都会被入口层的峰值抵消。

---

## 5.2 预览模式：继续同步，还是引入异步预览？

### 方案 A：所有预览都同步执行

优点：

- 前端交互最直接
- 对中小文件体验较好

缺点：

- 大文件仍会长时间占用 API worker
- 超大 Excel 即使只预览前 50 行，也可能慢
- 高并发下容易拖慢整体服务

### 方案 B：小文件同步预览，大文件异步预览

优点：

- 小文件体验好
- 大文件能避免阻塞 API 请求线程
- 更符合企业级导入平台设计

缺点：

- 前端需要兼容两种交互方式
- 预览任务也需要状态管理

### 结论

**推荐采用方案 B。**

理由：

- 当前用户最在意的是“不卡死、不频繁溢出”
- 同步预览可以保留，但必须加体积阈值
- 超阈值后自动降级为异步预览是更稳妥的企业级策略

---

## 5.3 Excel 解析：整 sheet 入内存，还是按批生成？

### 方案 A：保持 `smart_parse_sheet(...) -> rows[]`

优点：

- 复用当前逻辑最多
- 对现有转换函数侵入较小

缺点：

- 本质上无法解决大 Excel 内存问题
- 辅助维度放大时风险更高
- 与“流式导入”目标矛盾

### 方案 B：拆为“表头扫描 + 数据迭代器/批生成器”

优点：

- 真正做到有界内存
- 可以统一 CSV/Excel 的批式转换接口
- 更利于后续任务化、监控化、限流化

缺点：

- 改造范围较大
- 需要重新整理表头识别、企业/年度注入、自定义映射等逻辑

### 结论

**推荐采用方案 B。**

这是本轮修复最核心的技术改造点。

---

## 5.4 数据生效：先删旧数据，还是先导入新版本再切换？

### 方案 A：保持“先清旧数据，再导新数据”

优点：

- 逻辑简单
- 当前代码改动少

缺点：

- 导入失败会导致线上数据缺失
- 与企业级系统要求冲突

### 方案 B：引入“准备态数据集 -> 校验/汇总 -> 激活切换”机制

优点：

- 导入失败不影响线上数据
- 可以为后续回滚、重试、对账留出空间
- 符合生产系统的数据发布思路

缺点：

- 需要设计数据集版本或激活语义
- 对写库流程和汇总流程都有影响

### 结论

**推荐采用方案 B。**

这是数据安全层面的 P0。

---

## 5.5 后台任务：继续 `create_task`，还是迁移到独立 worker？

### 方案 A：继续使用 `asyncio.create_task(...)`

优点：

- 开发最快
- 对当前 API 改动少

缺点：

- Web 服务重启即丢任务
- 多实例一致性差
- 不利于限流、重试、取消和运维排障

### 方案 B：使用独立任务执行器 / worker 进程

优点：

- 生命周期独立于 Web 服务
- 更容易做并发控制、超时、重试和取消
- 更适合生产环境

缺点：

- 落地成本高于 `create_task`
- 需要规划任务状态与执行器协同方式

### 结论

**推荐采用方案 B。**

但不建议一口气重构完所有任务体系后再发布，应采用“先稳住入口与解析，再迁移执行器”的顺序。

## 6. 分阶段修复步骤

下面按推荐执行顺序给出具体步骤。每个阶段都包含：目标、操作步骤、注意事项、验收标准、回滚策略。

---

## 阶段 0：建立基线与防护栏

### 目标

在不改变主流程的前提下，先让系统具备“可观测、可限流、可拒绝危险请求”的能力。

### 操作步骤

#### 步骤 0.1：补充导入链路指标与日志

需要记录：

- 文件名、文件大小、文件类型
- sheet 数量
- 预览耗时、导入耗时、写库耗时
- 行数、辅助维度拆分后行数
- 失败阶段（上传、解析、映射、转换、写库、汇总、激活）

#### 步骤 0.2：增加上传保护阈值

至少增加以下限制：

- 单文件大小上限
- 单次总上传体积上限
- 单次文件数量上限
- 预估 sheet 数上限
- 预估辅助维度膨胀告警阈值

#### 步骤 0.3：前端展示容量提醒

前端在选择文件后，至少应提示：

- 当前已选文件数
- 总体积
- 是否超过推荐预览阈值
- 是否将自动走异步处理

### 注意事项

- 阈值要配置化，不能硬编码死在视图或路由里
- 日志不要打印敏感业务数据内容，只打印结构性指标
- 指标先做最小闭环，不要一开始就铺太大

### 验收标准

- 能从日志明确看到每个文件的处理阶段与耗时
- 大于阈值的请求会收到明确提示或被降级处理
- 没有改坏现有小文件导入流程

### 回滚策略

- 阈值放宽或关闭
- 新日志保留，不影响主流程

---

## 阶段 1：重构文件接入层，去掉请求内的整文件内存驻留

### 目标

把文件接入从“内存字节传递”改成“临时存储引用传递”。

### 操作步骤

#### 步骤 1.1：上传后落临时文件

预览与导入都不再直接把 `UploadFile` 全量 `read()` 后塞给解析器，而是：

- 逐块写入临时文件
- 保存文件元信息
- 后续处理只传递文件引用

#### 步骤 1.2：统一临时文件生命周期

至少要覆盖：

- 预览成功但未导入
- 导入成功
- 导入失败
- 用户取消
- worker 异常退出
- 过期任务无人处理

#### 步骤 1.3：让预览与导入复用同一份上传产物

避免：

- 预览上传一次
- 确认导入再上传一次

推荐思路：

- 预览完成后返回一个临时产物标识
- 导入时直接引用该产物，而不是要求前端重新上传文件

### 注意事项

- Windows 下要注意文件句柄释放，否则后续读取/删除会失败
- 清理任务要考虑进程崩溃后的孤儿文件
- 临时文件目录要和正式数据目录隔离
- 文件名不能直接信任用户输入，要做安全处理

### 验收标准

- 预览和导入接口不再以完整字节数组作为主数据载体
- 大文件上传时 API 进程内存峰值明显下降
- 同一批文件预览后再导入，无需重复上传

### 回滚策略

- 保留老的字节流路径作为 feature flag 下的回退方案
- 如临时文件机制不稳定，可临时回退到原接口实现

---

## 阶段 2：重构预览路径，建立“小文件同步、大文件异步”的双模式

### 目标

让预览从“统一同步处理”升级为“按文件规模分流处理”。

### 操作步骤

#### 步骤 2.1：定义同步预览阈值

建议依据以下指标组合判断：

- 文件大小
- 文件数量
- Excel sheet 数量
- 是否命中复杂表头/合并单元格特征

#### 步骤 2.2：同步预览只做轻量动作

同步预览仅保留：

- 表头识别
- 企业/年度提取
- 前 `N` 行样本解析
- 列映射诊断
- 样本级辅助维度膨胀比例估算

不要在同步预览中做：

- 全表转换
- 全量一致性校验
- 大批量辅助行生成

#### 步骤 2.3：超阈值预览改为异步任务

异步预览返回：

- 任务已创建
- 前端轮询或订阅状态
- 完成后再拿诊断结果

### 注意事项

- 样本估算必须在界面上标明“估算值”，不能误导为精确值
- 异步预览的任务表述要与导入任务区分清楚
- 同步与异步的返回结构应尽量统一，减少前端分支复杂度

### 验收标准

- 小文件预览体验不下降
- 大文件预览不会长时间占用 API worker
- 前端可以清晰区分“预览中”“预览完成”“预览失败”

### 回滚策略

- 可临时只保留同步预览，但阈值要严格限制

---

## 阶段 3：重构 Excel 解析器为真正的批式处理

### 目标

彻底消除“整 sheet `rows[]` 全量驻留内存”的模式。

### 操作步骤

#### 步骤 3.1：拆分表头扫描与数据扫描

将现有 `smart_parse_sheet(...)` 拆分为两个逻辑阶段：

- 第一阶段：扫描有限行数，识别表头、企业、年度、映射关系
- 第二阶段：基于映射结果，按批生成数据行

#### 步骤 3.2：统一 CSV/Excel 的批处理接口

目标是让后续转换逻辑接收统一的“批数据”，而不是依赖“完整 rows 列表”。

#### 步骤 3.3：保留现有业务能力

重构过程中必须保留：

- 多行表头合并
- 企业名称提取
- 年度提取
- 多 sheet 识别
- 用户自定义列映射
- 辅助维度识别与拆分

#### 步骤 3.4：把辅助维度拆分放到批处理中进行

避免先把原始行与拆分后的辅助行都放进超大内存结构。

### 注意事项

- 合并单元格识别不要粗暴让整本 workbook 进入 full mode
- 推荐只在“表头扫描窗口”里处理复杂表头，数据区仍尽量保持 `read_only`
- 自定义映射应用时要避免每批重复做高成本重映射
- 需要重点防止预览逻辑和导入逻辑走成两套不同识别规则

### 验收标准

- 大 Excel 导入时，解析阶段内存曲线明显平稳
- 单个超大 sheet 不再因 `rows[]` 全量驻留造成峰值暴涨
- 现有复杂表头样例、企业/年度提取样例不回归

### 回滚策略

- 保留老解析器作为 feature flag 下的兼容模式
- 新旧解析器在灰度期并行对比结果

---

## 阶段 4：重构写库与生效机制，禁止“先清旧数据再导新数据”

### 目标

建立“准备态导入 -> 完成校验/汇总 -> 激活切换”的数据发布机制。

### 操作步骤

#### 步骤 4.1：定义准备态数据集语义

新导入的数据在未完全成功前，不应直接替代当前有效数据。

至少需要明确：

- 哪一批数据处于 preparing 状态
- 哪一批数据处于 active 状态
- 哪一批数据处于 failed / obsolete 状态

#### 步骤 4.2：准备态导入完成后再执行汇总与校验

包括：

- 辅助余额汇总
- 必要的一致性校验
- 账号提取/补全

#### 步骤 4.3：激活切换必须是明确的最终步骤

只有在以下都成功后，才允许切换：

- 原始数据写入成功
- 辅助数据写入成功
- 汇总数据生成成功
- 必要校验通过

### 注意事项

- 不建议把“旧数据删除”放在导入最开始
- 激活步骤必须幂等，重复执行不能造成数据错乱
- 若短期内无法引入完整版本模型，也必须先实现“先导入新批次，再统一切换可见性”的最小安全闭环

### 验收标准

- 导入失败时，当前线上有效数据不受影响
- 导入成功后，新旧数据切换逻辑清晰可验证
- 辅助余额汇总与主表数据属于同一批次语义

### 回滚策略

- 激活失败时，系统应保持旧数据继续可见
- 新批次数据可标记为失败或废弃，等待运维处理

---

## 阶段 5：将后台执行从 Web 进程中解耦

### 目标

把长时间导入任务从 `create_task(...)` 迁移为独立执行模型。

### 操作步骤

#### 步骤 5.1：明确 API 与执行器边界

API 层只负责：

- 接收请求
- 校验参数
- 创建任务
- 返回任务标识

执行器负责：

- 解析文件
- 写入数据
- 更新进度
- 处理失败与重试

#### 步骤 5.2：保留现有队列状态模型，但替换执行方式

`ImportQueueService` 目前可以继续承担：

- 项目级互斥
- 进度查询
- 失败状态落库

但真正执行不应再依赖 Web 进程内存任务。

#### 步骤 5.3：补齐心跳、超时、取消、重试语义

至少需要：

- 心跳更新时间
- 超时失效处理
- 用户取消标记
- 失败重试策略

### 注意事项

- 不要在迁移初期同时把前端进度协议全部推翻
- 先保持查询接口兼容，再逐步优化进度事件推送
- 任务执行器要有最大并发控制，避免多个大任务同时打爆机器

### 验收标准

- Web 服务重启不再直接丢失导入任务执行能力
- 任务状态可持续查询
- 长任务有超时和失败语义

### 回滚策略

- worker 执行器不稳定时，可临时回退到单实例任务执行，但必须保留严格并发限制

---

## 阶段 6：补齐前端交互与运维能力

### 目标

让用户与运维都能看懂、控制、恢复导入流程。

### 操作步骤

#### 步骤 6.1：补充前端导入交互

建议补齐：

- 文件大小/数量提示
- 是否异步预览提示
- 当前任务阶段展示
- 取消任务入口
- 失败后的重试入口

#### 步骤 6.2：补充后台运维能力

至少需要：

- 清理过期临时文件
- 释放卡死任务
- 查看失败阶段
- 查看每批记录数、耗时、文件信息

#### 步骤 6.3：补充告警与巡检

关注以下告警：

- 单任务耗时异常长
- 单任务写库条数异常大
- 辅助维度膨胀比例异常
- 频繁 OOM/进程重启
- 连续导入失败

### 注意事项

- 进度文案要表达真实阶段，避免“卡在 88%”这种模糊感受
- 用户取消不能等同于系统失败，状态语义要区分清楚
- 运维工具必须能定位是上传、解析、映射还是写库出了问题

### 验收标准

- 用户能够理解任务当前阶段
- 运维能够单独清理僵尸任务与孤儿文件
- 失败问题可定位到具体阶段

### 回滚策略

- 新增运维功能原则上不需要回滚，只要不影响主流程即可

## 10. 最终建议

综合现状与风险，修复优先级建议如下：

### 第一优先级

- 去掉请求层整文件内存读取
- 建立临时文件接入层

### 第二优先级

- Excel 真正批式解析
- 消除整 sheet `rows[]` 全量驻留

### 第三优先级

- 改造数据生效机制，禁止“先清旧数据再导新数据”

### 第四优先级

- 把后台任务从 `create_task(...)` 迁移到独立执行器

### 第五优先级

- 补齐取消、重试、告警、运维能力

**总原则：** 先稳住内存与数据安全，再优化体验与任务体系；先分阶段落地，再逐步统一架构；先让系统不出事故，再追求更优雅的实现。

## 11. 全面修复补充：全链路修复矩阵

从全面修复角度看，本次改造不能只盯住某一个函数或某一个接口，而要把整个链路拆成多个治理面，分别明确问题、目标、优先级与依赖关系。

### 11.1 修复矩阵总览

| 维度 | 当前主要问题 | 修复目标 | 优先级 | 主要依赖 |
|------|--------------|----------|--------|----------|
| 上传接入层 | 请求内全量 `read()`，内存峰值高 | 文件先落临时存储，接口层不持有大字节 | P0 | 无 |
| 预览层 | 大文件同步预览阻塞 API | 小文件同步，大文件异步 | P0 | 上传接入层 |
| Excel 解析层 | 整 sheet 先落 `rows[]` | 真正按批扫描与产出 | P0 | 预览/导入解析统一策略 |
| CSV 解析层 | 整体较好，但接口层仍会放大风险 | 继续保留流式，统一接入层 | P1 | 上传接入层 |
| 转换层 | 辅助维度拆分可能放大内存 | 边解析边拆分边写入 | P0 | Excel 批式解析 |
| 写库层 | 导入与激活未解耦 | 准备态写入 + 激活切换 | P0 | 数据模型调整 |
| 任务执行层 | `create_task(...)` 依赖 Web 进程 | 独立 worker 执行 | P1 | 接入层、任务模型 |
| 前端交互层 | 无容量护栏、无取消、状态不细 | 任务分阶段状态机 | P1 | 后端任务状态增强 |
| 运维层 | 缺少容量告警与僵尸任务治理 | 具备清理、诊断、告警 | P1 | 观测指标、任务模型 |
| 数据安全层 | 失败可能破坏有效数据 | 导入失败零影响线上数据 | P0 | 准备态/激活机制 |

### 11.2 全链路必须同时守住的四条底线

- **[底线一]** 任何单个请求都不能因为文件过大而无限抬高 API 进程内存。
- **[底线二]** 任何一次导入失败都不能直接破坏当前已生效数据。
- **[底线三]** 任何一个长任务都不能只靠 Web 进程内存存活。
- **[底线四]** 任何一个高膨胀数据集都必须在“可观测”和“可中止”的前提下处理。

如果这四条里有任意一条没有满足，就不应认定为“全面修复完成”。

## 12. 依赖关系与实施先后顺序细化

全面修复最容易失败的地方，不是技术做不到，而是顺序搞反。下面给出推荐顺序和原因。

### 12.1 为什么不能先改 worker，再改解析器

如果先把任务执行迁到 worker，但 Excel 解析仍然是整 sheet 入内存，那么：

- 只是把 OOM 从 Web 进程挪到 worker 进程
- 没有解决根因
- 反而会让排障更复杂

所以正确顺序是：

- 先解决**接入层内存模型**
- 再解决**解析层内存模型**
- 最后再解决**任务执行模型**

### 12.2 为什么不能先改前端体验，再改后端生效机制

如果先做取消按钮、任务阶段 UI、预览优化，而导入仍然是“先删旧数据再导新数据”，那么：

- 用户体验看起来更好了
- 但真实数据安全问题没有解决
- 一旦导入失败，业务影响仍然严重

因此数据安全相关的改造优先级必须高于体验优化。

### 12.3 推荐的严格执行顺序

推荐执行顺序如下：

1. 上传接入层改造
2. 预览分流与容量护栏
3. Excel 批式解析与批式转换
4. 准备态写入与激活切换
5. 独立 worker 执行
6. 前端取消/重试/运维增强

任何跳步都需要额外说明理由，并评估是否会放大线上风险。

---

## 13. P0 / P1 / P2 实施项进一步细化

下面从“可以真正拆成工程任务”的角度，细化每个阶段的工作包。

### 13.1 P0：必须优先完成的工程项

#### P0-1 上传接入层重构

核心动作：

- 将上传文件写入临时存储，而不是在路由层整文件读入内存
- 为每个上传产物分配稳定标识
- 让预览与导入都基于该标识工作

验收重点：

- 多文件上传时 API 内存曲线明显下降
- 同一批文件不需要重复上传

#### P0-2 容量护栏与降级策略

核心动作：

- 配置单文件、总文件、总大小、sheet 数阈值
- 配置同步预览阈值
- 超阈值后自动改走异步任务

验收重点：

- 超大文件不再直接走同步全链路
- 用户收到清晰反馈而不是接口超时

#### P0-3 Excel 真正批式解析

核心动作：

- 将表头扫描与数据扫描分离
- 数据扫描按批输出统一结构
- 辅助维度拆分在批处理中完成

验收重点：

- 大 sheet 解析时内存峰值受控
- 样例识别能力不回归

#### P0-4 数据安全改造

核心动作：

- 新批次先写准备态
- 成功后再激活
- 失败不影响 active 数据

验收重点：

- 人工制造失败后，当前查询结果仍为旧数据
- 新旧版本切换可审计、可追踪

### 13.2 P1：系统工程能力补齐

#### P1-1 独立 worker 与任务生命周期

核心动作：

- 任务提交与任务执行解耦
- 增加心跳、超时、取消、重试语义
- 限制 worker 最大并发

#### P1-2 前端任务状态机升级

核心动作：

- 将状态拆成上传中、预扫描、预览中、等待确认、导入中、汇总中、激活中、完成、失败、取消
- 在界面上显示阶段而不是只显示百分比

#### P1-3 运维工具与巡检能力

核心动作：

- 补充过期临时文件清理
- 补充僵尸任务处理
- 补充导入失败分阶段诊断页或诊断数据

### 13.3 P2：长期优化项

#### P2-1 更完善的断点续传与断点恢复

适用于：

- 网络不稳定
- 文件特别大
- 用户上传成本高

#### P2-2 更细粒度的导入版本治理

例如：

- 多批次比对
- 历史版本回滚
- 版本间差异审计

#### P2-3 更智能的容量预估与告警模型

例如：

- 基于样本预测最终辅助维度膨胀倍数
- 提前预测导入耗时和写库压力

---

## 14. 数据模型与状态模型细化建议

全面修复不只是改逻辑，还需要明确数据层和状态层的承载方式。这里给出建议方向，分为“最小可行改造”和“理想形态改造”。

### 14.1 最小可行改造

如果希望尽快落地，可优先在现有模型基础上补充以下语义：

- 上传产物标识
- 导入任务标识
- 批次状态：`preparing`、`active`、`failed`、`obsolete`
- 上传文件路径/存储引用
- 任务最近心跳时间
- 是否允许取消

适用场景：

- 需要快速落地 P0
- 当前数据库改动窗口有限

### 14.2 理想形态改造

如果允许进行更系统的建模，建议拆成三个层次：

- **[文件资产层]** 记录上传产物、文件大小、哈希、存储位置、过期时间
- **[任务层]** 记录预览任务、导入任务、执行状态、执行器信息、心跳、失败原因
- **[数据集层]** 记录某项目某年度某次导入形成的数据集版本，区分 preparing/active

好处：

- 上传、任务、数据版本职责清晰
- 便于回滚、重试、审计、复用
- 更适合未来扩展多阶段处理

### 14.3 状态流转必须满足的约束

至少需要满足：

- `preparing -> active` 必须是明确的最终切换动作
- `preparing -> failed` 后不得影响当前 `active`
- `active -> obsolete` 必须发生在新版本成功激活之后
- `cancelled` 与 `failed` 必须是不同语义

---

## 15. 接口、前端状态机与用户交互细化

### 15.1 前端不能再把“预览”和“导入”理解成两个完全独立上传动作

推荐目标：

- 第一步：上传并创建上传产物
- 第二步：基于上传产物做预览
- 第三步：用户确认映射
- 第四步：基于同一上传产物发起导入

这样可以避免：

- 重复上传
- 上传结果不一致
- 前后两次文件不一致导致的诊断偏差

### 15.2 前端任务状态建议

建议前端至少识别以下状态：

- `uploading`
- `uploaded`
- `preview_queued`
- `preview_processing`
- `preview_completed`
- `import_queued`
- `import_processing`
- `summarizing`
- `activating`
- `completed`
- `failed`
- `cancelled`

### 15.3 用户文案必须避免误导

不建议只显示：

- “处理中”
- “导入中，请稍候”

建议显示：

- 当前阶段
- 已处理文件数 / sheet 数
- 是否正在写库
- 是否正在生成辅助汇总
- 失败所在阶段

---

## 16. 数据库写入、事务与性能控制细化

### 16.1 不建议一个超大导入任务只用一个超长事务

原因：

- 长事务会放大数据库压力
- 回滚代价高
- 更容易产生锁等待与恢复困难

推荐方向：

- 以批次为单位提交准备态数据
- 激活切换使用单独短事务

### 16.2 写库路径应统一遵循“批生成、批转换、批写入”

对于 Excel 和 CSV，应尽量统一成以下阶段：

1. 读取一批原始行
2. 应用列映射和元信息注入
3. 生成主表记录和辅助记录
4. 分表分批写入
5. 更新进度与监控数据

### 16.3 辅助维度膨胀必须有硬阈值与中止能力

当样本预测或实时统计发现辅助维度膨胀严重时，系统至少要支持：

- 告警
- 自动降速或限批
- 人工取消
- 失败后不污染线上数据

---

## 17. 反模式清单：修复过程中明确不要做的事情

为了避免“看似修了，实际上埋雷更多”，以下做法不建议采用。

### 17.1 不要继续把“流式导入”建立在整 sheet 内存列表之上

如果导入函数名字叫 streaming，但内部还是 `rows = []` 全量积累，那么这不是修复，只是命名伪装。

### 17.2 不要为了兼容复杂表头，直接把整个 workbook 全部改成 full mode

这会把表头问题扩大成全文件内存问题。

### 17.3 不要在导入成功前清空旧数据

这是企业级修复中必须摒弃的模式。

### 17.4 不要只补前端 loading，而不修复后端容量问题

前端转圈更顺滑，不等于系统更稳定。

### 17.5 不要把所有改造一次性合并上线

一次性合并会带来：

- 排障困难
- 回滚困难
- 风险不可控

---

## 18. 灰度发布与回滚策略细化

### 18.1 建议使用 feature flag 分流

建议至少有以下开关：

- 新上传接入层开关
- 新 Excel 批式解析器开关
- 新准备态激活机制开关
- 新 worker 执行器开关

### 18.2 建议的灰度顺序

建议依次灰度：

1. 仅内部测试项目
2. 小文件场景
3. 大 CSV 场景
4. 大 Excel 场景
5. 多文件 + 多 sheet + 辅助维度高膨胀场景

### 18.3 回滚必须按层回退，而不是整套推翻

例如：

- 新解析器异常，可回退到旧解析器，但保留接入层优化
- 新 worker 异常，可临时切回单实例执行，但保留数据生效安全机制
- 新前端状态机异常，可临时退化展示，但不影响后端批式导入

## 19. 开工前的最终决策清单

在正式进入修复编码前，建议逐项确认以下决策，不确认就不要开工。

### 19.1 容量与 SLA 决策

- 单文件允许多大
- 总上传体积允许多大
- 同时执行几个大任务
- 大文件预览最长允许多久
- 大文件导入最长允许多久

### 19.2 数据安全决策

- 是否接受准备态 + 激活切换
- 是否允许保留旧版本若干天
- 导入失败后是否允许手动重试激活

### 19.3 基础设施决策

- 临时文件落本地、共享盘还是对象存储
- worker 如何部署
- 清理任务如何调度

### 19.4 发布决策

- 是否启用 feature flag
- 是否先灰度到特定项目
- 是否保留旧路径作为紧急回退

### 19.5 组织协同决策

- 谁负责后端接入层
- 谁负责解析器重构
- 谁负责前端状态机与交互
- 谁负责运维脚本、监控、告警
- 谁负责验收样本与压测样本准备

当这五类决策被明确后，修复才真正进入“可执行状态”。
