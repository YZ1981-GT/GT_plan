# ADR-003: 复核意见↔工单联动的事务策略差异

## 状态

已接受 (2026-05-08)

## 上下文

Refinement Round 1 Task 4（正向联动）和 Task 6（反向同步）分别实现了
复核意见与工单之间的双向联动。两个方向采用了不同的事务策略：

- **Task 4 正向（review → issue 创建）**：使用 `db.begin_nested()` SAVEPOINT 隔离。
  复核动作（ReviewRecord 创建）是主事务，工单创建在嵌套事务中执行。
  工单创建失败时 SAVEPOINT 回滚，但 ReviewRecord 仍然提交成功。
  后续通过 `REVIEW_RECORD_CREATED` 事件做补偿重试。

- **Task 6 反向（issue → review/WP 状态回退）**：使用强一致回滚。
  工单状态变更与 ReviewRecord 状态更新、底稿 review_status 回退在同一事务中。
  任何一步失败则整体回滚。

## 决策

正向容错、反向强一致是**设计意图**，不是 bug。

### 正向容错的理由

"复核动作不能被工单服务故障拖死。"

合伙人/经理执行复核是高频操作（每天数十次），如果工单服务暂时不可用
（Redis 队列满、数据库锁等待等），复核动作不应被阻断。用户体验优先：
复核意见立即生效，工单异步补偿创建。

### 反向强一致的理由

"复验通过关工单时，底稿状态必须一起回退。"

当编制人整改完毕、工单切 `pending_recheck` 或 `closed` 时，
ReviewRecord.status 和底稿 review_status 必须同步变更。
如果只关了工单但底稿仍显示 `rejected`，会导致：
1. 编制人困惑（工单关了但底稿还是红色）
2. Gate 规则误判（底稿仍有未解决退回）
3. 数据不一致需要人工修复

因此反向同步必须强一致：要么全部成功，要么全部回滚。

## 后果

1. 两套联动行为不一致是设计意图，代码审查时不应视为"不统一"的问题。
2. 正向补偿路径需要监控：如果 `REVIEW_RECORD_CREATED` 事件长时间未消费，
   EventCascadeHealthRule 会在 gate 评估时发出 warning/blocking。
3. 反向强一致意味着工单状态变更的响应时间略长（多一次 flush），
   但频率远低于正向（每天个位数 vs 数十次），可接受。
4. 未来如果引入分布式事务（如 Saga），正向已天然适配；
   反向需要改造为补偿模式（但当前单库架构下无此需求）。
