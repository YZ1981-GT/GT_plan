## 1. 系统定位

面向会计师事务所的**本地私有化、轻量级、可平滑扩展**的审计全流程作业平台。

- 全程本地部署，数据不出内网，满足事务所数据合规与监管要求
- 以**统一底层数据中台**为唯一数据源，实现 账→证→表→调整→抵消→底稿→报表→附注 全链路联动
- 叠加**本地 LLM 大模型 AI 辅助**，坚持人机协同、关键节点人工确认、过程可视、不黑盒、可追溯
- 适配从单人作业到多分所规模化管理的全生命周期需求
- 遵循中国注册会计师审计准则体系（CSA）及《会计师事务所质量管理准则》要求，系统设计需满足事务所质量管理体系（ISQM1/CSQM1）对信息技术资源的管控要求
- 前端界面严格遵循致同GT品牌视觉识别系统（详见《致同GT审计手册设计规范》），确保系统外观与事务所品牌形象一致

全系统唯一数据源，所有模块依赖它，保证数据一致、不冲突、可联动、可同步。

| 引擎 | 职责 |
|------|------|

| 权限与安全引擎 | 用户/角色/项目权限、数据行级权限、加密、日志、防外泄 |

## 4. 核心数据架构

### 4.1 集团架构三码体系

| 字段 | 说明 |
|------|------|
| `company_code` | 本企业代码（唯一标识各法人主体） |
| `parent_code` | 上级企业代码（构建集团股权层级） |
| `ultimate_code` | 最终控制方代码（确定合并范围） |

辅助字段：`consol_level`（合并层级）、`shareholding`（持股比例）、`is_consol`（是否纳入合并）、`consol_method`（合并方法：全额合并/权益法/比例合并）、`acquisition_date`（取得控制权日期）、`disposal_date`（丧失控制权日期）

> **合伙人批注：** 原文档缺少合并方法字段。实务中同一集团内既有全额合并的子公司，也有权益法核算的联营/合营企业，还可能存在年中新纳入或退出合并范围的情况。必须记录取得/丧失控制权日期，否则无法正确处理当年新增/处置子公司的合并（只合并控制期间的损益）。

### 4.2 全局统一关键字段

所有数据表统一携带，保证全链路穿透：

- `project_id` — 项目ID（项目隔离根基）
- `year` — 年度
- `period` — 期间（月度/季度/年度）
- `company_code` — 公司代码
- `account_code` — 科目代码
- `voucher_date` — 凭证日期
- `voucher_no` — 凭证号
- `currency_code` — 币种代码（新增，支持外币审计）
- `exchange_rate` — 汇率（新增，外币报表折算必备）

> **合伙人批注：** 原文档完全没有考虑外币问题。事务所审计的集团客户中，境外子公司非常普遍。缺少币种和汇率字段，合并报表中的外币报表折算差额将无法处理。这是一个架构级的遗漏，后期补救成本极高。

### 4.3 多年度衔接设计

每张核心表强制携带 `year`、`prev_year`、`is_opening` 字段，实现：

- 年末余额自动结转为下年期初
- 前期调整自动更新本年年初数
- 跨年变动分析（同比、环比自动计算）
- 多年度余额在同一试算表中并存
- 首次审计时支持手工录入期初余额（非结转），标记为"未审计期初"或"已审计期初"（新增）

### 4.4 科目体系与映射

实务中不同被审计单位使用的科目编码体系差异极大，系统必须具备：

- **标准科目模板库**：内置企业会计准则一级/二级科目模板，支持小企业准则、金融企业等行业模板
- **科目映射表**（`account_mapping`）：将被审计单位的原始科目映射到标准审计科目，支持多对一、一对多映射
- **报表取数公式**：报表行次与标准科目的对应关系可配置，不同行业报表格式不同
- **科目辅助核算维度**：支持客户、供应商、部门、项目、个人等多维辅助核算

> **合伙人批注：** 这是原文档最大的实务缺陷之一。每个客户的科目编码都不一样，有的用4位，有的用6位，有的自定义了大量明细科目。如果系统不做科目映射，底稿模板和报表取数公式就无法标准化，每个项目都要重新配置，效率反而不如Excel。

> 1. **独立分录编辑表**：调整分录（AJE/RJE）和抵消分录需要在独立的编辑界面中录入，每笔分录包含多行借贷明细。科目选择必须基于报表一级科目（资产负债表/利润表行次）做下拉选择，选定一级科目后联动显示该科目下的二级明细科目（来自余额表映射后的标准科目），也支持手动输入科目编码。这样确保所有分录的科目口径与试算表、报表完全一致。
> 2. **底稿调用调整分录**：每个实质性程序底稿（D-N循环）对应一个或多个报表科目。底稿中的"审定表"sheet需要自动汇总该科目相关的所有AJE和RJE分录，展示为：未审数 + 各笔AJE调整明细 + 各笔RJE调整明细 = 审定数。这与致同底稿模板中E1（货币资金）、D2（应收账款）等底稿的审定表结构一致。
> 3. **分录→底稿双向穿透**：在调整分录列表中点击某笔分录，可跳转到引用该分录的底稿；在底稿审定表中点击某笔调整金额，可跳转到调整分录详情。

>
> 系统需要：在审计报告签发前强制检查期后事项审阅程序是否全部执行完毕；期后事项中的调整事项自动关联到审计调整分录；非调整事项自动关联到附注披露内容。这是近年来监管检查的重点领域。

## 6. AI 赋能与人机联动（合规版）

### 6.1 核心原则

- **AI = 提效工具，人工 = 最终责任**
- AI 不替代专业判断、不自动确认关键底稿、不隐藏过程、不黑盒生成
- AI 可写，但不能认；人工可以不改，但必须点确认
- 关键底稿强制确认，不允许自动跳过
- AI 辅助内容在审计档案中必须明确标注为"AI辅助生成"，满足监管对审计档案真实性的要求（新增）

### 6.2 自然语言对话操控

- 对话窗口支持语音/文字指令：切换项目、切换年度、打开底稿、查询数据、生成分析、展示差异
- 所有操作必须点击确认才执行，AI 只做理解与调度

#### 6.2.1 项目知识库索引

AI 对话不是无状态的通用问答，而是基于当前项目上下文的专业助手。系统为每个项目自动构建本地知识库索引：

**索引范围：**
- 当前项目的全部账表数据（余额表、序时账、辅助账）
- 已导入的合同文本、单据扫描件的结构化提取结果
- 已编制的底稿内容（含AI生成和人工确认的部分）
- 审计调整分录及说明
- 抵消分录及合并工作底稿
- 函证清单与回函结果
- 审计发现与复核意见
- 上年审计档案摘要（如有）
- **审计复核提示词库（TSJ/）**：约70个按报表科目组织的审计复核提示词Markdown文件，覆盖货币资金、应收账款、存货、固定资产、无形资产、长期股权投资、短期借款、长期借款、应付账款、应付职工薪酬、应交税费、收入、成本、费用等全部报表科目，以及审计方案（B60总体审计策略）提示词（v7新增）

#### 6.2.1.1 审计复核提示词库应用

TSJ/ 文件夹下的约70个Markdown文件是按报表科目组织的审计复核提示词，每个提示词包含完整的复核框架：

**提示词结构（以货币资金为例）：**
- 复核前准备（底稿清单确认、行业特性识别、特殊业务确认）
- 复核优先级指引（高/中/低风险领域分级）
- 审计认定复核要点（存在性/完整性/准确性/权利义务/分类，每个认定含必检项+条件检查项+风险检查项）
- 审计程序复核要点（风险评估程序/实质性程序/分析程序）
- 数据完整性及准确性复核
- 风险评估及应对措施复核（舞弊风险/内控风险/流动性风险等）
- 行业特定复核要点（金融/房地产/制造业/互联网）
- 问题报告格式模板（精准定位到sheet名称+单元格范围）
- 分级复核机制（初级/中级/高级复核）

**在审计作业平台中的三大应用场景：**

1. **底稿AI智能复核（Phase 1b QC引擎 + Phase 4 AI）**：当审计员提交底稿进行AI辅助复核时，系统根据底稿所属科目（如E1货币资金→加载"货币资金审计复核提示词.md"）自动选择对应的提示词作为LLM的system prompt，驱动AI按照完整的复核框架逐项检查底稿内容，输出结构化的复核发现（含风险等级、涉及底稿位置、证据关联、整改建议）。提示词中的 `{{#sys.files#}}` 占位符在运行时替换为实际上传的底稿文件列表。

2. **AI辅助底稿填充与分析性复核（Phase 4）**：AI生成分析性复核初稿时，参考对应科目提示词中的"分析程序"章节确定分析维度（如应收账款的周转率分析、账龄结构分析、客户集中度分析），确保AI生成的分析内容覆盖审计准则要求的关键复核点。

3. **审计方案自动生成（Phase 3 审计程序管理）**：B60总体审计策略提示词（TSJ/B60 总体审计策略及具体审计计划.md）和审计方案提示词（TSJ/审计方案提示词.md）驱动AI按照致同标准模板结构逐章节生成审计计划，包含审计工作范围、被审计单位基本情况、审计安排、财务报表分析、重要性水平、重大错报风险等15个标准章节。

**提示词管理机制：**
- 提示词文件存储在 `TSJ/` 目录下，按科目名称命名，系统启动时自动加载到知识库
- 支持在线编辑和版本管理（编辑/替换/追加/恢复），修改后即时生效
- 提示词与底稿模板的科目映射关系通过科目名称关键词自动匹配（如底稿编号E1→审计循环"货币资金"→匹配"货币资金提示词.md"）
- 支持事务所级自定义：可新增行业专用提示词、修改现有提示词的复核要点

**对话能力示例：**
- "这个项目应收账款前五大客户是谁？期后回款情况如何？"
- "帮我查一下XX公司的合同约定账期和实际回款周期"
- "上年审计对存货减值做了什么调整？今年情况有变化吗？"
- "汇总一下本项目所有未更正错报，累计影响多少？"
- "收入底稿里标注的异常交易有哪些还没处理？"

**索引机制：**
- 项目数据变更时自动增量更新索引（导入数据、编制底稿、录入调整等触发）
- 索引存储在本地，不外传，随项目归档一同锁定
- 支持跨年度项目知识关联（如查询上年同一客户的审计情况）

#### 6.2.2 文件/文件夹智能分析

支持用户指定某个文件或文件夹，AI 自动分析其内容并生成结构化摘要：

**单文件分析：**
- 拖入或选择一个文件（PDF合同、Excel报表、扫描件、Word文档等）
- AI 自动识别文件类型，执行对应分析：
  - 合同 → 提取关键条款摘要（参见18.2合同分析）
  - Excel报表 → 识别报表类型、提取关键数据、标注异常
  - 银行流水 → 自动解析交易明细、标注大额异常
  - 扫描件 → OCR识别后结构化提取
- 分析结果可一键插入当前底稿或保存为独立分析备忘录

**文件夹批量分析：**
- 选择一个文件夹（如客户提供的一整个资料包）
- AI 自动遍历文件夹，按文件类型分类统计
- 生成资料清单：文件名、类型、关键信息摘要、建议关联的审计领域
- 标注缺失资料（如"有采购合同但未见对应入库单"）
- 支持与审计资料需求清单（PBC清单）自动比对，标注已收到/未收到

### 6.3 AI 辅助底稿填充流程

1. AI 从底层中台读取数据（余额、发生额、大额凭证、调整分录、波动、异常）
2. AI 自动生成底稿内容（表格数据、测试说明、变动分析、核查说明、风险提示）
3. 全部进入**待确认状态**：底色标记（AI 生成）、来源可查、过程可见
4. 人工逐题/逐底稿确认：接受 / 修改 / 拒绝 / 重新生成
5. 确认后才进入审定数据，未确认不参与报表

### 6.4 关键底稿强制人工把关

以下底稿类型 AI 仅可辅助，严禁自动确认：

- 收入底稿、应收/应付底稿、存货减值底稿
- 金融资产/投资/长投底稿、预计负债/或有事项底稿
- 合并抵消底稿、重要会计政策/会计估计底稿、关键审计事项相关底稿
- 持续经营评价底稿、关联方交易底稿（新增）
- 期后事项底稿（调整事项/非调整事项的判断必须人工确认）（v6新增）
- 组成部分审计师工作评价底稿（集团审计场景）（v6新增）

### 6.5 AI 实务应用场景细化

| 场景 | AI 能做什么 | AI 不能做什么 |
|------|------------|--------------|
| 科目分析 | 自动计算变动率、标注异常波动、生成分析性复核初稿 | 判断波动是否合理、确定是否需要追加程序 |
| 抽凭测试 | 按规则自动抽样、提取凭证摘要和附件清单 | 判断凭证是否异常、确认业务实质 |

### 6.6 AI 过程透明

每段 AI 生成内容附带：数据来源、引用依据、计算逻辑、风险提示、生成时间、模型版本、人工确认状态

### 6.7 AI 不触碰的边界

AI 严禁自动处理：重大审计判断、减值/公允/预计负债、舞弊风险识别结论、合并范围判断、重要审计程序执行与否、审计意见类型、持续经营评价结论、关联方关系认定。最终判断 100% 人工。

## 7. 底稿↔报表联动逻辑

### 7.1 取数来源统一

报表、底稿全部从同一张试算表取数：
- `trial_balance` → 审计数（经科目映射后的标准科目口径）
- `consol_trial` → 合并数

### 7.2 正向联动

- 底稿变 → 试算变 → 报表变
- 调整变 → 试算变 → 底稿 + 报表同时变
- 抵消变 → 合并试算变 → 合并底稿 + 合并报表变
- 重要性水平变 → 抽样范围联动更新 → 未更正错报评价联动更新（新增）

### 7.3 反向穿透

报表单元格 → 取数公式 → 底稿行 → 科目余额 → 凭证

### 7.4 多年度联动

- 本年报表 ←→ 本年底稿
- 上年报表 ←→ 上年底稿
- 同期对比、变动分析自动生成到底稿

### 7.5 全链路穿透路径

```
合并报表项目
  → 合并数构成（汇总 + 抵消）
    → 各公司明细（含外币折算明细）
      → 组成部分审计师审计结果（如适用，含审计指令与质量评价）（v6新增）
        → 科目余额（标准科目 → 原始科目映射）
          → 序时账
            → 凭证
              → 辅助明细账
                → 对应底稿
                  → 对应调整/抵消分录
                    → 对应审计程序与抽样记录
                      → 对应原始单据/合同/函证回函（v6新增）
```

一条线到底，无断点。

## 8. 多年度衔接规则

### 8.1 余额衔接
- 上年期末余额 = 本年年初余额
- 前期调整自动更新本年年初数（区分：影响期初留存收益 vs 影响期初其他项目）（补充）
- 多年余额在同一试算表中并存
- 首次承接项目时，支持导入前任事务所审定的期初数（新增）

### 8.2 底稿衔接
- 底稿自动加载：本年审定数、上年审定数、年初调整数、变动额/变动率
- 跨年对比底稿自动生成
- 上年底稿可作为本年底稿的参考模板一键复制（新增）

### 8.3 权益衔接
- 未分配利润跨年滚动
- 上期少数股东权益自动结转
- 合并留存收益自动衔接
- 上年抵消分录自动带入本年作为连续编制基础（新增）

### 8.4 现金流量表衔接
- 年初现金余额 = 上年期末现金
- 全年流量自动联动变动

### 8.5 审计发现衔接
- 上年未更正错报自动带入本年，评估累计影响
- 上年管理建议书事项自动跟踪本年整改情况
- 上年识别的特别风险自动提示本年关注

### 8.6 期后事项衔接
- 上年期后事项中的"非调整事项"自动提示本年关注其后续发展
- 上年持续经营评价中识别的疑虑事项自动带入本年，评估是否仍然存在
- 上年期后事项审阅程序清单可作为本年的参考模板

## 9. 审计复核与质控流程

### 9.1 三级复核机制

| 层级 | 角色 | 职责 |
|------|------|------|
| 编制 | 审计员 | 执行审计程序、编制底稿、录入调整分录 |
| 一级复核 | 项目经理 | 复核底稿质量、审计程序充分性、调整合理性 |
| 二级复核 | 签字合伙人 | 复核关键判断、重大事项、审计结论与报告 |

### 9.2 复核流程规则

- 底稿状态拆分为两个独立维度（v10修订）：
  - **编制状态**：草稿(draft) → 编制完成(edit_complete) → 已提交(submitted)
  - **复核状态**：未复核 → 一级复核中 → 一级复核通过 → 二级复核中 → 二级复核通过 → 已退回
- 底稿只有编制状态为"编制完成"或"已提交"时才允许提交复核
- 复核意见必须书面记录，编制人必须逐条回复
- 未通过复核的底稿（已退回状态）不能标记为"完成"，不能进入报表取数
- 关键底稿（收入、应收、存货、长投、合并）必须经过二级复核
- 复核时间节点记录，支持质控部门事后检查复核及时性

### 9.3 项目状态管理

```
项目创建 → 计划阶段 → 执行阶段 → 完成阶段 → 报告阶段 → 归档锁定
```

**各阶段准入/准出条件：**

| 阶段 | 准入条件 | 准出条件（进入下一阶段的要求） |
|------|----------|-------------------------------|
| 计划阶段 | 项目已创建、成员已分配 | 审计计划已编制、重要性水平已确定、集团架构已配置、科目映射已完成 |
| 执行阶段 | 账表数据已导入并通过校验 | 所有底稿编制完成（状态≥编制完成）且通过质量自检（无阻断问题）、调整分录已录入、函证已完成（v6修订：增加质量自检要求） |
| 完成阶段 | 所有底稿通过一级复核 | 未更正错报已评价、试算表已平衡、单户报表已生成、期后事项审阅程序全部执行完毕、持续经营评价已完成（v6修订：增加期后事项和持续经营要求） |
| 报告阶段 | 关键底稿通过二级复核 | 审计报告已签发、管理建议书已出具 |
| 归档锁定 | 审计报告已签发 | 归档检查清单全部勾选完成（报告日后60天内） |

- 归档后项目数据全部锁定，任何修改需走"归档后修改"特殊审批流程
- 归档后修改必须记录原因、审批人，保留修改前后对比

> **合伙人批注：** 这是原文档最严重的缺失。审计质量的核心保障是复核机制，不是技术手段。没有复核流程的审计系统，在质控检查和监管检查中是不合格的。事务所每年被处罚的案例中，"复核不到位"是最常见的问题之一。

- 按项目/公司/科目/底稿区间导出标准 Excel/JSON 模板

- 支持从被审计单位财务软件（用友/金蝶/SAP等）直接导入账套数据的标准接口（新增）

## 11. 版本控制机制

### 11.1 全局版本号
每个项目维护：`local_version`（本地版本）、`cloud_version`（云端版本）、`last_sync_time`（最后同步时间）

### 11.2 同步校验规则
- 本地版本 < 云端版本 → 禁止上传，必须先拉取最新版
- 本地版本 == 云端版本 → 允许上传，版本号 +1
- 本地版本 > 云端版本 → 提示冲突，进入合并流程

### 11.3 冲突解决策略
- 不同科目底稿 → 自动合并
- 同一调整/抵消分录 → 提示冲突，展示差异，用户选择（覆盖本地/覆盖云端/手动合并）
- 同一底稿单元格 → 保留修改时间最新者，标注冲突标记

### 11.4 强制不可覆盖规则
- 原始账表数据不可覆盖（仅允许追加导入，标记导入批次）
- 已审核调整分录不可覆盖（需走冲回+重新编制流程）
- 已通过复核的底稿不可直接修改（需退回重新编制）（新增）
- 项目锁定后禁止同步写入
- 每次同步自动生成备份文件，可回滚

### 11.5 操作日志
同步人、同步时间、同步前后版本、修改内容摘要、文件备份路径，确保任何覆盖事故可追溯、可恢复。

### 11.6 版本控制与ONLYOFFICE协同编辑的边界

v6引入ONLYOFFICE后，底稿文件存在两种编辑模式，版本控制机制需要区分：

| 编辑模式 | 版本控制方式 | 冲突处理 | 说明 |
|----------|-------------|---------|------|
| 在线编辑（ONLYOFFICE） | ONLYOFFICE内部管理（自动保存+历史版本） | ONLYOFFICE原生协同编辑，多人实时编辑无冲突 | 不走11.1-11.3的版本号机制，ONLYOFFICE自带版本历史可回溯 |
| 离线编辑（下载→本地Excel→上传） | 走11.1-11.3的版本号校验机制 | 上传时比对`working_paper.file_version`，冲突时展示单元格级差异 | 下载时记录当前版本号，上传时校验是否过期 |
| 私有云同步（项目级） | 走11.1-11.3的全局版本号机制 | 项目文件夹整体同步，冲突时按11.3策略处理 | 同步粒度是项目文件夹，不是单个底稿 |

- ONLYOFFICE的历史版本功能可以替代底稿文件级别的版本回滚需求（在线编辑场景）
- 离线编辑场景下，后端在用户下载时记录`file_version`，上传时校验版本号是否匹配，不匹配则进入冲突处理流程
- 结构化数据（调整分录、试算表、报表等）的版本控制仍然走11.1-11.3的机制，不受ONLYOFFICE影响

> 3. 取数公式通过ONLYOFFICE自定义函数实现，函数内部异步调用后端API，审计员在单元格里写`=TB("1001","期末余额")`即可

### 12.2 前端架构设计

| 维度 | 方案 | 说明 |
|------|------|------|
| MVP阶段 | Python + Streamlit | 快速验证核心业务流程，2-3个月出原型。注：MVP阶段底稿编辑采用离线模式（下载.xlsx→本地Excel编辑→上传），不集成ONLYOFFICE（Streamlit对iframe支持有限）（v6补充） |
| 正式阶段 | Vue 3 + Element Plus（或 Ant Design Vue） | 支持复杂交互：底稿类Excel编辑、多标签页、拖拽、右键菜单、快捷键 |
| 底稿编辑器 | ONLYOFFICE Document Server（私有化部署）+ 自定义插件 | 通过WOPI协议集成，原生支持.xlsx/.docx编辑，审计员零学习成本；通过自定义函数实现取数公式，通过插件实现复核批注、AI标记、交叉索引跳转（v6修订） |
| Word底稿 | ONLYOFFICE Document Server（同一套服务） | 风险评估、审计计划、管理建议书、审计报告等文字性底稿使用.docx模板，同一套WOPI架构管理（v6新增） |

| AI对话 | 独立侧边栏面板，支持流式输出（SSE/WebSocket） | 不阻塞主界面操作，对话历史持久化 |
| 响应式 | 桌面端优先，不考虑移动端 | 审计作业场景以笔记本/台式机为主 |

> **关键决策：底稿编辑器选型（v6修订）。** 这是整个系统体验的核心。经过技术调研，放弃 Luckysheet（已停止维护）和 Univer（API不稳定）方案，改为集成 ONLYOFFICE Document Server。
>
> **选择ONLYOFFICE的核心理由：**
> 1. **原生Excel兼容** — 底稿就是.xlsx文件，审计员的使用体验就是在编辑Excel，零学习成本。事务所现有的Excel底稿模板可以直接迁移，不需要重新制作
> 2. **自定义函数机制** — 从9.0版本开始支持`AddCustomFunction`，且支持异步API调用。取数公式（`=TB("1001","期末余额")`）可以实现为自定义函数，在单元格里直接写公式，函数内部异步调用后端API获取数据，体验和Excel原生函数一样
> 3. **插件扩展体系** — 可以开发审计专用插件：复核批注面板、AI填充标记与确认交互、交叉索引跳转、底稿状态栏等
> 4. **私有化部署成熟** — 开源（AGPL），Docker一键部署，完全不出内网，符合数据合规要求
> 5. **同时支持Word** — 风险评估底稿、审计计划、管理建议书、审计报告等文字性底稿使用.docx模板，同一套WOPI架构管理，不需要额外的技术方案
> 6. **原生协同编辑** — 多人同时编辑同一底稿是原生支持的，不需要自己实现冲突处理
> 7. **离线兼容** — 审计员在客户现场没网时，下载.xlsx文件用本地Excel编辑，回到事务所后上传，系统解析回写
>
> **ONLYOFFICE集成架构：**
> ```
> 浏览器（Vue 3）
>   └─ ONLYOFFICE Editor（iframe嵌入）
>        │
>        ├─ WOPI协议 ──→ 后端（FastAPI）WOPI Host
>        │                  ├─ 文件存取（项目目录/{project_id}/{year}/{wp_id}.xlsx）
>        │                  ├─ 锁定/解锁、版本管理
>        │                  └─ 权限校验（项目级+角色级）
>        │
>        ├─ 自定义函数 ──→ 后端API
>        │                  ├─ =TB("科目","列名") → 试算表取数
>        │                  ├─ =WP("底稿编号","单元格") → 跨底稿引用
>        │                  ├─ =AUX("科目","维度","值","列名") → 辅助账取数
>        │                  └─ =PREV(公式) → 上年数据
>        │
>        └─ 审计插件
>             ├─ 复核批注面板（侧边栏，逐条回复，状态跟踪）
>             ├─ AI填充标记（紫色底色标记，确认/拒绝/重新生成）
>             ├─ 交叉索引跳转（点击打开被引用底稿并定位）
>             └─ 底稿状态栏（当前状态、编制人、复核人）
> ```
>
> **取数公式的在线/离线双模式：**
> - 在线模式：ONLYOFFICE自定义函数实时调后端API取数，数据始终最新
> - 离线模式：下载前后端先把所有取数公式的结果预填充到单元格（同时保留公式标记），上传回来时后端重新刷新取数
> - 数据变更时（如新增调整分录），后端通过ONLYOFFICE的Command Service API推送更新到已打开的底稿
>
> **需要开发的ONLYOFFICE插件清单：**
>
> | 插件 | 功能 | 优先级 |
> |------|------|--------|
> | 审计取数函数包 | TB()/WP()/AUX()/CTB()/PREV()等自定义函数，异步调后端API | P0（MVP必须） |
> | 复核批注面板 | 侧边栏显示复核意见列表，支持逐条回复，状态跟踪，与`review_records`表联动 | P0 |
> | AI填充标记 | AI生成的单元格加GT紫色底色标记`rgba(75,45,119,0.08)`，提供确认/拒绝/重新生成按钮 | P1 |
> | 交叉索引跳转 | 点击交叉索引标记，在新标签页打开被引用的底稿并定位到对应单元格 | P1 |
> | 底稿状态栏 | 底稿顶部显示编制状态（草稿/编制完成/已提交）、复核状态（未复核/一级复核中/一级复核通过/二级复核中/二级复核通过/已退回）、编制人、复核人、最后修改时间 | P1 |
> | 底稿质量自检 | 保存时自动执行自检规则（结论是否填写、取数是否勾稽等），结果显示在侧边栏 | P2 |

#### 12.2.1 致同GT品牌视觉规范（依据《致同GT审计手册设计规范》）

正式阶段的前端界面必须严格遵循致同GT品牌视觉识别系统，具体规范详见独立文档《致同GT审计手册设计规范.md》，以下为核心要点摘要：

**品牌色系：**

| 色彩角色 | 色值 | 用途 |
|----------|------|------|
| GT核心紫色 | `#4b2d77` | 主操作按钮、链接、卡片头部背景、章节头部渐变 |
| GT亮紫色 | `#A06DFF` | 渐变终点色、焦点样式 |
| GT深紫色 | `#2B1D4D` | 按钮hover状态 |
| 水鸭蓝 | `#0094B3` | 辅助强调色 |
| 珊瑚橙 | `#FF5149` | 错误/高风险状态 |
| 麦田黄 | `#FFC23D` | 警告/注意状态 |
| 成功绿 | `#28A745` | 完成/通过状态 |

**核心规则：**
- 紫色背景（`#4b2d77`）必须配白色文字，对比度 8.6:1，满足WCAG AA
- 所有CSS类名使用 `gt-` 前缀（Grant Thornton），命名格式 `gt-组件名-修饰符`
- 间距系统基于4px网格，主要节奏单位8px
- 圆角：小4px / 中8px / 大12px
- 阴影使用GT紫色调：`rgba(75, 45, 119, 0.075/0.15/0.175)`

**字体规范：**
- 中文：方正悦黑 → 微软雅黑 → 苹方（降级链）
- 英文：GT Walsheim → Helvetica Neue → Arial（降级链）
- 正文16px、行高1.6、主文字色 `#2C3E50`

**组件体系：**
- 卡片组件（`.gt-card`）：白色背景 + 紫色头部 + 阴影边框
- 表格组件（`.gt-table`）：紫色表头 + 白字 + 斑马纹hover
- 流程图组件（`.gt-flow-diagram`）：步骤编号圆形紫色底
- 按钮组件（`.gt-button`）：紫色底 + 白字 + hover深紫
- 状态标记：成功绿左边框 / 警告黄左边框 / 错误红左边框

**审计底稿中的特殊视觉标记：**

| 标记场景 | 视觉方案 | 说明 |
|----------|----------|------|
| AI生成内容（待确认） | 浅紫色底色 `rgba(75,45,119,0.08)` + 左侧紫色竖条 | 与GT品牌色一致，醒目但不刺眼 |
| AI生成内容（已确认） | 无特殊底色，保留小图标标记 | 确认后视觉降级，不干扰阅读 |
| AI生成内容（已排除） | 浅灰色底色 + 删除线 | 明确标识已排除 |
| 复核意见未处理 | 珊瑚橙左边框 `#FF5149` | 高优先级视觉提示 |
| 复核意见已回复 | 水鸭蓝左边框 `#0094B3` | 已处理但待复核人确认 |
| 未审数 | 无特殊标记 | 默认状态 |
| 审定数 | 加粗 + GT紫色文字 `#4b2d77` | 突出最终审定结果 |
| 数据来源穿透链接 | GT紫色下划线文字，hover变深紫 | 可点击跳转 |

**打印样式：**
- A4纸张，页边距上下20mm、左右15mm
- 打印时强制黑白配色，紫色头部改为灰底黑字
- 卡片去阴影加实线边框，避免分页断裂
- 隐藏导航、按钮等交互元素
- 链接后自动追加URL文字

**可访问性要求：**
- 符合WCAG 2.1 AA级标准
- 正文对比度 ≥ 4.5:1，大文字 ≥ 3:1
- 所有交互元素必须有 `:focus-visible` 焦点样式（亮紫色轮廓）
- 紫色背景上焦点样式改用麦田黄轮廓
- 表格必须有 `<caption>` 和 `scope` 属性
- 装饰性图标使用 `aria-hidden="true"`
- 章节使用 `aria-labelledby` 关联标题

**暗色模式：**
- 预留CSS变量映射方案（详见设计规范文档），当前不实现
- 启用前需重新验证所有配色的WCAG对比度

### 12.3 后端架构设计

| 维度 | 方案 | 说明 |
|------|------|------|
| 框架 | Python FastAPI | 异步支持好、性能优于Flask、自动生成API文档、类型校验 |
| ORM | SQLAlchemy 2.0 | 成熟稳定，支持PostgreSQL/SQLite双数据库，支持数据迁移（Alembic） |
| 任务队列 | Celery + Redis（或轻量替代：Huey + SQLite） | 处理耗时任务：数据导入校验、合并计算、PDF生成、AI批量分析 |
| 缓存 | Redis（多用户）/ 内存缓存（单机） | 试算表、报表等频繁读取的计算结果缓存 |
| 文件存储 | 本地文件系统 + 统一文件管理服务 | .xlsx/.docx底稿文件、附件、扫描件、PDF导出文件，按项目/年度/公司目录组织（v6修订：底稿文件为第一公民） |
| WOPI Host | FastAPI实现WOPI协议接口 | 管理ONLYOFFICE与后端的文件交互：文件存取、锁定/解锁、版本管理、权限校验（v6新增） |
| 底稿文件处理 | openpyxl / python-docx | 后端解析.xlsx/.docx文件：取数预填充、人工填写区域提取、结构化数据回写数据库（v6新增） |
| AI推理 | Ollama API（本地LLM）+ PaddleOCR（本地OCR） | 统一AI服务层，屏蔽具体模型差异 |
| 向量库 | ChromaDB / Milvus Lite（本地部署） | 项目知识库索引，支持语义检索 |
| 日志 | 结构化日志（JSON格式）+ 审计操作日志表 | 区分系统运行日志和业务审计日志 |

> **关键决策：为什么选FastAPI而不是Django？**
> - 审计系统的前端是独立的Vue应用，不需要Django的模板引擎
> - FastAPI的异步特性更适合AI推理（流式输出）和文件处理等IO密集场景
> - 自动生成的OpenAPI文档便于前后端协作开发
> - 类型注解+Pydantic校验减少数据错误

### 12.4 数据库设计原则

| 原则 | 说明 |
|------|------|
| 初期即用PostgreSQL | 避免SQLite→PostgreSQL迁移的数据风险和工作量，PostgreSQL单机性能完全够用 |
| 项目级数据隔离 | 所有业务表以 `project_id` 为第一分区键，查询时强制带项目ID条件 |
| 软删除 | 所有表增加 `is_deleted` + `deleted_at` 字段，不物理删除数据 |
| 操作审计 | 关键表（调整分录、底稿、报表）增加 `created_by`、`updated_by`、`created_at`、`updated_at` 字段 |
| 乐观锁 | 底稿、调整分录等多人可能并发编辑的表增加 `version` 字段，更新时校验版本号防止覆盖 |
| 数据归档 | 已归档项目的数据可迁移至归档库/归档表，主库只保留活跃项目 |
| 索引策略 | 核心查询路径（project_id + year + company_code + account_code）建立联合索引 |
| 外键约束 | 关键关联关系使用外键约束（如调整分录→科目表），保证数据完整性 |

### 12.7 ONLYOFFICE许可证风险说明

**建议策略：**

### 12.8 异常处理策略

| 异常场景 | 处理策略 | 用户感知 |
|----------|----------|----------|
| 数据导入中途失败 | 整批事务回滚，已入库的部分数据全部撤销，不留脏数据 | 前端显示导入失败原因（如第N行借贷不平衡），用户修正后重新导入 |
| ONLYOFFICE服务宕机 | 自动降级为离线模式：前端检测到ONLYOFFICE不可用时，显示"下载编辑"按钮替代在线编辑 | 提示"在线编辑暂不可用，请下载底稿后用本地Excel编辑，完成后上传" |
| 取数公式执行失败 | 单元格显示`#REF!`错误标记（模拟Excel行为），鼠标悬停显示具体原因（如"科目1001未配置映射"） | 底稿可继续编辑，失败的取数公式不影响其他单元格 |
| 合并计算中内部往来差异 | 差异金额低于重要性水平→自动生成警告标记，继续合并；差异金额超过重要性水平→阻断合并，要求先处理差异 | 合并结果页面显示差异明细，用户选择"忽略并继续"或"返回处理" |
| 试算表借贷不平衡 | 阻断报表生成，不允许在不平衡状态下出报表 | 试算表页面红色高亮不平衡差额，提示"借贷差额X元，请检查调整分录" |
| PDF导出超时 | 异步任务超时后自动重试一次，仍失败则标记为失败 | 通知中心显示"PDF导出失败，请重试"，附带错误日志链接 |
| 底稿文件损坏 | 上传时校验文件完整性（openpyxl尝试打开），损坏文件拒绝上传 | 提示"文件无法解析，请检查文件是否损坏或格式是否正确" |
| 并发编辑冲突（离线模式） | 上传时检测版本号，冲突时展示差异对比（按单元格级别），用户逐项选择保留哪个版本 | 弹窗显示冲突单元格列表，每个冲突项显示"本地值"和"服务器值"，用户逐项选择 |

**索引策略：**

**分区表策略：**

```sql
-- 按年度分区journal_entries表
CREATE TABLE journal_entries (
    id UUID PRIMARY KEY,
    project_id UUID NOT NULL,
    year INTEGER NOT NULL,
    company_code VARCHAR(50) NOT NULL,
    account_code VARCHAR(50) NOT NULL,
    date DATE NOT NULL,
    voucher_id VARCHAR(50),
    debit_amount NUMERIC(20,2),
    credit_amount NUMERIC(20,2),
    balance NUMERIC(20,2),
    description TEXT,
    is_deleted BOOLEAN DEFAULT false,
    created_at TIMESTAMP DEFAULT current_timestamp
) PARTITION BY RANGE (year);

-- 创建2024年分区
CREATE TABLE journal_entries_2024 PARTITION OF journal_entries
    FOR VALUES FROM (2024) TO (2025);

-- 创建2025年分区
CREATE TABLE journal_entries_2025 PARTITION OF journal_entries
    FOR VALUES FROM (2025) TO (2026);
```

**分区表优势：**
- 提升查询性能（查询只扫描特定年份分区）
- 便于数据归档（归档年份的分区可迁移至归档库）
- 提升维护效率（索引重建、统计信息更新只需针对单个分区）

```python
# 使用CTE一次性查询多层级数据
async def get_penetrate_data(
    project_id: UUID,
    year: int,
    company_code: str,
    account_code: Optional[str] = None,
    date_range: Optional[str] = None
):
    query = f"""
    WITH total AS (
        SELECT
            account_code,
            account_name,
            SUM(debit_amount) as debit_total,
            SUM(credit_amount) as credit_total,
            SUM(debit_amount) - SUM(credit_amount) as balance
        FROM journal_entries
        WHERE project_id = :project_id
          AND year = :year
          AND company_code = :company_code
          {f"AND account_code = :account_code" if account_code else ""}
          {f"AND date BETWEEN :start_date AND :end_date" if date_range else ""}
          AND is_deleted = false
        GROUP BY account_code, account_name
    ),
    ledger AS (
        SELECT
            date,
            voucher_id,
            account_code,
            account_name,
            debit_amount,
            credit_amount,
            balance,
            description
        FROM journal_entries
        WHERE project_id = :project_id
          AND year = :year
          AND company_code = :company_code
          {f"AND account_code = :account_code" if account_code else ""}
          {f"AND date BETWEEN :start_date AND :end_date" if date_range else ""}
          AND is_deleted = false
        ORDER BY date, voucher_id
    ),
    voucher AS (
        SELECT DISTINCT
            voucher_id,
            date
        FROM journal_entries
        WHERE project_id = :project_id
          AND year = :year
          AND company_code = :company_code
          {f"AND account_code = :account_code" if account_code else ""}
          {f"AND date BETWEEN :start_date AND :end_date" if date_range else ""}
          AND is_deleted = false
        ORDER BY date, voucher_id
    )
    SELECT * FROM total;
    SELECT * FROM ledger;
    SELECT * FROM voucher;
    """

    # 执行查询并返回结果
    results = await db.execute(query, {
        "project_id": project_id,
        "year": year,
        "company_code": company_code,
        "account_code": account_code,
        "start_date": start_date if date_range else None,
        "end_date": end_date if date_range else None
    })

    return {
        "total": results[0],
        "ledger": results[1],
        "voucher": results[2]
    }
```

## 14. 核心原则（修订为十二条铁律）

1. **数据唯一** — 中台一本账，不重复、不冲突
2. **联动到底** — 账→证→底稿→报表全程联动
3. **AI 辅助不代决** — 人机联动，关键底稿必须人工确认
4. **过程透明不黑盒** — 生成可看、依据可查、责任可追
5. **协作安全不乱** — 同步有版本、导入有校验、操作有留痕
6. **扩展无限** — 项目可海量、团队可扩大、架构不重构
7. **合规私有化** — 本地部署、模型离线、数据不出内网
8. **复核闭环** — 编制必复核、复核必留痕、意见必回复、关键必签核
9. **准则驱动** — 系统流程对齐审计准则要求，底稿结构符合质控检查标准
10. **导入为先** — 数据导入是一切的起点，导入不顺畅则系统不可用，必须严格校验、灵活适配
11. **模板为魂** — 底稿模板引擎决定系统实用性，取数公式必须灵活、模板必须可配置、版本必须可管理（v6新增）
12. **自检前置** — 提交复核前先过机器自检，低级问题不留给复核人，提升全链路质量效率（v6新增）

## 16. 现金流量表编制方案

> **复盘发现：** v2 版本虽然在报表自动生成中提到了现金流量表，但没有说明编制方法。现金流量表是审计中公认最难编制的报表，必须单独说明。

### 16.1 编制方法

现金流量表不能简单地从试算表取数，它需要对利润表和资产负债表的变动进行调整还原。系统需支持两种编制方式：

- **工作底稿法（推荐）**：基于资产负债表各项目的期初期末变动，结合利润表项目，通过调整分录还原现金流量。系统需提供现金流量表工作底稿模板，自动填入资产负债表变动数据，人工补充调整项
- **T型账户法（辅助）**：对特定复杂科目（如固定资产处置、债务重组）提供T型账户分析工具

### 16.2 自动化辅助

- 自动从余额表提取资产负债表各项目期初期末差额
- 自动识别常见调整项（折旧、摊销、资产减值损失、投资收益等）
- 非现金交易自动标注提示（如以资产清偿债务、非货币性资产交换）
- 合并现金流量表需额外抵消内部现金往来

### 16.3 勾稽校验

- 现金及现金等价物净增加额 = 期末现金余额 - 期初现金余额（强制校验）
- 间接法净利润调节到经营活动现金流量的勾稽检查
- 补充资料中的数据与主表、资产负债表、利润表的交叉勾稽

## 17. 函证管理模块

> 实务中函证由事务所**函证中心**统一管控，项目组不直接发函。系统需对接函证中心的发函/回函流程，项目组侧重于：编制函证清单、提交发函申请、下载回函结果、更新函证底稿、生成函证统计表。

### 17.1 函证类型

| 类型 | 对象 | 说明 |
|------|------|------|
| 银行函证 | 开户银行 | 确认银行存款余额、贷款余额、担保信息、银行承兑汇票等 |
| 往来函证 | 客户/供应商 | 确认应收/应付账款余额 |
| 律师函证 | 律师事务所 | 确认未决诉讼、或有负债 |
| 其他函证 | 保险/证券/海关等 | 根据业务需要 |

### 17.2 函证流程（对接函证中心）

```
项目组编制函证清单
  → 系统生成询证函（自动填充被审计单位信息、余额、截止日期）
    → 项目经理审核函证清单
      → 提交函证中心（导出标准格式数据包 / API对接）
        → 函证中心统一发函、统一收函
          → 项目组从函证中心下载回函结果（扫描件 + 结构化数据）
            → 系统自动匹配回函与原始函证清单
              → 项目组登记核对结果、差异分析
                → 未回函项执行替代程序
                  → 更新函证底稿 → 生成函证统计表
```

### 17.3 项目组侧功能

**编制函证清单：**
- 系统自动从应收/应付/银行存款余额表中提取函证候选对象
- 支持按金额大小、账龄、风险等级、抽样规则筛选
- 人工可增删调整函证对象（如新增非账面往来方、剔除已注销单位）
- 函证清单需项目经理审核后方可提交

**询证函生成：**
- 内置标准询证函模板（银行函证/往来函证/律师函证分别对应不同模板）
- 自动填充：被审计单位名称、对方单位名称、函证金额、截止日期、项目编号
- 支持批量生成，导出为 PDF 或 Excel 供函证中心使用

**发函前 AI 辅助地址核查：**
- AI 自动比对函证清单中的收件地址与工商登记地址、历史函证地址、公开信息（天眼查等缓存数据）
- 标注疑似异常地址：地址不完整、与工商登记不一致、与上年函证地址变更、疑似非经营场所（如住宅地址）
- 银行函证自动校验开户行名称与银行网点地址的匹配性
- 所有 AI 标注结果仅作为提示，项目组人工确认后方可提交函证中心
- 降低退函率，减少因地址错误导致的二次发函

**回函下载与匹配：**
- 从函证中心下载回函数据（支持手工导入扫描件 + Excel结构化数据，预留API对接接口）
- 系统自动按对方单位名称 + 函证编号匹配回函与原始清单
- 匹配失败的回函进入人工确认队列

**AI 辅助回函扫描件识别与比对：**
- AI 对回函扫描件进行 OCR 识别，自动提取关键信息：回函单位名称、确认金额、签章、回函日期
- 自动将 OCR 提取的金额与原始函证金额比对，标注相符/不符及差异金额
- AI 识别回函扫描件的形式要素：是否有对方单位盖章、是否有经办人签字、回函日期是否在合理范围内、是否存在涂改痕迹
- 对比回函件上的印章名称与函证对象名称是否一致，不一致时自动标注预警
- 银行回函自动识别银行业务专用章，校验章上银行名称与开户行是否匹配
- **所有 AI 识别结果标记为【AI辅助-待人工确认】，审计员必须逐份核对原始扫描件后确认**

**回函核对与差异处理：**
- 回函状态：已回函-相符 / 已回函-不符 / 未回函 / 退函（地址不详等）
- 不符项：自动计算差异金额，人工填写差异原因及是否需要调整
- 未回函项：系统自动标注需执行替代程序，记录替代程序类型及结论
- AI 辅助分析不符原因：自动匹配在途款项（结合期后银行流水/收付款记录）、识别记账时间差（结合凭证日期），生成差异原因建议供人工确认

### 17.4 函证统计表（核心产出）

函证统计表是函证底稿的核心，按函证类型分别生成，至少包含以下字段：

| 字段 | 说明 |
|------|------|
| 序号 | 自动编号 |
| 被函证单位名称 | 对方单位 |
| 函证金额（账面） | 被审计单位账面余额 |
| 发函日期 | 函证中心发函日期 |
| 回函日期 | 收到回函日期（未回函为空） |
| 回函状态 | 相符 / 不符 / 未回函 / 退函 |
| 回函确认金额 | 对方确认的金额 |
| 差异金额 | 账面金额 - 回函确认金额 |
| 差异原因 | 人工填写（在途款项/记账时间差/其他） |
| 是否需要调整 | 是/否 |
| 替代程序 | 未回函时执行的替代程序（检查期后回款/检查原始单据等） |
| 替代程序结论 | 替代程序执行结果 |
| 备注 | 其他说明 |

**统计汇总区域（自动计算）：**

| 统计项 | 公式 |
|--------|------|
| 函证总数 | COUNT(全部) |
| 已回函数 | COUNT(回函状态=相符 或 不符) |
| 未回函数 | COUNT(回函状态=未回函 或 退函) |
| 回函率 | 已回函数 / 函证总数 |
| 函证总金额 | SUM(函证金额) |
| 回函相符金额 | SUM(回函状态=相符 的函证金额) |
| 回函不符金额 | SUM(差异金额) |
| 金额覆盖率 | 函证总金额 / 对应科目期末余额 |
| 需调整金额 | SUM(是否需要调整=是 的差异金额) |

> 函证统计表按银行函证、应收函证、应付函证、律师函证分别生成，同时生成一张汇总统计表。统计表数据自动联动到对应科目底稿（如应收账款底稿中引用应收函证的回函率和差异金额）。

### 17.5 数据表

| 表名 | 说明 |
|------|------|
| `confirmation_list` | 函证清单（函证编号、函证类型、被函证单位、账面金额、科目、审核状态、提交函证中心状态） |
| `confirmation_letter` | 询证函记录（关联函证清单、模板类型、生成时间、PDF文件路径） |
| `confirmation_result` | 回函结果（回函日期、回函状态、回函确认金额、差异金额、差异原因、是否需调整、替代程序、替代程序结论） |
| `confirmation_summary` | 函证统计表（按类型汇总：函证数、回函数、回函率、金额覆盖率、差异金额、需调整金额） |
| `confirmation_attachment` | 回函附件（扫描件路径、关联函证编号，支持查看原始回函影像） |

## 十八、AI 深度应用场景：单据识别、合同分析与风险导向审计（新增章节）

> 本章将 AI 的定位从"辅助填表工具"提升为"风险导向审计的智能感知层"。围绕审计准则要求的风险评估→风险应对→审计证据获取全流程，系统性规划 AI 在单据识别、合同分析、异常检测、底稿联动等场景中的应用与预留。所有 AI 输出仍遵循"AI辅助-人工确认"原则。

### 18.1 单据智能识别与结构化（OCR + 语义理解）

审计证据的核心是原始单据。系统应具备对各类纸质/电子单据的 AI 识别能力，将非结构化的图片/PDF 转化为结构化数据，供后续分析和底稿填充使用。

#### 18.1.1 支持的单据类型与提取字段

| 单据类型 | 关键提取字段 | 关联审计领域 |
|----------|-------------|-------------|
| 增值税专用发票（销售） | 购方名称、金额、税额、开票日期、发票号码、货物名称 | 收入确认、应收账款、销项税 |
| 增值税专用发票（采购） | 销方名称、金额、税额、开票日期、发票号码、货物名称 | 成本费用、应付账款、进项税 |
| 银行回单/银行流水 | 交易日期、对方户名、金额、摘要、交易类型 | 货币资金、收入回款、付款真实性 |
| 出库单 | 出库日期、品名、数量、单价、金额、领用部门/客户 | 收入与成本配比、存货 |
| 入库单 | 入库日期、品名、数量、单价、金额、供应商 | 采购与存货、成本归集 |
| 物流单/快递单 | 发货日期、收货方、发货方、物流单号、签收日期 | 收入截止测试、发出商品确认 |
| 记账凭证 | 凭证号、日期、摘要、科目、借贷金额、附件张数 | 全科目审计、抽凭测试 |
| 差旅费报销单 | 出差人、日期、目的地、交通费、住宿费、补贴、审批人 | 管理费用/销售费用、关联方资金占用 |
| 过路费/加油费发票 | 日期、金额、车牌号（如有） | 费用真实性、车辆使用合理性 |
| 合同/协议 | 见 18.2 合同分析专节 | 收入确认、或有事项、关联方交易 |
| 房租/租赁发票 | 出租方、租赁期间、金额 | 使用权资产、租赁负债（新租赁准则） |
| 海关报关单 | 报关日期、品名、数量、金额、进出口类型 | 进出口收入/采购、存货 |

#### 18.1.2 AI 识别处理流程

```
扫描件/照片/电子PDF上传
  → AI OCR 识别（文字提取）
    → AI 语义理解（字段分类、金额提取、日期标准化）
      → 结构化数据入库（关联项目ID、公司、年度）
        → 自动与账面数据交叉比对
          → 异常标注 → 人工确认
```

- 批量上传支持：一次性导入整箱扫描件，AI 自动分类单据类型
- 识别置信度标注：每个提取字段附带置信度分数，低置信度字段高亮提示人工复核
- 支持手工修正 AI 识别结果，修正后的数据作为训练反馈（本地模型持续优化）

#### 18.1.3 单据数据表

| 表名 | 说明 |
|------|------|
| `document_scan` | 原始单据（文件路径、上传时间、单据类型、识别状态、关联项目/公司/年度） |
| `document_extracted` | AI 提取的结构化数据（字段名、字段值、置信度、人工确认状态） |
| `document_match` | 单据与账面数据的匹配记录（匹配的凭证号/科目/金额、匹配结果、差异） |

### 18.2 合同智能分析与审计底稿联动

合同是审计中判断收入确认、或有负债、关联方交易等关键事项的核心依据。

#### 18.2.1 合同关键信息提取

AI 从合同文本（扫描件OCR或电子PDF）中自动提取：

| 提取项 | 用途 | 关联底稿 |
|--------|------|----------|
| 合同双方名称 | 关联方识别、客户/供应商核实 | 关联方交易底稿、应收/应付底稿 |
| 合同金额/单价/数量 | 收入确认金额核对、大额合同识别 | 收入底稿、应收底稿 |
| 合同签订日期/生效日期/到期日期 | 收入确认时点、截止测试 | 收入截止测试底稿 |
| 付款条款（账期、分期、预付） | 应收账龄分析、坏账评估 | 应收底稿、坏账准备底稿 |
| 交货条款（交货方式、验收条件） | 收入确认时点判断（控制权转移） | 收入确认底稿 |
| 违约条款/赔偿条款 | 或有负债评估 | 或有事项底稿 |
| 担保/质押条款 | 担保事项披露 | 担保底稿、附注 |
| 关联方标识 | 关联方交易识别 | 关联方交易底稿 |
| 特殊条款（退货权、价格调整、回购义务） | 收入确认方法判断（是否含可变对价） | 收入底稿 |

#### 18.2.2 合同与底稿联动规则

- 合同金额 vs 对应收入/成本科目发生额 → 自动比对，差异标注
- 合同约定账期 vs 实际回款周期 → 自动计算逾期天数，联动应收账龄分析
- 合同到期日 vs 审计基准日 → 识别跨期合同，提示截止测试关注
- 合同对方名称 vs 关联方清单 → 自动匹配，命中则标注为关联方交易
- 合同特殊条款（退货权/回购/可变对价） → 自动提示收入确认方法需关注

#### 18.2.3 合同数据表

| 表名 | 说明 |
|------|------|
| `contracts` | 合同主表（合同编号、双方名称、金额、日期、类型、关联项目/公司） |
| `contract_extracted` | AI 提取的合同条款结构化数据（条款类型、内容、置信度） |
| `contract_wp_link` | 合同与底稿关联表（合同编号→底稿编号，实现合同到底稿的穿透） |

### 18.3 AI 驱动的风险导向审计分析

围绕风险导向审计的核心理念——识别风险、评估风险、应对风险，AI 在以下环节提供深度分析支持：

#### 18.3.1 风险识别阶段

| AI 分析场景 | 具体能力 | 输出 |
|------------|----------|------|
| 行业与客户画像 | 基于导入的财务数据自动生成客户财务画像（规模、盈利能力、偿债能力、成长性），与行业基准对比 | 风险评估底稿初稿 |
| 异常财务指标识别 | 自动计算数十项财务比率，标注偏离行业均值或历史趋势的异常指标 | 分析性复核底稿 |
| 舞弊三角分析辅助 | 基于财务数据特征（如收入与现金流背离、关联方交易占比高、期末突击交易）自动标注舞弊风险信号 | 舞弊风险评估底稿 |
| 关联方自动识别 | 从工商数据缓存、合同对方、大额往来对象中自动识别潜在关联方 | 关联方清单底稿 |

#### 18.3.2 风险应对阶段——单据交叉验证与审计证据链

这是 AI 最能发挥价值的环节：将多种单据数据交叉比对，自动构建审计证据链，识别不一致。

**收入循环证据链验证：**
```
合同 → 订单 → 出库单 → 物流单（签收记录） → 销售发票 → 记账凭证 → 银行回款
```
- AI 自动匹配上述链条中的金额、日期、客户名称、品名数量
- 任何环节缺失或不一致自动标注（如：有发票无出库单、出库日期晚于发票日期、物流签收日期在资产负债表日后但收入已确认）
- 输出：收入截止测试底稿、收入确认底稿的异常清单

**采购循环证据链验证：**
```
采购合同 → 采购订单 → 入库单 → 采购发票 → 记账凭证 → 银行付款
```
- AI 自动匹配链条中的金额、日期、供应商、品名数量
- 标注异常：有付款无入库、入库数量与发票数量不符、付款对象与合同供应商不一致
- 输出：采购与付款底稿、存货底稿的异常清单

**费用报销证据链验证：**
```
差旅申请 → 差旅费发票（交通/住宿/餐饮/过路费） → 报销单 → 审批记录 → 记账凭证
```
- AI 自动核对：出差日期与发票日期是否匹配、住宿地点与出差目的地是否一致、交通费路线是否合理、同一天是否有异地发票
- 标注异常：周末/节假日大额报销、同一人短期内频繁出差、发票连号、金额恰好卡在审批权限临界值
- 输出：管理费用/销售费用底稿的异常清单

**银行流水深度分析：**
- 自动识别大额异常交易（超过重要性水平的单笔交易）
- 自动标注资金体外循环特征（A→B→C→A 的闭环转账）
- 自动识别与关联方的资金往来（比对关联方清单）
- 期末/期初集中大额收付款标注（截止测试关注）
- 非营业时间交易标注
- 整数金额大额转账标注
- 输出：货币资金底稿、收入/成本截止测试底稿

#### 18.3.3 存货审计 AI 辅助

| 场景 | AI 能力 |
|------|---------|
| 出入库匹配 | 自动匹配入库单与采购发票、出库单与销售发票，标注不匹配项 |
| 存货周转分析 | 按品类计算周转天数，标注周转异常慢的品类（减值风险） |
| 库龄分析 | 基于入库日期自动计算库龄，标注超过一定期限的存货（减值风险） |
| 毛利率分析 | 按产品/客户计算毛利率，标注毛利率异常波动的品类 |
| 期末大额入库 | 标注资产负债表日前后的大额入库（采购截止测试） |
| 盘点差异分析 | 导入盘点表后自动与账面比对，生成差异清单 |

#### 18.3.4 应收账款 AI 辅助

| 场景 | AI 能力 |
|------|---------|
| 账龄自动计算 | 基于发票日期/记账日期自动计算账龄分布 |
| 期后回款匹配 | 自动从期后银行流水中匹配应收回款，计算期后回款比例 |
| 坏账风险评估 | 基于账龄、历史回款率、客户信用信息，自动计算预期信用损失模型参数建议值 |
| 大额应收异常 | 标注期末突增的大额应收、长期挂账未回款、与收入趋势背离的应收增长 |
| 函证与回款交叉 | 函证不符项自动与期后回款记录交叉验证 |

#### 18.3.5 收入审计 AI 辅助（舞弊风险重点领域）

| 场景 | AI 能力 |
|------|---------|
| 收入截止测试 | 自动提取资产负债表日前后N天的销售交易，比对出库单/物流单签收日期，标注可能跨期确认的收入 |
| 收入与现金流配比 | 计算收入与经营活动现金流入的比率，标注严重背离（收入虚增信号） |
| 新增客户分析 | 自动识别本年新增的大额客户，标注需重点关注的交易 |
| 期末突击收入 | 标注第四季度/12月收入占比异常高的情况 |
| 收入与合同匹配 | 大额收入自动匹配对应合同，无合同支撑的大额收入标注预警 |
| 信用期外销售 | 标注超出合同约定信用期仍继续赊销的客户 |

#### 18.3.6 关联方交易 AI 辅助

- 自动从合同、银行流水、往来余额中识别与关联方清单匹配的交易
- 自动识别"隐性关联方"：交易对手的注册地址/法人/股东与被审计单位存在关联（基于工商数据缓存）
- 关联方交易定价分析：与同类非关联方交易价格对比，标注定价异常
- 自动生成关联方交易汇总表，联动附注中关联方披露内容

#### 18.3.7 AI 分析结果与底稿的联动机制

所有 AI 分析结果不是孤立的报告，而是直接嵌入对应审计底稿：

| AI 分析输出 | 自动填入的底稿 | 填入方式 |
|------------|---------------|----------|
| 收入证据链异常清单 | 收入确认底稿、截止测试底稿 | 异常项自动填入"需关注事项"区域 |
| 采购证据链异常清单 | 采购与付款底稿 | 异常项自动填入"测试例外事项" |
| 银行流水异常交易 | 货币资金底稿 | 大额异常交易自动填入"需进一步核实"清单 |
| 存货周转/库龄异常 | 存货底稿、存货减值底稿 | 异常品类自动填入"减值风险评估"区域 |
| 应收账龄与期后回款 | 应收账款底稿、坏账准备底稿 | 账龄分布和回款率自动填入底稿表格 |
| 合同关键条款 | 收入底稿、或有事项底稿、关联方底稿 | 合同摘要自动填入底稿"合同核查"区域 |
| 费用报销异常 | 管理费用/销售费用底稿 | 异常报销清单自动填入"细节测试例外" |
| 关联方交易汇总 | 关联方交易底稿、附注 | 自动生成关联方交易明细表 |
| 财务指标异常 | 分析性复核底稿 | 异常指标自动填入"需解释的重大波动" |
| 舞弊风险信号 | 舞弊风险评估底稿 | 风险信号自动填入"识别的舞弊风险因素" |

**联动规则：**
- 所有 AI 填入内容标记为【AI生成-待确认】底色
- 每条 AI 分析结果附带：数据来源、分析逻辑、置信度、风险等级（高/中/低）
- 审计员确认后状态变为【已确认】，拒绝则标记为【已排除-附原因】
- 未确认的 AI 分析结果不影响底稿结论，但在复核时会提示"存在未处理的AI分析事项"

### 18.4 AI 能力预留接口（远期规划）

以下场景当前可能受限于模型能力或数据条件，但系统架构应预留接口：

| 预留场景 | 说明 | 预留方式 |
|----------|------|----------|
| 电子发票真伪验证 | 对接税务局发票查验接口，自动验证发票真伪 | 预留外部API调用接口 |
| 工商信息实时查询 | 查询交易对手的工商注册、股东、法人变更信息 | 预留天眼查/企查查API接口（或定期导入缓存） |
| 银行对账单自动对账 | 银行流水与账面逐笔自动对账，生成银行余额调节表 | 预留对账引擎接口 |
| 印章/签名真伪检测 | 对比历史印章样本，识别疑似伪造印章 | 预留图像比对模型接口 |
| 语音审计笔记 | 现场审计时语音记录，AI 自动转文字并归入底稿备注 | 预留语音转文字接口 |
| 审计底稿智能复核 | AI 辅助检查底稿完整性（是否有结论、交叉索引是否完整、数据是否勾稽） | 预留底稿质检模型接口 |
| 持续审计/实时监控 | 对接被审计单位ERP，实时监控异常交易 | 预留数据流接入接口 |
| 多人团队 AI 群聊协作 | 项目组成员在同一对话空间中与 AI 协同讨论审计问题，AI 基于项目知识库回答，所有对话留痕可追溯 | 预留多人会话引擎接口（新增） |

**多人团队 AI 群聊说明（远期扩展）：**
- 项目组成员（审计员、项目经理、合伙人）可在项目级 AI 对话空间中协同交流
- AI 作为"项目助手"参与对话，基于当前项目知识库索引回答问题、提供数据支持
- 典型场景：
  - 审计员："@AI 帮我查一下XX科目上年审定数和今年未审数的变动"→ AI 即时回复数据
  - 项目经理："@审计员A 收入截止测试做完了吗？"→ AI 自动补充当前底稿完成状态
  - 合伙人："这个项目的未更正错报累计多少了？"→ AI 汇总回复
- 所有对话记录自动归档到项目日志，可作为项目组内部沟通的审计证据
- 支持 @提及特定成员、@AI 触发智能回复
- 权限控制：对话内容受项目权限约束，非项目组成员不可见

### 18.5 AI 应用的数据安全补充说明

- 所有单据扫描件、合同文本仅在本地处理，不上传至任何外部服务
- OCR 和语义理解均使用本地部署的模型（如 PaddleOCR + 本地LLM）
- 工商数据等外部信息采用定期批量导入缓存的方式，而非实时联网查询
- 客户敏感信息（银行账号、个人身份信息等）在 AI 处理前自动脱敏，处理后还原

## 十九、数据导入规范与校验规则

> **复盘发现：** v2 版本提到了支持财务软件导入，但没有定义导入的数据规范和校验规则。这是系统能否真正落地的关键——如果导入环节不顺畅，后面所有功能都是空中楼阁。

### 19.1 支持的导入来源

| 来源 | 格式 | 说明 |
|------|------|------|
| 用友 U8/T+ | 标准导出 Excel/MDB | 科目余额表、序时账、辅助账 |
| 金蝶 K3/KIS | 标准导出 Excel | 科目余额表、序时账、辅助账 |
| SAP | 标准导出 CSV/Excel | 适配大型企业 |
| 通用格式 | 标准 Excel/CSV 模板 | 系统提供标准模板，手工填入 |
| 其他财务软件 | 插件适配 | 预留适配器接口，后续按需开发 |

### 19.2 导入数据校验规则（强制）

| 校验项 | 规则 | 失败处理 |
|--------|------|----------|
| 借贷平衡 | 每张凭证借方合计 = 贷方合计 | 拒绝导入，提示不平衡凭证 |
| 期初期末勾稽 | 期初 + 本期借方 - 本期贷方 = 期末（资产类）| 标注异常，允许导入但警告 |
| 科目完整性 | 余额表科目必须在科目表中存在 | 提示未映射科目，要求补充映射 |
| 年度一致性 | 导入数据年度与项目年度匹配 | 拒绝导入 |
| 重复检测 | 同一凭证号+日期不重复导入 | 提示重复，用户选择跳过或覆盖 |
| 余额表与序时账勾稽 | 序时账按科目汇总 = 余额表发生额 | 标注差异，允许导入但警告 |
| 辅助账与总账勾稽 | 辅助余额表合计 = 对应总账科目余额 | 标注差异，允许导入但警告 |

### 19.3 导入后自动处理

- 自动执行科目映射（已配置映射关系的科目）
- 未映射科目进入待映射队列，提示人工处理
- 自动生成导入日志（导入时间、数据量、校验结果、操作人）
- 支持导入回滚（整批撤销）

## 十九a、底稿模板引擎设计（v6新增章节）

> **复盘发现（v6）：** 底稿模板引擎是整个系统的灵魂。v6选定ONLYOFFICE Document Server作为底稿编辑器后，底稿模板的载体就是原生的.xlsx/.docx文件。模板引擎的核心职责变为：管理模板文件的生命周期、定义取数公式的执行规则、管理模板中的区域标记、以及在项目创建时从模板生成项目底稿文件。

### 19a.1 底稿模板结构定义

底稿模板就是一个标准的.xlsx或.docx文件，通过以下约定实现结构化管理：

| 要素 | 实现方式 | 说明 |
|------|----------|------|
| 模板元数据 | .xlsx文件的自定义属性（Document Properties） | 模板编号（如A1-货币资金）、模板名称、所属审计循环、适用准则、模板版本号，后端通过openpyxl读取 |
| 取数公式区 | Excel单元格中写ONLYOFFICE自定义函数，如`=TB("1001","期末余额")` | 在线编辑时实时调后端API取数；离线下载前后端预填充为静态值 |
| 人工填写区 | Excel单元格无保护，背景色为白色 | 审计员手工填写的区域，上传后后端解析提取 |
| AI填充区 | Excel单元格背景色为GT浅紫色`rgba(75,45,119,0.08)`，单元格批注标记为"AI填充" | AI生成内容填入后自动标记，审计员确认后去除标记 |
| 只读/锁定区 | Excel工作表保护功能（Sheet Protection） | 取数公式区、表头、底稿编号等不允许修改的区域 |
| 计算公式区 | Excel原生公式（如`=SUM(D5:D20)`） | 底稿内部的汇总计算，使用Excel原生公式即可 |
| 交叉索引区 | Excel单元格批注中标记`##XREF:底稿编号:单元格##` | ONLYOFFICE交叉索引插件识别此标记，提供点击跳转功能 |
| 底稿结论区 | Excel命名范围（Named Range）`WP_CONCLUSION` | 后端通过命名范围定位结论区域，用于质量自检（检查是否已填写） |

### 19a.2 取数公式——ONLYOFFICE自定义函数实现

取数公式通过ONLYOFFICE的`AddCustomFunction` API注册为Excel自定义函数。审计员在单元格中直接输入公式（如`=TB("1001","期末余额")`），ONLYOFFICE调用自定义函数，函数内部异步请求后端API获取数据并返回结果。体验与Excel原生函数完全一致。

**函数清单与语法：**

```
// 从试算表取数（标准科目口径）
TB(科目代码, 列名)
  TB("1001", "期末余额")          → 货币资金期末审定数
  TB("1001", "未审数")            → 货币资金未审数
  TB("1001", "AJE调整")           → 货币资金审计调整合计
  TB("1122", "期末余额") - TB("1122", "年初余额")  → 应收账款本年变动

// 从辅助余额表取数
AUX(科目代码, 辅助维度, 维度值, 列名)
  AUX("1122", "客户", "TOP5", "期末余额")  → 应收前五大客户期末余额

// 从合并试算表取数
CTB(科目代码, 列名)
  CTB("1001", "合并数")           → 合并后货币资金

// 跨底稿引用
WP(底稿编号, 单元格引用)
  WP("A1", "D15")                → 引用A1底稿D15单元格的值

// 从报表取数
RPT(报表类型, 行次)
  RPT("资产负债表", "第1行")       → 资产负债表第1行金额

// 聚合函数
SUM_TB(科目代码范围, 列名)
  SUM_TB("6001~6099", "期末余额")  → 收入类科目期末余额合计

// 上年数据
PREV(任意取数公式)
  PREV(TB("1001", "期末余额"))     → 上年货币资金期末审定数
```

**设计原则：**
- 公式中使用标准科目代码（经科目映射后），不使用原始科目代码
- 公式执行时自动带入当前项目ID、年度、公司代码等上下文
- 公式结果缓存，底层数据变更时自动失效重算
- 公式错误（如科目不存在、映射缺失）在底稿中以红色标注提示

### 19a.3 模板版本管理

| 规则 | 说明 |
|------|------|
| 版本号格式 | 主版本.次版本（如 1.0、1.1、2.0） |
| 主版本升级 | 模板结构变更（增删行列、修改取数公式）时升级主版本 |
| 次版本升级 | 模板格式微调（文字修改、样式调整）时升级次版本 |
| 已使用模板不可删除 | 已有项目使用的模板版本永久保留，确保历史底稿可追溯 |
| 新项目自动使用最新版本 | 创建项目时自动关联最新版本的模板集 |
| 项目内模板锁定 | 项目创建后，该项目使用的模板版本锁定，不受后续模板升级影响 |

### 19a.4 底稿模板数据表

| 表名 | 说明 |
|------|------|
| `wp_template` | 底稿模板主表（模板编号、名称、所属循环、适用准则、版本号、状态：草稿/已发布/已废弃、模板文件路径：指向.xlsx/.docx模板文件） |
| `wp_template_meta` | 模板区域元数据（模板ID、命名范围名称、区域类型：取数公式/人工填写/AI填充/结论区/交叉索引、说明）。注：区域标记主要通过Excel命名范围和单元格批注实现在.xlsx文件内部，此表作为后端快速索引的辅助（v6修订） |
| `wp_template_set` | 模板集（模板集名称：如"企业会计准则-一般企业-年审"、包含的模板编号列表、适用审计类型、适用准则）（v6新增） |

> **合伙人批注（v6修订）：** 选定ONLYOFFICE方案后，底稿模板的核心载体是.xlsx文件本身，而不是数据库中的单元格定义表。模板中的取数公式直接写在Excel单元格里（如`=TB("1001","期末余额")`），区域标记通过Excel命名范围和工作表保护实现。数据库只存储模板的元数据索引，不存储单元格级别的定义——这大幅简化了模板管理的复杂度。事务所现有的Excel底稿模板只需要加上取数公式和命名范围标记就能迁移到系统中，迁移成本很低。
>
> 建议MVP阶段先实现TB()和WP()两个取数函数，后续再扩展AUX()、CTB()等。取数函数的后端API是核心组件，需要充分的单元测试覆盖。

### 19.5 底稿文件生命周期

底稿文件（.xlsx/.docx）在系统中的完整生命周期：

```
模板库中的.xlsx模板文件
  → 项目创建时，从模板复制到项目目录：/{project_id}/{year}/{wp_id}.xlsx
    → 后端预填充：执行取数公式，将结果写入对应单元格（openpyxl）
      → 审计员在线编辑（ONLYOFFICE，通过WOPI协议自动保存）
        或 下载到本地用Excel编辑 → 编辑完上传回系统
          → 后端解析：提取人工填写区域内容，回写到数据库（working_paper表）
            → 底稿数据变更时（如新增调整分录），后端重新执行取数公式更新文件
              → 复核通过 → 归档锁定（文件设为只读，禁止修改）
                → 归档导出（原始.xlsx文件 + PDF转换版本）
```

**在线编辑 vs 离线编辑的数据同步：**

| 场景 | 取数公式 | 数据保存 | 冲突处理 |
|------|---------|---------|---------|
| 在线编辑（ONLYOFFICE） | 自定义函数实时调API，数据始终最新 | WOPI协议自动保存，后端实时解析 | ONLYOFFICE原生协同编辑，无冲突 |
| 离线编辑（本地Excel） | 下载前后端预填充为静态值 | 上传后后端解析回写 | 比对文件版本号，同一单元格冲突时提示人工选择 |
| 数据变更推送 | 后端通过ONLYOFFICE Command Service API推送更新 | 自动更新已打开的底稿 | 仅更新取数公式区，不覆盖人工填写区 |

## 十九b、项目初始化向导（v6新增章节）

> **复盘发现（v6）：** 新项目创建是用户接触系统的第一个环节，体验好坏直接决定用户的第一印象。当前文档只定义了`projects`表的字段，但没有描述项目创建的引导流程。实务中一个审计项目的初始化涉及大量配置工作，如果没有向导引导，用户很容易遗漏关键步骤。

### 19b.1 向导步骤

```
步骤1：基本信息
  → 客户名称、企业代码（统一社会信用代码）、审计期间（年度）、项目类型（年审/专项/IPO/内控/验资/税审）、适用准则（企业会计准则/小企业准则/金融企业/政府会计/IFRS）
  → 附注模板类型（国企版/上市版）、报表类型（单户/合并）
  → 选择合并报表时，展开集团架构折叠面板：上级企业名称/代码、最终控制方名称/代码（三码体系：company_code/parent_code/ultimate_code）
  → 合并报表项目自动创建差额表（用于合并抵消及其他调整分录），子公司清单和持股比例在「合并项目」模块中配置
  → 签字合伙人、项目经理指定

步骤2：集团架构（如适用）
  → 是否集团审计？
  → 是 → 配置集团架构（录入/导入子公司清单、持股比例、合并方法）
  → 否 → 跳过，单户审计模式

步骤3：科目体系（含文件预览与列自动映射）
  → 选择标准科目模板（企业会计准则一级/二级科目）
  → 导入被审计单位数据文件（科目表/序时账/余额表/辅助账，从财务软件导出或手工上传）
  → 上传后进入预览阶段：
    - 调用 POST /api/projects/{id}/account-chart/preview 解析文件
    - 展示前20行数据预览（el-table，可滚动）
    - 系统自动识别文件类型（科目表/序时账/余额表/辅助账/未知）
    - 系统自动匹配列名到标准字段（基于中英文列名映射表）
    - 每列上方显示 el-select 下拉框，按分组展示可选字段：
      · 科目信息：account_code/account_name/direction/parent_code
      · 金额信息：debit_amount/credit_amount/opening_balance/closing_balance
      · 凭证信息：voucher_date/voucher_no/summary/counterpart_account/preparer
      · 辅助核算：aux_type/aux_code/aux_name
    - 已匹配列显示绿色✓图标，未匹配列显示橙色⚠图标
    - 用户可调整任意列的映射关系
    - 显示总行数提示："共 N 行数据，预览前 20 行"
  → 确认映射后执行导入（column_mapping 作为 JSON 参数传入导入接口）
  → 系统自动匹配建议科目映射 → 人工确认/调整未匹配科目

步骤4：重要性水平
  → 选择基准（税前利润/营业收入/总资产）
  → 输入基准金额，系统按经验比例自动计算：
    - 整体重要性水平
    - 实际执行的重要性水平（通常为整体的50%-75%）
    - 明显微小错报临界值（通常为整体的5%）
  → 人工可调整计算结果

步骤5：团队分工
  → 分配项目组成员
  → 按审计循环分工（如：审计员A负责货币资金+应收，审计员B负责存货+固定资产）
  → 设置各成员权限级别

步骤6：底稿模板集
  → 系统根据项目类型+适用准则自动推荐底稿模板集
  → 人工可增删调整（如IPO项目需要额外的IPO专用底稿）
  → 确认后自动生成项目底稿索引

步骤7：数据导入
  → 选择导入来源（用友/金蝶/SAP/通用格式）
  → 上传数据文件
  → 系统执行校验规则（详见第十九章）
  → 校验通过后确认导入

步骤8：确认与创建
  → 展示项目配置摘要（基本信息、集团架构、科目映射完成率、重要性水平、团队分工、底稿模板集）
  → 确认创建 → 项目进入"计划阶段"
```

### 19b.2 向导设计原则

- 每个步骤可独立保存，支持中途退出后继续
- 非必填步骤可跳过（如单户审计跳过集团架构）
- 步骤间有依赖关系的自动校验（如科目映射依赖科目表导入）
- 首次承接项目 vs 连续审计项目的差异处理：连续审计项目可从上年项目一键复制配置（集团架构、科目映射、底稿模板集、团队分工）
- 向导完成后生成"项目初始化检查清单"，标注已完成/未完成的配置项

## 十九c、底稿质量自检（v6新增章节）

> **复盘发现（v6）：** 原文档18.4将"审计底稿智能复核"列为远期AI预留功能，但实际上大部分底稿质量检查可以用规则引擎实现（不依赖AI），应提前到第二或第三阶段。底稿质量自检是提交复核前的"预检"，可以大幅减少复核人发现低级问题的概率，提升复核效率。

### 19c.1 自检规则

底稿提交复核前，系统自动执行以下检查：

| 检查类别 | 检查项 | 严重级别 |
|----------|--------|----------|
| 完整性检查 | 底稿结论区是否已填写 | 阻断（不填写不允许提交复核） |
| 完整性检查 | 人工填写区是否有空白未填项 | 警告（允许提交但标注） |
| 完整性检查 | AI填充区是否全部已确认（无"待确认"状态） | 阻断 |
| 勾稽检查 | 取数公式结果是否与试算表一致（防止手工修改后不一致） | 阻断 |
| 勾稽检查 | 底稿内部合计数是否正确（如明细合计=汇总行） | 警告 |
| 勾稽检查 | 交叉索引引用的数据是否与被引用底稿一致 | 警告 |
| 索引检查 | 底稿编号是否在索引表中登记 | 警告 |
| 索引检查 | 交叉索引引用的底稿是否存在 | 警告 |
| 程序检查 | 审计程序清单中关联本底稿的程序是否全部标记为"已执行" | 警告 |
| 抽样检查 | 如底稿涉及抽样，抽样记录是否完整（方法、样本量、结果、结论） | 警告 |
| 调整检查 | 底稿中标注的需调整事项是否已录入调整分录 | 警告 |
| 时间检查 | 底稿编制日期是否在合理范围内（不早于进场日、不晚于报告日） | 提示 |

### 19c.2 自检结果展示

- 自检结果以清单形式展示，按严重级别排序（阻断→警告→提示）
- 阻断级别的问题必须全部解决后才能提交复核
- 警告级别的问题允许提交，但在复核人界面中高亮标注
- 自检结果记录存档，复核人可查看编制人提交时的自检状态

### 19c.3 项目级自检汇总

项目经理可在项目看板中查看所有底稿的自检状态汇总：

| 统计项 | 说明 |
|--------|------|
| 底稿总数 | 项目底稿索引中的底稿总数 |
| 已通过自检 | 自检无阻断问题的底稿数 |
| 存在阻断问题 | 有阻断级别问题未解决的底稿数 |
| 未编制 | 尚未开始编制的底稿数 |
| 自检通过率 | 已通过自检 / (底稿总数 - 未编制) |

## 二十一、审计档案归档规范

> **复盘发现：** v2 版本提到了"归档锁定"，但没有展开归档的具体要求。根据《中国注册会计师审计准则第1131号——审计工作底稿》，审计档案归档有严格的规范要求。

### 21.1 归档要求

- 审计档案应在审计报告日后60天内完成归档
- 归档后的审计档案不得删除或废弃
- 归档后如需修改，必须记录修改原因、修改内容、修改人、修改日期、审批人
- 审计档案保存期限不少于10年（自审计报告日起算）

### 21.2 归档内容清单

系统应自动生成归档检查清单，确认以下内容完整：

- [ ] 审计计划与风险评估底稿
- [ ] 重要性水平确定底稿
- [ ] 各科目/循环审计底稿（含索引）
- [ ] 审计调整分录汇总（AJE + RJE）
- [ ] 未更正错报汇总表
- [ ] 试算平衡表（未审数→审定数）
- [ ] 财务报表（单户/合并）
- [ ] 附注
- [ ] 审计报告
- [ ] 管理建议书
- [ ] 复核记录
- [ ] 函证底稿
- [ ] 关键审计事项底稿（上市公司审计）
- [ ] 期后事项审阅底稿（含审阅程序清单和期后事项记录）（v6新增）
- [ ] 持续经营评价底稿（含风险指标检查清单和评价结论）（v6新增）
- [ ] 组成部分审计师相关文件（审计指令、审计结果、质量评价，集团审计适用）（v6新增）
- [ ] 抽样记录（含抽样方法、样本量计算依据、测试结果和推断结论）（v6新增）
- [ ] 底稿质量自检报告（各底稿提交复核时的自检结果汇总）（v6新增）

### 21.3 电子档案导出

- 支持按项目一键导出完整电子档案（PDF格式）
- 底稿按索引编号自动排序
- 生成档案目录页
- 支持加密导出（防止未授权访问）
- 导出的PDF中保留穿透链接（点击可跳转到关联底稿页面）
- PDF封面、页眉页脚使用致同GT品牌元素（GT紫色标题栏、事务所Logo、项目信息水印）
- 打印样式遵循《致同GT审计手册设计规范》中的打印规范（A4、黑白配色、避免分页断裂）

## 附录A：原文档主要问题总结（合伙人审阅意见）

以下是本次修订中识别的原文档核心缺陷，按严重程度排序：

### 架构级缺陷（不修复则系统不可用）

1. **缺少科目映射机制** — 不同客户科目编码体系不同，没有映射层，底稿模板和报表取数无法标准化，每个项目都要重新配置
2. **缺少外币处理** — 集团审计中境外子公司普遍存在，缺少币种/汇率字段，外币报表折算无法实现
3. **缺少审计复核流程** — 这是审计质量的核心保障，没有复核机制的系统不符合质量管理准则要求
4. **调整分录未区分AJE/RJE** — 审计调整和重分类调整性质完全不同，混在一起会导致试算表结构错误
5. **缺少重要性水平** — 重要性水平贯穿整个审计，决定抽样范围、错报评价、审计意见

### 功能级缺陷（不修复则效率低下）

6. **缺少财务软件数据导入适配** — 不能对接用友/金蝶等主流软件，数据导入环节仍需大量手工
7. **缺少底稿索引与交叉引用** — 底稿缺乏组织结构，无法体现审计逻辑的完整性
8. **缺少抽样记录管理** — 抽样是核心审计程序，需要完整记录方法、样本、结果
9. **缺少未更正错报汇总** — 这是出具审计意见前的必要评价步骤
10. **缺少审计报告生成** — 系统的最终产出应该是审计报告，而不仅仅是报表
11. **缺少PDF/打印导出** — 审计档案需要打印装订或生成电子档案

### 设计级建议（优化体验与长期演进）

12. **合并方法字段缺失** — 需区分全额合并/权益法/比例合并
13. **商誉和少数股东权益需独立管理** — 不能混在抵消分录中
14. **抵消分录连续编制机制** — 上年抵消本年需重新编制，这是合并报表最易出错的环节
15. **前端技术选型需评估** — Streamlit 适合原型，不适合重交互的底稿编辑场景
16. **建议初期即用PostgreSQL** — SQLite多用户并发写入会锁库
17. **AI模型需做抽象层** — 不绑定具体模型，预留切换能力

### v3 复盘新增发现

18. **现金流量表编制方案缺失** — 现金流量表是审计中最难编制的报表，不能简单从试算表取数，需要工作底稿法支持，需要间接法勾稽校验
19. **函证管理模块缺失** — 函证是最重要的外部审计证据，涉及生成、发送、跟踪、回函分析、替代程序的完整流程，不能只在AI场景中一笔带过
20. **数据导入规范未定义** — 只说"支持导入"远远不够，必须定义具体的校验规则（借贷平衡、期初期末勾稽、科目完整性等），否则脏数据进入系统会导致后续所有计算错误
21. **项目管理看板缺失** — 合伙人和项目经理最需要的是项目全局视图（进度、风险、工时），而不是单个底稿的操作界面
22. **审计档案归档规范缺失** — 归档不只是"锁定"，涉及60天归档期限、10年保存期、归档检查清单、电子档案导出等准则硬性要求
23. **工时管理未考虑** — 事务所的成本核算和绩效考核依赖工时数据，这是项目管理的基础
24. **试算表结构仍需细化** — 合并试算表应分列展示：各公司审定数汇总 → 合并调整 → 合并抵消 → 合并数，与单户试算表的四列结构对应

### v4 复盘新增发现（技术架构与工程化）

25. **技术架构过于笼统** — 原五层架构图没有体现前后端分离、API层、异步任务队列、向量检索等关键组件，无法指导实际开发
26. **前端选型未落地** — 只说"Streamlit→Vue迁移"但没有定义底稿编辑器选型（v6已解决：选定ONLYOFFICE Document Server + 自定义插件方案）
27. **后端框架未明确** — 没有选定Web框架、ORM、任务队列等基础组件，无法开始编码
28. **API设计规范缺失** — 前后端分离的系统必须先定义API规范，否则前后端无法并行开发
29. **数据库设计原则缺失** — 没有定义软删除、乐观锁、操作审计字段、索引策略等数据库层面的工程规范
30. **异步任务未考虑** — 数据导入、合并计算、PDF生成、AI批量分析等耗时操作如果同步执行会阻塞前端，必须有异步任务队列
31. **部署方案未定义** — 私有化部署需要一键安装能力，Docker Compose是最合适的方案，原文档完全没提
32. **开发优先级缺少第零阶段** — 技术基础设施（数据库、API框架、认证、日志）是所有功能的地基，必须先搭好骨架再写业务

### v5 复盘新增发现（非功能性需求与工程完整性）

33. **缺少非功能性需求定义** — 没有性能指标（查询响应时间、合并计算耗时）、容量规划（并发用户数、项目数上限、数据量）、可用性要求（备份策略、灾难恢复），开发团队无法做技术决策
34. **缺少通知提醒机制定义** — 文档多处提到"自动提醒"、"预警"，但没有统一定义通知的触发条件、通知方式、实现方案，也没有消息中心的数据表
35. **缺少权限矩阵** — 定义了6种角色但没有一张完整的功能×角色权限矩阵表，开发时无法确定每个接口的权限校验规则
36. **缺少数据备份与灾难恢复方案** — 审计数据保存10年是准则硬性要求，但文档只在同步章节提到了"备份文件"，没有系统级的备份策略（频率、保留周期、恢复流程）
37. **操作日志缺少"改前值/改后值"** — 审计系统的操作日志不能只记录"改了什么"，必须记录改前值和改后值，否则无法追溯和恢复
38. **`notifications` 消息表缺失** — 通知提醒功能需要持久化存储，之前没有定义对应数据表

### v6 复盘新增发现（业务完整性与工程深化）

39. **期后事项管理缺失** — CSA 1501明确要求审计师关注资产负债表日至审计报告日之间的重大事项（调整事项和非调整事项），系统缺少专门的模块来记录和跟踪期后事项，也没有期后事项审阅程序清单
40. **组成部分审计师管理缺失** — 大型集团审计中，主审事务所需要向组成部分审计师发送审计指令、接收审计结果、评估工作质量（ISA 600/CSA 1401），这是集团审计流程的核心环节，原文档完全没有涉及
41. **抽样方法定义不足** — `sampling_records`表只记录了抽样结果，但没有定义抽样方法的配置（统计抽样vs非统计抽样、MUS/属性抽样/随机抽样等）、样本量计算公式、从样本推断总体错报的计算逻辑
42. **底稿模板引擎设计缺失** — 这是整个系统的灵魂，但原文档只有`wp_template`一张表，没有定义取数公式的语法（DSL）、单元格区域类型划分、模板版本管理机制、交叉索引绑定规则。没有模板引擎，底稿就只是一个空壳表格
43. **项目初始化向导缺失** — 新项目创建涉及大量配置工作（审计类型、准则选择、集团架构、科目映射、重要性水平、团队分工、底稿模板集），没有引导式向导，用户容易遗漏关键步骤

## 附录C：审计专业术语数据字典

> 本附录为开发团队提供审计专业术语的中英文对照和释义，避免因术语理解偏差导致实现错误。

| 缩写/术语 | 英文全称 | 中文释义 | 在系统中的对应 |
|-----------|----------|----------|---------------|
| AJE | Adjusting Journal Entry | 审计调整分录。审计师发现被审计单位账面存在错报，建议其调整账面的分录。被审计单位需要入账。 | `adjustments` 表，`adj_type = 'AJE'` |
| RJE | Reclassifying Journal Entry | 重分类调整分录。仅影响报表列报，不影响被审计单位账面。如将"预付账款贷方余额"重分类到"应付账款"。 | `adjustments` 表，`adj_type = 'RJE'` |
| TB | Trial Balance | 试算平衡表。列示所有科目的未审数、调整数、审定数，是审计的核心工作表。 | `trial_balance` 表 |
| PBC | Prepared By Client | 客户准备的资料。审计项目启动后向被审计单位发出的资料需求清单。 | `pbc_checklist` 表 |
| KAM | Key Audit Matter | 关键审计事项。上市公司审计报告中必须披露的、审计师认为最重要的事项。 | `audit_report` 表中的关键审计事项字段 |
| MUS | Monetary Unit Sampling | 货币单元抽样。一种统计抽样方法，以货币金额为抽样单元，大金额项目被选中的概率更高。 | `sampling_config` 表 |
| WP | Working Paper | 审计工作底稿。记录审计程序执行过程和结论的文档。 | `working_paper` 表 |
| CSA | China Standards on Auditing | 中国注册会计师审计准则。中国审计工作的法定标准体系。 | 系统流程设计的准则依据 |
| ISA | International Standards on Auditing | 国际审计准则。由IAASB发布的国际审计标准。 | 扩展阶段的准则适配 |
| ISQM1 / CSQM1 | International/China Standard on Quality Management 1 | 质量管理准则第1号。要求事务所建立质量管理体系。 | 系统的复核流程、权限控制、操作日志等设计依据 |
| IPO | Initial Public Offering | 首次公开发行。IPO审计有特殊的底稿要求和报告格式。 | `projects` 表的项目类型字段 |
| PM | Performance Materiality | 实际执行的重要性水平。通常为整体重要性的50%-75%，用于确定审计程序的范围。 | `materiality` 表 |
| SUD | Summary of Unadjusted Differences | 未更正错报汇总表。汇总被审计单位未调整的错报，评估累计影响是否重大。 | `unadjusted_misstatements` 表 |
| OCR | Optical Character Recognition | 光学字符识别。将图片/扫描件中的文字转化为可编辑的电子文本。 | AI服务层的PaddleOCR组件 |
| LLM | Large Language Model | 大语言模型。系统中用于对话、分析、生成等AI功能的核心模型。 | Ollama本地部署 |

### 审计业务流程术语

| 术语 | 释义 |
|------|------|
| 审计循环 | 将被审计单位的业务按性质分组（如收入循环、采购循环、投资循环等），每个循环对应一组相关的科目和底稿 |
| 认定 | 管理层在财务报表中做出的明示或暗示的声明（如存在性、完整性、计价和分摊、权利和义务、列报和披露） |
| 实质性程序 | 用于发现认定层次重大错报的审计程序，包括细节测试和实质性分析程序 |
| 控制测试 | 用于评价内部控制运行有效性的审计程序 |
| 分析性复核 | 通过分析财务数据之间的关系和趋势来识别异常波动的审计程序 |
| 穿透测试 | 从报表数字追溯到原始凭证的完整验证路径 |
| 截止测试 | 验证交易是否记录在正确的会计期间（如收入是否在控制权转移时确认） |
| 函证 | 审计师直接向第三方（银行、客户、供应商等）发函确认信息的审计程序 |
| 替代程序 | 当函证未回函时，采用其他方式（如检查期后回款、核对原始单据）获取审计证据 |
| 持续经营 | 被审计单位在可预见的未来（通常12个月）能否继续经营的评估 |
| 期后事项 | 资产负债表日至审计报告日之间发生的事项，分为调整事项和非调整事项 |
| 三级复核 | 审计底稿的质量控制机制：编制人→项目经理复核→签字合伙人复核 |
| 归档 | 审计项目完成后，将全部审计档案整理归档的过程，准则要求在报告日后60天内完成 |
| 组成部分审计师 | 集团审计中，受主审事务所委托审计部分子公司的其他事务所 |
| 商誉 | 企业合并中，合并成本超过被合并方可辨认净资产公允价值份额的差额 |
| 少数股东权益 | 子公司所有者权益中不属于母公司的份额 |
| 连续编制 | 合并报表中，上年的抵消分录在本年需要重新编制并调整的过程 |
| 功能货币 | 企业经营所处的主要经济环境中的货币 |
| 列报货币 | 编制财务报表时使用的货币 |
| 外币报表折算 | 将以功能货币编制的财务报表折算为以列报货币表示的过程 |

## 附录D：电子签名方案说明

> 9.2提到"电子签核留痕"，但未定义技术实现方案。本附录明确电子签名的分阶段实现策略。

### 方案分级

| 阶段 | 方案 | 说明 | 法律效力 |
|------|------|------|----------|
| MVP阶段 | 用户名+密码确认 | 用户在系统中点击"确认/签核"按钮，系统记录操作人、时间、IP地址 | 内部管理效力，不具备法律签名效力 |
| 正式阶段 | 手写签名图片+时间戳 | 用户在触摸屏/手写板上签名，系统保存签名图片并绑定时间戳和操作记录 | 内部管理效力，可作为审计档案的辅助证据 |
| 远期扩展 | CA数字证书签名 | 对接第三方CA机构（如CFCA中金认证），使用数字证书对PDF档案进行电子签名 | 具备《电子签名法》认可的法律效力 |

### 设计原则

- MVP阶段的"确认"操作必须记录完整的操作链（谁、何时、确认了什么、IP地址），即使没有电子签名也要确保可追溯
- 系统架构预留电子签名接口（`sign_service`），后续接入CA证书时不需要修改业务逻辑
- 归档导出的PDF在远期阶段支持嵌入数字签名，确保档案的完整性和不可篡改性
- 私有化部署场景下，CA证书的密钥存储在本地HSM（硬件安全模块）或加密文件中，不上传外部

## 附录E：持续经营评价底稿设计

> 6.4将持续经营评价底稿列为"强制人工把关"的底稿类型，但缺少专门的数据表和评价流程定义。

### 持续经营评价流程

```
管理层提供持续经营评估报告
  → 审计师独立评价管理层评估的适当性
    → 识别可能导致持续经营重大疑虑的事项和情况
      → 评估管理层的应对计划
        → 考虑期后事项对持续经营的影响
          → 形成审计结论（无重大疑虑 / 存在重大不确定性但披露充分 / 持续经营假设不适当）
            → 结论影响审计报告类型（标准无保留 / 增加强调事项段 / 保留/否定意见）
```

### 数据表

| 表名 | 说明 |
|------|------|
| `going_concern` | 持续经营评价记录（project_id、评价日期、管理层评估摘要、识别的疑虑事项：财务指标恶化/重大诉讼/主要客户流失/行业衰退等、管理层应对计划、审计师评价意见、结论类型：无重大疑虑/存在重大不确定性/持续经营假设不适当、对审计报告的影响、复核状态） |
| `going_concern_indicators` | 持续经营风险指标（指标类型：财务指标/经营指标/其他指标、指标名称：如"连续亏损"/"资不抵债"/"主要银行授信到期未续"、当前状态：存在/不存在、评价说明）（v6新增） |

> **合伙人批注（v6）：** 持续经营是近年来监管处罚的高频领域。很多事务所被处罚的原因是"未充分关注持续经营风险"或"持续经营评价程序不到位"。系统应在项目创建时自动生成持续经营风险指标检查清单，审计员逐项评估后形成结论。如果识别到重大疑虑事项，系统应自动提示需要在审计报告中增加强调事项段或修改审计意见类型。

D-N类（实质性程序）

  → 底稿编号：{循环字母}{序号}（如D4-1收入审定表、E1-1货币资金审定表）

### G.4 附注模版体系（实际结构）

系统内置两套附注模版，按企业类型区分：

| 模版类型 | 适用范围 | 差异点 |
|----------|---------|--------|
| 国企版 | 国有企业、国资委监管企业 | 按国资委披露要求，含特有科目（如专项储备、安全生产费） |
| 上市版 | 上市公司、拟上市公司 | 按证监会/交易所披露要求，含特有科目（如设定受益计划净资产、每股收益） |

每套附注模版包含4个配置文件：

#### G.4.1 科目对照模板（account_mapping_template）

定义报表科目与附注表格的对应关系，每个科目下的表格标注校验角色：

```
科目：应收账款
  ├─ ① 按账龄披露应收账款 [其中项]
  ├─ ② 按坏账准备计提方法分类 [交叉]
  ├─ ③ 按单项计提坏账准备 [交叉]
  ├─ ④ 按信用风险特征组合计提 [交叉]
  ├─ ⑤ 坏账准备变动表 [宽表][交叉]
  ├─ ⑥ 本期实际核销 [其中项]
  ├─ ⑦ 前五名应收账款 [其中项]
  └─ ⑧ 终止确认的应收账款 [其中项]
```

**校验角色类型：**

| 角色 | 含义 | 校验方式 |
|------|------|---------|
| `[余额]` | 报表vs附注余额核对 | 取合计行与报表科目余额比对 |
| `[宽表]` | 横向公式校验 | 期初±变动=期末（movement型） |
| `[交叉]` | 同科目多表交叉校验 | 如坏账准备变动表期末=总表坏账准备 |
| `[其中项]` | 明细行之和=合计行 | 通用规则：sum(非合计行)=合计行 |
| `[描述]` | 纯文字描述 | 不参与数值校验，可LLM审核 |

**互斥规则：** `[余额]`不与`[其中项]`/`[宽表]`共存——含其中项或宽表的表格，余额核对从计算结果自然得出。

#### G.4.2 校验公式预设（check_presets）

逐科目、逐表格定义每一条校验公式，公式类型包括：

| 公式类型 | 说明 | 示例 |
|----------|------|------|
| 余额 | 报表余额vs附注合计行 | 货币资金报表数 = 附注①合计行 |
| 宽表 | 横向公式 | 期初+本期增加-本期减少=期末 |
| 纵向 | 多段表纵向勾稽 | 原值-折旧-减值=账面价值 |
| 交叉 | 同科目多表核对 | 坏账准备变动表期末 = 总表坏账准备列 |
| 跨科目 | 不同科目间核对 | 信用减值损失 = 各科目坏账准备本期计提之和 |
| 其中项 | 明细行求和=合计行 | 通用规则，不硬编码子项名称 |
| 二级明细 | 报表子明细vs附注 | 其他应收款-应收利息 = 附注应收利息表合计 |
| 账龄衔接 | 账龄段跨期比较 | 期末"1至2年" ≤ 期初"1年以内" |
| 完整性 | 数据行非空检查 | 账面余额≠0时其他列不应为空 |
| LLM审核 | 文本合理性判断 | 会计政策描述是否与准则一致 |

**核心设计原则：本地规则引擎优先，LLM兜底。**

#### G.4.3 宽表公式预设（wide_table_presets）

定义常见宽表（多列变动表）的标准列结构和横向公式：

```
示例：固定资产变动表
  列结构：
    项目(label) | 期初原值(opening) | 本期增加(+) | 本期减少(-) | 期末原值(closing=)
                | 期初折旧(opening) | 本期计提(+) | 本期减少(-) | 期末折旧(closing=)
                | 期初减值(opening) | 本期计提(+) | 本期减少(-) | 期末减值(closing=)
                | 期初账面价值(skip) | 期末账面价值(skip)
  
  横向公式：期初 + 本期增加 - 本期减少 = 期末
  纵向公式：期末原值 - 期末折旧 - 期末减值 = 期末账面价值
```

每列标注角色（opening/movement/closing/skip/label）和符号（+/-/=），引擎按角色自动构建校验公式。

#### G.4.4 报表附注正文模版（disclosure_template）

完整的附注正文模版（.docx格式），包含：

| 章节 | 内容 | 取数方式 |
|------|------|---------|
| 公司基本情况 | 企业历史沿革、注册地、业务性质 | 人工填写 |
| 财务报表编制基础 | 持续经营声明 | 模板文字+人工确认 |
| 遵循企业会计准则的声明 | 合规声明 | 模板文字 |
| 重要会计政策、会计估计 | 会计期间/记账本位币/合并范围/金融工具/收入确认等 | 模板文字+人工按实际修改 |
| 税项 | 增值税/企业所得税/其他税种 | 人工填写税率 |
| 财务报表主要项目注释 | 逐科目披露（核心章节） | 从试算表/底稿自动取数+AI辅助文字 |
| 合并财务报表项目注释 | 合并范围变动/子公司信息 | 从合并数据自动取数 |
| 其他重要事项 | 或有事项/承诺/期后事项/关联方 | 人工填写+AI辅助 |

### 18.9 第三方系统集成
