# 审计底稿系统深度开发方案

> 基于代码审查的现状分析 + 修改方案 + 功能拓展，按角色维度组织

---

## 一、现状诊断（代码审查结论）

### 1.1 已做实的能力

| 模块 | 状态 | 关键文件 |
|------|------|----------|
| 模板生成（363个致同模板） | ✅ 完整 | template_engine.py, gt_template_library.json |
| 预填充引擎（5种公式 TB/SUM_TB/AUX/PREV/WP） | ✅ 完整 | prefill_engine.py, formula_engine.py |
| WOPI 在线编辑（Redis锁+版本快照+哈希校验） | ✅ 完整 | wopi_service.py, WorkpaperEditor.vue |
| 离线下载/上传回传（冲突检测+版本递增） | ✅ 完整 | wp_download_service.py, working_paper_service.py |
| QC 自检（14条规则，5阻断+8警告+1提示） | ✅ 完整 | qc_engine.py |
| 复核批注（open→replied→resolved 状态机） | ✅ 完整 | wp_review_service.py |
| 编制/复核双状态机 | ✅ 完整 | workpaper_models.py (WpFileStatus + WpReviewStatus) |
| 底稿-科目四级映射（38条） | ✅ 完整 | wp_account_mapping.json, wp_mapping_service.py |
| 事件驱动级联（DATA_IMPORTED→prefill_stale） | ✅ 完整 | event_handlers.py |
| scope_cycles 循环级权限过滤 | ✅ 完整 | working_paper.py 路由层 |
| 底稿表头自动填充 | ✅ 完整 | wp_header_service.py |
| 交叉索引自动生成 | ✅ 完整 | wp_header_service.py get_cross_ref_text() |

### 1.2 框架已有但未打实的能力

| 模块 | 问题 | 影响 |
|------|------|------|
| LLM 分析性复核 | wp_ai_service.py 调用 LLM 但无 prompt 工程优化，返回质量不稳定 | 审计助理看到的 AI 分析可能无意义 |
| 底稿对话（wp_chat_service） | SSE 流式框架在，但上下文注入不够（缺底稿 parsed_data 注入） | 问答与当前底稿脱节 |
| ONLYOFFICE 插件 | audit-formula + audit-review 代码存在，但未部署到 ONLYOFFICE 容器 | 在线编辑时无法使用取数公式和复核批注 |
| 审计说明 LLM 填充 | 无专门的审计说明生成服务，底稿中"审计说明"区域需手动填写 | 审计助理最耗时的工作之一 |
| 底稿解析回写 | parse_workpaper_real 只提取审定数/结论/交叉引用，不提取审计说明文本 | 复核人看不到底稿中的说明内容摘要 |
| 多用户并发编辑 | WOPI 锁是互斥锁（同一时间只有1人可编辑），无只读协同浏览 | 复核人想看底稿时必须等编制人关闭 |
| 附件与底稿映射 | attachment_working_paper 关联表存在，但前端无可视化映射界面 | 审计证据与底稿的关联不直观 |

### 1.3 完全缺失的能力

| 模块 | 说明 |
|------|------|
| 审计说明智能生成 | 根据科目特征+试算表变动+审计程序结果，LLM 自动草拟审计说明段落 |
| 底稿数据刷新通知 | prefill_stale=true 时前端无主动提醒，用户不知道数据已过期 |
| 角色化底稿视图 | 四种角色看到的底稿界面完全相同，无差异化裁剪 |
| 底稿批量操作面板 | 项目经理无法一次性查看/操作多个底稿的状态 |
| 底稿智能审阅（LLM） | 复核人无 AI 辅助审阅底稿内容（如检查数据一致性、说明完整性） |
| 底稿版本对比 | 无法对比两个版本之间的差异（数值变化、说明修改） |
| 人机协同工作流 | 无明确的"AI 草拟→人工确认→AI 修正"闭环机制 |

## 二、修改方案（修复现有问题）

### 2.1 LLM 调用链路打实

**现状：** wp_ai_service.py 的 analytical_review() 直接拼接简单 prompt 调用 LLM，无科目特征注入、无 TSJ 提示词融合、无输出格式约束。

**修改方案：**

```
修改文件：backend/app/services/wp_ai_service.py

1. analytical_review() 重写 prompt 构建：
   - 注入科目特征（从 wp_account_mapping.json 取科目类别、关联底稿）
   - 注入 TSJ 审计要点（从 tsj_prompt_service 按科目名加载）
   - 注入试算表变动数据（未审数→审定数，变动率）
   - 注入上年同期数据（year-1 的 trial_balance）
   - 输出格式约束：JSON schema（变动原因/风险等级/建议程序/关注事项）

2. 新增 generate_audit_explanation() 方法：
   - 输入：wp_code + project_id + year
   - 从 trial_balance 取本科目审定数/未审数/调整额
   - 从 adjustments 取本科目 AJE/RJE 明细
   - 从 wp_account_mapping 取关联附注章节
   - 拼接 system prompt（角色=资深审计员，要求=简洁专业）
   - 输出：审计说明草稿（200-500字）

3. 新增 review_workpaper_content() 方法（供复核人使用）：
   - 输入：wp_id（从 parsed_data 取审定数/结论/说明）
   - LLM 检查：数据一致性、说明完整性、结论合理性
   - 输出：审阅意见列表（问题描述+建议修改+严重程度）
```

**新增路由：**
- POST /api/projects/{id}/wp-ai/{wp_id}/generate-explanation — 生成审计说明
- POST /api/projects/{id}/wp-ai/{wp_id}/review-content — AI 审阅底稿内容

### 2.2 底稿对话上下文增强

**现状：** wp_chat_service.py 的 SSE 流式对话只注入了基础项目信息，缺少当前底稿的 parsed_data、试算表数据、审计程序状态。

**修改方案：**

```
修改文件：backend/app/services/wp_chat_service.py

1. _build_context() 方法增强：
   - 注入当前底稿 parsed_data（审定数/未审数/AJE/RJE/结论）
   - 注入关联科目的 trial_balance 行
   - 注入关联的 adjustments 明细
   - 注入 TSJ 审计要点（按科目名匹配）
   - 注入底稿 QC 结果（如有未通过项）
   - 注入复核意见（如有未回复项）

2. fill_suggestion 提取增强：
   - 识别 LLM 回复中的数值建议（如"建议调整为 xxx"）
   - 返回 structured suggestion（cell_ref + suggested_value + reason）
   - 前端可一键应用到底稿
```

### 2.3 ONLYOFFICE 插件部署自动化

**现状：** audit-formula 和 audit-review 插件代码在 onlyoffice/plugins/ 目录，但 docker-compose.yml 的 volumes 挂载可能未生效。

**修改方案：**

```
1. docker-compose.yml 验证挂载路径：
   onlyoffice:
     volumes:
       - ./onlyoffice/plugins/audit-formula:/var/www/onlyoffice/documentserver/sdkjs-plugins/audit-formula
       - ./onlyoffice/plugins/audit-review:/var/www/onlyoffice/documentserver/sdkjs-plugins/audit-review

2. 新增 scripts/check_onlyoffice_plugins.py：
   - 检查 ONLYOFFICE 容器内插件目录是否存在
   - 验证 config.json 格式正确
   - 测试 XHR 调用 /api/formula/execute 是否可达

3. audit-formula 插件增强：
   - 新增 EXPLAIN() 函数：调用 /wp-ai/generate-explanation 返回审计说明
   - 新增 REFRESH() 函数：触发当前底稿预填充刷新
   - 错误提示从 #REF! 改为中文提示（如"科目编码不存在"）
```

### 2.4 多用户协同增强

**现状：** WOPI 互斥锁导致同一时间只有1人可编辑，复核人想查看底稿时必须等编制人关闭。

**修改方案：**

```
修改文件：backend/app/services/wopi_service.py

1. check_file_info 增加只读模式判断：
   - 如果当前用户不是锁持有者 → UserCanWrite=False（只读浏览）
   - 如果当前用户是复核人角色 → 始终只读（复核人不应直接编辑底稿）
   - 返回 ReadOnly=True + ReadOnlyReason="其他用户正在编辑" 或 "复核模式"

2. 新增 GET /wopi/files/{id}/viewers 端点：
   - 返回当前正在查看此底稿的用户列表
   - 前端 WorkpaperEditor.vue 底部状态栏显示"当前查看者：张三、李四"

3. 锁超时自动释放优化：
   - 当前 TTL=30min，改为 15min（减少等待时间）
   - 新增心跳检测：前端每 5min 发送 REFRESH_LOCK
   - 超过 2 次未收到心跳 → 自动释放锁
```

### 2.5 底稿数据刷新通知

**现状：** prefill_stale=true 时前端无感知，用户不知道底稿中的取数公式结果已过期。

**修改方案：**

```
1. 后端：working_paper.py list_workpapers 返回值增加 prefill_stale 字段（已有）

2. 前端 WorkpaperList.vue：
   - prefill_stale=true 的底稿行显示橙色 ⚠ 图标 + tooltip "数据已变更，建议刷新"
   - 新增"批量刷新"按钮：选中多个 stale 底稿 → POST /prefill/batch

3. 前端 WorkpaperEditor.vue：
   - 打开 stale 底稿时顶部显示黄色横幅："试算表数据已变更，点击刷新预填充数据"
   - 点击刷新 → 调用 POST /prefill/{wp_id} → 重新执行公式 → 页面 reload

4. 前端 WorkpaperWorkbench.vue：
   - 左栏底稿树节点增加 stale 标记（橙色圆点）
   - 中栏数据卡片显示"数据可能已过期"提示
```

### 2.6 底稿解析回写增强

**现状：** parse_workpaper_real 只提取审定数、结论文本、交叉引用，不提取审计说明段落。

**修改方案：**

```
修改文件：backend/app/services/prefill_engine.py

parse_workpaper_real() 增强：
1. 新增审计说明提取：
   - 搜索关键词："审计说明"/"审计结论"/"执行情况"/"审计发现"
   - 提取关键词下方连续非空单元格的文本内容
   - 写入 parsed_data.audit_explanation（TEXT，最长 2000 字）

2. 新增审计程序执行状态提取：
   - 搜索关键词："审计程序"/"检查程序"/"执行程序"
   - 识别 ✓/√/是/已执行 等标记
   - 写入 parsed_data.procedure_status（dict: {程序名: 已执行/未执行}）

3. 新增附件引用提取：
   - 搜索 "见附件"/"参见"/"详见" 等引用文本
   - 写入 parsed_data.attachment_refs（list[str]）
```

## 三、功能拓展 — 审计助理视角

> 审计助理是底稿的主要编制者，80%的时间花在查账→填数→写说明→自检→提交复核。

### 3.1 审计说明智能生成（核心新功能）

**痛点：** 审计助理每张底稿需手写 200-500 字审计说明，描述执行了哪些程序、发现了什么、结论是什么。这是最耗时的环节。

**方案：**

```
新增服务：backend/app/services/wp_explanation_service.py

        - 从 wp_account_mapping 取科目特征（循环/关联底稿/附注章节）

        - 从 sampling_records 取抽样结果（如有）

        - system: 你是资深审计员，根据以下数据撰写审计说明
        - 模板: 致同标准审计说明格式（审计目标→执行程序→审计发现→审计结论）

        - draft_text: 审计说明草稿

        - 用户可附加反馈（如"语气太正式"/"需要补充抽样结果"）

        """

        批量生成：项目经理一键为所有未填写说明的底稿生成草稿
        """
```

**前端交互：**

```
WorkpaperWorkbench.vue 右栏新增"审计说明"面板：

1. 空白状态：显示"AI 生成审计说明"按钮
2. 点击生成 → SSE 流式输出草稿 → 实时显示在编辑区
3. 编辑区：TipTap 富文本编辑器（支持加粗/列表/引用数据标签）
4. 底部操作栏：
   - "采纳并写入底稿" → 调用 WOPI 或直接写入 parsed_data
   - "重新生成" → 附加反馈文本框 → 调用 refine_draft
   - "手动编写" → 切换为纯手动模式
5. 数据引用标签：点击可跳转到试算表/调整分录对应行
```

### 3.2 底稿编制智能助手

**痛点：** 审计助理不确定某张底稿该执行哪些程序、填哪些数据。

**方案：**

```
WorkpaperWorkbench.vue 右栏"AI 助手"面板增强：

1. 打开底稿时自动加载：
   - TSJ 审计要点（已有，从 tsj_prompt_service 加载）
   - 审计程序检查清单（已有框架，需做实交互）
   - 本科目常见风险提示

2. 新增"智能引导"模式：
   - 根据底稿模板类型（审定表/明细表/分析表）显示不同引导
   - 审定表：引导填写未审数→检查调整→确认审定数→写结论
   - 明细表：引导逐行核对→标记差异→记录发现
   - 分析表：引导计算变动率→识别异常→解释原因

3. 新增"一键检查"按钮：
   - 调用 QC 引擎的子集规则（不是全部14条，只检查当前底稿相关的）
   - 实时显示通过/未通过项
   - 未通过项可点击跳转到底稿对应位置
```

### 3.3 数据提取与刷新可视化

**痛点：** 审计助理不清楚底稿中哪些单元格是公式取数、哪些是手动输入，刷新后不知道哪些值变了。

**方案：**

```
WorkpaperEditor.vue / WorkpaperWorkbench.vue 增强：

1. 公式单元格高亮：
   - 预填充后的单元格背景色浅蓝（区分手动输入的白色）
   - hover 显示公式来源（如 "=TB('1001','期末余额') → 来自试算表"）

2. 刷新差异对比：
   - 刷新预填充后，显示"变更摘要"弹窗
   - 列出所有值发生变化的单元格（旧值→新值）
   - 用户可逐项确认或一键全部接受

3. 数据来源追溯：
   - 底稿中的审定数可点击 → 跳转到试算表对应行
   - 底稿中的调整额可点击 → 跳转到调整分录明细
   - 底稿中的附注引用可点击 → 跳转到附注编辑器对应章节
```

### 3.4 附件证据链可视化

**痛点：** 审计助理上传了附件但不清楚哪些附件关联了哪些底稿，复核人也看不到证据链。

**方案：**

```
WorkpaperWorkbench.vue 中栏新增"证据链"区域：

1. 已关联附件列表（从 attachment_working_paper 查询）：
   - 文件名 + 类型图标 + OCR 状态标签
   - 点击预览（调用 /api/attachments/{id}/preview）
   - 拖拽排序（调整证据顺序）

2. 快速关联：
   - "上传并关联"按钮 → 上传文件 → 自动关联当前底稿
   - "从已有附件关联"按钮 → 搜索下拉选择 → 关联

3. 证据链图谱（可选，项目经理/合伙人视角）：
   - 力导向图：底稿节点 ↔ 附件节点 ↔ 科目节点
   - 点击节点跳转到对应页面
```

## 四、功能拓展 — 项目经理视角

> 项目经理关注的是进度管控、质量把关、团队协调，不直接编制底稿但需要全局视野。

### 4.1 底稿进度总览面板

**痛点：** 项目经理只能逐个点开底稿查看状态，无法一眼看到全项目底稿完成情况。

**方案：**

```
新增页面：ProjectProgressBoard.vue 底稿进度 Tab 增强

1. 进度矩阵（行=审计循环 D-N，列=状态）：
   - 每个格子显示该循环该状态的底稿数量
   - 颜色编码：草稿灰/编制中蓝/待复核橙/已通过绿/已退回红
   - 点击格子 → 展开该组底稿列表

2. 关键指标卡片：
   - 总底稿数 / 已完成数 / 完成率（环形进度条）
   - 待复核数（需要项目经理关注）
   - 逾期底稿数（超过预期完成日期）
   - prefill_stale 数（数据已过期需刷新）

3. 时间线视图：
   - 横轴=日期，纵轴=底稿编号
   - 甘特条显示每张底稿的编制周期（开始→提交→复核→通过）
   - 红色标记逾期底稿

4. 人员负荷视图：
   - 每个审计助理负责的底稿数量和完成率
   - 识别负荷不均（某人 20 张底稿，某人 3 张）
   - 一键重新分配按钮
```

### 4.2 批量操作面板

**痛点：** 项目经理需要批量分配、批量刷新、批量审阅底稿，当前只能逐个操作。

**方案：**

```
WorkpaperList.vue 增强（项目经理角色可见）：

1. 批量选择模式：
   - 顶部工具栏新增"批量操作"按钮 → 进入多选模式
   - checkbox 全选/按循环选/按状态选

2. 批量操作菜单：
   - 批量分配编制人 → 弹窗选择人员 → 调用 POST /assign/batch
   - 批量分配复核人 → 同上
   - 批量刷新预填充 → 调用 POST /prefill/batch → 进度条
   - 批量生成审计说明 → 调用 POST /wp-ai/batch-explanation → SSE 进度
   - 批量下载 → 调用 POST /download-pack → ZIP 下载
   - 批量提交复核 → 检查 QC 门禁 → 通过的提交，不通过的列出原因

3. 新增后端端点：
   - POST /api/projects/{id}/working-papers/assign/batch
   - POST /api/projects/{id}/working-papers/prefill/batch
   - POST /api/projects/{id}/wp-ai/batch-explanation
```

## 七、人机协同机制设计

### 7.1 核心原则

```
AI 不替代审计判断，AI 是审计助理的"副驾驶"：
- AI 草拟 → 人工审阅确认 → AI 根据反馈优化 → 人工最终定稿
- 所有 AI 生成内容必须有明确标记（parsed_data.ai_content 字段）
- 未经人工确认的 AI 内容 → QC 规则 QC-02 阻断提交复核
- AI 建议可一键采纳或一键忽略，不强制
```

## 十、补充建议（7项延伸功能）

### 10.1 底稿模板热更新

**问题：** 363 个致同模板是静态文件，致同每年修订底稿模板（2025版→2026版），当前没有模板版本迁移机制。项目进行中如果模板更新了，已生成的底稿无法升级。

**方案：**

```
新增服务：backend/app/services/wp_template_migration_service.py

1. 模板差异对比：
   - 上传新版模板 → 与旧版逐 sheet 对比（结构差异/公式变化/新增行列）
   - 生成差异报告（新增了哪些行、删除了哪些行、公式变化了哪些单元格）
   - 前端展示差异对比视图（左旧右新，差异高亮）

2. 批量升级：
   - 项目经理选择"升级到新版模板"
   - 系统自动识别哪些项目还在用旧版（通过 wp_template.version 匹配）
   - 升级策略：保留已填数据（审定数/说明/结论），只替换格式和公式
   - 升级前自动备份旧版底稿到 .versions/ 目录

3. 新增端点：
   - POST /api/wp-templates/{code}/compare — 新旧版本对比
   - POST /api/wp-templates/{code}/migrate — 批量升级项目底稿
```

### 10.2 底稿离线工作包

**问题：** 审计现场经常没网。当前离线模式是单个底稿下载/上传，审计助理需要的是一次性打包整个循环的工作资料。

**方案：**

```
新增服务：backend/app/services/wp_offline_pack_service.py

1. 打包内容：
   - 当前循环所有底稿 Excel 文件
   - 关联科目的试算表数据（导出为 Excel）
   - 关联的调整分录明细（导出为 Excel）
   - TSJ 审计要点（导出为 Markdown/PDF）
   - 底稿索引清单（编号+名称+状态+编制人）
   - 离线元数据文件 manifest.json（版本号+打包时间+文件清单）

2. 回传机制：
   - 上传离线工作包 ZIP → 解析 manifest.json
   - 逐个底稿对比版本号 → 冲突检测（本地版本 vs 服务器版本）
   - 无冲突：自动合并
   - 有冲突：列出冲突文件，用户选择保留哪个版本
   - 回传后自动触发 parse_workpaper + prefill 一致性检查

3. 新增端点：
   - POST /api/projects/{id}/working-papers/offline-pack — 打包下载
   - POST /api/projects/{id}/working-papers/offline-sync — 回传同步
```

### 10.3 底稿编制时间统计

**问题：** 完全没有记录审计助理在每张底稿上花了多少时间。WOPI 有 lock/unlock 时间戳可以利用。

**方案：**

```
数据采集（自动，无需用户操作）：
1. WOPI lock 时记录 session_start
2. WOPI unlock/put_file 时记录 session_end
3. 离线模式：下载时记录 checkout_at，上传时记录 checkin_at
4. 写入新表 wp_edit_sessions（wp_id, user_id, start_at, end_at, mode）

聚合展示：
- 项目经理看板：每张底稿累计编制时长 + 每人总工时
- 合伙人看板：按循环汇总编制时长（识别哪个循环最耗时）
- 智能提醒：某张底稿编制时长超过同类底稿平均值 2 倍 → 提示"是否需要帮助？"

新增表：
  wp_edit_sessions (id, wp_id, user_id, start_at, end_at, 
                    edit_mode, duration_minutes)
```

### 10.4 底稿间数据穿透（跨底稿超链接）

**问题：** 当前交叉索引只是文本引用（如"见 E1-2"），点击不能跳转。WP() 公式已有基础但前端未配合。

**方案：**

```
1. 解析交叉引用为可点击链接：
   - parse_workpaper_real 提取 cross_refs 时，同时记录引用位置（sheet+cell）
   - 前端 WorkpaperWorkbench 中栏显示交叉引用列表
   - 点击引用 → 跳转到被引用底稿的对应位置（router.push + 锚点）

2. WP() 公式前端可视化：
   - 底稿 A 中 =WP('E1-2','B5') 引用底稿 B 的 B5 单元格
   - hover 公式单元格 → tooltip 显示"引用 E1-2 审定表 B5 = 12,000"
   - 点击 → 在新 Tab 打开被引用底稿，定位到 B5

3. 反向引用查询：
   - 底稿详情页显示"被以下底稿引用"列表
   - 帮助复核人了解修改当前底稿会影响哪些其他底稿
```

### 10.5 审计程序与底稿双向绑定

**问题：** procedure_instances 和 working_paper 目前通过 wp_code 间接关联，裁剪程序时不会自动影响底稿，底稿完成时不会自动更新程序状态。

**方案：**

```
数据模型变更：
  procedure_instances 新增 working_paper_id UUID FK → working_paper.id

双向联动：
1. 程序裁剪 → 底稿联动：
   - 裁剪程序为 skip/not_applicable → 关联底稿标记为"不适用"（新状态）
   - 恢复程序为 execute → 关联底稿恢复为 draft

2. 底稿完成 → 程序联动：
   - 底稿 review_status 变为 level1_passed → 关联程序 execution_status 自动更新为 completed
   - 底稿被退回 → 关联程序状态回退为 in_progress

3. 前端联动：
   - ProcedureTrimming.vue 裁剪时显示关联底稿名称和当前状态
   - WorkpaperList.vue 显示关联程序的执行状态标签
```

### 10.6 底稿推荐反馈闭环

**问题：** recommend_workpapers 是静态规则推荐，无法学习用户行为优化推荐质量。

**方案：**

```
1. 采纳率统计：
   - 推荐 N 张底稿 → 用户实际生成 M 张 → 采纳率 = M/N
   - 记录每次推荐的 accepted/skipped 状态
   - 按行业/企业规模/科目类别分组统计

2. 推荐规则优化：
   - 采纳率 < 30% 的推荐规则降权
   - 用户手动添加的非推荐底稿 → 反馈为"遗漏"，补充到推荐规则
   - 按项目类型（年审/专项/IPO）差异化推荐权重

3. 新增端点：
   - POST /api/projects/{id}/wp-mapping/recommend-feedback — 记录采纳/跳过
   - GET /api/wp-mapping/recommend-stats — 推荐效果统计
```

### 11.2 三层模板架构

```
第一层：致同标准模板（只读，随年度版本更新）
  ├── 底稿 xlsx 360个 + md 51个
  │   索引：gt_template_library.json
  │   文件：磁盘目录，不入数据库
  ├── 报表行次 JSON（report_config_seed.json，5种准则×4张报表）
  │   公式驱动：TB()/SUM_TB()/ROW()
  └── 附注模板 JSON（note_template_soe/listed.json）
      配置驱动：科目对照+校验公式+宽表公式+正文模板

第二层：项目级模板（从第一层克隆，可按项目修改）
  ├── 底稿：generate_project_workpapers 复制到 storage/projects/{id}/workpapers/
  │   已有机制：shutil.copy2 真实复制 + 表头自动填充
  ├── 报表：clone_report_config 克隆到 report_config 表
  │   已有机制：ReportConfigService.clone_report_config()
  └── 附注：note_section_instances 表（裁剪机制）
      已有机制：note_trim_service 裁剪保留/跳过/不适用

第三层：用户自定义模板（完全自由，不受版本更新影响）
  ├── 底稿：用户上传自定义 xlsx（wp_template_custom 表）
  ├── 报表：用户修改行次公式（update_config API）
  └── 附注：用户自定义章节（note_template_service CRUD）

版本继承规则：
  - 第一层更新时 → 第二层可选择性升级（P4-1 模板热更新）
  - 第二层修改不影响第一层
  - 第三层完全独立，不受任何更新影响
```

### 11.3 底稿模板：51个 md 接入 LLM 知识库

**问题：** 51 个 md 文件包含极有价值的内容（操作手册、底稿填写规范、Excel 制作规格），但当前只是静态文件，AI 助手无法引用。

**处理方案：**

```
1. 操作手册 md（11个，每循环1个）→ 接入 LLM 知识库
   - 类似 TSJ 提示词的 tsj_prompt_service 模式
   - 新增 wp_manual_service.py：按循环/科目名加载操作手册
   - AI 助手回答"这张底稿怎么填"时引用操作手册内容
   - 复核人 AI 审阅时参照操作手册检查底稿完整性

2. 底稿模板库 md（11个）→ 驱动底稿结构识别
   - 每个 md 列出了该循环所有底稿的索引号、名称、sheet 结构
   - parse_workpaper_real 解析时参照模板库 md 知道每个 sheet 的含义
   - 交叉引用验证时参照模板库 md 知道哪些底稿之间应该有引用关系

3. Excel 制作规格 md（目前仅 E 循环有 1 个）→ 驱动自动化
```

### 12.2 致同标准排版规范（从附注模板提取，适用于全部导出文档）

```
一、页面设置
  页边距：左 3cm、右 3.18cm、上 3.2cm、下 2.54cm
  页眉：1.3cm
  页脚：1.3cm

二、字体字号
  中文（含页眉页脚）：仿宋_GB2312、小四（12pt）
  数字和英文（含页眉页脚）：Arial Narrow
  表格中金额过大时：自行缩小字号（建议最小 9pt）

三、三色字体规则（附注专用，导出时必须处理）
  黑色：正式披露内容（保留）
  蓝色：编写者指引，括号内标注（出具报告时必须删除）
  红色：披露格式提示（出具报告时必须改为黑色）
  → Word 导出时自动执行：删除蓝色文本 + 红色文本转黑色

四、文字排版
  标题行（含次级）：左缩进 -2 字符，首行不缩进，居左对齐
  段落间距：段前 0 行、段后 0.9 行，单倍行距
  表格下一行：段前 0.5 行、段后 0.9 行，单倍行距

五、表格排版（三线表）
  缩进：左右不缩进，首行不缩进
  间距：段前段后 0 行，单倍行距
  对齐：垂直居中；标题列左对齐，其他列右对齐
  宽度：同一页表格宽度尽量一致
  边框：上下边框 1 磅，标题行下边框 1/2 磅（无左右边框）
  加粗：标题行 + 合计行加粗
  单行表格：不需要合计行

六、页眉页脚
  页眉：事务所名称（左）+ 项目名称（右），仿宋_GB2312 小四
  页脚：页码居中，格式"第 X 页 共 Y 页"
```

### 12.7 推荐方案：Word 模板填充 + ONLYOFFICE 在线编辑（替代从零生成）

> 上述 12.3-12.6 的 python-docx 从零生成方案作为降级备选。
> 优先采用以下"模板填充 + 在线编辑"方案，格式由模板保证，程序只管填数。

**核心思路：**

```

python-docx 填充数据到占位符/书签
```

```
   - 存储位置：backend/data/word_templates/
```
