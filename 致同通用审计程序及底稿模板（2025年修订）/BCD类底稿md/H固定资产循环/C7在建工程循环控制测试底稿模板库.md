# C7 在建工程循环控制测试底稿模板库

## 📖 文档概述

本文档为在建工程循环控制测试底稿模板库，包含在建工程相关的所有控制测试程序模板，适用于致同会计师事务所的审计工作。

## 🚀 使用说明

1. **控制测试选择**：根据被审计单位的实际情况选择合适的控制测试模板
2. **程序执行**：按照底稿中的控制测试程序逐步执行
3. **记录完整**：确保所有控制测试程序都有完整的执行记录
4. **偏差处理**：及时识别和处理控制偏差

## 📋 标准化选项清单

### 🎯 认定维度选项
- **存在/发生**：确认在建工程交易和事项确实发生，在建工程确实存在
- **完整性**：确认所有应当记录的在建工程交易和事项均已记录
- **准确性、计价和分摊**：确认在建工程交易和事项以恰当的金额记录
- **权利和义务**：确认企业拥有或控制在建工程相关的权利
- **截止**：确认在建工程交易和事项记录于正确的会计期间
- **分类**：确认在建工程交易和事项记录于恰当的账户
- **列报**：确认在建工程在财务报表中得到恰当列报和披露

### ⚙️ 控制属性选项
- **人工的**：完全由人工执行的控制活动
- **自动化的**：由信息系统自动执行的控制活动
- **人工依赖信息系统控制**：依赖信息系统产生信息的人工控制

### ⏰ 频率选项
- **每笔交易**：针对每笔在建工程交易执行的控制
- **每天**：每个工作日执行一次的控制
- **每周**：每周执行一次的控制
- **每半月**：每半个月执行一次的控制
- **每月**：每月执行一次的控制
- **每季度**：每季度执行一次的控制
- **每年**：每年执行一次的控制
- **非常规/低运行频率**：不定期或根据特定情况执行的控制
- **其他（描述）**：其他特定频率的控制

### 🎚️ 与控制相关的风险等级
- **高**：控制失效可能导致重大错报风险
- **中**：控制失效可能导致中等程度错报风险
- **低**：控制失效导致的错报风险相对较低

### 🔍 控制测试方法详细说明

#### 1. 询问
**定义**：是指向被审计单位相关人员进行询问
**适用情况**：了解控制设计和执行情况
**执行要点**：
- 选择合适的询问对象
- 准备结构化的询问提纲
- 获取书面确认（如需要）

#### 2. 检查
**定义**：是指对被审计单位相关文件或记录进行检查
**适用情况**：验证控制执行的书面证据
**执行要点**：
- 检查原始凭证和记录
- 关注审批签字和日期
- 验证文件的完整性和真实性

#### 3. 观察
**定义**：是指察看相关人员执行某项控制活动的过程
**适用情况**：了解控制的实际执行过程
**执行要点**：
- 在正常业务环境下观察
- 记录观察到的关键控制步骤
- 注意控制执行的一致性

#### 4. 重新执行
**定义**：是指独立执行作为被审计单位内部控制组成部分的程序或控制
**适用情况**：验证控制的有效性
**执行要点**：
- 使用相同的数据和参数
- 比较执行结果的一致性
- 分析差异原因

#### 5. 前期
**定义**：利用以前审计获取的某些控制测试的审计证据
**适用情况**：控制未发生变化的情况
**执行要点**：
- 评估控制变化情况
- 确定可依赖的程度
- 补充必要的当期测试

#### 6. 前推
**定义**：利用期中已获取的控制测试证据推断至期末
**适用情况**：期中测试基础上的期末推断
**执行要点**：
- 评估剩余期间的控制变化
- 执行补充测试程序
- 考虑控制环境变化

#### 7. 利用内部审计工作
**定义**：在某些情况下，注册会计师可以利用内部审计人员的工作
**适用情况**：内部审计工作质量可靠的情况
**执行要点**：
- 评估内部审计的客观性和胜任能力
- 复核内部审计工作质量
- 确定利用程度

#### 8. 利用服务机构的审计师的报告
**定义**：服务注册会计师对服务机构控制的报告
**适用情况**：使用服务机构处理在建工程业务的情况
**执行要点**：
- 评估报告的相关性和可靠性
- 关注报告涵盖的期间
- 考虑补充控制的必要性

- **复杂工程计算**：重点测试"准确性、计价和分摊"认定

## 📊 在建工程循环业务流程图及描述

### 在建工程循环主要业务流程

```
工程需求 → 工程立项 → 立项审批 → 施工准备 → 施工执行 → 进度监控 → 质量控制 → 资金支付 → 工程验收 → 工程结算 → 转固处理
    ↓         ↓         ↓         ↓         ↓         ↓         ↓         ↓         ↓         ↓         ↓
  使用部门    工程部     董事会     工程部     施工方     工程部     质检部     财务部     工程部     财务部     财务部
```

### 详细流程描述

**A. 工程立项阶段**
1. **需求识别**：根据业务发展需要识别工程建设需求
2. **立项申请**：编制工程立项申请，明确建设目标和预算
3. **可行性研究**：进行详细的可行性研究和技术论证
4. **立项审批**：按照审批权限进行工程立项审批
5. **预算确定**：确定工程建设预算和资金来源

**B. 施工管理阶段**
1. **施工准备**：进行施工前的各项准备工作
2. **施工执行**：按照施工计划执行工程建设
3. **进度监控**：定期监控工程建设进度
4. **质量控制**：严格控制工程建设质量
5. **安全管理**：确保施工过程的安全

**C. 资金管理阶段**
1. **资金计划**：制定工程资金使用计划
2. **支付审批**：按照合同和进度审批资金支付
3. **支付执行**：执行工程款项支付
4. **资金监控**：监控工程资金使用情况
5. **成本控制**：控制工程建设成本

**D. 验收结算阶段**
1. **工程验收**：组织工程竣工验收
2. **验收确认**：确认工程验收结果
3. **工程结算**：进行工程最终结算
4. **结算审核**：审核工程结算的准确性
5. **转固处理**：将在建工程转为固定资产

### 关键控制点识别

| 业务环节 | 关键风险 | 控制措施 | 控制类型 |
|----------|----------|----------|----------|
| 工程立项 | 立项决策失误、预算不准确 | 立项审批、可行性研究、预算审核 | 预防性 |
| 施工管理 | 进度延误、质量问题、安全事故 | 进度监控、质量控制、安全管理 | 预防性+检查性 |
| 资金管理 | 资金超支、支付风险、成本失控 | 资金审批、支付控制、成本监控 | 预防性+检查性 |
| 验收结算 | 验收不严、结算错误、转固不当 | 验收制度、结算审核、转固控制 | 检查性 |

### 容易出错的领域

1. **工程立项**：可行性研究不充分，预算编制不准确
2. **进度控制**：进度监控不及时，延期风险控制不力
3. **质量管理**：质量控制标准不明确，验收程序不严格
4. **资金管理**：资金支付审批不规范，成本控制不到位
5. **工程结算**：结算审核不充分，转固时点把握不准

### 问题1：如何确定控制测试范围？

**解决方案**：分析偏差原因，评估偏差影响，必要时调整审计策略。

## 🔄 控制偏差评价步骤

### 步骤一：
确认发现的控制例外情况（选定的样本无法执行控制测试）是否属于控制偏差（内控制运行偏离设计）？

| 是/否 | 评价分析过程 | 备注 |
|-------|-------------|------|
| | 该控制例外情况影响在建工程计价的金额，且金额与其他关紧要。进入步骤二 | |

### 步骤二：
确定控制偏差的性质：系统性偏差？人为有意造成的偏差？随机性偏差？

| 随机性偏差 | 注册会计师确定这只是众多工程项目中唯一一个没有报告进度记录的，而且其他所有项目都完成了进度记录的报告。确定应对措施：进入步骤三 |

| 注2：控制偏差的性质 |  |  |  |  |  |
| 系统性偏差是指在相似情形下，预计都会发生的错误。 |  |  |  |  |  |
| 随机性偏差是指偶然发生的错误。随机性偏差是由某一孤立事件引起的，该事件只有在特定条件下才会重复发生，因而 |  |  |  |  |  |
| 随机性偏差对总体误差不具有代表性。 |  |  |  |  |  |
| 如果发现的控制偏差是系统性偏差或人为有意造成的偏差，注册会计师应当考虑是否存在舞弊的迹象以及对审计方案的影响。 |  |  |  |  |  |
| 注册会计师在将控制偏差确认为随机性的时候需要谨慎，在得出该结论之前，应当实施追加的审计程序，以高度确信该偏差 |  |  |  |  |  |
| 对总体误差不具有代表性。 |  |  |  |  |  |
| 在评价控制偏差的性质时，需要考虑： |  |  |  |  |  |
| （1）控制偏差是否具有共同的特征。例如，控制偏差发生在某些特定类型的业务、地点或生产线中，或者发生在同样的人员 |  |  |  |  |  |
| 执行控制时，抑或是发生在某一特定的时期。 |  |  |  |  |  |
| （2）控制偏差是否是由于基础工作错误造成的。例如，控制偏差是由于错误的公司或计算造成的，或者是基于不完整或 |  |  |  |  |  |
| 不准确的数据造成的。 |  |  |  |  |  |
| 如果确认偏差时系统性的或人为有意造成的，应直接将该控制偏差视为一项内部控制缺陷。 |  |  |  |  |  |
| 注3：随机性偏差的应对措施 |  |  |  |  |  |
| 如果认为一项控制偏差是随机的，则应当确定适当的审计应对措施。包括： |  |  |  |  |  |
| （1）扩大样本量进行测试，从而确定测试发现的错误率在可容忍范围内； |  |  |  |  |  |
| （2）直接将该项控制偏差视作一项内部控制缺陷。 |  |  |  |  |  |
| 如果决定扩大样本量进行测试，注册会计师应当根据发现的控制偏差的数量来确定扩大样本量的范围。例如，对于每日发生 |  |  |  |  |  |
| 多次的控制，如果初始样本规模为25个，当测试发现一项控制偏差，且该偏差不是系统性偏差时，注册会计师扩大样本规模 |  |  |  |  |  |
| 进行测试，所增加的样本规模至少为15个。 |  |  |  |  |  |
| 注4：扩大样本量后的测试结果 |  |  |  |  |  |
| 如果扩大样本量测试后再次发现偏差，则注册会计师可以得出该控制无效的结论。 |  |  |  |  |  |
| 如果扩大样本量测试没有再次发现偏差，则注册会计师可以得出该控制有效的结论。即尽管存在一项控制偏差，但测试范围 |  |  |  |  |  |
| 内的错误率在可容忍范围内且该控制偏差不具有代表性。 |  |  |  |  |  |
| 单项控制并非一定要毫无偏差地运行，才被认为有效。 |  |  |  |  |  |
| 注5：评价是否构成设计缺陷 |  |  |  |  |  |
| 若确定发现的控制例外情况不属于控制偏差，则可以选取替代样本以获得证明控制运行有效的充分、适当的审计证据。 |  |  |  |  |  |
| 但需确定控制例外情况是否表明相关控制的设计存在缺陷。 |  |  |  |  |  |
| 设计缺陷是指缺少为实现控制目标所必需的控制，或现有控制设计不适当、即使正常运行也难以实现预期的控制目标。 |  |  |  |  |  |
