# 上下文压缩策略

来源：[Agent-Skills-for-Context-Engineering](https://github.com/muratcankoylan/Agent-Skills-for-Context-Engineering)

## 核心原则

优化目标是 tokens-per-task（完成任务的总 token），不是 tokens-per-request。压缩丢失关键信息导致的重新探索，比压缩节省的 token 更多。

## 压缩方法选择

| 方法 | 适用场景 | 压缩率 | 质量 |
|------|---------|--------|------|
| 锚定迭代摘要 | 长会话、需要文件追踪 | 98.6% | 3.70/5 |
| 再生式全量摘要 | 有明确阶段边界的会话 | 98.7% | 3.44/5 |
| 不透明压缩 | 短会话、重取成本低 | 99.3% | 3.35/5 |

## 结构化摘要模板

压缩时必须保留以下结构：

```markdown
## 会话意图
[用户要完成什么]

## 已修改文件
- 文件路径: 具体改了什么（函数名级别）

## 已做决策
- 决策内容及原因

## 当前状态
- 测试/构建状态

## 下一步
1. 具体待办
```

## 关键规则

1. 上下文利用率达 70-80% 时触发压缩，不要等到 90%+
2. 永远不要压缩工具定义和 schema
3. 先压缩工具输出（占 80%+ token），再压缩对话历史
4. 文件路径、函数名、错误码必须原样保留，不要改写
5. 调试进行中时，不要掩码错误输出
6. 会话前几轮包含不可重新推导的约束，优先保护
7. 代码和文本需要不同的压缩策略——文本可以大幅压缩，代码必须原样保留
