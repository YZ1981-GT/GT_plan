## UI 视觉偏好（详 #conventions）

- 表格列宽充足不折行不省略号；数字列统一 `.gt-amt`；**所有 el-table 必须加 `border` + 各列 `resizable`**（用户可拖拽调整列宽），固定宽度列优先用 `min-width` 而非 `width`
- 表格选中行 ≥14% 透明度 + 左 3px 紫竖线 + hover 浅反馈
- 工具栏按钮合并到 Tab 栏右侧（不独占行）；简单 CRUD 不用 GtPageHeader 紫渐变
- **GtToolbar slot 契约**（2026-05-24 沉淀，调整分录全屏按钮死代码教训）：`GtToolbar` 只定义 `#left` / `#right` / `#right-extra` 三个具名 slot，**没有默认 slot**；直接写在 `<GtToolbar>` 标签下的子节点会被 Vue 静默丢弃（DOM 里压根不渲染），用户看不到按钮就以为"没生效"；正确做法 = 自定义按钮必须显式包 `<template #right-extra>` 或 `#left`；判定 = 任何"插槽组件子节点没渲染"问题先核对组件 template 是否定义了对应 slot
- **全屏按钮归位规约**（2026-05-24 沉淀）：表格类页面全屏按钮统一放**表格自身工具栏**（`GtEditableTable` 已内置 `useFullscreen` + `⛶ 全屏` 按钮），不要挂横幅 GtToolbar——理由 = 全屏语义是"只放大核心数据区"，横幅/KPI 卡/筛选器都不需要看；`.gt-fullscreen` CSS class（`position:fixed; inset:0; z-index:2000`）是视口级全屏（盖侧栏顶栏）已够用，不要走 `element.requestFullscreen()` Fullscreen API（SPA 里 ESC 退出后状态恢复差）；判定 = 表格页 `useFullscreen` 实例化只能存在一处（要么页面级要么表格级），重复实例化是死代码
- **Tab 栏 + 同行工具按钮模式**（2026-05-24 沉淀，调整分录"全屏挪到 tab 行"实现）：当表格只读视图（editable=false）GtEditableTable 内置工具栏只剩孤零零一个全屏按钮时，不挤独占行而是合并到 tab 栏右侧——三件套 = ①外层 `<div class="X-tabs-row" style="display:flex;align-items:center;gap:12px">` + el-tabs `flex:1` + 按钮 `flex-shrink:0` + 按钮 `margin-bottom:8px` 对齐 tabs 下边线 ②`:show-toolbar="false"` 关闭表格内置工具栏避免重复 ③通过 `tableRef.value.fullscreen.toggleFullscreen()` 代理（GtEditableTable 已 `defineExpose({ fullscreen, ... })`）+ `tableFullscreen = computed(() => tableRef.value?.fullscreen?.isFullscreen?.value)` 读状态；判定 = 任何"工具栏只剩 1-2 个按钮"场景都用此模式而不是独占行
- **全屏退出兜底铁律**（2026-05-24 沉淀，调整分录 show-toolbar=false 翻车修复）：`GtEditableTable` 工具栏 `v-if` 条件必须 `showToolbar || fullscreen.isFullscreen.value`——只用 `showToolbar` 时关闭工具栏 + 进入全屏 = 用户只能靠 ESC 退出（外部按钮被全屏盖住看不见，不知道 ESC 即卡死）；正确语义 = 全屏强制显示工具栏（至少有"退出全屏"按钮）哪怕业务关闭了工具栏；判定 = 任何"`v-if=showToolbar`"控制的工具栏组件，只要内含全屏按钮就必须 OR `isFullscreen`
- **全屏 fixed 失效铁律**（2026-05-24 沉淀，GtEditableTable 真因修复）：CSS 规范 = 任何祖先有 `transform` / `filter` / `perspective` / `will-change` 都会让子孙的 `position: fixed` **不再相对视口**而是相对该祖先（成为 containing block）；本项目隐藏陷阱 = ①`.gt-fade-in` 动画用 `transform: translateY(16px)→0` ②路由 `slide-left / slide-right` Transition 用 `transform: translateX` ③hover 卡片普遍用 `transform: translateY(-2px)`——任何一个挂在祖先链上 `position:fixed` 全屏元素都会被夹在 detail 区域内（看到侧栏顶栏没被盖住即此症）；正确解 = 全屏组件用 `<Teleport to="body" :disabled="!isFullscreen">` 切换时传送到 body 下彻底脱离 transform 祖先链，el-table 配 `:height` 不是 `:max-height`（max-height 不强制撑满，只能算上限）+ 全屏切换后 `nextTick(() => tableRef.value?.doLayout?.())` 重算列宽（父容器换祖先后列宽错位）；判定 = 任何 `position:fixed` 行为异常先 `grep "transform:" .` 在祖先链上找肇事者，找不到再考虑 z-index 或 stacking context
- **CSS 全屏三件套**（2026-05-24 沉淀）：基础组件做"真正全屏"必须 = ①`<Teleport to="body" :disabled="!fullscreen">` 切换式传送（避免 transform 祖先夹住） ②`:height` 计算式撑满（`calc(100vh - 110px)` 减工具栏 + footer + padding），非全屏时回退 `:max-height` 弹性 ③`watch(isFullscreen) → nextTick → doLayout()` 重算列宽；缺一即视觉异常（不全屏 / 表格高度未撑满 / 列宽错位）
- **长流程加载阶段提示铁律**（2026-05-24 沉淀，WorkpaperEditor Phase 5.1）：多步异步加载场景（详情→元数据→工作簿→引擎→渲染等 ≥3 步）必须配 `loadingHint` ref + `GtLoadingOverlay` 的 `:hint` prop 让用户感知卡在哪一步，不要只挂"加载中..."黑盒（用户看 5 秒不动就以为卡死）；模式 = ①`const loadingHint = ref('')` 挂顶部 ②每个 await 前 `loadingHint.value = '当前阶段名'` ③加载完 `loadingHint.value = ''`；判定 = 任何"超过 1 秒、≥3 阶段"的加载流程都加阶段提示，单步快加载（<500ms）可省略
- **加载失败友好引导铁律**（2026-05-24 沉淀，WorkpaperEditor Phase 4.1）：API 加载失败禁止"`goBack()` 直接踢回去 + ElMessage toast"——用户被踢回前根本看不清错误，且失去重试机会；正确做法 = 用 `loadErrorState` 状态机（no_file/no_index/invalid_id/error 四态）+ 全屏 overlay 卡片（图标+标题+消息+操作按钮：返回列表/前往生命周期/重试），让用户看清问题再决定下一步；UUID 格式校验前置拦截避免后端 404 的"底稿不存在"误导信息；判定 = 任何"加载失败 = goBack/router.push 强制跳转"模式都改为状态展示 + 用户主动选择
- **el-table 全屏高度铁律**（2026-05-24 沉淀，CSS flex 终稿，废弃 JS viewportH 减常数方案）：`:height="number"` viewportH 减常数方案在 padding/字号/工具栏行高变化时基线会失准（90、120、140 来回调都不准），**必须改 CSS flex 自适应** — 三件套 = ①父容器 `.gt-fullscreen { display: flex; flex-direction: column }` ②el-table 选择器 `:deep(.el-table) { flex: 1; min-height: 0 }` 自动吃剩余空间 ③el-table prop `:height="'100%'"` 字符串（el-table 走"父容器高度"代码分支，跟数字硬编码不同）；不依赖 ResizeObserver / window.resize 监听 / 任何扣减常数，浏览器 resize / DevTools 打开 / 字号缩放全部自动响应；Teleport 切换 + 全屏切换仍需保留三阶段 doLayout（`nextTick + requestAnimationFrame + setTimeout(100)`）让 el-table 重算列宽；`.gt-fullscreen` padding 全方位 8px 12px（不要 16px 太占空间）；判定铁律 = 任何"全屏后表格只占视口一半"症状不要再纠结 JS 扣减数字精确度，直接换 flex column + flex:1 + height:100% 三件套
- 所有表格统一 el-table（底稿编辑器仍 Univer，不引入 AG Grid）
- 标准分页非"加载更多"；四表联查支持全屏+行选择+右键菜单
- 溯源/穿透跳转支持 Backspace 返回（DefaultLayout `initGlobalBackspace`）

- **底稿模块 Tab 顺序**（2026-05-24）：生命周期→委派矩阵→列表→工作台→看板→依赖图→手册（生命周期第一位=先裁剪程序，委派紧跟=裁剪后立即分配）；树默认折叠（去掉 `default-expand-all`）；新增"手册"视图（体系总览/审计流程/底稿关系/循环详解 4 子页 + 面包屑返回 + 交叉索引跳转到列表视图选中底稿）

## 操作铁律（标题级，详细规则见 #conventions）

- **彻底解决不绕开**：错误必须复现+根因+修主代码+防御测试
- **三层一致校验**：DB 迁移 + ORM `Mapped[]` + service 方法，任一缺失即伪绿（AT-3 实战教训）
- **可复用脚本沉淀**：批量入库/UAT/迁移类工具放 `backend/scripts/{name}.py`（非 `_{tmp}.py`）配 docstring + 多场景；判定 = 操作目标可能再发生即保留
- **PG 运维**：SET 不支持绑定参数（用 set_config）/ superuser bypass RLS / CONCURRENTLY 必须 asyncpg raw conn + lock_timeout / 失败留 _ccnew 残骸需先 cleanup
- **后台 worker 心跳完整性铁律**（2026-05-24 沉淀，DegradedBanner "服务响应较慢"诊断修复）：`event_cascade_health_service._WORKER_NAMES` = (`sla_worker`, `import_recover_worker`, `outbox_replay_worker`, `import_worker`) 4 个 worker 心跳缺一即 degraded（前端 DegradedBanner 黄色横幅永久挂着）；本地默认 `LEDGER_IMPORT_IN_PROCESS_RUNNER_ENABLED=True` 时 `ImportJobRunner.run_forever()` 才会写 `import_worker` 心跳但**长期漏挂**在 `main.py::_start_workers` 列表里（pre-existing bug，commit 修复）—— `ImportJobRunner.enqueue()` 把任务塞队列后无人主动消费，导入任务只能靠 `import_recover_worker` 兜底慢恢复；正确做法 = `_start_workers` 按 `settings.LEDGER_IMPORT_IN_PROCESS_RUNNER_ENABLED` 条件 `tasks.append(asyncio.create_task(ImportJobRunner.run_forever(stop_event=stop_event)))`，生产关闭后改用 `python -m app.workers.import_worker` standalone 进程；判定铁律 = 任何"DegradedBanner 持续报 worker 异常"先 `docker exec audit-redis redis-cli keys "worker_heartbeat:*"` 看真实心跳 key 列表，对照 `_WORKER_NAMES` 找缺失的，再回溯 `main.py::_start_workers` 是否真的启动了对应循环（不要只看 worker 文件存在就以为已启动）
- **year 必传参数三级 fallback 铁律**（2026-05-24 沉淀，"一键重算"422 修复）：项目维度业务接口（trial-balance/recalc / report/* / disclosure/* / 全量重算等）后端普遍声明 `year: int = Query(...)` **必填 query**，漏传爆 422 错误顶部显示原始 pydantic JSON `[{type:missing,loc:[query,year],msg:Field required}]`；前端取 year 标准三级 fallback = `projectStore.year || Number(route.query.year) || new Date().getFullYear() - 1`（projectStore 已在 DefaultLayout 实例化全局可用，syncFromRoute 自动同步路由 query；最后一档兜底当前年-1 因为审计常对上年度数据）；`apiProxy.api.post` 第三参是 `AxiosRequestConfig` → `{ params: { year } }` 自动拼 `?year=2025`；判定 = 任何写 `api.post('/api/projects/${pid}/...')` 路径都先 grep 后端 router 看是否 `Query(...)` 必填，是则三级 fallback 取 year + params 传；本次实例 = `DefaultLayout::onRecalcStale` 黄色横条"一键重算"按钮长期漏传 year（pre-existing bug）
- **router_registry 必查铁律**（2026-05-23 沉淀）：新建 router 文件后必须在 `backend/app/router_registry/{system,collaboration,workpaper,...}.py` 对应分组 `from app.routers.X import router as X_router` + `include_router(X_router, ...)`；漏挂时 endpoint 写好但前端 404（如 independence.py / `/api/my/pending-independence` 漏挂事件）；判定 = `grep -r "from app.routers.NAME"` 在 router_registry 下找得到即 OK
- **本地 Redis 容器名**（2026-05-24 沉淀）：实际容器名 `audit-redis`（不是 `gt-redis`），暴露端口 6380→6379（容器内 6379 但宿主映射 6380）；命令 = `docker exec audit-redis redis-cli ping` / `docker exec audit-redis redis-cli keys "worker_heartbeat:*"`；其它容器 = `audit-postgres` (5432) / `audit-metabase` (3000)；判定 = 所有 docker 排查脚本/文档都用 `audit-*` 前缀名
- **WpFileStatus 完成语义**（2026-05-23 沉淀）：「底稿已完成」= `status in (review_passed, archived)`；不要猜 `locked / done / completed` 等不存在枚举值（PG 会爆 InvalidTextRepresentationError）；完整枚举见 `backend/app/models/workpaper_models.py::WpFileStatus`
- **临时文件不进 commit**（2026-05-23 沉淀）：写长 commit message 时若用 fsWrite 创建 `commit-msg.txt` 中转，必须 `git rm --cached commit-msg.txt` 后 `--amend` 清掉；更稳的做法 = 用 `git commit -m "..." -m "..."` 多 -m 拼接或反斜杠续行
- **SQLite 测试 set_rls_context 兼容**（2026-05-23 沉淀）：set_rls_context 调 PG 的 `set_config(...)` 在 SQLite in-memory 测试会爆 `no such function`；admin 路径走 require_project_access 仍会触发；测试侧 `patch("app.deps.set_rls_context", new=AsyncMock())` 绕开（不在 conftest 全局短路，按需测试 mock 保留生产语义）
- **FastAPI dep_overrides 闭包陷阱**（2026-05-23 沉淀）：`require_project_access("readonly")` 工厂每次返回新闭包，`app.dependency_overrides[require_project_access("readonly")]` 不会命中路由实际 Depends 对象；正确做法 = 仅 override `get_current_user` + `get_db`，让 admin 路径自身短路，配合上一条 mock set_rls_context 即可
- **PowerShell**：写中文/emoji 用 fsWrite 工具；`Out-File` 文件锁需先 Stop-Process powershell + 用 `cmd /c "... > log 2>&1"`；多 `-m` 长 commit message 含 `()` / `→` / 中文冒号会被 pwsh 当子表达式或 pathspec 错切，必须 `git commit --% -m "..." -m "..."` 用 stop-parsing token 把后续参数原样交给 git（`commit-msg.txt` 临时文件方案不进 commit 是底线，`--%` 是优选）
- **agent 调 service 优于 Playwright UI**：大文件入库直调 ledger_import 管线快 10x，Playwright 仅做前端可见性验证
- **历史档案不回填修改铁律**（2026-05-23 沉淀）：`.kiro/steering/dev-history.md` / `.kiro/specs/*/tasks.md` 等历史记录是 append-only 审计轨迹；做目录重组/路径迁移时**不回填**这些文档中的旧路径（保持时点准确性），只更新活跃代码 + 当前文档；判定边界 = "记录写入时该路径是真的吗" → 是即保留
- **vue-tsc 类型债务清理 SOP**（2026-05-23 沉淀，143→0 实战）：mitt `Events` 类型表是中央枢纽，新事件必须显式补 key（漏一个则订阅/发布两端都爆）；`SyncEventPayload` 扩展用 `[key: string]: any` escape hatch + `project_id` optional 兼容编辑锁/导出等带顶层自定义字段的 SSE 推送；spec test 引用的 composable 导出（如 `useEditTransition` / `EDIT_TRANSITION_MS` / `LockConflictInfo` / `transitioning`）必须真的实现并导出，不是占位；element-plus ElTag/ElBadge `type` 不接受 `''`，要用 `undefined` + 联合改具体字面量；el-radio-group `@change` 签名是 `(string|number|boolean|undefined)` 不能直接收窄到业务枚举，需在 handler 内 cast；el-dialog `@close` 是无参 `() => any`（`:before-close` 才有 `done` 参数，别混淆）；FUniver / xlsx SDK 有未对外暴露的方法（如 `createUnit`）用 `(api as any).method()` cast 避免阻塞编译；判定 = 任何"理论上对"但 vue-tsc 报错的位置都先看类型源头不要绕开 ts-ignore
- **Vue setup ref 定义顺序铁律**（2026-05-24 沉淀，WorkpaperEditor "Cannot access wpDetail before initialization" 实测教训）：`<script setup>` 中的 ref/computed 按代码顺序执行，computed 引用未定义的 ref 会爆 `ReferenceError: Cannot access 'X' before initialization`；本次实例 = 第 943 行的 `componentType` computed 链路依赖 `wpDetail`，但 `wpDetail` 在第 1370 行才定义；判定铁律 = 业务 ref（`wpDetail` / `loading` / 表单数据等）必须在 `<script setup>` **最顶部**先 `ref()` 定义（紧跟 `import` 和 `route/router` 之后），所有 computed/composable 调用之后；典型错误模式 = "改动模板加了 `v-if="!data"` 守卫但 data 在 setup 后部定义，触发 setup 阶段未初始化的 computed 重算"；测试纪律 = 任何 WorkpaperEditor 这种千行组件改动必须 **Playwright 实测点击触发**，仅 `getDiagnostics` 通过不代表运行时无错
- **顶层 v-if 守卫拦 init 死锁铁律**（2026-05-24 沉淀，WorkpaperEditor "加载底稿中..." 永久卡死实测）：在依赖 template ref 挂载触发 init 的组件（如 Univer 编辑器需 `univerContainer` ref 挂载后 `initUniver` 才能执行到 `loading.value = false`），**不能在顶层加 `v-if="loading"` 守卫**——会形成死锁：守卫等 loading=false / loading=false 需要 init 跑完 / init 需要 container 渲染 / container 渲染需要守卫不拦；正确做法 = ①顶层守卫只用于"分支选择"（如 `componentType !== 'univer' && !wpDetail` 仅守子编辑器路径） ②univer 路径直接渲染容器 div，由内部 absolute-positioned `.loading-overlay`（`v-if="loading"`）覆盖加载状态；判定铁律 = 任何加 `v-if="loading"` 守卫前先确认"该 v-if 拦截的 DOM 是否包含 init 必需的 ref"，包含则改 overlay 模式（容器永远渲染 + 内部蒙层）
- **useWpDetailGuard 三态 case 默认接入铁律**（2026-05-25 沉淀，WorkpaperEditor Phase 4 实测）：任何依赖 wpId 的视图（编辑器/列表详情面板/子编辑器）必须调用 `useWpDetailGuard(wpId)` 获取三态（loading/error/ready），不允许直接 `goBack()` 跳转处理异常；overlay 模式取代顶层 v-if 守卫 = 容器永远渲染 + GtLoadingOverlay/GtError overlay 覆盖状态（参见"顶层 v-if 守卫拦 init 死锁铁律"）；已接入：WorkpaperEditor + WorkpaperList + FormEditor/WordEditor/TableEditor/HybridEditor
- **Vue setup const 声明顺序铁律**（2026-05-25 沉淀，commit 065c91e WorkpaperEditor 真因修复教训扩展）：原"setup ref 顺序铁律"扩展到 **所有 const 声明** — 任何 `const X = useY(..., Z)` 引用的参数 Z 必须在 X 之前已定义；本次实例 = `const iCycle = useICycleEditor(..., cycleDialogs)` 等 6 个 cycle composable 实例化放在 `const cycleDialogs = useCycleDialogs(...)` 之前，触发 `ReferenceError: Cannot access cycleDialogs before initialization`，下游表现为 `Cannot read properties of undefined (reading 'data')` 被 ErrorBoundary 兜底；**判定 = const 链式依赖必须按拓扑顺序写**，cycleType → cycleDialogs → 各 cycle composable，不要分散到不同代码块；**imports 是 hoisted 不受此约束**，但 const 不是；**子代理实施时声称"placed in correct order"不能盲信，必须 Playwright MCP 真浏览器验证 setup 阶段 console**
- **append-to-body :deep() 失效铁律**（2026-05-24 沉淀，审计导航图 dialog 样式实测）：`<style scoped>` 中的 `:deep()` 选择器从组件 root 开始查找，但 `el-dialog` / `el-drawer` / `Teleport to="body"` 的内容已被传送到 `<body>` 下脱离了组件作用域，`:deep()` 选不到目标元素（CSS 编译后变成 `[data-v-xxx] .target` 形式但 body 下元素没有该 data 属性）；正确做法 = ①新增独立全局 `<style>` 块（不带 scoped）放传送出去的元素样式 ②或用 `<style scoped>` + `<style>` 双块组合；判定 = `:deep()` 写了但样式不生效先用 DevTools 检查目标元素是否在组件 DOM 树外，是则迁到全局块
- **前端硬编码假数据铁律**（2026-05-24 沉淀，WorkpaperAuditNav 关系图 E1 假数据被用户发现）：UI 组件中的"演示数据/示例数据"（如硬编码的 `graphNodes` / `graphEdges`）必须在产品成熟阶段全部移除，改为后端 API 数据驱动；典型反模式 = 注释写"简化版"/"示例"但实际生产环境跑 = 用户在 D 循环底稿看到 E1 循环节点；判定铁律 = 任何 wp_code 维度的可视化（关系图/进度图/节点图）必须根据 `props.wpCode` 动态获取数据，硬编码节点数组 = 假数据等于 bug；现有数据源 = `cross_wp_references.json` 400 条 + 后端 `/relation-graph` 端点（按 wp_code 实时计算上游/下游 + 查 DB 验证存在性）
