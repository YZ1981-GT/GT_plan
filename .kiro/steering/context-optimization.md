# 上下文优化技术

来源：[Agent-Skills-for-Context-Engineering](https://github.com/muratcankoylan/Agent-Skills-for-Context-Engineering)

## 四大优化策略（按优先级）

### 1. KV-Cache 优化（零风险，优先做）
稳定内容放前面，动态内容放最后：
1. System prompt（最稳定）
2. 工具定义（跨请求稳定）
3. 复用模板和 few-shot 示例
4. 对话历史（增长但共享前缀）
5. 当前查询和动态内容（最后）

关键：system prompt 中不要插入时间戳、session ID 等动态值，一个空格的变化都会使缓存失效。

### 2. 观测掩码（最大收益）
工具输出占 80%+ token，选择性掩码：
- 永不掩码：当前任务关键输出、最近一轮输出、调试中的错误输出
- 3 轮后掩码：已提取要点的冗长输出，替换为 `[Obs:{ref_id} 已省略. 要点: {摘要}]`
- 立即掩码：重复输出、样板头尾

### 3. 压缩（有损，谨慎用）
- 70-80% 利用率时触发，不要等到 90%+
- 目标：50-70% token 缩减，<5% 质量下降
- 先压缩工具输出，再压缩旧对话，最后压缩检索文档
- 永不压缩 system prompt

### 4. 上下文分区（最后手段）
预估上下文超过窗口 60% 时考虑分区：
- 拆分为独立子任务，分配给子 agent
- 每个子 agent 在干净的上下文中工作
- 协调开销是真实的——少于 3 个子任务通常不值得分区

## 预算分配

会话开始前分配 token 预算：
- System prompt + 工具定义：固定预留
- 检索文档：按需分配
- 对话历史：滑动窗口
- 工具输出：掩码后保留
- 缓冲区：总量的 5-10%

## 性能目标

| 技术 | Token 缩减 | 质量影响 | 延迟影响 |
|------|-----------|---------|---------|
| KV-Cache | 50%+ 成本降低 | 零 | -40% |
| 观测掩码 | 60-80% | <2% | 近零 |
| 压缩 | 50-70% | <5% | 压缩步骤本身有开销 |
| 分区 | 视任务而定 | 协调开销 | 增加 |
