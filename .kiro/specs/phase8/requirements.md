# Phase 8 — 需求文档

## 概述
Phase 8 聚焦于数据模型优化、性能提升和用户体验完善。修复 Phase 7 遗留的数据模型问题，优化系统性能，完善核心功能的细节体验。

## Phase 0-7 已实现功能回顾

为确保 Phase 8 需求与主需求文档对齐，以下列出 Phase 0-7 已实现的核心功能，Phase 8 需在此基础上进行优化完善：

### Phase 0：基础设施
- 基础架构搭建、Docker 容器化、数据库设计
- FastAPI 后端框架、Vue 3 前端框架
- PostgreSQL 16、Redis 7、Alembic 迁移

### Phase 1a：核心功能
- 科目体系（account_chart）、试算表（trial_balance）、余额表（tb_balance）
- 序时账（tb_ledger）、凭证表、辅助余额表（tb_aux_balance）、辅助明细账（tb_aux_ledger）
- 科目映射（account_mapping）、报表行次映射（report_line_mapping）
- 审计调整分录（adjustments、adjustment_entries）、重要性水平（materiality）、未更正错报（unadjusted_misstatements）

### Phase 1b：底稿管理

- 附注自动生成（disclosure_notes）
- 报表格式配置（report_config）、财务报表数据（financial_report）

### Phase 2：合并报表

### Phase 3：协作功能

## 需求优先级

- **P0（必须实现）**：需求1（数据模型字段缺失修复）、需求2（查询性能优化，含2d索引补齐/2e事件去重/2f超时控制/2g流式导入）
- **P1（重要）**：需求3（底稿编辑体验优化）、需求4（报表导出优化）、需求7（数据校验增强）、需求10（安全增强，含10d权限缓存）
- **P2（可选）**：需求5（移动端适配）、需求6（审计程序精细化）、需求8（性能监控）、需求9（用户体验优化）

## 需求依赖关系

- 需求1（数据模型字段缺失修复）是需求2（查询性能优化）的前置依赖
- 需求2（查询性能优化）是需求3（底稿编辑体验优化）和需求4（报表导出优化）的基础
- 需求8（性能监控）依赖需求2（查询性能优化）的性能指标
- 需求10（安全增强）独立于其他需求，可并行开发

## 需求验收标准

每个需求的验收标准在各子需求中已标注性能目标，最终验收需满足：
1. 所有P0需求100%完成并通过测试
2. 所有P1需求至少90%完成并通过测试
3. 所有P2需求根据资源情况灵活调整
4. 性能目标必须达到预期指标
5. 跨阶段兼容性测试通过
6. 回归测试通过

### 需求 2：查询性能优化

#### 2a 穿透查询性能优化（✅缓存部分已在 Phase 6 完成，游标分页为增量）
- ~~2a.1 优化 LedgerPenetrationService 的查询逻辑~~

- 2a.4 分页优化：大数据量场景下使用游标分页替代 OFFSET
- 2a.5 性能目标：穿透查询响应时间从平均 2s 降低到 1s 以内

#### 2b 四表联查性能优化
- 2b.1 优化四表（余额表/序时账/凭证表/辅助表）联查的 SQL 查询
- 2b.2 使用 CTE（Common Table Expression）优化复杂查询
- 2b.3 查询计划分析：使用 EXPLAIN ANALYZE 分析并优化慢查询
- 2b.4 批量查询优化：减少 N+1 查询问题
- 2b.5 性能目标：四表联查响应时间提升 50% 以上

#### 2c 报表生成性能优化
- 2c.1 优化 ReportEngine 的公式计算性能
- 2c.2 使用缓存策略：报表公式计算结果缓存（Redis TTL=10min）
- 2c.3 增量更新：只重新计算受影响的报表行
- 2c.4 报表生成完成后发布 REPORT_GENERATED 事件（供附注/导出等下游服务监听）
- 2c.5 性能目标：报表生成时间从平均 5s 降低到 3s 以内

#### 2d 核心表复合索引补齐
- 2d.1 trial_balance 表补充复合索引 (project_id, year, standard_account_code)——试算表重算和报表生成的核心查询路径
- 2d.2 tb_balance 表补充复合索引 (project_id, year, is_deleted)——导入服务计数和软删除查询
- 2d.3 adjustments 表补充复合索引 (project_id, year, account_code)——调整分录按科目查询（已有按 type 的索引，缺按 code 的）
- 2d.4 import_batches 表补充复合索引 (project_id, year)——导入批次查询
- 2d.5 合并到 034 迁移脚本中一次性创建

#### 2e 事件总线去重与合并
- 2e.1 EventBus 新增事件去重机制：相同 (event_type, project_id, account_codes) 在 debounce 窗口内（默认 500ms）只触发一次
- 2e.2 批量调整分录操作（如一次创建 10 条 AJE）合并为单次 ADJUSTMENT_CHANGED 事件，避免 N 次试算表全量重算
- 2e.3 事件去重窗口可配置（config.py EVENT_DEBOUNCE_MS）

#### 2f 公式引擎超时控制
- 2f.1 FormulaEngine.execute() 添加 asyncio.wait_for 超时包装（默认 10s）
- 2f.2 超时后返回 FormulaError(code="TIMEOUT") 而非卡死
- 2f.3 记录超时公式到日志（含公式内容、project_id、account_code）供排查

#### 2g 数据导入流式处理
- 2g.1 import_service.start_import() 文件解析阶段改为流式读取（openpyxl read_only=True）
- 2g.2 避免一次性加载整个 Excel 到内存（26 万行凭证表可能占用 500MB+）
- 2g.3 分批解析+分批校验+分批写入，每批 CHUNK_SIZE 行
- 2g.4 解析过程中发布 SSE 进度事件（已解析行数/总行数）
