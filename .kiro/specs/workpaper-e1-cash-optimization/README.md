## 一、为什么做(基于真实模板核验的痛点)

合伙人/审计助理实际进入 E1 货币资金底稿时遇到 **6 类核心问题**(每条都有真实模板数据支撑):

### 1.1 双总控台并存,场景驱动机制完全缺失

- **真实事实**:E1 底稿含**两个总控台** —— E1A(常规审计,25 项程序) + E26A(IPO/上市/新三板/重组/舞弊应对,11 项程序)
- 普通项目根本用不到 E26A 及其下属的 E1-26~E1-32 共 **8 个 sheet**,但全部塞在底稿里需要审计助理手工跳过
- E1A 程序分类列**只有 3 档**(常规★/备选/IPO 上市新三板重组舞弊应对),且实测没驱动后续 sheet 显示

### 1.2 prefill_formula_mapping 与 Univer 内部 193 公式职责未分清

- **真实事实**:E1-1 审定表本身**内置 193 个公式**(实测精确分类:54 cross_sheet + 95 加减 + 26 IF + 12 SUM + 6 ref_dir),Univer 加载 xlsx 时自带,**无需后端 prefill**
- prefill_formula_mapping.json 的 E1 条目实测 **13 cell**(2 entry:审定表 11 + 分析程序 2)是**额外补给**,作用是让 Univer 内部公式有取数源
- 当前痛点:
  1. 用户混淆"prefill 13 cell"vs"Univer 内部 193 公式"两套机制
  2. **prefill 只覆盖 E1-1 自身,完全没覆盖明细表 E1-2/E1-3/E1-4**(数据链路真正缺口)
  3. 第二条 prefill 的 sheet 名 `分析程序E1-3` 不是真实 sheet 名(真实是 `银行存款及其他货币资金明细表(仅人民币)E1-3` 等),需清理

### 1.3 总控台 E1A 的 25 项程序完成情况不可见

- **真实事实**:E1A 表 R17-R44 共 **25 项程序**(15 项常规★ + 8 项备选 + 2 项 IPO 应对),分 7 段:本账户科目/审计目标/总体获取/IPE/期初/库存现金/银行存款其他/其他
- 每项有"程序分类"+"5 项财务报表认定 √"+"底稿索引号"列
- 当前 E1A 是**纯静态模板**,审计助理填完 E1-1 不会自动在 E1A 标记完成;合伙人复核时看不到"还有几项程序未做"

### 1.4 E1-1 审定表对称双区结构未利用

- **真实事实**:E1-1 实际是 **47 行 × 10 列**,含**两套对称区**:
  - 上区(R7-R21,15 行):"项目名称(外币及人民币)" - 含本金/外币换算/合计 三列对照
  - 下区(R23-R39,17 行):"项目名称(仅人民币)" - 单一币种简化版
- 当前痛点:无外币业务的项目仍然加载完整双区,徒增审阅负担

### 1.5 历史遗留 sheet "F1-6 修订前" 来源已查实

- **真实事实**:`货币资金分析表F1-6 (修订前)`实际存在于 **`E1-14至E1-15 货币资金 -分析程序(Leap 应对措施-分析程序).xlsx`** 模板原文件中
- **不是** chain_orchestrator 跨 cycle 合并 bug(之前 v1 README 误判)
- 是致同模板原作者保留的历史遗留 sheet,所有项目都会带上,需在 chain 生成时按 sheet 名前缀过滤

### 1.6 跨底稿引用提示缺失但 E1A 已显式列出

- **真实事实**:E1A R38(序号 14)和 R44(序号 4)明确写"底稿索引号 = A1-1/A1-15/A1-16",但编辑器内没有跳转链接
- E1A R29(银行函证)指向 E0,R34(回单抽样)指向 E1-23,R23/R43 指向 E26A —— 全部需要 cross_wp_references.json 补条目

### 1.7 与 B/C 类底稿的循环联动完全缺失

- **真实事实**:E1 货币资金实质性程序**不是孤立执行**,它与 B 类(风险评估)和 C 类(控制测试)有**强前后依赖**:

| 底稿 | 类型 | 与 E1 的联动关系 | 当前状态 |
|------|------|----------------|---------|
| **B23-2** 了解评价并测试货币资金循环业务层面控制 | B 风险评估 | E1A 程序执行的**前置条件**(先了解控制→再做实质性程序)| ❌ 无联动 |
| **B23-2-2** 货币资金循环流程图及描述(.docx)| B 风险评估 | 描述货币资金业务流程,E1A 裁剪依据(流程复杂度→程序范围)| ❌ 无联动 |
| **B51-3** 识别对货币资金需保持警觉的情形 | B 风险评估 | 舞弊风险评估结论→**触发 E26A IPO 应对程序**(B51-3 标"高"→E26A 必做)| ❌ 无联动 |
| **C3** 货币资金循环业务层面控制测试 | C 控制测试 | 控制测试结论→**影响 E1 实质性程序范围**(控制有效→减少抽样量/截止测试天数)| ❌ 无联动 |
| **C3-2** 货币资金循环评价控制偏差 | C 控制测试 | 控制偏差→**增加 E1 实质性程序**(偏差严重→加大截止测试范围/增加函证比例)| ❌ 无联动 |

- **审计逻辑链路**:B23-2(了解控制)→ C3(测试控制)→ C3-2(评价偏差)→ E1A(裁剪实质性程序范围)→ E1-1~E1-32(执行)
- **当前痛点**:
  1. 审计助理打开 E1 时看不到"B23-2 控制了解是否已完成"的前置状态
  2. C3 控制测试结论(有效/无效)没有自动影响 E1A 程序范围(如"控制无效→截止测试从 ±5 天扩大到 ±10 天")
  3. B51-3 舞弊风险评估"高"没有自动触发 E26A 加载(当前靠 scenario 字段手动设)
  4. 这 5 个 B/C 类底稿与 E1 之间没有 cross_wp_references 条目,LinkageBus 依赖图中 E1 是孤岛

### 1.8 货币资金完整生态圈(24 个相关底稿)未统一管理

- **真实事实**:全模板库中与货币资金/现金/银行/函证相关的底稿共 **24 个文件**,分布在 **7 个循环目录**:

| 循环 | 底稿 | 与 E1 的关系 |
|------|------|-------------|
| **E 货币资金**(核心)| E0 银行询证函(20 sheet) / E1 四文件(33 sheet)| E1A R29 指向 E0;E0 函证结果直接影响 E1-1 审定数 |
| **A 完成阶段** | A5-1 现金流量表审计(9 sheet) / A17-5-5 函证程序核对表(2 sheet)| A5-1 的"现金及现金等价物"= E1-1!R18 合计(CFS 勾稽);A17-5-5 是函证程序完成度总检 |
| **B 风险评估** | B23-2 货币资金循环控制了解 / B23-2-2 流程图 / B51-3 舞弊警觉 | 前置条件+裁剪依据+触发器(§1.7 已详述)|
| **C 控制测试** | C3 货币资金循环控制测试 / C3-2 评价控制偏差 | 范围驱动(§1.7 已详述)|
| **D 收入循环** | D0 收入循环函证 | 与 E0 共享函证管理框架(函证程序表结构相同);应收账款函证回函可能涉及银行账户信息 |
| **F/G/H/K/L 其他循环** | F0/G0/H0/K0/L0 各循环函证 | 与 E0 共享函证管理框架;银行函证回函可能包含这些循环的担保/质押信息 |
| **S 特定项目** | S33-8 现金收付交易 / S34-15 现金交易 / S34-37 现金流异常 / S35-3 现金分红核查 / S4 非货币性资产交换 | IPO/上市场景下与 E26A 联动(现金交易分析/现金流异常检查)|

- **当前痛点**:
  1. **E0 银行询证函是 E1 的核心配套底稿**(E1A R29 明确指向),但两者之间没有数据联动——E0 函证回函确认的银行余额应自动回填到 E1-3 银行存款明细表
  2. **A5-1 现金流量表审计**的"列示于现金流量表的现金及现金等价物"(sheet A5-1-1)应与 E1-1!R18 合计数勾稽,当前完全独立
  3. **A17-5-5 函证程序核对表**是合伙人复核函证完成度的总检表,应联动 E0+D0+F0+G0+H0+K0+L0 全部 7 个循环函证的完成状态
  4. **S 类 IPO 底稿**(S33-8/S34-15/S34-37)与 E26A 的现金交易分析(E1-26)内容高度重叠,但没有互引避免重复工作
  5. 6 个循环函证(D0/F0/G0/H0/K0/L0)与 E0 共享相同的函证管理框架(程序表/结果汇总/跟函控制/舞弊评价),但各自独立无法统一查看"全项目函证完成率"

## 二、E1 底稿真实结构(4 文件 / 33 sheet 完整核验)

### 2.0 按内容特征的组件选型(裁剪后实际使用视角)

#### 2.0.0 设计原则:保持模板专业内容 + 强化风险导向逻辑

**铁律**:组件选型分流**不能丢掉**致同模板的专业内容结构。每个 sheet 的审计目标、认定、程序文本、结论格式必须原样保留,只是用更适合的 UI 组件呈现。

**D-N 类实质性测试底稿通用架构**(E1 不是特例):

致同 D-N 全部 89 个实质性测试底稿都遵循**相同的"程序表总控台"逻辑**:

| 底稿 | 总控台 sheet | 程序数 | 结构 |
|------|------------|-------|------|
| D0 收入循环函证 | 函证程序表D0A | ~10 项 | 与 E0A 同款 |
| D2 应收账款 | 应收账款实质性程序表D2A | ~20 项 | 与 E1A 同款 |
| F2 存货 | 存货实质性程序表F2A | ~30 项 | 与 E1A 同款(最复杂)|
| H1 固定资产 | 固定资产实质性程序表H1A | ~25 项 | 与 E1A 同款 |
| ... | ...A | ... | 全部同款结构 |

**通用结构**(每个 XxA 总控台都有):
- R1-R4:表头(致同 logo / 底稿名称 / 项目信息 / 索引号+页次)← **自动填充**
- R5-R13:审计目标与认定对应关系(5 项认定 ABCDE)
- R14:章节标题"三、计划实施的实质性程序"
- R15-R16:列标题(序号/LEAP 程序/审计程序/程序分类/认定/底稿索引号)
- R17+:程序行(常规★/备选/IPO 应对 三档分类)

**因此本 spec 的 E1A 总控台设计必须是通用方案**,未来可直接推广到 D-N 全部 89 个底稿(O1 独立 spec),不能做成 E1 专属逻辑。

**数据刷新 + 底稿表头自动填充(通用需求)**:

每个底稿的 R1-R4 表头区域都有相同的自动填充需求:

| 表头 cell | 内容 | 数据来源 | 填充时机 |
|----------|------|---------|---------|
| R1 | "致同会计师事务所" | 固定文本(模板自带)| 不需要填充 |
| R2 | 底稿名称(如"货币资金实质性程序表")| 固定文本(模板自带)| 不需要填充 |
| R3 左 | 被审计单位名称 | `Project.company_name` | 底稿生成时 + 项目信息变更时刷新 |
| R3 中 | 审计期间(如"2025 年度")| `Project.year` | 底稿生成时 |
| R3 右 | 索引号(如"E1A")| `wp_index.wp_code` + sheet 后缀 | 底稿生成时 |
| R4 左 | 编制人/日期 | `current_user.name` + 当前日期 | 首次打开时自动填(可修改)|
| R4 中 | 复核人/日期 | L1 复核人 + 复核日期 | L1 复核通过时自动填 |
| R4 右 | 页次 | 自动计算(当前 sheet 在底稿中的序号)| 底稿生成时 |

**数据刷新机制**(一键刷新按钮已在 WorkpaperEditor 工具栏):

| 刷新场景 | 触发方式 | 刷新内容 |
|---------|---------|---------|
| 试算表数据变更 | eventBus `trial-balance:updated` + 手动"🔄 刷新取数" | E1-1 的 prefill 13 cell 重新从 TB 取数 → Univer 内部 193 公式自动重算 |
| 调整分录变更 | eventBus `adjustment:saved` | E1-1 的 AJE/RJE cell 重新取值 → 审定数自动重算 |
| 项目信息变更 | eventBus `project:updated` | 所有 sheet 的 R3-R4 表头区域重新填充(公司名/审计期间/编制人)|
| 函证回函 | eventBus `confirmation:received` | E1-3 银行存款明细表对应行标记"已函证" + E1-6 余额调节表更新 |
| 上年数据导入 | eventBus `prior-year:imported` | E1-1 的 PREV 公式重新取值(上年审定数)|
| 手动全量刷新 | 工具栏"🔄 刷新取数"按钮 | 以上全部 + 重新检测数据异常(变动率>30%高亮)|

**预设公式从四表直接提取 + 手动公式编辑**:

底稿中很多程序可以直接从入库的四表(tb_balance/tb_ledger/tb_aux_balance/tb_aux_ledger)提取数据,不需要审计助理手工填写:

> **⚠ 实施原则**:以下 10 种公式类型是**系统能力清单**(定义系统支持哪些取数方式)。
> 具体"哪个 cell 用哪个公式"**必须在实施阶段最后确认**——先逐 sheet 核验表样格式(表头列项目如"未审数/账项调整/审定数/变动额/变动率"等列名 + 行结构如"库存现金/银行存款/其他货币资金"等业务项目),
> 然后才能精确定义每个 cell 的公式映射。**不能脱离表样格式空谈公式设计**。
>
> 实施顺序:Sprint 1 先完成表样格式分析(openpyxl 提取每个 sheet 的列头+行结构)→ 输出"cell 级公式映射表"→ 再写入 prefill_formula_mapping.json。

| 公式类型 | 语法 | 数据来源 | 适用场景 |
|---------|------|---------|---------|
| `=TB('科目','列名')` | 单科目取数 | tb_balance | 审定表期初/期末/未审数 |
| `=TB_SUM('起始~结束','列名')` | 科目范围汇总 | tb_balance | 多科目合计(如 1001~1012)|
| `=LEDGER('科目','方向','期间')` | 序时账汇总 | tb_ledger | 本期借方/贷方发生额;按月汇总 |
| `=AUX('科目','维度类型','维度编码','列名')` | 辅助明细取数 | tb_aux_balance | 按银行账户/客户/部门细分取数 |
| `=LEDGER_DETAIL('科目','日期范围','金额条件')` | 序时账明细筛选 | tb_ledger | 截止测试(截止日±N天 + 金额>阈值)|
| `=ADJ('科目','类型')` | 调整分录净额 | adjustments | AJE/RJE 调整金额 |
| `=PREV('底稿','sheet','字段')` | 上年同底稿取数 | 上年 wp parsed_data | 上年审定数/上年结论 |
| `=WP('底稿','sheet','字段')` | 跨底稿取数 | 其他 wp parsed_data | 折旧分摊/薪酬分摊等跨循环引用 |
| `=NOTE('章节','字段')` | 附注取数 | disclosure_notes | 附注披露金额反查 |
| `=COUNT_LEDGER('科目','日期范围')` | 序时账笔数统计 | tb_ledger | 截止测试抽样基数(期间内交易笔数)|

**手动公式编辑支持**:
- 审计助理可在任何 cell 输入自定义公式(以 `=` 开头)
- 公式管理弹窗(FormulaManagerDialog 已有)支持:
  - 查看当前 sheet 所有预设公式 + 手动公式
  - 新增/修改/删除自定义公式
  - 公式语法校验(输入时实时提示错误)
  - 公式执行预览(输入后立即显示计算结果)
- 手动公式与预设公式**共存不冲突**:
  - 预设公式(来自 prefill_formula_mapping.json)标记为"系统预设"(蓝色背景)
  - 手动公式标记为"用户自定义"(绿色背景)
  - 用户可覆盖预设公式(覆盖后标记为"已修改",保留原始预设供恢复)
- 新增公式类型 `=LEDGER` / `=AUX` / `=LEDGER_DETAIL` / `=COUNT_LEDGER` / `=NOTE`:
  - 当前 prefill_engine 只支持 TB/TB_SUM/ADJ/PREV/WP 5 种
  - 需扩展支持从 tb_ledger/tb_aux_balance 直接取数(截止测试/辅助明细等场景)
  - `=LEDGER_DETAIL` 返回符合条件的明细行列表(用于 E1-21/E1-22 截止测试自动填充 el-table)

**每页底稿 LLM 辅助按钮("✨ AI 审计说明")**:

每个 sheet/弹窗的结论区域(如 E1-1 R40-R46 / E1-2 R24-R26 / 各检查表结论行)旁边增加 **"✨ AI 审计说明"按钮**:

| 功能 | 触发 | LLM 输入 | LLM 输出 | 用户操作 |
|------|------|---------|---------|---------|
| 审计说明生成 | 点击"✨ AI 审计说明" | 当前 sheet 全部数据(金额/变动率/异常项)+ 审计目标 + 认定 | 结构化审计说明草稿(含变动原因分析+异常说明+结论建议)| 确认/修改/拒绝 |
| 差异原因分析 | 点击"✨ 分析差异" | E1-1 变动率>30%的行 + 对应序时账明细(LEDGER_DETAIL)| "货币资金增长 45% 主要因:①12月收到预收款 XX 万;②新增定期存款 XX 万" | 确认后填入 R41 |

**统计**:
- 真正需要 Univer 的:**9 个 sheet**(A 类数据表格)
- 应改为弹窗/表单/el-table 的:**21 个 sheet**(B+C+D+E+F 类)
- 应隐藏的:**2 个 sheet**(G 类)
- 不可见(F4 裁剪):**8 个 sheet**(普通项目)→ 其中 E 类 7 个 + C 类 1 个(E26A)

**架构决策**:
- **Univer 只渲染 A 类 9 个 sheet**,其余用前端原生组件(el-form/el-table/弹窗)
- B/C/D/E 类 sheet 的数据存储仍在 `wp.parsed_data` JSONB 中(不依赖 xlsx 文件)
- 用户点击左侧 sheet 导航时:A 类切换 Univer sheet;B/C/D/E 类弹出对应弹窗/面板
- **全屏模式必须支持**(useFullscreen 已接入 3 视图):
  - A 类 Univer 区域:全屏后 Univer 占满视口(隐藏左侧导航+顶部工具栏),审计助理处理大表时必须
  - B/C/D/E 类弹窗:使用 el-dialog `fullscreen` 属性或 `useFullscreen` composable,避免弹窗内 el-table 行数多时挤压
  - 全屏切换按钮统一放在每个面板/弹窗右上角(与 TrialBalance/ReportView/Adjustments 已有的全屏按钮风格一致)
  - **全屏弹窗交互规则**:
    1. 全屏弹窗内**不保留**左侧 sheet 导航(全屏 = 聚焦单一任务,导航在退出全屏后恢复)
    2. 全屏弹窗底部 **sticky footer** 固定"保存"+"取消"按钮(避免长表格滚动后找不到保存)
    3. 从全屏弹窗返回(保存或取消)时,**自动刷新 Univer 数据**(如 D 类盘点数据影响 E1-2 明细表 → E1-1 合计)
    4. 全屏状态下 ESC 键退出全屏(不关闭弹窗),再次 ESC 关闭弹窗(两步退出防误操作)
- 这样做的好处:
  1. **裁剪更自然**:弹窗类 sheet 按 scenario 条件"不显示按钮"即可,无需操作 Univer sheet 列表
  2. **附件管理天然集成**:D 类盘点弹窗直接内嵌附件上传区域
  3. **程序状态联动**:C 类总控台用 el-table 渲染,每行加 el-tag 完成状态 + 点击跳转
  4. **截止测试自动化**:E 类从 ledger 模块自动抽样填充 el-table,审计助理只需逐笔核对标记
  5. **附注关联**:F 类直接嵌入 DisclosureEditor 的只读预览(已有组件),不重复渲染

**与附注关联的 sheet 处理**:
- 附注披露(上市公司) / 附注披露(国企) 两个 sheet 内容 = 附注模块 `disclosure_notes` 的子集
- 不应在 Univer 中独立编辑(会导致数据源分叉)
- 正确做法:只读预览面板 + "跳转附注编辑器"按钮(复用 cross_wp_ref E1→NOTE 链路)
- 按 `Project.template_type` 自动选择显示哪个版本(soe/listed),另一个完全不渲染
- **P4 核验结论**:E1-1 的 193 公式中**没有引用附注披露 sheet**(openpyxl grep 实测 0 命中),F 类可安全移出 Univer

### 4.3 公式分布(193 个实测精确分类)

| 公式类型 | **实测数** | 说明 | 样本 |
|---------|----------|------|------|
| arith_only(纯加减汇总)| **95** | =B+C / =G-D / 双区对称汇总 | D7=B7+C7;F7=G7-E7;H7=G7-D7 |
| cross_sheet(跨 sheet 引用明细表)| **54** | 从 E1-2/E1-3/E1-4 汇总到 E1-1 | B7='现金明细表E1-2'!B22;B9='E1-3'!H18 |
| if_only(IF 变动率)| **26** | I 列变动率防 0 除 | I7=IF(AND(D7=0,G7=0),0,IF(AND(D7=0,G7>0),1,H7/D7)) |
| other(SUM/SUMIF 等)| **12** | 应计利息 SUM 子项 | B13=SUM(B14:B17);C13=SUM(C14:C17) |
| ref_dir(引用底稿目录)| **6** | 项目名/索引号自动填 | A3=底稿目录!A2;D3=底稿目录!A4 |
| **合计** | **193** | 完全分类零剩余 | - |

**重要发现**:
- v2 README 旧版猜测"~80 跨 sheet"实际是 54;猜测"~25 加减"实际 95;模板**没有 SUMIF**(被识别为 other 类的实际是 SUM)
- E1-1 的核心数据流只有两类:54 个明细表引用拉数据 + 95 个加减自动汇总,合计占 77%
- 26 个 IF 变动率公式让 I 列(本期与上期变动率)在期初/期末为 0 时不报错

### 4.4 prefill 与 Univer 内部公式职责分离

**关键架构事实**:E1-1 数据填充靠**两套机制并存**:

| 机制 | 来源 | 覆盖范围 | 当前状态 |
|------|------|---------|---------|
| 1. Univer 内部公式(模板内置)| xlsx 文件本身的 193 个公式 | E1-1 ↔ E1-2/E1-3/E1-4/底稿目录 引用 + 加减/IF/SUM 自算 | ✅ 模板自带,Univer 加载即生效 |
| 2. prefill_formula_mapping.json E1 条目 | 后端 prefill_engine 写入 | **实测 13 cell**(分 2 sheet 2 entry)| ✅ 已存在但仅汇总级 |

**prefill 现状实测**(`backend/data/prefill_formula_mapping.json` 中 wp_code='E1' 条目):

```
entry 1: sheet='审定表E1-1' (11 cells)
  - 期初余额          → =TB_SUM('1001~1012','期初余额')
  - 未审数            → =TB_SUM('1001~1012','期末余额')
  - 库存现金_期初/未审数  → =TB('1001','期初/期末余额')
  - 银行存款_期初/未审数  → =TB('1002','期初/期末余额')
  - 其他货币资金_期初/未审数 → =TB('1012','期初/期末余额')
  - AJE 调整          → =ADJ('1001','aje_net')
  - RJE 调整          → =ADJ('1001','rje_net')
  - 上年审定数         → =PREV('E1','审定表E1-1','审定数')

entry 2: sheet='分析程序E1-3' (2 cells)
  - 上年审定数 / 本年未审数(从分析程序入口)
```

**prefill 真正的作用**:不是替代 193 个 Univer 内部公式,而是**给 E1-1 自身的语义行(库存现金/银行存款/其他货币资金/AJE/RJE/上年)提供 TB 取数源**;Univer 内部 193 公式负责**从 E1-2/E1-3/E1-4 明细表汇总**到 E1-1。

**当前真正缺口**:prefill 只对 E1-1 自身,**没有对 E1-2/E1-3/E1-4 三个明细表定义任何 prefill**;明细表填空 → Univer 内部 cross_sheet 公式取到空 → 数据链路断。

**注**:prefill_formula_mapping 第二条 `sheet='分析程序E1-3'` 实际不是真实 sheet 名(真实是"银行存款及其他货币资金明细表(仅人民币)E1-3" / "(人民币及外币)E1-3"),应在三件套实施时清理或修正。

| R1 prefill 公式覆盖 Univer 内部公式(尤其 B22=SUM 合计公式)| 数据错乱(合计行被 TB 取数覆盖,变成单一币种值)| prefill_engine 必须实现"不覆盖已有公式 cell"逻辑(TD-9);F2 实施时单测覆盖 cell 类型识别(空 cell vs 公式 cell vs 数值 cell)|

| R4 cross_wp_ref 跳转死循环(A1 → E1 → A1)| 体验问题 | LinkageBus 已防 BFS 循环,复用其检测 |

| R7 普通项目误标 IPO 后 F4 文件加载耗时陡增 | 用户体验下降 | scenario 切换前提示"将额外加载 8 个 sheet,预计 +N 秒";切换异步进行不阻塞 UI |

**核验方法论沉淀(v2.1 升级)**:
1. 任何 spec 起草涉及具体底稿内容时**必须 openpyxl 实加载 xlsx 提取**真实程序文本/列结构/公式样本
2. 不能仅依赖 metadata.sheets 字段推测(metadata 只有 sheet 名列表,没有行列内容)
3. 公式分类必须**实测精确统计**,不能凭"~80"估算(本次 cross_sheet 实测 54 vs 估算 80 差 48%)
4. prefill 目标 cell 必须**先核验该 cell 是否已有公式**(B22=SUM 合计公式被覆盖是 P0 级 bug)
5. 脚本模式 = openpyxl load_workbook + 遍历 R/C 写 UTF-8 dump(Windows 必须直接写文件不走 PowerShell 管道避开 GBK 编码陷阱)+ 用完即删
