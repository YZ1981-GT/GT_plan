## 一、问题背景

### 1.1 D2A 实测痛点

D2A「应收账款实质性程序表」用 Univer Sheets 渲染：

- **B 列长文本压缩**：20 条程序，每条 200~500 字描述被强行塞进 1 行单元格，截断/重叠无法阅读
- **D-H 五项认定列宽塞不下**：表头「存在/完整性/权利和义务/准确性、计价和分摊/列报」占两行，内容区每行只显 1 个 √，信息密度极低
- **没法过滤分类**：常规★/IPO 加项/备选/舞弊应对 4 类混杂，用户只想看本项目适用的程序，没有筛选
- **没法批量裁剪**：审计程序裁剪是核心决策，但 Univer 里只能逐行手动改单元格，没有"全选不适用 + 填理由"
- **决策状态不可见**：执行/裁剪/已完成 = 状态根本没在表里
- **关联底稿不可点**：I 列「D2-1/D2-2/D2-3」是文本不是链接
- **进度不可追**：谁负责、什么时候做、做到哪一步，全看不到

### 1.2 不仅是 D2A — 同款问题广泛存在（基于全量实测）

## 二、用户偏好与设计原则

### 2.1 用户已表达的偏好

- **想做一套通用组件**（不是 D2A 一处定制）
- **HTML 格式更适合**：表格 + 表单混合的结构化文档
- **必须支持一键导出 Excel**：完成填写后能导回客户期望的 xlsx 格式（合规归档要求）
- **不动 ProcedureTrimming**：项目已有独立程序裁剪页面（`backend/data/wp_account_mapping.json` 206 条 + 委派矩阵），D2A 中控台与之协同非替代
- **本次不实施，先收集需求**：希望先通过 md 文档汇总，再评估技术方案

### 2.2 设计原则候选

1. **数据 ≠ 渲染**：底稿数据存 JSON（Univer snapshot 已落 PG `working_paper.parsed_data`），渲染分两路（HTML 中控台 / Univer 表格）
2. **配置驱动**：每类底稿（程序表/目录/附注/检查表）提供 schema 描述渲染规则，新增底稿不改代码只加 schema
3. **导出 Excel 是 first-class 能力**：填写完成后必须能导出符合致同模板版式的 xlsx（已有 `python-docx` + LibreOffice 转换链路，xlsx 用 openpyxl 写更直接）
4. **决策可追踪**：每个决策（裁剪/执行/完成）落 DB 有审计轨迹（who/when/why）
5. **支持 Excel 模板更新**：致同每年修订程序表，模板变更时不破坏已填用户数据

### 3.0.2 真假底稿明确区分（347 假底稿不进渲染层）

> **关键概念**：模板 xlsx 里有"非底稿"sheet（不该渲染、不该编辑、不计入底稿数），design 阶段必须明确剔除

| 类型 | 数量 | 占比 | 处理 |
|------|------|------|------|
| **真底稿**（要渲染 + 编辑 + 归档 + 计入完成率） | 2148 | 82.6% | HTML 或 Univer 渲染 |
| **假底稿**（不渲染 / 静态 / 跳过 / 不归档） | 347 | 13.3% | 详见下表 b |
| **合并/母公司专属**（跳转专属模块） | 450 | 17.3% | 由现有合并模块处理 |
| **pending（全在合并模板）** | 7 | 0.3% | 不在本 spec |
| **合计** | 2602 | 100% | — |

> 注：真底稿 2148 包含合并/母公司 450（由专属模块负责）。**本 spec 实际渲染单体真底稿 = 1788（2148 - 450 母公司合并 + 不算合并的 pending）**

#### b. 假底稿 4 子类（明确剔除）

| 子类 | 数量 | 实例 | 处理铁律 |
|------|------|------|---------|
| **I 占位** | 243 | GT_Custom / Data / List / Sheet1 / 参数 / temp_ | 跳过渲染 + 导出 xlsx 时保留（Univer 兼容） + **不计入底稿完成率** |
| **H 辅助说明** | 81 | 修订说明 / 编制说明 / 填表说明 / 文号规则 / 参考-XX 准则 / 准则及应用指南 | 静态 markdown 展示 + **只读不可编辑** + **不计入底稿完成率** |
| **不归档标记** | ~50 | "选项清单列表（不归档）" / "示例" / "（不打印）" / "示例-评价控制偏差（不打印）" | 渲染但**导出时移除** + **不归档存储** |
| **删除 / 重复** | ~15 | "...-删除" / "...G2-9-3-删除" / "(2)" 后缀 / "首发业务解答二-删除" | **完全跳过** + 模板清理时物理删除 |

**判定规则**：
- 含 `不归档` / `不打印` / `示例` / `参考` / `temp_` / `Sheet\d+` / `GT_Custom` → 假底稿
- 末尾 `-删除` / `(2)` / `(3)` → 假底稿
- `修订说明` / `编制说明` / `填表说明` / `文号规则` / `准则` / `指南` / `规定` → 假底稿（H 辅助说明）

#### c. 模板提示框架（~50 sheet）— 介于真假之间

```
表头（请先填写）  × 16   ←  S 循环模板里"请用户先填头部信息"占位
提示              × 13   ←  S32-X 各专项核查的"风险提示语"
相关资源/相关资源2 × 4    ←  B 循环资源参考
准则/指南/规定    × ~10   ←  财政部/证监会原文引用
IPO 提示/解答     × ~5    ←  IPO 业务参考
2025 假期清单     × 1     ←  C 循环辅助
```

**处理铁律**：
- "表头（请先填写）" → 移到主底稿头部信息区（B 目录类已有），**单独 sheet 删除**
- "提示" 类长文本 → **嵌入对应主 sheet 顶部 collapse 区**（点击展开），**不单独成 sheet**
- "准则/指南/规定/IPO 解答" → 归 H 辅助说明（链接到原文 / 折叠展示）
- "2025 假期清单" → **跨项目共享数据**，应放系统级配置（不属于底稿）

#### d. 真假底稿在 implementation 的标记

`workpaper_sheet_classification` 表新增字段：

```sql
CREATE TABLE workpaper_sheet_classification (
  ...
  is_real_workpaper BOOLEAN NOT NULL DEFAULT TRUE,    -- 是否真底稿
  exclude_from_archive BOOLEAN NOT NULL DEFAULT FALSE, -- 不归档（标"不打印"）
  exclude_from_progress BOOLEAN NOT NULL DEFAULT FALSE, -- 不计入完成率
  is_static_doc BOOLEAN NOT NULL DEFAULT FALSE,       -- 静态文档（不可编辑）
  ...
);
```

**前端渲染规则**：
- `is_static_doc = TRUE` → 显示为只读 markdown / iframe 嵌入 PDF
- `is_real_workpaper = FALSE` AND `exclude_from_archive = TRUE` → 编辑器隐藏 + 导出时不写入归档 zip
- `exclude_from_progress = TRUE` → Dashboard 进度卡 / 委派矩阵不计入

**铁律**：底稿完成率 = 真底稿已完成数 / 真底稿总数（**不含假底稿**），避免完成率被占位 sheet 稀释失真

### 3.0.3 项目实例层级（L5）— 模板 vs 实例的差异

> **关键概念**：模板（template）是源头，项目内的底稿（instance）是模板的"裁剪 + 自定义"实例。两者数据结构相同但生命周期分离

#### a. 4 层数据模型

```
Layer 1：致同模板库（template repository）
  └── backend/wp_templates/{cycle}/*.xlsx  ← 349 xlsx 全集（致同总所发布）
  └── workpaper_template_registry 表       ← 模板元数据（版本/适用条件/sheet 列表）
  └── workpaper_sheet_classification 表    ← sheet 9 类归属（HTML/Univer/H/I + 真假底稿）

Layer 2：项目模板订阅（project subscription）
  └── 每个项目按 (project.template_type, report_scope, industry_type) 订阅相应模板子集
  └── chain_orchestrator 步骤 5b 已实现：按需启用 / 不适用裁剪
  └── 行业特定模板（H5 油气 / H7 生物资产 / M7 专项储备）按 industry 启用

Layer 3：项目级裁剪（ProcedureTrimming，已实现）
  └── ProcedureInstance 表 status 字段：not_started / in_progress / not_applicable / completed
  └── 用户在项目启动阶段决定哪些程序适用（项目级一次决策）
  └── chain_orchestrator 步骤 5b 跳过 status='not_applicable'

Layer 4：项目实例底稿（wp_index + working_paper）
  └── wp_index 表：项目内底稿索引（按 wp_code 全局唯一）
  └── working_paper 表：xlsx 文件落地 + parsed_data JSONB（含 univer_snapshot）
  └── 底稿状态机：draft / submitted / review_passed / archived
```

#### b. 项目实例与模板的差异类型

| 差异类型 | 说明 | 实例 |
|---------|------|------|
| **裁剪缺失**（项目无此底稿） | ProcedureTrimming 标记 not_applicable | 非 IPO 项目无 D4-22~D4-32 |
| **行业不启用** | industry_type 不匹配 | 非农业项目无 H7 生产性生物资产 |
| **维度差异** | template_type/report_scope 切版 | 国企版无"上市公司专属"sheet |
| **自定义新增**（用户上传） | 用户自定义模板 | 客户独有的特殊业务底稿 |
| **自定义修改**（用户改 sheet 内容） | 在模板基础上扩展行/列 | 增加自定义关联方 |
| **跨年度实例**（数据延续） | 同 wp_code 不同 audit_year | 2024 年 D2 vs 2025 年 D2 |

#### c. 项目级"自定义新增"机制（已有）

`ProcedureTrimming.vue` 已支持：
- **自定义新增程序**：弹窗表单（名称 + 编码 + Excel 模板上传）
- **创建路径**：`custom-with-template` 端点 → 创建 ProcedureInstance（is_custom=True）+ WpIndex 占位 + 文件上传
- **chain_orchestrator 步骤 5c**：扫描 is_custom=True 的程序加入项目底稿清单

#### d. 自定义底稿的归类挑战

| 场景 | 挑战 | 处理方案 |
|------|------|---------|
| 用户上传致同未发布的新程序 xlsx | 没有 sheet_classification 记录 | **运行时扫描 + 自动归类**（复用 `analyze_wp_templates.py` 的 60+ 条规则） |
| 用户改 sheet 名（如复制致同模板再改） | 归类规则失效 | **fallback 到内容特征启发式**（公式数 + 合并 + 长文本） |
| 极端 case：用户上传非审计 xlsx（如 Excel 数据透视表） | 归类规则全部不命中 | **降级到 Univer 兜底**（保留现状）+ 标记 `_pending` 通知运维人工归类 |
| 客户特殊业务底稿（如金融机构特色） | 不在标准 9 类中 | **支持"X-自定义"类别**：项目级新建归类，落 PG 表 |

#### e. 实例层级数据流

```
项目启动
  ↓
[L1 模板库] 致同 v2025 模板（349 xlsx）
  ↓
[L2 订阅] 按维度订阅（如国企单体）：扣除上市/合并 sheet → 280 xlsx 候选
  ↓
[L3 裁剪] ProcedureTrimming 项目级一次裁剪：用户标 not_applicable → 230 xlsx
  ↓
[L4 实例化] chain_orchestrator 步骤 5b/5c：
  ├─ 5b：copy 模板 xlsx 到 storage/projects/{pid}/workpapers/
  ├─ 5b：写 wp_index + working_paper 记录
  ├─ 5c：自定义程序追加 + 用户上传 xlsx 落地
  └─ 完成：项目实例底稿数 230 + 自定义 ~5 = 235
```

#### f. 实例底稿渲染策略

**继承 + 覆盖**模式：
- 默认按 `workpaper_sheet_classification` 中模板级归类渲染
- **覆盖点 1**：项目级"裁剪后实例"自动隐藏 not_applicable sheet（chain_orchestrator 已处理）
- **覆盖点 2**：用户自定义 xlsx 实例 → 项目级归类（落 `project_workpaper_sheet_override` 表）
- **覆盖点 3**：用户改 sheet 内容（如增加行/列） → Univer 兜底（HTML schema 不强行重新匹配）

**铁律**：
- HTML 渲染层 = 模板归类驱动 + 项目实例数据填充
- 同一 wp_code 在不同项目里**渲染策略一致**（来自模板归类）
- 实例数据差异（裁剪/自定义）由 chain_orchestrator + ProcedureTrimming 处理，**渲染层不感知**

### 3.0.4 模板版本管理（致同年度修订 + 项目绑定 + 数据迁移）

> **关键概念**：致同每年发布修订版（v2024 → v2025 → v2026），同一 wp_code 在不同版本里 sheet 数量 / 字段 / 公式可能差异；项目立项后绑定一个版本，跨年度审计需要数据迁移

#### a. 版本管理需求

| 场景 | 需求 |
|------|------|
| 致同发布 v2026 修订版 | 系统支持上传新版本 + 不破坏 v2025 老项目 |
| 项目立项时选择版本 | 默认绑定最新版，可手动选历史版 |
| 已立项项目跨年度延续 | 用户决定"沿用旧版" vs "升级新版数据迁移" |
| 同一 wp_code 跨版本字段差异 | 自动检测差异 + 用户确认迁移规则 |
| 模板归类规则跨版本变化 | sheet 名变更 → 重新跑 `analyze_wp_templates.py` |

#### b. 数据模型扩展

```sql
-- 模板版本表
CREATE TABLE workpaper_template_version (
  id UUID PRIMARY KEY,
  version VARCHAR(20) NOT NULL,             -- 'v2025-R5' / 'v2026-R1'
  release_date DATE NOT NULL,
  source VARCHAR(50) NOT NULL,              -- '致同总所' / '客户定制'
  is_current BOOLEAN NOT NULL DEFAULT FALSE,
  parent_version_id UUID REFERENCES workpaper_template_version(id),  -- 升级关系
  changelog TEXT,                            -- 版本变更说明
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 项目 ↔ 模板版本绑定
ALTER TABLE projects ADD COLUMN template_version_id UUID REFERENCES workpaper_template_version(id);

-- sheet 归类按版本独立（同一 sheet 在不同版本可能不同分类）
ALTER TABLE workpaper_sheet_classification ADD COLUMN template_version_id UUID REFERENCES workpaper_template_version(id);

-- 跨版本 sheet 映射（用于数据迁移）
CREATE TABLE workpaper_sheet_version_mapping (
  id UUID PRIMARY KEY,
  from_version_id UUID NOT NULL,
  to_version_id UUID NOT NULL,
  from_wp_code VARCHAR(50) NOT NULL,
  from_sheet_name VARCHAR(255) NOT NULL,
  to_wp_code VARCHAR(50),                    -- NULL = 新版本删除此 sheet
  to_sheet_name VARCHAR(255),                -- NULL = 新版本删除此 sheet
  field_mapping JSONB,                       -- 字段级映射 {"old_cell": "new_cell"}
  migration_strategy VARCHAR(50),            -- 'auto_map' / 'user_confirm' / 'fresh_start'
  notes TEXT
);
```

#### c. 模板上传 + 自动归类流程

```
用户上传 v2026 模板 xlsx (致同新发版本)
  ↓
[1] 解压到 backend/wp_templates_v2026/
  ↓
[2] 跑 analyze_wp_templates.py --version v2026-R1
  ├─ 60+ 条规则自动归类
  ├─ 与 v2025 对比 diff（sheet 增删改）
  └─ 输出 v2026 版 workpaper_template_analysis.json
  ↓
[3] 中/低置信度走人工审核
  ↓
[4] 落 PG workpaper_sheet_classification (template_version_id=v2026)
  ↓
[5] 自动生成 workpaper_sheet_version_mapping (v2025 → v2026)
  ├─ sheet 名相同 → auto_map
  ├─ sheet 名变更但 wp_code 相同 → user_confirm
  └─ sheet 删除 → fresh_start
  ↓
[6] 通知所有正在进行的项目 "新版本已发布"
  ├─ 新立项项目默认用 v2026
  └─ 老项目可选"沿用 v2025"或"升级 v2026 数据迁移"
```

#### d. 项目升级数据迁移策略

**3 种迁移策略**：

| 策略 | 适用场景 | 处理 |
|------|---------|------|
| **auto_map** | 字段名 / cell 位置完全一致 | 直接 copy parsed_data 到新 sheet |
| **user_confirm** | 字段重命名 / cell 位移 | 弹窗显示新旧字段映射 + 用户确认每条 |
| **fresh_start** | sheet 已删除 / 结构剧变 | 清空数据 + 提示用户重新填写 |

**用户体验流程**：

```
项目 A（v2025-R5）→ 用户点"升级到 v2026-R1"
  ↓
[1] 系统计算迁移影响：
    - 234 个底稿可 auto_map（无需用户操作）
    - 5 个底稿需 user_confirm（弹窗确认）
    - 1 个底稿需 fresh_start（提示用户重填）
  ↓
[2] 用户预览影响 + 确认/取消
  ↓
[3] 后台执行迁移（事务性）：
    - 备份 v2025 数据到 archive/{project_id}/v2025-snapshot.json
    - 复制新版 xlsx 模板到项目 storage
    - 按 mapping 迁移 parsed_data
    - 更新 wp_index + project.template_version_id
  ↓
[4] 完成 + 显示变更摘要
```

#### e. 跨年度审计数据延续

```
2024 年项目（v2025-R5 模板）已签发归档
  ↓
2025 年新立项（同一被审计单位）
  ↓
[1] 系统自动检测同一 client_name + 上一年度项目
  ↓
[2] 弹窗询问："是否沿用上年度审定数 + 复用底稿框架？"
  ├─ 是 → 自动 carry forward：
  │     - 期初余额 = 上年期末审定数（trial_balance.audited_amount）
  │     - 关联方清单 / 银行账户清单 / 业务模式判定 等延续性数据自动填充
  │     - 上年函证模板复用（仅修改"截止日"等字段）
  └─ 否 → 全新空白底稿
  ↓
[3] 模板版本选择：
  ├─ 默认升级到当前最新版（如 v2026-R1）
  └─ 可选"沿用 v2025-R5"（保持版式一致便于跨年比对）
```

#### f. 版本管理的边界 / 不在范围

- **不做模板自由编辑**：致同模板版本由总所统一发布，不允许项目级自由改模板（防止合规风险）
- **不做版本回退**：升级后不支持回滚到旧版（备份在 archive/ 但不可复原项目）
- **不做版本分叉**：每个项目仅绑定一个版本，不允许混合（避免数据一致性问题）

#### g. 实施优先级（在本 spec 内）

| 优先级 | 功能 | 实施时机 |
|--------|------|---------|
| P0 | 当前版本（v2025-R5）的 schema 落库 | 本 spec implementation 第 1 周 |
| P1 | 多版本数据模型（workpaper_template_version 表） | 本 spec implementation 第 4 周 |
| P2 | 上传新版本 + 自动归类 | 后置（独立 spec） |
| P3 | 跨版本数据迁移 | 后置（独立 spec） |
| P4 | 跨年度审计数据延续 | 后置（独立 spec） |

**铁律**：本 spec 只做 P0 + P1（基础数据模型），P2~P4 后置避免范围爆炸

    │   含"实质性程序表" 或 末尾"[A-S]\d+A" 或 含"程序表"
    │   → A 类 HTML 中控台
    │
    ├─ ④ B 底稿目录
    │   含"底稿目录" 或 末尾"目录"
    │   → B 类 HTML 表单
    │

    │
    ├─ ⑨ F 数据型表格
    │   含"审定表"/"明细表"/"分析表"/"调整分录"/"汇总"
    │   → F 类 Univer
    │
    ├─ ⑩ "其他" 子区按所属主底稿前缀归类（基于实测）
    │   ├─ D4-* 子区 → 多数归 F3 分析表
    │   ├─ F2-* 子区 → 多数归 F2 明细表

    │   ├─ 含"替代程序" → A 类程序表
    │   ├─ 含"风险评估"/"舞弊"/"特别风险" → D 类问答型
    │   ├─ 含"复核"/"签发"/"项目负责经理复核"/"合伙人复核" → D7 复核记录表

#### 3.11.10 索引号边缘 case + 解析规则

**特殊情况处理**：

| Case | 处理规则 | 例子 |
|------|---------|------|
| **中文索引号** | 中文小节按 `Note:` 前缀解析 | `Note:五、(1)货币资金` → disclosure_notes section |
| **带空格** | trim 后按规则匹配 | `D2-1 ` / ` D2-1` 都匹配 D2-1 |
| **大小写** | 不区分大小写但归一化为大写 | `d2-1` → `D2-1` |
| **多目标** | 一个 chip 可对应多个目标，hover 弹菜单选 | `D2-1/D2-2/D2-3` chip 展开 3 选项 |
| **不存在** | 灰显 + tooltip "底稿不存在或被裁剪" | 模板有但项目未生成此 sheet |
| **被裁剪** | 灰显 + tooltip "已裁剪：理由" | A 程序表中标记为不适用的程序 |
| **跨项目** | 禁止跳转，提示"仅同项目内联动" | 防止数据隔离泄漏 |
| **GT_Custom** | 不可跳转 | 占位 sheet 无实际内容 |
| **空 sheet** | 跳转但提示"sheet 为空，请先填写" | 用户尚未编辑 |

**解析正则**（前端工具函数 `parseIndexRef`）：

```typescript
// 严格模式
const STRICT_RE = /^(Note|TB|Adj|Att|EQCR|wp|sheet|cell):(.+)$/
// 宽松模式（识别文本中的底稿编码）
const LOOSE_RE = /\b([A-S]\d+(?:-\d+)*[A-Z]?)\b/g

function parseIndexRef(s: string): IndexRef | null {
  const trimmed = s.trim().replace(/^([a-z])/i, (m) => m.toUpperCase())
  // 先尝试命名空间格式
  const ns = trimmed.match(STRICT_RE)
  if (ns) return { ns: ns[1], target: ns[2] }
  // 再尝试底稿编码
  const wp = trimmed.match(LOOSE_RE)
  if (wp) return { ns: 'wp', target: wp[0] }
  return null
}
```

**前端实现 `<GtIndexChip>` 组件**：
- props: `value: string`（索引号文本） / `validate?: boolean`（默认 true）
- emit: `@click(target)` 携带解析结果
- 自动调 `wp_index` 表校验目标存在性
- hover 显示目标底稿状态卡（编制人/状态/最后修改）

## 四、技术方案候选（待 design 阶段确认）

### 4.1 方案 A：纯 HTML + 后端 openpyxl 导出

- 前端：el-table/el-form/自定义组件渲染
- 数据：存 PG JSONB（沿用 working_paper.parsed_data 结构）
- 导出：后端调 openpyxl 按 schema 把 JSONB 映射回 xlsx
- 优点：前端轻、互动好、移动友好
- 缺点：需维护 schema → xlsx 映射规则；用户已熟悉 Excel 操作（如复制粘贴大量数据）会有学习成本

### 4.2 方案 B：HTML + Univer 双视图切换

- 同一份数据，提供两种视图：HTML（决策/填表）+ Univer（编辑/计算）
- 用户按需切换，导出走 Univer 已有 xlsx 链路
- 优点：兼容现有用户习惯
- 缺点：双轨维护成本高、状态同步复杂

### 4.3 方案 C：仅 HTML + 一键转 xlsx（推荐方向）

- 前端只做 HTML 渲染
- 导出 = 后端 openpyxl 加载致同模板 xlsx → 按 JSONB 数据填值 → 输出
- 模板更新 = 替换模板 xlsx 文件，schema 映射不变
- 优点：模板版式不破坏（致同可直接上传新版 xlsx）；前端统一；导出体积小
- 缺点：需为每个底稿类型写一份 schema + 写入规则

### 4.3.1 方案 C 的导出 xlsx 还原版式技术约束

为保证"HTML 编辑 → 导出 xlsx 与原致同模板版式 1:1 还原"，必须满足：

#### a. 模板 xlsx 占位符约定

每个致同模板 xlsx 提供 schema 描述：

```yaml
template: D2A.xlsx
sheets:
  应收账款实质性程序表D2A:
    fixed_cells:        # 固定头部信息（自动填充）
      A1: '致同会计师事务所'
      H3: '${index_no}'         # 索引号
      H4: '${page_no}'          # 页次
      A5: '${entity_name}'      # 被审计单位
    dynamic_table:      # 程序清单（用户填写）
      start_row: 17
      end_row: dynamic   # 用户决定有多少行
      columns:
        A: program_no              # 序号
        B: program_desc            # 程序描述
        C: program_category        # 程序分类
        D-H: assertions            # 5 项认定打勾
        I: linked_workpapers       # 底稿索引号
        J: trim_reason             # 裁剪说明
    static_text:        # 静态说明文字（不可改）
      rows: [5-13, 14-16, end+]
    formulas:           # 保留模板中的公式
      preserve: true
```

#### b. 公式列保留规则

- HTML 计算的值（如审定数 = 未审 + 调整）写入 xlsx 时**保留原模板公式**，不直接写值
- 例：xlsx 模板第 K10 是 `=H10+I10+J10`，HTML 渲染时用同样公式计算显示，导出时保留 `=H10+I10+J10`
- 用户在 HTML 中修改 H10/I10/J10，导出后 xlsx 打开时 K10 自动重算
- **零差异保证**：HTML 显示值 ≡ xlsx 公式重算值 ≡ 致同模板原始公式（三处口径一致）

#### c. 合并单元格保留规则

- 致同模板大量合并单元格（国企附注 355 行有 187 处合并），导出时**完整保留 merged_ranges**
- HTML 渲染时把合并区显示为单一 cell（rowspan/colspan）
- 用户在合并区填写 → 导出时写入合并区"主单元格"（左上角）

#### d. 国企/上市版本切换

- 模板按 `(template_type, report_scope)` 选择对应 xlsx 文件：
  - `(soe, standalone)` → 国企单体模板
  - `(soe, consolidated)` → 国企合并模板
  - `(listed, standalone)` → 上市单体模板
  - `(listed, consolidated)` → 上市合并模板
- 用户切换版本 = 切换模板 + 数据自动迁移（保留共有字段，差异字段提示）
- ✅ 已有 `applicable_standard` 派生（commit da36de1）

#### e. 致同模板年度更新

- 每年致同总所发布修订版（如 2025-R5 → 2026-R1）
- 升级流程：
  1. 上传新版模板 xlsx 到 `backend/data/wp_templates/{year}/{wp_code}.xlsx`
  2. 跑 `analyze_wp_templates.py` + `generate_template_analysis_md.py`
  3. 中/低置信度走人工审核
  4. 落 PG `workpaper_template_version` 表（含旧版兼容）
  5. 已填项目可选"沿用旧版" / "升级新版数据迁移"

#### f. 字符级 diff 校验

- 测试要求：导出 xlsx 与"用户在原致同模板上手填"的 xlsx 字符级 diff = 0（除用户填写位置）
- 自动化测试：`pytest tests/exports/test_d2a_xlsx_diff.py` 跑 9 个循环各 1 个底稿样本
- 失败示例：模板 cell B5 是字符串 "审计目标"，导出后变成 None 或多空格，diff 失败

#### g. 复合写入策略

- 同一 cell 在不同场景下写入策略不同：
  - **fixed_cells**（头部 + 静态文字）：导出时强制写入 schema 默认值，覆盖用户改动
  - **dynamic_table**（数据行）：按 schema 字段映射写入
  - **formula_cells**（公式 cell）：保留公式，不写值
  - **note_cells**（"勿删勿改"区 + 财政部引用）：原样保留

### 4.4 关键技术依赖

- 已有：openpyxl（read_only 经验）/ univer_snapshot_helper.py（slim JSON 数据结构）/ python-docx（程序化生成附注/EQCR）
- 缺口：模板 xlsx → schema 自动提取工具 / 用户数据 → schema 双向映射 / HTML 组件库扩展（程序表中控/嵌套表格/检查表）

## 五、影响面评估

### 5.1 数据迁移

- 现有 working_paper.parsed_data 存的是 Univer IWorkbookData 格式（行/列 cell 索引）
- HTML 渲染需要的是结构化数据（programs[].id/desc/category/assertions/status...）
- 需要**双向转换器**：Univer JSON ⇄ HTML schema JSON

### 5.2 已存在模块协同

- **ProcedureTrimming**：程序裁剪页面已存在，处理项目级"哪些程序保留/不适用"。D2A 中控台是底稿级（项目内每个 D2A 都可能有差异），关系 = 一对多继承（项目级裁剪 → 底稿级再细化）
- **disclosure_notes 模块**：附注披露已有独立模块（`backend/data/consol_note_sections_{soe,listed}.json` 91 大类）。D2 底稿里的附注披露信息 sheet 应该剥离出来还是双写？需确认
- **chain_orchestrator 步骤 5b**：已尊重 ProcedureInstance.status='not_applicable' 跳过。D2A 中控台决策需写回 ProcedureInstance 表

### 5.3 测试要求

- 一键导出 xlsx 必须**字符级 diff** 与原致同模板 = 0 差异（除用户填写位置）
- HTML 填写 → 导出 xlsx → 重新导入 = 数据无丢失
- 上市/国企版本切换 = 用户数据保留
- 模板年度修订（致同每年改）= 用户已填数据自动迁移

### 5.4 性能要求

- D2 底稿 18 sheet，HTML 渲染 < 500ms（Univer 现在 ~3s）
- 一键导出 xlsx < 5s（含模板加载 + 写入）
- 6000 并发场景下导出走异步队列 + 信号量
