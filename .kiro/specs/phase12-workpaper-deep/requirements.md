# Phase 12: 底稿深度开发 - 需求文档

> 基于人机协同理念的审计底稿系统深度开发，实现AI草拟→人工确认→AI优化→人工定稿的闭环。

---

## 1. 背景与目标

### 1.1 现状
底稿系统核心能力（模板生成、WOPI编辑、QC自检、复核批注）已做实，但存在以下关键缺口：
- **审计说明纯手工编写**：每张底稿200-500字，占编制时间50%+
- **AI能力未激活**：LLM调用无prompt工程，返回质量不稳定
- **角色视图无差异**：四种角色看到相同界面，无法聚焦各自关注
- **数据一致性无保障**：底稿审定数与试算表差异无法及时发现

### 1.2 核心目标

| 目标 | 量化指标 | 实现方式 |
|------|---------|---------|
| 审计助理效率提升 | 单张底稿编制时间 -40% | AI生成审计说明草稿 |
| 项目经理复核效率提升 | 单张底稿复核时间 -30% | AI预审+复核工作台 |
| 数据一致性保障 | 底稿-试算表差异率 <1% | 自动校验+冲突提示 |
| 质量可控 | QC通过率从 85% → 95% | 内容级QC规则+AI辅助 |

### 1.3 本期交付策略

| 层级 | 范围 | 目标 |
|------|------|------|
| MVP（本期必须上线） | P0-1~P0-4、P1-1、P1-2、P1-4、P1-6、P1-7 | 形成底稿编制→复核→签字前检查闭环 |
| 增强（MVP稳定后插入） | P1-3、P1-5、P2-3、P2-5、P2-6 | 提升管理效率与编辑体验 |
| 延后（下一子阶段） | P2-1、P2-2、P2-4，以及离线工作包、模板热更新、归档导出、推荐规则自优化 | 不阻塞本期上线，待核心闭环稳定后迭代 |

> 本文保留完整需求池，但默认排期与上线验收以MVP为准；增强项需在MVP稳定后单独确认资源。

## 2. 需求分层

### 2.1 P0 — 阻塞问题（立即修复）

| 序号 | 需求 | 用户故事 | 验收标准 |
|------|------|---------|---------|
| P0-1 | prefill_stale前端通知 | 作为审计助理，当底稿数据过期时，我希望在多处看到明显提示，以便及时刷新 | 底稿列表橙色⚠图标、编辑器顶部黄色横幅、工作台树节点标记 |
| P0-2 | WOPI只读浏览模式 | 作为项目经理，我想在编制人编辑时也能查看底稿，而不是被锁挡住 | check_file_info返回ReadOnly=True，复核人只读模式打开 |
| P0-3 | 审计说明提取 | 作为复核人，我想在底稿列表看到审计说明摘要，而不是打开Excel | parse_workpaper_real增加说明提取（关键词搜索+连续文本） |
| P0-4 | 底稿列表角色裁剪 | 作为审计助理，我只想看到自己负责的循环底稿，而不是全部300+张 | 前端配合scope_cycles后端过滤，只显示负责循环 |

### 2.2 P1 — 核心需求池（按本期交付策略分批落地）

 > 说明：P1-1、P1-2、P1-4、P1-6、P1-7 纳入本期MVP；P1-3、P1-5 作为增强项保留，在MVP稳定后再插入。

| 序号 | 需求 | 用户故事 | 验收标准 |
|------|------|---------|---------|
| P1-1 | 审计说明智能生成 | 作为审计助理，我希望AI根据试算表数据自动草拟审计说明，我只需修改确认 | 数据采集→Prompt构建→流式输出→人工编辑→写回底稿→刷新parsed_data；首字节<3秒，完整流程<10秒 |
| P1-2 | 复核工作台 | 作为项目经理，我希望在一个界面连续复核多张底稿，有AI预审辅助 | 待复核队列+只读预览+AI预审+批量通过/退回，快捷键支持 |
| P1-3 | 进度总览面板 | 作为项目经理，我想一眼看清全项目底稿完成情况 | 进度矩阵+指标卡片+甘特图+人员负荷视图 |
| P1-4 | 数据一致性监控 | 作为项目经理，我想及时发现底稿与试算表数据不一致 | 矩阵对比+自动校验+重新预填/保留人工差异说明，误差>0.01元即标红 |
| P1-5 | 批量操作面板 | 作为项目经理，我想一次性分配/刷新/生成说明多张底稿 | checkbox选择+5个批量API（分配/刷新/生成说明/提交复核/下载）+job_id+进度订阅 |
| P1-6 | 内容级QC规则 | 作为质控人员，我想自动检查审计说明完整性和数据一致性 | 新增4条规则：QC-15说明完整性、QC-16数据一致性、QC-17附件充分性、QC-18交叉引用完整性；规则准确率>95%，误报率<5% |
| P1-7 | 签字前底稿专项检查 | 作为合伙人，签字前必须确认所有底稿已完成复核 | 5项检查清单：复核状态/QC通过/说明非空/数据一致/证据充分；检查耗时<10秒（500张底稿） |

### 2.3 P2 — 体验优化（默认延后）

| 序号 | 需求 | 用户故事 | 验收标准 |
|------|------|---------|---------|
| P2-1 | 风险底稿聚焦 | 作为合伙人，我只想关注高风险科目底稿 | 自动筛选：金额>重要性/有调整/QC阻断/被退回过；筛选响应<2秒 |
| P2-2 | 独立抽查工作台 | 作为质控人员，我想独立于项目团队抽查底稿质量 | 智能抽样+检查清单+抽查报告生成；抽样覆盖率>80%，高风险科目100%覆盖 |
| P2-3 | 证据链可视化 | 作为审计助理，我想清楚看到哪些附件关联了当前底稿 | 关联列表+快速关联+证据图谱；附件关联加载<1秒，图谱渲染<3秒 |
| P2-4 | 底稿推荐反馈闭环 | 作为系统，我希望学习用户行为优化推荐质量 | 采纳率统计+推荐规则优化+反馈端点；采纳率计算准确率100%，规则优化触发条件：采纳率<30%降权 |
| P2-5 | 底稿编制智能引导 | 作为审计助理，我不确定某张底稿该执行哪些程序 | 根据底稿类型（审定表/明细表/分析表）显示不同引导；底稿类型识别准确率>95%，引导覆盖率100% |
| P2-6 | 数据提取可视化增强 | 作为审计助理，我想清楚看到哪些单元格是公式取数 | 公式单元格高亮+hover显示来源+刷新差异对比弹窗；公式识别准确率>99%，hover响应<100ms |

### 2.4 角色权限矩阵

| 角色 | 可查看 | 可编辑 | 可提交/审批 | 关键限制 |
|------|--------|--------|-------------|---------|
| 审计助理 | 自己负责循环的底稿、关联附件、AI草稿 | 底稿内容、审计说明、附件关联、复核回复 | 提交复核 | 不可绕过QC门禁，不可直接改写试算表/调整来源数据 |
| 项目经理 | 全部底稿、进度、差异监控、批量任务 | 分配、复核意见、批量操作、例外说明 | 通过/退回复核、发起批量任务 | 他人持锁时仅可只读浏览 |
| 合伙人 | 风险底稿、签字前检查结果、关键差异 | 标记已阅、补充签字前说明 | 发起签字前专项检查 | 不直接修改底稿正文或审计说明 |
| 质控人员 | 全部底稿、QC结果、抽查记录 | QC意见、抽查结论 | 出具QC阻断项/解除建议 | 独立于项目团队，仅保留只读正文权限 |

### 2.5 数据权威源与同步要求

| 对象 | 权威源 | 缓存/镜像 | 同步要求 |
|------|--------|-----------|---------|
| 审计说明正文 | 底稿工作簿中的最终单元格内容 | `parsed_data.audit_explanation` | AI生成或人工修改后必须先写回工作簿，再刷新 `parsed_data` |
| AI生成轨迹 | `wp_ai_generations` / `parsed_data.ai_content.latest_*` | 前端草稿态 | 仅用于追踪生成、确认、拒绝历史，不作为最终正文 |
| 底稿审定数/结论 | 工作簿解析结果 | `parsed_data.audited_amount` / `parsed_data.conclusion` | 作为底稿视图和QC缓存，更新后需记录最后同步时间 |
| 试算表与调整分录 | `trial_balance` / `adjustments` | 预填充缓存 | 仅单向流向底稿预填充与一致性校验，不反向改写源账数据 |

提交复核前必须满足：审计说明已确认且写回成功、`parsed_data` 刷新完成、AI草稿不存在 `pending` 状态。

## 3. 非功能需求

### 3.1 性能需求

| 指标 | 目标值 |
|------|--------|
| AI生成首字节时间 | <3秒 |
| 单张底稿AI生成完成时间 | <10秒 |
| 500张底稿批量刷新 | <120秒（10并发） |
| 500张底稿签字前检查 | <10秒 |
| 底稿列表加载（100张） | <1秒 |
| WOPI只读模式打开 | <2秒 |
| 并发用户支持 | 50人同时在线 |

> 说明：所有性能指标默认按P95统计，且长任务必须支持基于 `job_id` 的进度恢复。

### 3.2 可靠性需求

| 场景 | 要求 |
|------|------|
| LLM服务故障 | 自动降级为手动模式，不阻断操作 |
| Redis故障 | 降级为内存锁，性能下降但可用 |
| ONLYOFFICE故障 | 自动切换为离线下载模式 |

### 3.3 安全需求

- 所有AI生成内容必须人工确认后才能提交复核
- 未经确认的AI内容QC-02规则阻断
- 底稿编辑时间自动采集，防止倒签

### 3.4 可用性需求

| 场景 | 要求 | 验收标准 |
|------|------|---------|
| AI生成失败 | 显示"AI服务暂不可用"，提供"重试"和"手动编写"选项，不阻断操作流程 | 降级检测<1秒，手动模式可正常填写，重试成功后恢复AI选项 |
| 批量操作中断 | 已完成的保存结果，未完成的标记失败并基于`job_id`提供重试机制 | 中断点记录准确，重试从失败项继续，进度条正确显示 |
| 页面刷新恢复 | 长任务开始后即生成`job_id`，页面刷新或网络闪断后可恢复查看进度 | 页面刷新后30秒内恢复任务视图，已完成/失败项与服务端一致 |
| 网络中断恢复 | 自动重连，未同步数据本地暂存，恢复后自动同步 | 本地暂存<10MB，重连后自动同步，无数据丢失 |
| 底稿锁定冲突 | 提示"其他用户正在编辑"，提供"只读浏览"和"稍后提醒"选项 | 只读模式<2秒打开，"稍后提醒"定时器准确触发 |

### 3.5 AI质量评估与上线门槛

| 项目 | 要求 | 上线门槛 |
|------|------|---------|
| 评估样本集 | 至少30-50张典型底稿，覆盖货币资金、往来、收入、存货、固定资产等主要循环 | 样本覆盖MVP范围内的高频底稿类型 |
| 事实准确性 | AI草稿中的金额、比率、结论需与输入数据一致 | 事实错误率<1% |
| 幻觉控制 | 不允许虚构未提供的证据、程序、结论 | 幻觉率<2% |
| 人工可采纳率 | 允许用户直接采纳或少量修改后采纳 | 可采纳率>60% |
| Prompt可回溯性 | 每次生成都记录prompt_version、model、generation_id | 生产环境100%留痕 |

## 4. 范围边界

> 以下功能本期不做，但预留扩展接口

| 功能 | 原因 | 后续版本 |
|------|------|---------|
| 风险底稿聚焦 / 独立抽查 / 推荐规则自优化 | 属于增强运营能力，依赖MVP稳定数据和角色工作台 | 下一子阶段 |
| Word导出引擎（审计报告/报表/附注） | 全局功能，与底稿关联度偏弱 | Phase 13 独立迭代 |
| 离线工作包 | 需补版本冲突与同步机制，实施成本高 | 下一子阶段 |
| 模板热更新 | 需先稳定底稿状态机与差异对比基线 | 下一子阶段 |
| 归档导出标准目录包 | 依赖归档口径和索引规则统一 | 下一子阶段 |
| 底稿版本对比 | 存储成本高 | 远期规划 |
| 移动端编辑 | 审计场景以PC为主 | 暂不考虑 |
| 自动审计程序执行 | 需对接客户ERP | 专项迭代 |
| 语音输入 | 准确率不足 | 等待ASR成熟 |
| 实时协同编辑 | 冲突解决复杂 | 远期规划 |
> P2需求池保留，但默认不进入当前承诺排期；是否插入由MVP交付质量和资源情况再决定.
