## 一、为什么做（基于真实模板核验的痛点）

合伙人/审计助理实际进入 D 销售循环底稿时遇到 **8 类核心问题**，且 D 循环比 E1 复杂得多——因为它是**收入审计的核心**，涉及 **D0 函证 / D1 应收票据 / D2 应收账款 / D3 预收账款 / D4 营业收入 / D5 应收款项融资 / D6 合同资产 / D7 合同负债** 共 8 个并列主底稿，每个底稿独立但相互联动（D2↔D0 函证、D4↔D2 收入对应应收）。

### 1.1 真实数据规模（实测 2026-05-18）

| 主底稿 | 物理文件数 | sheet 总数 | 核心地位 |
|--------|----------|-----------|---------|
| **D0** 收入循环函证 | 1 | 11 | 函证管理总枢纽（与 E0 同款）|
| **D1** 应收票据 | 1 | 21 | 业务模式分析最复杂 |
| **D2** 应收账款 | **3**（审定 11 + 分析 4 + 检查 12）| **27** | D 循环最大底稿 |
| **D3** 预收账款 | 1 | 12 | 业务相对简单 |
| **D4** 营业收入 | **8**（审定 10 + 政策 1 + 分析 7 + 合同 1 + 检查 9 + 关联 1 + IPO 14 + 其他 5）| **48** | 8 个文件最复杂；含 IPO 应对 14 sheet |

对比 E1 货币资金（4 文件 33 sheet），D 循环规模是 E1 的 **4.7 倍**，复杂度更高。

### 1.2 D5/D6/D7 wp_code 三连错位（数据 bug，必须 P0 修复）

**真实事实**: 致同 2025 修订版 D 循环编码已重排：
- D5 = **应收款项融资**（旧版可能是 D7）
- D6 = **合同资产**（旧版可能是 D5）
- D7 = **合同负债**（旧版可能是 D6）

但当前 `backend/data/prefill_formula_mapping.json` 第一段（审定表）仍按**旧编码**配置：

| 当前 wp_code | 当前 wp_name | 当前 sheet | 真实编码应为 |
|-------------|------------|-----------|------------|
| D5 | 合同资产审定表 | 审定表D5 | D6 ❌ |
| D6 | 合同负债审定表 | 审定表D6-1 | D7 ❌ |
| D7 | 应收款项融资审定表 | 审定表D7-1 | D5 ❌ |

**有趣的是**: 第二段（分析程序）和第三段（子明细）的 wp_code 是**对的**——D5=应收款项融资分析 / D6=合同资产分析 / D7=合同负债分析。

**结论**: 同一个 JSON 文件里 D5/D6/D7 在不同段落自相矛盾。审定表段是错的，分析程序段是对的。

**直接后果**:
- 审计助理打开 D5 应收款项融资底稿，prefill 取数走 1141（合同资产）而非 1124（应收款项融资）
- 数据完全错位，所有 D5-1/D6-1/D7-1 审定表都填错数
- E1 spec 实施时如果引用 prefill mapping 没核验也会沿用错误

### 1.3 D2 应收账款的"3 文件合并 27 sheet"是 D 循环最复杂场景

**真实事实**: D2 模板物理文件 **3 个**（不是 1 个），按 LEAP 程序分类拆分：
- `D2-1至D2-4 应收账款-审定表明细表（Leap-常规程序）.xlsx` — 11 sheet（含 4 套附注披露 + 1 个 GT_Custom）
- `D2-5 应收账款-分析程序（Leap应对措施-分析程序）.xlsx` — 4 sheet（含 GT_Custom）
- `D2-6至D2-13 应收账款-检查（Leap应对措施-检查）.xlsx` — 12 sheet

chain_orchestrator 必须按 wp_code 前缀（D2/D2-X）合并 3 文件为**一个 D2 底稿**（27 sheet）。当前 chain 已实现合并逻辑（E1 spec 验证过），但合并后 sheet 命名冲突需处理：
- 3 文件都有 `底稿目录` sheet（重复 3 次需合并）
- 3 文件都有 `附注披露信息(上市公司）D2-1` 和 `附注披露信息（国企）D2-1` sheet（**重复 3 次**，命名带圆括号差异 `(上市公司）` vs `(上市公司)` 容易踩字符编码坑）
- 3 文件都有 `GT_Custom` sheet（重复 3 次）

**当前痛点**: 合并后审计助理看到 27 sheet 含 6+ 重复，导航混乱，需要去重逻辑。

**当前痛点**:

### 1.5 D 循环 prefill 完全只覆盖审定表，明细表/检查表全靠手工

**真实事实**（grep 实测）: prefill_formula_mapping.json 中 D 类条目分 3 段：
- **段 1 审定表**（D0/D1/D2/D3/D4/D5/D6/D7 审定表期初/期末/AJE/RJE）—— 共 ~32 cell
- **段 2 分析程序**（D0/D1/D2/D3/D4/D5/D6/D7 分析程序上年数+本期未审）—— 共 ~16 cell
- **段 3 部分子明细**（D2-6~8 客户/账龄/坏账 + D1-4 票据明细 + D4-4 收入明细 + D5-1 子科目 + D0-2 函证覆盖率）—— 共 ~20 cell

**总计 D 循环 prefill ≈ 70 cell**，按 8 主底稿平均每个底稿 ~9 cell。

**关键缺口**（与 E1 的 13 cell prefill 类似）:
1. **D2 检查表 12 sheet 完全没 prefill**（D2-6~D2-13 关联方/检查表/坏账测算/账龄/质押/业务模式 等全靠审计助理手工填）
2. **D4 检查程序 9 sheet 完全没 prefill**（D4-13~D4-20 ERP 核对/发生/完整性/截止测试/折扣/退货 全手工）
3. **D4 IPO 应对 14 sheet 完全没 prefill**（D4-22~D4-32 包括程序表+客户访谈+第三方回款检查 全手工）
4. **D0 函证 11 sheet 中只有 1 个 prefill**（函证覆盖率 D0-2），D0-1 函证结果汇总/D0-3 跟函过程/D0-4 差异调节/D0-5/D0-6 替代程序/D0-7 邮件回函可靠性/D0-8 函证舞弊评价 全手工

**审计助理的真实痛点**: 打开 D2 后只有 D2-1 审定表有数填好，D2-2 明细表/D2-3 坏账明细/D2-6~D2-13 检查表等 9 sheet 全部空白，需要从序时账逐笔筛选+按客户聚合手工填，工作量巨大。

### 1.6 D 循环与 B/C 类底稿的循环联动完全缺失（与 E1 同款问题）

**真实事实**: D 循环实质性程序**不是孤立执行**，与 B 类（风险评估）和 C 类（控制测试）有强前后依赖：

| 底稿 | 类型 | 与 D 循环的联动关系 | 当前状态 |
|------|------|-------------------|---------|
| **B23-1** 销售收款循环控制了解 | B 风险评估 | D2/D4 实质性程序的**前置条件**（先了解控制→再做实质性程序）| ❌ 无联动 |
| **B23-1-2** 销售收款循环流程图（.docx）| B 风险评估 | 描述销售业务流程，D4A 裁剪依据（流程复杂度→程序范围）| ❌ 无联动 |
| **B51-5** 识别收入确认方面可能存在舞弊风险的迹象程序 | B 风险评估 | 收入舞弊风险评估→**触发 D4-22~D4-32 IPO 应对程序**（B51-5 标"高"→IPO 应对必做）| ❌ 无联动 |
| **C2** 销售收款循环控制测试 | C 控制测试 | 控制测试结论→**影响 D2/D4 实质性程序范围**（控制有效→减少 D2 抽样量+D4 截止测试天数）| ❌ 无联动 |
| **C2-2** 销售收款循环评价控制偏差 | C 控制测试 | 控制偏差→**增加 D 实质性程序**（偏差严重→加大截止测试范围+增加函证比例）| ❌ 无联动 |

**审计逻辑链路**: B23-1（了解控制）→ C2（测试控制）→ C2-2（评价偏差）→ D2A/D4A（裁剪实质性程序范围）→ D2-1~D2-13/D4-1~D4-36（执行）

**与 E1 不同的复杂点**: D 循环涉及 **8 个主底稿**，B/C 联动需要按底稿类型分流：
- B23-1/C2 控制了解和测试**对所有 D2/D3/D4/D6/D7 都适用**（销售收款循环整体）
- B51-5 舞弊警觉**主要触发 D4 IPO 应对**（D4 是收入审计核心，舞弊风险最高）
- D0/D1/D5 是函证/票据/融资类**特殊底稿**，B/C 联动可弱化（D0 直接关联函证程序框架）

#### F2 详细 — D2 三文件合并去重（chain_orchestrator 修改点）

```python
# backend/app/services/chain_orchestrator.py - _merge_sheets_from_other_files
# 现状：按 wp_code 加载所有匹配文件，sheet 名碰撞时后载覆盖前载
# 修复：sheet 名 normalize（去中英文括号差异 + GT_Custom 特殊处理）

import re

def _normalize_sheet_name(name: str) -> str:
    """归一化 sheet 名做去重比较（保留原始名作为最终展示）"""
    # 中英文括号统一
    n = name.replace("（", "(").replace("）", ")")
    # GT_Custom 类视为同一 sheet（多文件版本以第一个为准）
    if "GT_Custom" in n:
        return "GT_Custom"
    # 底稿目录 / 修订前 / （原）类视为同名
    if "底稿目录" in n:
        return "底稿目录"
    return re.sub(r"\s+", "", n.strip())

def _merge_sheets_dedup(file_workbooks: list[Workbook]) -> Workbook:
    """合并多个 xlsx 工作簿，sheet 名归一化后保留首次出现"""
    seen_normalized: set[str] = set()
    merged = openpyxl.Workbook()
    merged.remove(merged.active)  # 删除默认 sheet
    for wb in file_workbooks:
        for sheet_name in wb.sheetnames:
            norm = _normalize_sheet_name(sheet_name)
            if norm in seen_normalized:
                continue  # 跳过重复
            seen_normalized.add(norm)
            # 复制 sheet 到 merged
            src_sheet = wb[sheet_name]
            new_sheet = merged.create_sheet(sheet_name)
            for row in src_sheet.iter_rows(values_only=False):
                for cell in row:
                    new_cell = new_sheet[cell.coordinate]
                    new_cell.value = cell.value
                    if cell.has_style:
                        new_cell._style = copy.copy(cell._style)
    return merged
```

## 十二、架构决策记录（基于 E1 经验沉淀）

### ADR-1: 复用 E1 spec 已建组件（不重复造轮子）

**决策**: D 循环 spec 直接复用 E1 spec 的 9 个核心组件:
- WorkpaperAuditNav（审计导航图）
- WorkpaperSidePanel + 9 Tab
- ProcedureControlPanel（程序总控台 D0A/D2A/D4A 等通用）
- ProcedureDialogLauncher（弹窗按钮入口）
- BCheckListDialog / DCountDialog / E1CutoffDialog / E2ManualDialog（4 类弹窗）
- AiConclusionButton（AI 审计说明按钮）
- SignatureBlock（双人签字）
- ItemAnnotation + ItemAttachment（逐项批注+附件）
- ReviewLayerBadges（5 层复核 badge）

**理由**: E1 已 UAT 通过（13/19 通过），组件成熟稳定。D 循环只需扩展 sheet 类型映射规则。

### ADR-2: D 循环 chain_orchestrator 多文件合并策略

**决策**: chain_orchestrator 按 wp_code 前缀（如 D2/D2-X）合并多文件为同一 wp_index，但增加去重逻辑:
- 按 sheet 名 normalize 后去重（处理圆括号差异）
- GT_Custom / 底稿目录 / 附注披露 等通用 sheet 只保留第一份

**理由**: 物理文件按 LEAP 程序拆分是致同模板设计选择，但用户使用时只需要"主底稿单一编辑器"。

### ADR-3: 双总控台（D4A + D4-22A）切换机制

**决策**: 复用 E1 spec 的 scenario 字段机制
- 普通项目（scenario=normal）: 仅加载 D4A 25 项程序
- IPO 项目（scenario=ipo/listed/restructure）: 同时加载 D4A + D4-22A
- 舞弊应对项目（scenario=fraud_response）: 同时加载 D4A + D4-22A，并自动勾选所有"舞弊应对"程序行
- B51-5 conclusion="高" 自动触发: 即使 scenario=normal，也加载 D4-22A（参照 E1 的 B51-3→E26A 机制）

### ADR-4: D 循环组件选型按"sheet 内容特征"分流（不按 sheet 编号）

**决策**: 沿用 E1 spec 的 5 类组件分流（A Univer / B 检查清单 / C 程序总控台 / D 盘点访谈 / E 截止抽样）

**理由**: D 循环 155 sheet 按特征分流后，Univer 加载从 155 → ~75 sheet（-52%），明显减负。

### ADR-5: prefill 预设公式只覆盖审定表+分析程序+核心明细，不覆盖检查表

**决策**: D 循环 prefill 重点覆盖
- 8 主底稿审定表（已有，需修复 D5/D6/D7 错位）
- 8 主底稿分析程序（已有）
- D2/D1/D4 核心明细表（部分已有，扩展中）
- D2/D4 截止测试 LEDGER_DETAIL 自动列出跨期凭证

**不覆盖**:
- 检查表 B 类（问答式，按业务判断）
- 客户访谈 D 类（现场记录）
- 关联方检查 B 类（业务判断）

**理由**: 检查表/访谈/判断类 sheet 强行 prefill 反而误导审计师。
