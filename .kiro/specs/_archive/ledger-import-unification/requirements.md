# 账表导入统一方案 — 需求文档

## 背景术语

- **四表**：余额表（tb_balance）、序时账（tb_ledger）、辅助余额表（tb_aux_balance）、辅助序时账 / 辅助明细账（tb_aux_ledger）
- **核算维度 / 核算项目**：在科目下挂的多维度细分，如"客户/供应商/项目/部门/员工/存货"
- **探测（Detect）**：只读前 20 行，快速判断文件结构
- **识别（Identify）**：确定每个 sheet 归属哪张表、每列对应哪个字段
- **置信度（Confidence）**：识别的可信度 0-100，决定是否需要人工确认

---

## 需求清单

### 需求 1：多文件多 sheet 智能探测

**用户故事**：作为审计助理，我希望上传 1 个或多个 Excel/CSV 后，系统能快速告诉我每个 sheet 被识别成什么表，让我在入库前确认一下。

**验收标准**：

1. WHEN 用户上传 1-20 个文件（支持 .xlsx / .xls / .csv / .zip）AND 总大小 ≤ 500MB，系统 THEN 在 30 秒内返回所有 sheet 的探测结果
2. WHEN 文件 > 500MB，系统 THEN 以流式方式分批上传并在 60 秒内返回探测结果
3. 探测结果必须包含每个 sheet 的：文件名、sheet 名、判定为哪张表（balance / ledger / aux_balance / aux_ledger / account_chart / unknown）、置信度、前 20 行预览
4. 探测阶段**不写入任何业务数据**，失败可直接重来

---

### 需求 2：三级识别策略 + 列分层

**用户故事**：不同财务软件导出的文件 sheet 名和列名千差万别，系统应逐级降级识别，永远不空手而归。核心：识别严格度按列重要性分层，只对"四表联动关键列"严格要求，次关键列宽松，非关键列保留不校验。

**列分层定义**（贯穿整个 spec）：

| 层级 | 列名 | 识别门槛 | 未识别时行为 |
|-----|------|---------|------------|
| **关键列（key）** | 余额表：account_code / opening_balance / closing_balance / debit_amount / credit_amount；序时账：voucher_date / voucher_no / account_code / debit_amount / credit_amount；辅助表：+aux_type / aux_code | 置信度 ≥ 80 | 强制弹窗人工确认，不填不允许 submit |
| **次关键列（recommended）** | account_name / summary / preparer / currency_code / voucher_type / level / entry_seq / company_code / accounting_period | 置信度 ≥ 50 | 自动映射；低于门槛 → 值置 NULL，生成 warning |
| **非关键列（extra）** | 用户文件中不在上面两类的任何列 | 无门槛 | 原样存入 `raw_extra` JSONB 列，保留完整字段名+值 |

**验收标准**：

1. **Level 1 — Sheet 名识别**：Sheet 名含"余额"、"序时"、"明细账"、"总账"、"试算"等关键词时直接判定，置信度 90+
2. **Level 2 — 表头特征识别**：前 20 行扫描列名，命中关键列组合（见识别规则表）时判定，置信度 60-89
3. **Level 3 — 内容样本识别**：前几行数据的特征（日期格式、借贷方向值、金额列数量）作为兜底，置信度 30-59
4. **Level 4 — 人工确认**：关键列置信度 < 80 或多级冲突时，前端弹窗请求用户选择；次关键列无需人工
5. **Level 5 — AI 兜底（可选）**：所有 Level 1-3 对关键列仍无结果时，调用 LLM 精准识别"这列是不是 account_code / voucher_date / ..."（不让 LLM 漫无目标地识别整个 sheet）
6. 所有级别结果必须记录 `detection_evidence`（哪些特征命中/未命中），便于调试和用户申诉
7. `ColumnMatch.is_key_column: bool` 字段明确标记该列属于哪一层，前端据此分区展示

**架构原则（v2.1 补充）**：

8. **L1/L2/L3 并行联合打分**：三级识别始终同时执行，按可配置权重（默认 sheet名 0.2 + 表头 0.5 + 内容 0.3）加权聚合最终置信度，而非串行降级
9. **表头+内容双重验证**：列映射不仅看列名匹配，还用前 N 行数据做内容验证（日期列 ≥ 50% 可解析为日期、金额列 ≥ 80% 可解析为数字），验证结果参与置信度计算
10. **识别规则声明式配置**：表类型签名（sheet 名 pattern、关键列别名、内容信号、负向信号）抽离为 JSON 配置文件，支持热加载和用户自定义，新增财务软件不改 Python 代码
11. **合并表头通用算法**：基于"行间值多样性变化"自动判断表头边界（标题行 = unique_value_count ≤ 2；表头行 = fill_ratio ≥ 0.5 且 unique_value_count ≥ 3），不依赖硬编码阈值

### 需求 3：财务软件兼容适配器

**用户故事**：用友/金蝶/SAP 等主流财务软件的固定列名模式应开箱即用，不应每家都要人工配对。

**验收标准**：

1. 内置适配器至少覆盖 8 家：用友 U8/NC/T+、金蝶 K3/EAS、SAP、Oracle EBS、浪潮 GS、Kingdee Cloud、新中大、SAP B1
2. 每个适配器定义列名到标准字段的映射表（如用友"摘要"→ summary，金蝶"凭证摘要"→ summary）
3. 适配器识别优先级：Sheet 名或文件名含软件标识 → 列名特征命中 → fallback 通用规则
4. 适配器定义放在 `backend/data/ledger_adapters/*.json`（单文件单软件），支持 hot reload
5. 允许用户自定义适配器（`ledger_adapters/custom_{tenant}.json`）

---

### 需求 4：辅助表的虚拟派生

**用户故事**：很多财务软件只导出 2 张表（余额表+序时账），其中"核算项目"列里隐式带多维信息；系统应能从这 2 张表派生出辅助表，不必强求用户上传 4 张。

**验收标准**：

1. 当余额表里含"核算项目"、"辅助核算"、"客户"、"供应商"、"项目"等列时，系统识别为**多维余额表**
2. 当序时账里含同类列时，识别为**多维序时账**
3. 解析时按行检查该列是否非空：非空→写入辅助表，空→写入主表（不重复写）
4. 辅助维度格式兼容：`类型: 编码 名称`、`类型: 名称`、`客户编号|客户名称`、用友"项目 -> 研发部"、JSON
5. 识别无法拆分时，前端提供 UI 让用户指定"哪列是核算维度、类型是什么"

---

### 需求 5：大文件流式处理

**用户故事**：序时账经常超过 500MB，现有的全量加载容易 OOM；希望支持分块流式解析和入库。

**验收标准**：

1. 单文件 > 50MB 时自动切换 read_only 模式（openpyxl）
2. 单 sheet > 100 万行时按 50,000 行分批 COPY 写入 PG
3. 峰值内存 ≤ 2GB，保证小内存部署环境可运行
4. 进度回调至少每 100,000 行或每 5 秒上报一次（SSE 或轮询）
5. CSV 文件直接用 `csv.reader` + generator 流式处理，不加载到内存
6. 提供 `LEDGER_IMPORT_FULL_MODE_MAX_FILE_MB` 环境变量控制阈值

---

### 需求 6：多 sheet 跨 sheet 聚合识别

**用户故事**：某些软件把 1-3 月序时账拆成 3 个 sheet（`1月凭证`、`2月凭证`、`3月凭证`），系统应把它们合并为同一张序时账，而不是分 3 张表。

**验收标准**：

1. 同文件内多 sheet 若表头结构相同且均被识别为同一类型（如 ledger），自动合并为 1 个逻辑表
2. 合并时按"凭证日期 + 凭证号"去重，避免重复行
3. 合并策略可配置（`auto` / `by_month` / `manual`），默认 `auto`
4. 合并后的 ImportBatch 记录合并信息（source_sheets 数组），便于审计追溯

---

### 需求 7：前端两阶段交互 UI

**用户故事**：作为用户，我希望先看到"这几个 sheet 被识别成什么"的预览，确认后再触发正式入库；中途可以修改列映射或放弃。

**验收标准**：

1. 上传后弹出"预检对话框"，展示每个 sheet 的识别结果 + 置信度 + 前 20 行预览
2. 置信度 < 60 的 sheet 标红提醒，用户可点击修改表类型或列映射
3. 列映射支持拖拽和下拉选择，实时校验必填字段是否齐全
4. 确认后显示正式导入进度条（百分比 + 当前处理文件/sheet）
5. 正式导入失败时，进度条停在失败位置，弹出错误详情但保留已解析的预检数据，用户可修正重试

---

### 需求 8：分级错误反馈（按列分层严格度）

**用户故事**：导入失败时，当前只看到一行"智能导入失败: xxx"，用户无法判断是自己文件问题还是系统问题。

**验收标准**：

错误必须分 3 级；同一类错误在关键列/次关键列上的严重级不同：

| 级别 | 定义 | UI 呈现 |
|------|------|---------|
| **致命（fatal）** | 文件无法打开、格式不支持 | 红色弹窗，阻断流程 |
| **阻塞（blocking）** | 关键列缺失、关键列金额非数字、借贷不平 | 橙色弹窗，列出具体 sheet+列+行号 |
| **警告（warning）** | 次关键列缺失/解析失败、辅助维度格式不标准、多识别出一列 | 黄色提示条，可继续，建议用户修正 |

**同错误在不同列分层的严重级对照**：

| 错误码 | 关键列 | 次关键列 | 非关键列 |
|--------|-------|---------|---------|
| MISSING_COLUMN | blocking | warning | 不报（进 raw_extra） |
| AMOUNT_NOT_NUMERIC | blocking | warning（值置 NULL） | 不报（原样入 raw_extra） |
| DATE_INVALID | blocking（仅 voucher_date） | warning（值置 NULL） | 不报 |
| EMPTY_VALUE | blocking | 不报 | 不报 |

每条错误必须包含：
- `code`：错误码（`MISSING_KEY_COLUMN` / `AMOUNT_NOT_NUMERIC_KEY` / `MISSING_RECOMMENDED_COLUMN` / ...）
- `severity`：fatal/blocking/warning
- `column_tier`：key/recommended/extra（便于前端筛选）
- `message`：人类可读描述（"第 3 个 sheet 缺少'科目编码'列（关键列），请检查导出时是否勾选"）
- `file` / `sheet` / `row` / `column`：精确定位
- `suggestion`：修复建议（"请重新导出时勾选'科目编码'列" / "点这里手动指定哪列是科目编码"）

---

### 需求 9：列映射持久化与复用

**用户故事**：同一个客户每季度都要导一次账套，列映射不应每次手工配；系统应记住上次的映射。

**验收标准**：

1. 确认列映射后，按 (project_id, 软件特征指纹) 存入 `import_column_mapping_history`
2. 下次同一项目上传时，优先用历史映射自动匹配，命中率 100% 时置信度设为 95+
3. 映射可跨项目复制（UI 提供"从其他项目导入映射"入口）
4. 提供"重置此客户的历史映射"按钮，避免历史误映射污染新导入

---

### 需求 10：断点续传与重试

**用户故事**：大文件导入中途失败（网络断、服务重启），不应让用户从头重传。

**验收标准**：

1. 文件上传分 5MB chunk，已上传 chunk 存 `import_artifacts` 表，支持 7 天内续传
2. 入库失败时 ImportJob 状态设为 `failed`，但 ImportArtifact 保留，用户点"重试"直接复用原文件
3. 服务重启后，`import_recover_worker` 自动从 `processing` 状态恢复为 `queued`，重新入队
4. `retry_count` 上限 3，超限标 `failed_final`，必须人工介入

---

### 需求 11：数据校验规则层（按列分层）

**用户故事**：导入后的数据需要业务校验（借贷平衡、科目合法、期间连续），校验不过不应激活数据集。

**验收标准**：

1. **Level 1（解析期）**：按列分层校验
   - **关键列**：金额字段必须数值、日期字段必须可解析、必填列不为空 → blocking
   - **次关键列**：同类校验失败 → warning，值置 NULL，不阻断
   - **非关键列**：不校验，原样入 `raw_extra`
2. **Level 2（全量后）**：借贷平衡（sum(debit) == sum(credit)）、序时账日期全部在年度范围内、科目编码在 account_chart 中存在 → 默认 blocking，可 force 跳过（除 `L2_LEDGER_YEAR_OUT_OF_RANGE`）
3. **Level 3（跨表）**：余额表期末 = 序时账累计（容差 1 元）、辅助表与主表科目一致 → blocking 可 force 跳过；辅助维度不一致 → warning
4. Level 1 关键列失败直接阻塞入库；次关键列失败生成 `validation_findings` 但允许继续
5. 所有校验规则代码化在 `ImportValidationService`，可单独运行、可扩展

---

### 需求 12：导入历史与回滚

**用户故事**：如果激活后发现数据错了，应能回滚到上一版本。

**验收标准**：

1. 每次激活生成 `LedgerDataset` 版本记录（staged → active → superseded）
2. 只有 1 个 active 版本，历史版本保留 90 天
3. 提供"回滚到版本 X"按钮，一键切换 active 指针，不物理删除数据
4. 回滚触发 `LEDGER_DATASET_ROLLED_BACK` 事件（复用 `EventType` 现有枚举），下游（试算表、报表）自动重算
5. 回滚动作记录审计日志（谁、何时、为什么）

---

### 需求 13：审计年度自动识别

**用户故事**：用户不应手填年度，系统应从文件内容自动推断。

**验收标准**：

1. 按优先级识别：文件名（"2025 年度"）→ sheet 名 → 表头元信息（"会计期间: 2025-01 ~ 2025-12"）→ 内容抽样（序时账日期众数年份）
2. 识别失败时弹窗让用户选择年度
3. 年度与项目 audit_period 冲突时提示"文件年度是 2024，项目审计期是 2025，是否确认？"
4. 用户可强制覆盖（勾选"确认使用此年度"）

---

### 需求 14：编码与文件类型兼容

**用户故事**：客户的导出文件可能是 GBK / UTF-8 / UTF-8 BOM，系统应自适应。

**验收标准**：

1. CSV 编码检测：依次尝试 utf-8-sig / utf-8 / gbk / gb18030 / latin1，首个成功的用作 encoding
2. 支持 .xls（Excel 97-2003，用 xlrd 降级读）、.xlsx、.xlsm、.csv、.tsv、.zip（内含上述任意）
3. .xls 不支持时，弹窗明确告知"请另存为 .xlsx"并提供教程链接
4. 文件名乱码自适应：Windows ZIP 包内文件名默认 CP437，检测到乱码自动尝试 gbk 重编码

---

### 需求 15：合并表头与空行处理

**用户故事**：财务软件常输出带 1-3 行合并表头（如上行"期初余额"下行"借方|贷方"），或开头几行是"XX 公司 科目余额表 2025 年度"标题；系统要能识别并跳过。

**验收标准**：

1. 前 10 行中若某行有合并单元格 > 50% 列宽，视为辅助标题行，跳过
2. 合并表头识别：连续 2-3 行都是文本且第 1 行单元格合并跨列，合并为单行表头（用"."拼接）
3. 标题行（单行跨表宽度、含"公司"或年度）自动跳过
4. 表头下第 1 个非空数据行用于内容特征识别
5. 无法识别时，UI 让用户手工指定"表头在第几行"

---

### 需求 16：辅助维度解析的通用引擎

**用户故事**：辅助核算格式五花八门，系统要能从各种写法里拆出类型+编码+名称。

**验收标准**：

支持的格式至少包括：

| 格式 | 示例 | 说明 |
|------|------|------|
| `类型:编码 名称` | `客户:001 北京某科技` | 常见，冒号分隔 |
| `类型/编码/名称` | `供应商/V001/上海某某` | 斜杠分隔 |
| `类型|编码|名称` | `项目|P01|新产品开发` | 竖线分隔 |
| `编码 名称` | `C001 北京某科技` | 无类型（系统推断） |
| JSON | `{"客户":"001","项目":"P01"}` | 结构化 |
| 多列分拆 | 核算项目分到 3 列（类型/编码/名称） | 多列合并为维度 |

解析失败时，将原始字符串存入 `aux_dimensions_raw`，维度字段为 NULL，不阻断入库。

---

### 需求 17：前端预览性能

**用户故事**：预检弹窗应秒开，不应等 30 秒。

**验收标准**：

1. 预检 API 只读前 20 行 + 表头元信息，响应时间 < 3 秒（单文件 < 50MB）
2. 预览表格虚拟滚动（最多展示 20 行，每列限 200 像素宽）
3. 多文件并发预检，前端批量展示，不等最慢文件
4. 预检结果缓存到 sessionStorage，用户中途刷新页面可恢复

---

### 需求 18：进度反馈与可中断

**用户故事**：导入大文件需要等几分钟，用户希望能看到实时进度和能中途取消。

**验收标准**：

1. SSE 或轮询每 2 秒更新进度：当前阶段（解析中/写入中/校验中/激活中）、已处理文件/sheet、行数、百分比
2. "取消"按钮触发 `ImportJob.status = canceling`，后台 worker 检测到后终止解析并回滚 staged 数据
3. 取消后用户可查看"已解析多少、在哪里停止"，不留脏数据
4. 浏览器关闭不中断导入（后台 job 继续跑），用户可在导入历史查看结果

---

### 需求 19：诊断与调试模式

**用户故事**：支持人员要能在用户无法自助解决时快速定位问题。

**验收标准**：

1. 每次导入保留完整的 `detection_evidence`（识别决策树）+ `sheet_diagnostics`（每 sheet 的识别细节）
2. 支持/管理员可在后台查看任意 ImportJob 的诊断详情
3. 提供"下载诊断包"（包含原始文件前 100 行 + 识别结果 + 错误堆栈）的入口
4. 诊断日志脱敏（金额用 `***` 替换，客户名用 `[client_N]`）

---

### 需求 20：向后兼容既有导入

**用户故事**：既有的导入作业数据不应被破坏。

**验收标准**：

1. 旧表 `import_batches` / `tb_balance` / `tb_ledger` / `tb_aux_balance` / `tb_aux_ledger` 结构保持
2. 旧的 `ImportJob` 继续可查看，不需要迁移
3. 新识别引擎与旧 `smart_import_engine` 并存一段时间，通过 feature flag 切换
4. 新引擎 GA 后，旧 `_resolve_sheet_mapping` 标注 `@deprecated`，下个版本移除

### 需求 21：raw_extra 原始列保留

**用户故事**：作为审计员，我希望客户文件里的所有列都被保留，即使系统没识别出来——这样事后发现"某列其实有用"不用重新上传。

**验收标准**：

1. `tb_balance` / `tb_ledger` / `tb_aux_balance` / `tb_aux_ledger` 四张表新增 `raw_extra JSONB` 列（nullable）
2. 导入时所有未被映射到标准字段的原始列，按 `{ "原列名1": "原值1", "原列名2": "原值2", ... }` 存入 raw_extra
3. 关键列+次关键列已被映射的**不**重复存入 raw_extra（避免数据膨胀）
4. 每行 raw_extra 最大 8KB，超限时保留前 N 列 + warning 提示（避免某些表有 100+ 列时爆炸）
5. 查询接口 `GET /api/projects/{pid}/ledger/raw-extra-fields` 返回本项目所有 raw_extra 中出现过的字段名（聚合，给用户参考）
6. 前端在导入预检时明确展示"识别到 X 个关键列、Y 个次关键列、Z 个列会保留到 raw_extra"，让用户确认
7. 后续若发现某个 raw_extra 字段其实是标准字段，写一次性迁移脚本：`UPDATE tb_xxx SET standard_col = (raw_extra->>'原列名')::type WHERE raw_extra ? '原列名'`，不用重新导入

### 需求 22：序时账增量追加导入

**用户故事**：作为审计助理，我在预审阶段已导入 1-11 月的序时账，年审时客户只给了 12 月的新数据；我不想重新导入全年 12 个月（浪费时间且大文件容易超时），只想追加 12 月即可。余额表需要重新导入（因为期末数变了），但序时账应该支持增量。

**验收标准**：

1. 导入时可选择"全量覆盖"（默认）或"序时账增量追加"模式
2. 增量模式下，系统自动扫描本次文件的凭证日期，提取覆盖的月份集合
3. 与已入库的序时账期间做 diff，只导入新增月份的行
4. 如果有重叠月份，弹窗让用户选择"跳过已有"或"覆盖该月"
5. 余额表在增量模式下仍然全量覆盖（期末数每次都变）
6. 支持陕西华氏这种"12 个月度文件"的场景：预审导 1-11 月，年审只追加 12 月
7. 支持和平药房这种"按日期段拆分 CSV"的场景：先导 0101-1011，再追加 1012-1031
8. 增量追加后的回滚粒度为"按期间回滚"（可删除指定月份的序时账行）
9. 前端预检阶段显示"已入库期间 vs 本次文件期间 vs 增量期间"的对比信息
