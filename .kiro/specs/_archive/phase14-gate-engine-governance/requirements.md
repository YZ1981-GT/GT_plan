# Phase 14: 统一门禁引擎与治理收敛 - 需求文档

> 基于 Phase12/13 已落地主链路，补齐门禁口径分散问题，确保“提交复核/签字/导出”三入口规则一致、可解释、可回放。

---

## 1. 背景与目标

### 1.1 现状

- `QC-15~QC-18` 已上线，但扩展规则与入口判定仍分散在多个服务。
- `WP-P2-3A`（LLM证据链/确认冲突/版本映射门禁）在 `5.8` 为未开发。
- `WP-P0-2` 在 `5.8` 为部分完成，`UserCanWrite` 未完全动态化。

### 1.2 核心目标

| 目标 | 量化指标 | 实现方式 |
|------|---------|---------|
| 三入口规则一致 | 同条件判定一致率 = 100% | 统一 `gate_engine` 入口 |
| 风险零漏放 | `QC-21~QC-26` 漏检 = 0 | 扩展规则统一执行 |
| 用户可修复 | 阻断项可定位率 = 100% | 统一 `location + suggested_action` |
| 决策可追溯 | 门禁留痕覆盖率 >= 95% | 落地 `gate_decisions` |

### 1.3 本期交付策略

| 层级 | 范围 | 目标 |
|------|------|------|
| MVP（本期必须放行） | `WP-P2-3A`、`WP-P0-2`、`WP-P2-3` | 完成统一门禁引擎与扩展规则阻断 |
| 增强（MVP稳定后插入） | `WP-P1-9` 门禁交叉留痕 | 完成版本链相关门禁回放 |
| 延后（下一子阶段） | 任务树与事件补偿、取证不可抵赖 | 转入 Phase15/16 |

> 本文保留完整需求池，但默认排期与上线验收以 MVP 为准；增强项需在 MVP 稳定后单独确认资源。

---

## 2. 需求分层

### 2.1 P0 — 核心阻断能力

| 序号 | 需求 | 用户故事 | 验收标准 |
|------|------|---------|---------|
| P0-1 | 统一门禁引擎 | 作为研发，我希望三入口走同一评估逻辑 | 提交/签字/导出统一调用 `gate_engine` |
| P0-2 | 扩展门禁规则落地（QC-21~QC-26） | 作为质控，我希望关键风险在提交前被阻断 | `QC-21~QC-26` 全部可执行 |
| P0-3 | WOPI只读策略动态化 | 作为复核人，我不应在只读场景写入 | `UserCanWrite` 基于锁态+角色动态判定 |
| P0-4 | 统一错误返回协议 | 作为前端，我希望阻断项可定位可修复 | 返回 `rule_code/error_code/location/suggested_action/trace_id` |
| P0-5 | 统一过程留痕模型（trace_events） | 作为审计，我希望所有关键动作有统一留痕 | 新增 `trace_events` 表，核心链路留痕覆盖率 >= 95%（对齐 v2 WP-ENT-01） |
| P0-6 | 程序裁剪门禁（QC-19/QC-20） | 作为质控，我希望 mandatory 程序裁剪被阻断 | `QC-19` mandatory 裁剪阻断 + `QC-20` conditional 裁剪无证据阻断（对齐 v2 3.2 校验9/10） |
| P0-7 | SoD 职责分离硬约束 | 作为安全，我希望编制/复核/签字/放行互斥 | 编制人≠复核人≠签字人服务端强校验，SoD 冲突拦截率=100%（对齐 v2 WP-ENT-04 + 4.5.14） |
| P0-8 | Trace 回放 API | 作为运维，我希望按 trace_id 回放决策链路 | `GET /api/trace/{trace_id}/replay` 可用，回放成功率>=99%（对齐 v2 5.9.3 A-02） |

### 2.2 P1 — 治理增强

| 序号 | 需求 | 用户故事 | 验收标准 |
|------|------|---------|---------|
| P1-1 | 门禁留痕表 | 作为审计，我希望每次判定有证据 | `gate_decisions` 覆盖核心链路 >=95% |
| P1-2 | 规则配置分层 | 作为平台管理员，我希望强制阈值不可篡改 | 平台强制与租户可配分层生效 |
| P1-3 | 冲突解释模板 | 作为用户，我希望知道怎么修复 | 冲突类规则均返回修复建议 |
