# Phase 16: 取证包与版本链闭环 - 需求文档

> 以 `WP-P1-9/WP-EXT-3/WP-EXT-4` 为主线，完成“版本戳贯通 + 取证完整性 + 离线冲突治理”三条链路闭环，满足审计追溯与监管取证要求。

---

## 1. 背景与目标

### 1.1 现状

- 导出主链路可用，但取证包 hash/签名摘要和不可抵赖证明不完整。
- 报表快照与 stale 检测已具备，版本链仍存在跨对象断点。
- 离线冲突当前以版本检查为主，缺少 `procedure_id + field_name` 粒度合并治理。

### 1.2 核心目标

| 目标 | 量化指标 | 实现方式 |
|------|---------|---------|
| 版本链闭环 | 版本戳贯通率 = 100% | 统一 `version_line_stamps` |
| 取证完整性 | hash 校验通过率 = 100% | `manifest + sha256 + signature_digest` |
| 冲突零漏检 | 关键字段漏检 = 0 | 细粒度冲突检测与合并队列 |
| 链路可回放 | 回放成功率 >= 99% | `trace_id` 贯穿处置流程 |

### 1.3 本期交付策略

| 层级 | 范围 | 目标 |
|------|------|------|
| MVP（本期必须放行） | `WP-P1-9`、`WP-EXT-3`、`WP-EXT-4` | 版本链与取证、冲突主链路可用 |
| 增强（MVP稳定后插入） | `WP-P4-1` 导出治理深水区 | 双通道模板化与可复算报告 |
| 延后（下一子阶段） | 全租户级成本核算与历史全量补录 | 独立专项治理 |

> 本文保留完整需求池，但默认排期与上线验收以 MVP 为准；增强项需在 MVP 稳定后单独确认资源。

---

## 2. 需求分层

### 2.1 P0 — 主链路能力

| 序号 | 需求 | 用户故事 | 验收标准 |
|------|------|---------|---------|
| P0-1 | 统一版本戳模型 | 作为审计，我希望对象变更可串成完整版本链 | 四表/附注/正文/底稿均写入版本戳 |
| P0-2 | 取证完整性校验 | 作为质控，我希望导出包可校验未被篡改 | `manifest + hash +校验结果` 一致 |
| P0-3 | 离线冲突细粒度检测 | 作为助理，我希望冲突识别到具体字段 | 支持 `procedure_id + field_name` 检测 |
| P0-4 | 冲突合并队列 | 作为经理，我希望冲突可分配可回放可复验 | 合并后自动触发 QC 重跑 |

### 2.2 P1 — 增强能力

| 序号 | 需求 | 用户故事 | 验收标准 |
|------|------|---------|---------|
| P1-1 | 可复算一致性报告 | 作为合伙人，我希望导出附带关键口径复算 | 导出结果附复算摘要 |
| P1-2 | 双通道模板化 | 作为项目经理，我希望阅读包/取证包字段统一 | 双通道字段字典统一 |
| P1-3 | 回放中心增强 | 作为运维，我希望快速定位冲突与导出链路 | 支持按 `trace_id` 跳转回放 |
